Kernel Fusions in vLLM

A guided tour of the popular fusions — RMSNorm+Quant, SiLU+Mul+Quant, AllReduce+Norm, Attention+Quant, and the comm/compute overlap fusions — what they merge, why it's faster, and the torch.fx machine that rewrites your graph to use them. Grounded line-by-line in vLLM source on branch dao/understand.

TL;DR Most of these layers are memory-bound: the GPU spends more time shuffling tensors to and from HBM than doing math. A fusion takes two adjacent ops — say rms_norm then fp8_quant — and replaces them with a single kernel that reads the input once, does both computations in registers, and writes the output once. That kills one kernel launch and a full round-trip to HBM of the intermediate tensor. vLLM does this automatically at compile time: a stack of torch.fx pattern-matcher passes scans the traced graph, finds known op sequences, and swaps in pre-written fused CUDA/Triton kernels. The same idea extends to collectives (fold the norm into an AllReduce) and to overlap (pipeline a GEMM with its ReduceScatter so communication hides behind compute). Newer models push it furthest: MiniMax M3 fuses a cross-GPU AllReduce into its QK-norm (§11), and DeepSeek V4 welds entire MLA prologues — compress + RMSNorm + RoPE + fp8 quant + KV-cache write — into single Triton kernels (§12) — both hand-written, since they run with the auto-fusion passes off (§10).

1. Why fuse? The HBM round-trip

On a modern GPU, arithmetic is cheap and memory bandwidth is the bottleneck for "small" pointwise ops. An RMSNorm or a quantization touches every element once — it does almost no FLOPs per byte. So its runtime is dominated by how many times the tensor crosses the HBM↔SM boundary, plus the fixed cost of launching the kernel.

Consider the most common pair in a quantized model: normalize a hidden state, then quantize it to fp8 before the next matmul. Unfused, that's two kernels, and the bf16 intermediate is written out and immediately read back:

Unfused: rms_norm → fp8_quant
read x (bf16) → RMSNorm → write y (bf16) ↓HBM
read y (bf16) ↑HBM → find scale + quant → write q (fp8)
2 kernel launches. The full-size bf16 tensor y makes a wasted round-trip: written by kernel 1, re-read by kernel 2.
Fused: rms_norm_static_fp8_quant
read x (bf16) → RMSNorm + scale + quant (in registers) → write q (fp8)
1 kernel launch. y never touches HBM — it lives in registers between the two steps. saved: 1 launch + 1 write + 1 read of a full bf16 tensor
The mental model in one line A fusion turns read → compute → write → read → compute → write into read → compute → compute → write. For memory-bound ops, removing the middle write+read roughly halves the time.
Two flavors of "fusion" Most of this page is vertical / epilogue fusion — merge adjacent pointwise ops to cut HBM traffic (§4, §5, §7). The collective fusions (§6, §8, §9) are different: they merge a communication op with its neighbors, and the big win is often overlap — letting the network and the SMs run at the same time — not just saving a write.

2. The fusion machine (torch.fx)

A fusion reaches the GPU one of two ways, and which one applies depends on whether the model runs under torch.compile. This section is the automatic route — you write plain RMSNorm(...) then quant(...) in the model and vLLM rewrites it for you. §10 is the hand-written route — you call a fused op directly in forward. Every fusion in §4–§9 is automatic; the model spotlights in §11–§12 are hand-written.

The automatic rewriter is a stack of post-grad passes over the traced FX graph, orchestrated by PostGradPassManager. You emit the unfused ops; the matcher swaps in the fused kernel.

Source vllm/compilation/passes/pass_manager.py — PostGradPassManager; base classes in .../vllm_inductor_pass.py (VllmInductorPass, VllmPatternMatcherPass). Each fusion lives in its own file under .../passes/fusion/.
This machine only runs under torch.compile The pass manager is an inductor post-grad step. A model that runs eager — or under breakable CUDA graph (VLLM_USE_BREAKABLE_CUDAGRAPH=1, which forces compilation mode = NONE, i.e. no dynamo / no inductor) — never invokes it, so none of §4–§9 fire. Those models must fuse by hand instead (§10). In vLLM, DeepSeek V4 and MiniMax M3 default to exactly this regime — which is why they ship the hand-written kernels in §11–§12.

Pattern → replacement, registered against the graph

Each fusion is a pair of small Python functions traced into FX subgraphs and handed to PyTorch's PatternMatcherPass via register_replacement. The pattern traces the unfused ops; the replacement traces the single fused op. At compile time the matcher finds every occurrence of the pattern in the graph and splices in the replacement. Here is the real static-fp8 case, lightly trimmed: 

# vllm/compilation/passes/fusion/rms_quant_fusion.py — RMSNormStaticQuantPattern
def pattern(input, weight, scale):
    result_rms = vllm.ir.ops.rms_norm(input, weight, self.epsilon)
    return self.quant_matcher(result_rms, scale)[0]   # rms_norm → fp8 quant

def replacement(input, weight, scale):
    result = torch.empty(input.shape, dtype=self.quant_dtype, ...)
    at = auto_functionalized(
        self.FUSED_OP,                  # torch.ops._C.rms_norm_static_fp8_quant
        result=result, input=input, weight=weight,
        scale=scale, epsilon=self.epsilon,
    )
    return at[1]                          # the fp8 result

pm.register_replacement(pattern, replacement, inputs, pm.fwd_only, pm_pass,
                       extra_check=_rms_input_weight_dtype_match)
Why extra_check matters A pattern can match structurally but be unsafe to fuse. Here _rms_input_weight_dtype_match blocks the rewrite when the norm's input and weight dtypes differ (e.g. a Gemma fp32 weight.float()+1 gamma), because the fused kernel assumes matching dtypes. Most passes carry similar guards.
Why auto_functionalized wraps the fused op The fused kernels are in-place: the schema is (Tensor! result, …) -> () — they write into a caller-allocated buffer and return nothing. But the FX graph the matcher rewrites is functional (every node yields fresh values, no mutation, no aliasing), so a raw mutating op can't sit in the replacement. auto_functionalized — a higher-order op (HOP), i.e. an op that takes another op as its first argument (from torch._higher_order_ops) — presents it as a pure function, returning the mutated buffers as new SSA outputs in a tuple (orig_return, *Tensor!-args-in-schema-order). Index 0 is the op's declared return (empty here, since -> ()), so the mutated tensors start at index 1 — which is why this replacement does return at[1] (just result), while the residual variant does return at[1], at[2] (result, residual, matching its Tensor! result, …, Tensor! residual schema). It's also why replacement allocates result itself and passes it in. A later pass (FixFunctionalizationPass, shown next) restores the true in-place call — functional for the compiler, in-place on the GPU.

Example: turning it back in-place (fix_functionalization)

The fusion pass leaves the functional form in the graph; the very last pass, FixFunctionalizationPass, rewrites it into the in-place call that actually runs. Same static-fp8 fusion, before and after that pass:

Before — functional (what the fusion pass emits)
result = torch.empty(...)                       # output buffer, allocated in replacement()
at = auto_functionalized(                       # HOP wrapper — presents a pure op
        torch.ops._C.rms_norm_static_fp8_quant,
        result=result, input=x, weight=w, scale=s, epsilon=eps)
q  = at[1]                                       # getitem(1): the "new" result value
# ... downstream ops read q ...
After — in-place (FixFunctionalizationPass, mutated_args = {1: "result"})
result = torch.empty(...)
torch.ops._C.rms_norm_static_fp8_quant(         # direct in-place call, returns ()
        result=result, input=x, weight=w, scale=s, epsilon=eps)
# every reader of q is rewired to read `result` directly;
# the auto_functionalized + getitem(1) nodes are erased — no clone, no copy.
The pass keys each fused op to a mutated_args map (getitem-index → kwarg name) and does three things: rewire at[i] readers to that kwarg, insert the plain in-place call, erase the HOP + getitem nodes.

The mutated_args map is literally the inverse of the return at[…] in the replacement — one entry per Tensor! arg, in schema order:

Fused opreplacement returnsmutated_args
rms_norm_static_fp8_quantat[1]{1: "result"}
fused_add_rms_norm_static_fp8_quantat[1], at[2]{1: "result", 2: "residual"}
Why it must run last — and why no DCE after it Every fusion pass needs the functional form to match against, so defunctionalization is pinned to the end of the pipeline. And once an op produces its result by mutation instead of a consumed SSA value, dead-code elimination would see the call as unused and wrongly delete it — so the pass's own docstring warns "After this pass, DCE should never be run."
Source vllm/compilation/passes/utility/fix_functionalization.py — FixFunctionalizationPass with the defunctionalize / replace_users_with_mutated_args / insert_defunctionalized helpers and the per-op mutated_args table; wired to run last in pass_manager.py (self.fix_functionalization(graph)).

The op names matter: IR ops vs. fused C++ ops

What the model emits

Custom ops in the vllm_ir namespace — these are the matchable "before" tokens:

  • torch.ops.vllm_ir.rms_norm
  • torch.ops.vllm_ir.fused_add_rms_norm
  • quant matchers from matcher_utils.py

What fusion swaps in

Pre-compiled fused kernels in torch.ops._C (CUDA) — the "after":

  • rms_norm_static_fp8_quant
  • fused_add_rms_norm_static_fp8_quant
  • silu_and_mul_quant

Where do those op names come from?

Key subtlety: register_replacement never matches on a string. It traces the pattern and replacement callables (with the example inputs and the fwd_only tracer) into FX subgraphs, then matches on the op-overload identity of the nodes — the OpOverload objects themselves. The "names" in the snippet are just Python handles to ops registered in two different libraries:

"Before" ops → the vllm_ir library

vllm.ir.ops.rms_norm is a vLLM IR op, declared with a decorator whose function name becomes the op name:

# vllm/ir/ops/layernorm.py
@register_op
def rms_norm(input, weight, epsilon): ...
# vllm/ir/op.py — registers it into the vllm_ir torch library
Library("vllm_ir", "FRAGMENT").define("rms_norm" + schema)
# ⇒ matchable node:  torch.ops.vllm_ir.rms_norm.default

The model's RMSNorm layer emits the same IR op, so pattern and graph share the node target — that's the match.

vllm/ir/ops/layernorm.py, vllm/ir/op.py

"After" ops → the torch.ops._C library

rms_norm_static_fp8_quant is a hand-written CUDA kernel registered in the C++ bindings — that def string is where the name is born:

// csrc/.../torch_bindings.cpp
ops.def("rms_norm_static_fp8_quant(Tensor! result, "
        "Tensor input, Tensor weight, ...) -> ()");
ops.impl("rms_norm_static_fp8_quant", TORCH_BOX(&rms_norm_static_fp8_quant));
// ⇒ in Python:  torch.ops._C.rms_norm_static_fp8_quant.default

The pass doesn't hard-code it inline — it looks one up from the FUSED_OPS dict by FusedRMSQuantKey(quant_kind, fused_add) → self.FUSED_OP.

csrc/…/torch_bindings.cpp, layernorm_quant_kernels.cu

So when you write your own fusion The before ops are whatever your model's forward already emits — you don't invent them; read the layer (the RMSNorm layer emits vllm_ir.rms_norm; the quant layer emits a torch.ops._C.*_quant) and make your pattern call the exact same ops. The after op is the registered fused kernel: grep the C++ bindings for its def, or list [n for n in dir(torch.ops._C) if "rms" in n] at a REPL, then reference torch.ops._C.<name>.default. Because the match is by node identity, a stray reshape between your two before-ops breaks it — which is exactly what the NoOp pass below exists to prevent.

Order of passes (and why NoOp must go first)

The manager runs passes in a fixed order. NoOp elimination runs first and is mandatory: models are full of reshape/view ops that sit between the two ops you want to fuse, and a pattern like rms_norm → reshape → quant won't match rms_norm → quant. Collapsing those redundant reshapes is what lets the rest of the patterns hit.

4. RMSNorm + Quant — the workhorse

This is the single most-used fusion in a quantized vLLM model. Every transformer block has an input-layernorm and a post-attention layernorm, and in an fp8 model each is immediately followed by a quant before the next GEMM. That's two fusable sites per layer, so 80 layers → 160 fused kernels instead of 320 separate ones.

Four variants, one pattern family

The pass registers a Cartesian set of patterns keyed by (has residual-add, quant kind). The "has residual-add" axis is why there are paired classes everywhere: the residual stream is usually added inside the norm kernel (fused_add_rms_norm), so the fused quant op must also fold in the add.

By residual

  • plain — rms_norm → quant
  • + add — fused_add_rms_norm (residual) → quant

By quant kind

  • static per-tensor fp8 (scale known ahead)
  • dynamic per-token fp8 (scale from the data)
  • per-group fp8 (block size 128 / 64)
  • NVFP4 (Blackwell)
# rms_quant_fusion.py — the fused-op table (which kernel each key maps to)
FUSED_OPS = {
  (kFp8StaticTensorSym, False): torch.ops._C.rms_norm_static_fp8_quant,
  (kFp8StaticTensorSym, True ): torch.ops._C.fused_add_rms_norm_static_fp8_quant,
  (kFp8DynamicTokenSym, False): torch.ops._C.rms_norm_dynamic_per_token_quant,
  (kFp8Dynamic128Sym,   ...): torch.ops._C.rms_norm_per_block_quant,
  # ... 64-group, +residual rows ...
}
With residual: fused_add_rms_norm → quant ⟹ fused_add_rms_norm_static_fp8_quant

Unfused — 2 kernels

read x, residual
x += residual; RMSNorm
write y, new residual ↓
read y ↑ quant write q (fp8)
⟹

Fused — 1 kernel

read x, residual
add + RMSNorm + quant
write q (fp8) + new residual

The updated residual is still written (the next block needs it) — but y never hits HBM.

5. SiLU·Mul + Quant (SwiGLU MLP)

The gated MLP (SwiGLU) computes silu(a) * b where a, b are the two halves of the up-projection. In an fp8 model that activation is immediately quantized for the down-projection GEMM — the exact same read-compute-write-read-compute-write shape as §4, so the same fix applies.

gate·up (bf16) → SiLU(a)·b → act (bf16) ↓ → read ↑ → quant → fp8
⟹ gate·up (bf16) → silu_and_mul_quant → fp8 −1 launch, −1 round-trip

6. AllReduce + RMSNorm (+ Quant)

With tensor parallelism, each transformer block ends with an AllReduce that sums the partial outputs across GPUs. The very next thing is the next block's rms_norm (usually with the residual add). Naively that's: comm kernel → write summed tensor → read it back → norm kernel.

On NVIDIA GPUs with FlashInfer available, vLLM fuses the whole tail into one one-shot allreduce kernel that does the cross-GPU reduction and then the norm (and optionally the fp8/fp4 quant) in the same launch's epilogue — the reduced tensor is consumed in-place instead of bouncing through HBM and a second kernel.

TP block tail — fused all the way to fp8

Unfused — 3 kernels

AllReduce sum ↓
read ↑ add + RMSNorm y ↓
read ↑ quant fp8
⟹

Fused — 1 collective kernel

trtllm fused_allreduce_norm (+quant)

pattern code kARResidualRMSNormFP8Quant selects the all-in-one variant.

Size-gated, not always on The fused one-shot allreduce only wins for small messages (latency-bound). The pass checks the tensor size against a per-(device-capability, world-size) threshold (fi_allreduce_fusion_max_size_mb, e.g. ~1–64 MB) and falls back to the regular AllReduce + separate norm for large tensors, where NCCL's ring/tree algorithms are bandwidth-optimal.
Source allreduce_rms_fusion.py — AllReduceFusionPass with AllReduceRMSNormPattern, AllReduceFusedAddRMSNormPattern, and the ...StaticQuantFP8/NVFP4Pattern family; requires the flashinfer.comm module. ROCm has a parallel RocmAiterAllReduceFusionPass. Flag: fuse_allreduce_rms.

7. Attention + Quant (epilogue)

The attention kernel normally writes its output in bf16/fp16; a separate kernel then quantizes it for the output projection GEMM. This fusion is slightly different in shape: instead of replacing two ops with a third, it pushes the scale into the attention op itself. The matcher rewrites unified_attention_with_output(...) to take an output_scale, and the kernel quantizes in its epilogue — the result is written straight to fp8.

unified_attention → out (bf16) ↓ → reshape + read ↑ → quant → fp8
⟹ unified_attention_with_output(output_scale=scale) → writes fp8 directly
Why it's its own pass (and constrains the graph) Because the quant happens inside the attention kernel, the attention op cannot be split off into its own piecewise-cudagraph region the usual way. fuse_attn_quant therefore interacts with splitting_ops and graph partitioning (see the asserts in config/compilation.py). There is a separate MLAAttnQuantFusionPass for the MLA path.
Source attn_quant_fusion.py — AttnFp8StaticQuantPattern, AttnNvfp4QuantPattern; passes AttnQuantFusionPass / MLAAttnQuantFusionPass. The fused target is torch.ops.vllm.unified_attention_with_output. Flag: fuse_attn_quant.

8. Sequence parallelism: AllReduce → ReduceScatter + AllGather

This one doesn't save a kernel — it changes which collective you use so that the work between collectives shrinks, and so that the collectives can later fuse with neighboring GEMMs (§9). An AllReduce leaves the full tensor on every rank. The rewrite splits it into a ReduceScatter (each rank keeps only its 1/TP slice) and an AllGather (re-assemble before the next op that needs the whole thing). The total bytes moved by RS+AG equal the AR — but in between, the residual-add and RMSNorm run on only 1/TP of the rows per rank.

Same comm volume, 1/TP the norm work per rank
AllReduce → add + RMSNorm (full tensor)
⟹ ReduceScatter → add + RMSNorm (1/TP slice) → AllGather
Each rank normalizes only its shard → less compute and less activation memory. The real payoff is that RS and AG now sit next to GEMMs and can be overlapped with them (§9).
Gated by hidden size & token count SP only helps when the tensor is big enough that sharding the norm beats the extra collective bookkeeping. The pass disables itself below device-dependent thresholds on hidden_size (e.g. 8192 on H100/Blackwell, 4096 on XPU) and token count (sp_min_token_num). Requires TP > 1.
Source sequence_parallelism.py — SequenceParallelismPass with FirstAllReduceRMSNormPattern / MiddleAllReduceRMSNormPattern; comm ops torch.ops.vllm.reduce_scatter / all_gather. Flag: enable_sp.

9. Async TP: overlapping GEMM with its collective

Once §8 has turned the block boundaries into ReduceScatter / AllGather, the matmuls that feed or follow them become fusable. The point here is not saving HBM — it's latency hiding. A standard matmul → reduce_scatter runs the GEMM to completion, then sits idle on the SMs while the network does the scatter. The fused fused_matmul_reduce_scatter instead tiles the GEMM and pipelines it: as soon as one output tile is ready, its slice starts streaming over the interconnect (via PyTorch symmetric memory / P2P) while the next tile computes.

Serial vs. pipelined (GEMM → ReduceScatter)
unfused
GEMM (SMs)
ReduceScatter (net)
fused
GEMM tiles (SMs)
RS tiles (net) — overlapped
Communication slides under compute. Wall-clock ≈ max(GEMM, comm) instead of their sum.

10. Writing a fusion by hand (eager / breakable-CG)

When the pass manager is off (eager, or VLLM_USE_BREAKABLE_CUDAGRAPH=1) there is no matcher to rewrite your graph, so a fusion only happens if you call the fused op yourself from the model's forward. That sounds harder than it is — every hand-written fusion follows the same three-part recipe. The worked example below is the real MiniMax-M3 AllReduce + GemmaRMSNorm fusion (the spotlight in §11 is what it buys you).

  1. Producer — defer the intermediate. For a collective fusion, stop the producer from all-reducing so its raw per-rank partial flows downstream: a RowParallelLinear takes reduce_results=False. For a plain pointwise pair (norm→quant, act→quant) there is no producer change — just keep the two ops adjacent so the helper can see both. 
    # minimax_m3/nvidia/model.py — attention o_proj stops reducing
# "the attention all-reduce is fused with the following GemmaRMSNorm"
self.o_proj = RowParallelLinear(..., reduce_results=False, ...)
  2. Write the fused helper. One function that drives the merged kernel and returns exactly what the unfused pair returned. It must obey three rules — same return contract, a capability gate, and a fallback that is the unfused code: 
    # model_executor/layers/fused_allreduce_gemma_rms_norm.py (condensed)
def fused_allreduce_gemma_rms_norm(hidden_states, residual, norm):
    if tp_size == 1:
        return norm(hidden_states, residual)        # nothing to fuse

    ok, max_token_num = _can_use_flashinfer(hidden_states, tp_size)   # ← gate
    if ok:
        norm_out = torch.empty_like(hidden_states)
        flashinfer_trtllm_fused_allreduce_norm(       # the single fused kernel
            allreduce_in=hidden_states, residual=residual,
            rms_gamma=norm.weight, rms_eps=norm.variance_epsilon,
            weight_bias=1.0, norm_out=norm_out, ...)      # weight_bias=1.0 → Gemma (1+w)
        return norm_out, hidden_states               # (normed, new_residual) — same contract

    reduced = tensor_model_parallel_all_reduce(hidden_states)  # ← fallback = the unfused path
    return norm(reduced, residual)
  3. Wire it into forward, behind a guard that mirrors the producer. Replace the two ops with the helper. The guard must answer "is there actually a deferred all-reduce waiting for me?" — each clause maps back to a producer decision. 
    # minimax_m3/nvidia/model.py — decoder layer
self.fuse_input_allreduce = (
    layer_id > 0                                  # layer 0 input came from embeddings
    and not _is_moe_layer(config, layer_id - 1)  # MoE reduces internally → nothing deferred
    and pp_size == 1)                            # PP can't carry a partial across stages
...
if self.fuse_input_allreduce and residual is not None:
    hidden_states, residual = fused_allreduce_gemma_rms_norm(
        hidden_states, residual, self.input_layernorm)
else:
    ... # the original unfused add + norm, untouched
The three rules that keep a hand-written fusion correct (1) Identical contract — same return shapes/dtypes/meaning as the ops you replaced, so callers can't tell. (2) A fallback that is the unfused code — when the fast kernel can't apply, degrade to bit-for-bit the old path; that fallback is also your numerical reference. (3) A capability gate — dtype, shape, device, batch-size budget, workspace availability; opt out, never crash, when unmet.
Residual bookkeeping is the subtle part The fused kernel does AllReduce → + residual → RMSNorm and returns (normed, all_reduce(hidden)+residual) — identical to the unfused norm. It only stays correct because the deferring producer also did not fold itself into the residual yet. Break one half of that contract and the math drifts. Always diff logits against the fallback to confirm.

Which route should you use?

Automatic pass (§2)

  • You write a pattern/replacement pair + register_replacement
  • Matcher applies it across many models, for free
  • Needs torch.compile — dead under eager / breakable CG
  • Must express the before-pattern exactly; NoOp pass must clear reshapes first
vs

Hand-written op (this section)

  • You write a helper + edit the model's forward
  • Runs in any regime — no compiler dependency
  • Full control of guards & fallback
  • Only fuses where you put the call; logic lives in model code

11. Spotlight: MiniMax M3 — fused AllReduce + QK-norm

MiniMax M3 (and M2) shard each attention head's channels across the TP group, then RMSNorm Q and K per head before RoPE. Because a head's channels are split across GPUs, the norm's variance must be summed across ranks — so the natural code is: compute a partial per-token variance, AllReduce it, then normalize. That AllReduce sits on the critical path of every attention layer, at tiny message sizes (one fp32 scalar pair per token) where launch + latency dominate.

The fix is a hand-written custom op, minimax_allreduce_rms_qk, that does the cross-GPU variance reduction and the Q/K normalization in a single Lamport one-shot allreduce kernel. Lamport flags (set/poll in peer GPU memory over CUDA IPC) replace NCCL's handshake, so the reduction and the norm epilogue happen in one launch with no separate collective.

# minimax_rms_norm/rms_norm_tp.py — the fused QK-norm entry point
q, k = torch.ops.vllm.minimax_qk_norm_fusion(qkv, q_weight, k_weight, ...)
# dispatches to the fused C++ op only when it pays off:
if workspace is not None and tp_world > 1 \
   and num_tokens <= MINIMAX_QK_NORM_MAX_TOKEN_NUM:   # 2048
    return torch.ops._C.minimax_allreduce_rms_qk(qkv, q_weight, k_weight, workspace, ...)
return _minimax_qk_norm_fallback(...)     # eager allreduce + RMSNorm
When it falls back The fused kernel is decode/small-batch only: it bails to the eager allreduce+RMSNorm path above 2048 tokens (large prefills are bandwidth-bound, not latency-bound), when TP=1 (no reduction needed), or when the Lamport workspace can't be built — it needs CUDA IPC P2P peer access, so on consumer PCIe cards with P2P disabled, workspace allocation raises and M3 logs a warning and falls back instead of failing model load.
Why fp32 variance avoids the FlashInfer workspace The per-token variance is reduced in fp32 for stability. The general FlashInfer fused allreduce (§6) caches one global workspace keyed to the model's 16-bit activation dtype — routing an fp32 reduction through it would read a mismatched workspace and corrupt results. So _all_reduce_variance reduces a flattened (1D) fp32 view, which keeps it on custom all-reduce / pynccl. MiniMax's own Lamport workspace is separate, built specifically for the QK kernel.

Wired into attention at vllm/model_executor/models/minimax_m2.py (q_norm/k_norm), shared by M3. Fused C++ op: torch.ops._C.minimax_allreduce_rms_qk.

12. Spotlight: DeepSeek V4 — MLA mega-fusions

DeepSeek V4 is the most aggressively fused model in the tree. Its attention is sparse MLA (DeepSeek Sparse Attention with a "lightning indexer") over a compressed latent KV cache, and the pre- and post-attention plumbing is full of small pointwise ops — compress, normalize, rotate, quantize, write-to-cache. Left unfused, a single MLA step would launch a dozen tiny memory-bound kernels. V4 ships hand-written Triton/CUDA kernels (under vllm/models/deepseek_v4/common/ops/) that weld each of these chains into one launch, keeping every intermediate in registers.

Different from §4–§9 Those were generic compile-time pattern matches. These are model-specific custom ops called directly from the V4 model code (the hand-written recipe in §10) — chosen by hand because the op chains are unique to MLA + sparse attention and too specialized for the generic matcher. Each has NVIDIA CuTe-DSL, common-Triton, AMD, and XPU variants.

The KV compressor: the widest fusion in vLLM

Writing a token into the latent KV cache means: gather the raw state, softmax-compress it, RMSNorm the result, split into NoPE / RoPE parts, fp8-quant the NoPE part per block, GPT-J-rotate the RoPE part, and store both (plus the UE8M0 block scales) into the paged cache. That entire chain is one kernel:

gather state→ softmax compress→ RMSNorm→ split NoPE/RoPE→ fp8 quant (NoPE)→ RoPE→ store fp8 + bf16 + scales
⟹ _fused_kv_compress_norm_rope_insert_sparse_attn ~10 ops → 1 kernel, intermediates stay in registers

14. Turning fusions on

These flags govern the automatic fusions only. They are PassConfig flags inside CompilationConfig and take effect only under torch.compile (the default -O3/inductor path); in eager or breakable-CG mode there is no graph to rewrite, so they are no-ops. Most default to None and are resolved per platform/model during config post-init — you usually let vLLM decide, and override only to debug.

Hand-written fusions ignore these flags The §10–§12 fusions are explicit calls in model code, so they fire regardless of PassConfig. On a breakable-CG model (DeepSeek V4, MiniMax M3) the flags below do nothing — the fusion you get is whatever the model hand-wrote.
Pointwise: fuse_norm_quant fuse_act_quant fuse_attn_quant eliminate_noops (default True)
Collective: fuse_allreduce_rms enable_sp fuse_gemm_comms
RoPE/MLA: enable_qk_norm_rope_fusion fuse_rope_kvcache_cat_mla fuse_rope_kvcache 
# Example: force the pointwise fusions on from the CLI
vllm serve <model> \
  --compilation-config '{"pass_config": {"fuse_norm_quant": true, "fuse_act_quant": true}}'
eliminate_noops is load-bearing It defaults to True and you should leave it on: turn it off and the reshape ops between your norm and quant block the patterns from matching, silently disabling most of the fusions above.
Verify before trusting a number Flag names, defaults, and platform gating change often. The flags here were read from vllm/config/compilation.py (PassConfig) on branch dao/understand. Check VLLM_LOGGING_LEVEL=DEBUG compile logs — each pass logs how many patterns it matched — to confirm a fusion actually fired.

15. Reading tips & file map

How do I find which fused kernel a pattern maps to?

Open the pass file and look for the FUSED_OPS dict (or equivalent). The dict keys are QuantKey/FusedRMSQuantKey tuples describing the quant kind and whether a residual add is folded in; the values are the concrete torch.ops._C.* kernels. Then grep that op name in csrc/ for the actual CUDA implementation.

How do I confirm a fusion actually fired at runtime?

Each VllmPatternMatcherPass tracks a match count and logs it. Run with VLLM_LOGGING_LEVEL=DEBUG and watch for the per-pass "matched N patterns" lines during compilation. Zero matches means the pattern never hit — usually a stray reshape, a dtype-mismatch extra_check bailing out, or the flag being off.

Why are there separate CUDA and ROCm passes for the "same" fusion?

The fused kernels are backend-specific. CUDA paths lean on FlashInfer and torch.ops._C; AMD paths use rocm_aiter_ops with their own kernels and extra variants (e.g. bf16-norm fan-out for the DeepSeek indexer GEMM). The matcher logic is analogous, but the "after" op differs, so they're distinct classes.
