MSA Sparse Top-K Tile Selection

How the MSA indexer picks the 16 most relevant KV tiles per query — the kernel that turns a dense attention pass into a sparse one. A line-by-line walkthrough of sparse_topk_select, with worked examples.

TL;DR After the dense FMHA pass emits a max_score tensor (the peak QK logit of every KV tile, per query), this kernel selects the top-16 tiles by score for each (head, token) row so the second, sparse attention pass only touches those 16. It is not a sort — sorting up to 12 288 tiles per row would dominate the runtime. Instead it runs a multi-pass histogram top-k (vendored from TRT-LLM's indexerTopK): a monotonic float→bin trick lets a prefix-sum over bin counts find the score threshold below which exactly ~16 tiles sit, in a handful of O(K) passes. Two kernels run back-to-back: a transpose (lays K contiguous) and the histogram top-k + a 32-lane bitonic sort that injects the forced sink/window tiles and emits the 16 indices ascending, with out-of-range pages clamped to -1. A newer THK input layout lets the dense pass hand the indexer K-contiguous rows directly, skipping the transpose kernel entirely (§5).

1. What it computes & where it sits

MSA (MiniMax Sparse Attention) runs attention in two passes. The first is a cheap dense pass that, for every query token, scores all of its KV tiles and records the single highest QK logit in each tile. The second is the real sparse pass — but it only attends over a small subset of tiles. This kernel is the bridge: it reads the per-tile peak scores and decides which 16 tiles each query gets to keep.

A "tile" is one KV page (128 KV tokens). max_score[h][k][t] = the largest QK score that query token t (query head h) saw anywhere in tile k — a cheap proxy for "how much does this tile matter to this query".

Why two passes? A cheap scan, then an expensive read

Attending twice sounds wasteful — but the two passes do very different amounts of work per tile. Pass 1 is deliberately stripped down so it can afford to look at every tile; pass 2 runs the real, expensive attention math on only the 16 that survive.

Concretely: take a 32K-token context — 256 tiles of 128 KV tokens — with head dim d = 128, keeping topk = 16 tiles. For one query token:

Pass 1 — dense scan · all 256 tiles
  • QK + max only: dot the query (128-d) against each of a tile's 128 KV tokens, then keep the single largest logit — one fp32 out per tile.
  • No softmax, no PV, no output: it never reads V and never writes an attention vector.
  • MQA proxy KV (num_kv_heads_dense = 1): one shared key head, so K is read once rather than once per KV head.
vs
Pass 2 — sparse attention · 16 tiles
  • The full computation: QK and a softmax and the PV matmul that weighs all 128 value vectors and accumulates the result.
  • Reads V across the real (GQA) KV heads and writes the full output vector.
  • Per tile ≈ two matmuls + softmax + V traffic + output write.

So a pass-1 tile is far lighter than a pass-2 tile: it does only the QK half of the math, skips the softmax entirely, and its memory traffic is just one proxy key head — no V, no output. That asymmetry is the whole bet: spend a sliver on each of the 256 tiles to rank them (pass 1), then pay the full softmax-and-PV cost on only the 16 that matter (pass 2). Note pass 1 is still dense — at long context it touches far more tiles — so it is cheaper per tile, not necessarily in total; the win is that a plain dense attention would pay the full per-tile cost on all 256 tiles (256 / 16 = 16× more of the expensive work). sparse_topk_select is the hinge between the two passes: it turns pass 1's max_score into the 16-tile shortlist that pass 2 consumes.

The shapes (the whole contract in two lines)

in max_score : (num_qo_heads, max_k_tiles, total_qo_len) fp32, contiguous
out indices : (total_qo_len, num_qo_heads, topk=16) int32, ascending by tile idx, −1 padded

A "row" is one (head, token) pair; there are num_qo_heads × total_qo_len of them, and each is solved independently by one CTA. Within a row we choose the 16 largest of max_k_tiles scores. A few facts baked into the contract:

  • topk must be exactly 16. The host binding asserts it (sparse_topk_select.cu); the kernel is templated on MAX_TOPK≤64 but only the 16-instantiation is launched.
  • Padding slots are pre-filled with −inf. max_k_tiles is rounded up (to a multiple of 128 tiles), so the columns past the real KV length must already be −inf — the dense pass does this via torch.full. A −inf tile can never out-score a real one.
  • Output is ascending by tile index, not by score. The downstream sparse-FMHA kernel walks the selected tiles in index order, so the top-k result is sorted by index, and unused / invalid slots are −1 at the tail.

Made concrete: shapes end-to-end (k = 3)

Picture a concrete request: a ~768-token document already sits in the KV cache — that's max_k_tiles = 6 tiles of 128 KV tokens — and the user now sends a short 4-token query (say "what are the payment terms?"). This step therefore scores total_qo_len = 4 query tokens against the K = 6 cached tiles. The model has Hq = 2 query heads, and we keep topk = 3 tiles (16 in the real kernel). We trace the row (h = 0, t = 1) — query head 0, query token 1 — through all three tensors.

Two things these dimensions are not
  • qo and K are independent axes — there is no "6 keys per 4 queries" pairing. qo = 4 is how many query tokens we score now; K = 6 is how many tiles the KV history already spans. Scoring 4 query tokens in one shot like this is a prefill step (it ingests the whole query chunk at once); plain decode scores just qo = 1 token per step. Either way total_qo_len is the batch-flattened query-row count — so qo = 4 could equally be 4 sequences each decoding one token, and a long chat could have K in the thousands while qo stays 1. Every one of the 4 query rows looks at all 6 cached tiles and keeps its own best 3.
  • Hq counts query heads, and both passes have all of them. The dense (pass-1) "MQA proxy" collapses only the KV side to one shared key head — it does not drop query heads. So max_score's head dim is num_qo_heads (here 2), each query head scores against that single proxy key and picks its own top-k. The sparse pass uses the same num_qo_heads query heads (but real GQA KV heads, not the proxy).

In the real kernel only the magnitudes change: K climbs to ~12k tiles, topk is fixed at 16, and all Hq × qo rows are solved in parallel — one CTA each.

If qo = 4 is 4 different sequences, they have different keys — so what does a single K = 6 axis mean?

Nothing here is shared across sequences; the batch is just flattened into independent rows. Three facts make it work:

  • Each row is bound to one sequence. The varlen cu_seqlens (or the page table) say which sequence query row t belongs to, and the dense pass is varlen — so when it filled max_score[h][k][t] it scored token t against its own sequence's KV, not a shared key set. Four rows from four sequences already hold scores over four different key sets.
  • K is the padded maximum tile count across the batch. A sequence with only 3 tiles leaves tiles 3–5 as −inf padding and carries num_valid_pages = 3 — supplied per row via the num_valid_pages_per_token array for exactly these mixed-length batches (§5). Those padding tiles can never be selected and are clamped to −1.
  • A selected index is logical within that row's sequence. "Tile 2" for sequence A and "tile 2" for sequence B resolve — via each sequence's page table in the sparse pass — to completely different physical memory.

So K = 6 just means "no row in this batch has more than 6 tiles"; what each row actually attends to is private to its own sequence.

The kernel never reasons about sequences — it sees a flat list of total_qo_len rows. Batching is just concatenating rows; a bigger batch is more CTAs, nothing else. What changed going from one sequence to two — and what didn't:

  • Seq A's row is unchanged. It is solved by its own CTA and never sees that B exists — batching is pure parallelism, no cross-sequence interaction.
  • total_qo_len: 1 → 2. Rows are concatenated; cu_seqlens_q marks where Seq A's rows end and Seq B's begin.
  • max_k_tiles: stays 6 — the max over the batch. Seq B has only 3 tiles, so its row carries −inf in tiles 3–5 (padding it never had).
  • num_valid_pages: scalar 6 → per-row [6, 3]. This is the whole reason the per-token array exists: it tells Seq B's row that tiles 3–5 are padding, so they are clamped to −1 rather than selected. Pass a single scalar 6 here and Seq B could "select" tile 4 — a page its sequence doesn't have — and the sparse pass would read out of bounds.

(With Hq query heads, each token contributes Hq rows, so the count is Hq × (qoA + qoB) — but the batching logic above is identical.)

2. Why histogram top-k, not a sort

The obvious implementation — sort each row and take the front 16 — is the wrong shape for this problem:

Full sort per row
  • O(K log K) with K up to 12 288 tiles.
  • Produces a total order we don't need — we only want the set of 16, plus a cheap final ordering of those 16.
  • cub::BlockRadixSort over 512 threads sorts mostly-garbage lanes; v2.0 measured this at ~13 µs just for the final ordering.
vs
Histogram top-k
  • A few O(K) streaming passes — one fp16 pass resolves most rows.
  • Finds the score threshold that admits ~16 elements; only the handful of ties at the boundary need finer work.
  • Final 16 are ordered by a 32-lane warp bitonic sort (< 1 µs), not a block sort.

The algorithm is the standard "radix select / histogram top-k": treat each float as a sortable integer, bucket all K scores into a histogram, and use a prefix sum to find the bin where the cumulative count first reaches topk. Everything in higher bins is definitely in the top-16; only the elements inside that boundary bin are still contested, and we refine just those.

Background: radix select / histogram top-k from scratch

If you've not seen this before, here's the whole idea. To get the top-k you don't need a sorted array — you need selection: the set of the k largest, ignoring their order. The textbook serial answer is quickselect: pick a pivot, partition the values into "bigger" and "smaller", and recurse into whichever side still contains the cutoff. That's O(n) on average — but it's an awful fit for a GPU: the pivot is data-dependent, the partition is a sequential scan, and every row would branch differently.

Histogram select (a.k.a. radix select) keeps quickselect's "narrow down to the region that holds the cutoff" idea, but swaps pick-a-pivot-and-partition for two operations a GPU loves — counting into buckets and a prefix sum:

  1. Bucket. Drop all n values into B bins by their leading digit. Each value just does one atomicAdd into its bin — no value is ever compared against another.
  2. Scan from the best end. Walk the bins from largest to smallest, accumulating a running count. The first bin where the cumulative count reaches k is the threshold bin — the k-th-largest value lives somewhere inside it.
  3. Confirm the clear winners. Every value in a bin above the threshold is unambiguously in the top-k (there are fewer than k values better than it). Emit those directly.
  4. Refine the tie. The threshold bin may hold more candidates than the slots left to fill. Re-bucket only those by the next, finer digit and repeat from step 2 — until the bin fits the remaining slots, or you run out of digits (then the survivors are genuinely equal and any of them fill).

Worked example — top-3 of eight scores, bucketing by decimal digit (hundreds first, then tens):

742, 738, 251, 790, 700, 305, 759, 712

Pass 1 — bucket by the hundreds digit, scan from the top:

No bin sits above 7xx, so 0 winners are confirmed yet and all 3 must come from the six candidates inside 7xx. Six > 3, so we refine just those six on the next digit.

Bins above the threshold confirm 790 and 759 (2 winners). The threshold bin 74x holds exactly 1 candidate and we need exactly 1 more, so 742 takes the last slot — done. Result: [790, 759, 742], found in two passes. We never sorted — and never even looked twice at 305, 251, 700, 712, 738.

Why "radix", and why it isn't a sort We process the number one digit at a time, most-significant first — that's the "radix" part. But unlike a radix sort, which fully buckets every value at every digit, radix select only ever descends into the one bucket that straddles the cutoff. So the work collapses fast: pass 1 touches all n values, but each later pass touches only the threshold-bin survivors. A handful of O(survivors) passes beats an O(n log n) sort — and needs zero element-to-element comparisons, which is exactly why it parallelizes.

The kernel is this algorithm with two substitutions. A "digit" is a ~10-bit slice of the score's bit pattern, so each bin table has B = 1024 bins (§6b) and it finishes in at most four passes — one coarse fp16 pass plus three fp32 refinements (§6d). And because bins must be indexed by an integer, each float is bit-twiddled so its integer order matches its float order — inverted so that bin 0 holds the largest score, which is why "scan from the best end" becomes simply "scan bins 0, 1, 2, …" (§6a). "Confirm the winners" is the direct-emit path and "refine the tie" is the staging path (§6c).

Why transpose first?

Kernel ② solves one row — a fixed (query head, query token) — by streaming that row's K tile-scores. The snag is the axis order: max_score is (Hq, K, qo) = (query head, KV tile, query token), outermost to innermost. So the innermost (stride-1) axis is the query token (query heads are the outermost axis) — which means a single row's K tile-scores sit one every qo elements apart: strided, not contiguous. Scattered reads can't use the wide float4 loads the histogram relies on, so kernel ① reshuffles the data to put each row's K scores side by side first.

4. Kernel ①: the transpose

Kernel ① has exactly one job: a textbook 32×32 SMEM-tiled transpose per head that lays each row's K tile-scores contiguous (the layout fix from §3). It does not read or rewrite any score and does not force any tiles — forcing happens later, inside kernel ② (§6g), which is exactly why the THK path can skip this kernel entirely (§5). 5. Skipping the transpose: the THK fast path

Everything above assumes max_score arrives in HKT layout — (num_qo_heads, max_k_tiles, total_qo_len), token innermost. That layout is the only reason kernel ① exists: the transpose pays a full read+write of an Hq×K×qo fp32 scratch buffer purely to lay each row's K scores contiguous so the histogram can float4-load them. It computes nothing — it just moves data.

The fix is to ask the dense FMHA pass to emit max_score in the layout the indexer already wants: THK — (total_qo_len, num_qo_heads, max_k_tiles), with the tile dimension innermost. Now every (token, head) row is already K-contiguous, so the indexer reads the input directly and kernel ① is never launched.

Now suppose the histogram (§6) keeps the three highest-scoring tiles — {1: .90, 3: .70, 2: .50}, dropping tile 0 — and emits them ascending by index as [1, 2, 3]. The stride-aware write drops them into the (qo, Hq, topk) = (3, 2, 3) output with out_stride_t = 6, out_stride_h = 3, out_stride_k = 1: 

row_base = t*out_stride_t + h*out_stride_h = 2*6 + 1*3 = 15;
out[15] = 1;  out[16] = 2;  out[17] = 3;   // out_stride_k = 1 → three consecutive int32 slots

Same numbers either way — but the THK path produced [1, 2, 3] with one kernel and zero scratch, while HKT spent a full 2·4·3-element transpose first just to reach the same contiguous layout.

What had to move for the transpose to disappear

Forcing. Forced sink/local tiles used to be stamped at transpose store-time — which a THK row never visits. So the FLT_MAX rewrite lives inside the histogram kernel now (§6g): each processHistogramStep runs every score through an effective_logit() that returns FLT_MAX when IsForcedBlock(k, force_begin, force_end_start, …) is true. Forced tiles still land in bin 0 and always win — on both the HKT and THK paths, with no transpose needed to stamp them.

Output is now stride-aware. Every write to the index buffer goes through explicit out_stride_{t,h,k} instead of the old hard-wired (t·Hq + h)·topk + k offset. That lets the 16 indices be written directly into a strided slice of a larger topk_indices_buffer — no separate copy or scatter afterward.

SMEM config left the launch path. The per-launch cudaFuncSetAttribute (dynamic-SMEM size) that used to run inside the dispatcher now runs once at module load (ConfigureSparseTopKSelect / sparse_topk_select_init()). That drops a host API call per invocation and — more importantly — lets the whole selection be captured into a CUDA graph, since cudaFuncSetAttribute is not capturable.

Per-row valid-page counts + a short-row bypass. The kernel also gained an optional num_valid_pages_per_token array (per-query page counts for mixed-length batches) and an early exit: when a row's valid-page count is ≤ topk, every valid page must be emitted exactly once, so the histogram is skipped and the row is filled [0,1,…,nvp−1,−1,…] directly. This removes work for short rows and stops −inf padding ties from manufacturing duplicate or out-of-range selections.

Using it Pass max_score_layout="THK" to sparse_topk_select(...), and have the FMHA pass emit THK — its epilogue now writes max_score through strided max_score_stride_{t,h,k} rather than assuming a token-contiguous layout. HKT remains the default and is unchanged, so the fast path is fully opt-in.
Net effect on the critical path For a decode step, choosing THK removes one kernel launch and one full Hq×K×qo global-memory round-trip from the selection stage, drops the scratch allocation, and makes the stage graph-capturable. Kernel ② — the histogram top-k itself — is byte-for-byte the same algorithm described in §6.

6. Kernel ②: the histogram top-k

IndexerTopKWithSortKernel<16> launches one CTA of 512 threads per row (grid = total_qo_len × num_qo_heads). Each CTA owns a contiguous run of max_k_tiles scores and produces 16 indices. Here is the whole flow before we zoom in:

  1. Histogram a coarse fp16 view of the row into 1024 bins (§6a–b).
  2. Prefix-sum the bins to find the threshold bin where the cumulative count crosses 16 (§6b).
  3. Classify every element: above-threshold → emit straight to the output accumulator; in the threshold bin → stage it for tie-breaking (§6c).
  4. Refine the threshold bin with up to three fp32 passes, only if it held too many ties to stage (§6d).
  5. Parallel insertion-sort the staged ties and fill the remaining output slots (§6e).
  6. Warp bitonic sort the 16 winners ascending by index, clamp out-of-range pages to −1, write to GMEM (§6f).

6a. The float→bin monotonic trick

Histograms work on integers, and we need bin order to track value order. The classic trick: flip a float's bits so the unsigned integer compares the same way the float does. This kernel wants largest-first, so it arranges it so that a larger score maps to a smaller bin index: 

uint32_t bits = __float_as_uint(x);
bits = (bits & 0x80000000) ? bits          // negative: keep as-is (→ large bins)
                          : ~bits & 0x7fffffff; // positive: invert (bigger x → smaller bits)

So scanning bins from index 0 upward visits scores from highest to lowest — exactly what a "take the top 16" prefix sum wants. Forced FLT_MAX tiles fall in bin 0; −inf padding falls in the last bin and is structurally unreachable.

The bits are consumed most-significant first, across four stages, so each stage refines the previous one. Stage 0 is a fast, lossy fp16 cut; stages 1–3 walk the fp32 mantissa:

6b. Histogram + prefix-sum threshold

processHistogramStep does three things per stage: (1) every thread atomic-adds its elements into the 1024-bin SMEM histogram; (2) a cub::BlockScan turns bin counts into an exclusive prefix sum (count of elements in strictly-higher bins); (3) it finds the threshold bin — the first bin where the cumulative count reaches topk: 

// prefixSum = # elements in higher bins (strictly better scores)
if (prefixSum < topK) {
  int nextPrefixSum = prefixSum + binCount;  // includes this bin
  if (nextPrefixSum >= topK) {                // crossing point!
    smemThresholdBinIdx[0] = idx;             // THIS is the threshold bin
    smemFinalBinSize[0]   = nextPrefixSum - prefixSum;
  }
}

Everything in bins below the threshold is unambiguously in the top-16 (there are fewer than 16 better scores). The threshold bin is where the cutoff falls — it may hold more candidates than the remaining slots, so those are the only contested elements.

6c. Direct-emit vs. staging the boundary bin

A second streaming pass (processBins) classifies each element against the threshold:

bin < threshold

Definitely top-16. atomicAdd a slot in smemFoundTopKValues and write the index straight to smemOutput.

bin == threshold

Contested tie. If the bin fits (≤ 2048), stage (logit, index) into smemFinal.items for sorting. If it overflows, fall through to the next refinement stage.

bin > threshold

Worse than the cutoff. Ignored.

The output accumulator smemOutput lives in dynamic SMEM, sized at exactly topk × 4 bytes. The staging buffer smemFinal.items (capacity kNumFinalItems = 2048) shares a SMEM union with the histogram — they are never live at the same time.

6d. Multi-stage refinement (fp16 → fp32)

Why four stages? Because a single coarse bin can lump together more tied elements than the 2048-slot staging buffer can hold. When that happens, the kernel narrows its focus to only the elements that landed in the threshold bin (using isPartialMatch to test their high bits against the pattern fixed so far) and re-histograms them on the next slice of bits:

Refinement cascade — each stage only re-bins the previous threshold bin
stage 0fp16, all K

only that bin's elems

continueToNextStep is true iff the threshold bin still overflows kNumFinalItems. Most rows stop after stage 0. After stage 3 every survivor shares bits 31:2, so they're treated as equal and the topk window is filled directly.

Stage 0 working in fp16 is the speed win: converting to half halves the histogram-read bandwidth and resolves the vast majority of rows in one pass. The fp32 stages exist only as a correctness backstop for densely-tied rows.

6e. The parallel insertion sort

If we stopped before stage 3 (the threshold bin fit), the staged ties need ranking so we can take just the best topK − foundSoFar of them. It's an O(n²) rank computation, but parallel — each thread computes its own element's rank by counting how many staged elements beat it (ties broken by source index for determinism): 

for (int i = threadIdx.x; i < finalCount; i += blockDim.x) {
  int outIndex = 0;
  for (int j = 0; j < finalCount; j++)
    if (logit[i] < logit[j] || (logit[i] == logit[j] && i < j)) outIndex++;
  if (outIndex + baseIdx < topK)        // only the slots we still need
    smemOutput[outIndex + baseIdx] = index[i];
}

n here is the threshold-bin size (≤ 2048), and only the first few ranks survive the < topK guard, so the quadratic loop is cheap in practice.

6f. Warp bitonic sort + OOB clamp + write

At this point smemOutput[0..15] holds the 16 winning tile indices but in arbitrary order. The downstream sparse-FMHA kernel wants them ascending by index. Only warp 0 (32 lanes) does this — the other 480 threads exit. v2.1 replaced a full cub::BlockRadixSort<512> (which sorted 512 lanes, ~480 of them sentinel garbage, at ~13 µs) with a 32-lane bitonic sort that touches only the real slots (< 1 µs): 

uint32_t key = ~0u;                       // sentinel → sorts to the tail
if (lane < topK) {
  int32_t idx = smemOutput[lane];
  bool valid = (idx >= 0) && (idx < num_valid_pages); // ← OOB clamp folded in here
  key = valid ? idx : ~0u;
}
WarpBitonicSortAsc32(key, lane);            // 5-stage shuffle network
if (lane < topK)
  row_out[lane] = (key == ~0u) ? -1 : key;   // sentinels become -1
The num_valid_pages clamp (v2.5) A selected index in [num_valid_pages, max_k_tiles) points at a padding tile that has no entry in the real page table. Folding the clamp into the sort key (rewrite to ~0u → sorts to tail → written as −1) means the sparse-FMHA pass never dereferences an out-of-bounds page. This replaced an ~84–101 µs torch.where + sort + torch.where post-process that used to run in Python. Pass num_valid_pages = max_k_tiles to disable it.

Finally warp 0 writes the row to GMEM at the qo-outermost offset (t·num_qo_heads + qo_head)·topk + k, matching the (total_qo_len, num_qo_heads, topk) output layout. For topk=16 ≤ 32 the WarpBitonicSortAsc32 path (32 lanes × 1 key) is used; a WarpBitonicSortAsc64 (2 keys/lane) exists for the MAX_TOPK=64 template but isn't launched here.

6g. Forced tiles (sinks & local window)

Some tiles must be selected regardless of score: the first force_begin tiles (attention sinks) and the last force_end tiles of the row's valid range (the local / diagonal window, which includes the block the query token sits in). This is the part that used to live in the transpose (§4); it now runs right here, in processHistogramStep, so it behaves identically whether kernel ① ran (HKT) or the THK path skipped it.

The mechanism is to make a forced tile look like the best possible score. Before any logit is binned, it passes through effective_logit(), which swaps in FLT_MAX for a forced index — so it falls in bin 0 (§6a) and always wins, with zero special-casing in the rest of the histogram, insertion-sort, or final sort: 

// applied inside processHistogramStep, before extractBinIdx
auto effective_logit = [&](float logit, int idx) {
  uint32_t k = rowStart + idx;
  return IsForcedBlock(k, force_begin, force_end_start, row_num_valid_pages)
           ? FLT_MAX : logit;          // forced → bin 0 → always selected
};
// IsForcedBlock(k): (k < force_begin) || (k >= force_end_start && k < row_num_valid_pages)

The window is anchored per row, in the kernel, at force_end_start = row_num_valid_pages − force_end — pinned to that row's real KV extent (num_valid_pages, or its per-token value for mixed-length batches, §5), not the padded max_k_tiles. So "the last N tiles" means the last N real tiles — each row's own diagonal block — never padding. The host caps force_begin + force_end ≤ topk.

Worth stressing: force_begin / force_end are config constants (the model's init_blocks / local_blocks), not computed per call. MiniMax-M3 ships with sparse_init_block = 0 and sparse_local_block = 1 — so force_begin = 0 (no sink) and force_end = 1: each token forces exactly one block, its own diagonal block {nvp−1}, and the top-k picks the other 15 by score. What varies per token is num_valid_pages = pos / 128 + 1, which the kernel turns into the window via force_end_start = num_valid_pages − force_end.

To exercise both knobs at once, the table below uses a hypothetical force_begin = 1 (one sink) + force_end = 2 (a two-block window) — not M3's real (0, 1) — at page 128:

token posnvp = pos/128 + 1force_end_start = nvp − 2forced block idstop-k fills the rest

The pos = 100 row shows the clamp: force_end = 2 > nvp = 1, so force_end_start floors to 0 instead of underflowing into padding. Under M3's actual config force_begin = 0, force_end = 1, token 5000 forces just {39} (its own block) and the top-k picks the other 15.

Why forcing must live in the top-k, not the score kernel A caller's decode score kernel can bake forcing into the scores it writes (huge sentinels), but a fast prefill score path may write raw max-scores and force nothing. Doing the forcing here, in the shared top-k, is what guarantees every token still keeps its sink + diagonal blocks no matter which score kernel filled the buffer — drop the diagonal block and a token wouldn't attend to its own local context.

7. End-to-end worked example

Take one row of K=10 tiles, and (for legibility) pretend topk=3 instead of 16. These are the per-tile peak scores for one (head, token):

tile k 012 345 6789
score .10.97.32 .95.84.88 .05.40.20.15
emitted top-3 threshold bin (loses tie)

Bucket into descending-ordered bins of width 0.1 and prefix-sum from the top:

bin (desc)tiles in bincountprefix (higher bins)verdict
[.9,1.0)1 (.97), 3 (.95)200<3, 2<3 → not threshold; emit both
[.8,.9)5 (.88), 4 (.84)222<3 and 2+2≥3 → THRESHOLD bin
[.4,.5)7 (.40)14below cutoff → ignored
…………ignored
  1. Direct-emit the above-threshold bin: tiles 1 and 3 go straight to smemOutput. foundSoFar = 2.
  2. Stage the threshold bin (size 2 ≤ 2048): {5: .88, 4: .84}. No refinement needed — skip stages 1–3.
  3. Insertion-sort the staged pair: .88 ranks 0, .84 ranks 1. We need topK − foundSoFar = 1 more, so smemOutput[2] = 5; tile 4 is dropped.
  4. smemOutput = [1, 3, 5] (discovery order).
  5. Warp sort ascending by index → [1, 3, 5]. All < num_valid_pages, so no clamp.
  6. Write [1, 3, 5, −1, −1, …] (16 slots; here padded after 3).

Result: tiles 1, 3, 5 = scores .97, .95, .88 — the three largest, exactly. Note that the histogram never sorted the seven losing tiles; it spent work only on the four tiles at and above the cutoff.

What if tile 4 had been forced (local window)?

The transpose would have written FLT_MAX into tile 4's slot. It would then land in bin 0, get direct-emitted alongside tiles 1 and 3 (foundSoFar = 3), and the threshold would shift up. Tile 5 (.88) would now be the one dropped. Forcing is just "make the score infinite", so it falls out of the same machinery with no special case.

What if 5000 tiles all scored exactly 0.84 (a huge tie)?

The threshold bin would hold 5000 > 2048 elements, overflowing the staging buffer (continueToNextStep = true). The kernel would re-histogram only those 5000 on the fp32 high bits (stage 1), then mid (stage 2), then low (stage 3). Since they're genuinely equal, stage 3 ends with all survivors sharing bits 31:2 and the remaining slots are filled directly from the bin — any 16 of the tied tiles, deterministically by source index.

8. Edge cases & the contract

Trivial path (K ≤ 16)

Nothing to choose — every tile is kept. SparseTopKIdentityFillKernel writes [0,1,…,K−1] then −1 padding, applying the same num_valid_pages clamp. Skips the transpose and the histogram entirely.

Upper limit (K ≥ 12288)

Returns cudaErrorNotSupported. The insertion-sort staging path only covers K < 12288 tiles (< ~1.5M tokens at page_size 128); the radix-sort path for larger rows isn't implemented.

Forced blocks

force_begin sink tiles + force_end window tiles are pinned via FLT_MAX inside the histogram kernel (§6g), not the transpose. Host caps force_begin + force_end ≤ topk, and the window anchors to the row's num_valid_pages, not the padded length.

Padding vs. OOB — two different guards

−inf pre-fill keeps padding tiles from winning on score; num_valid_pages keeps any that are selected (via forcing or ties) from producing an index the sparse pass would dereference out of bounds. Both are needed.
