Reading the KDA Kernel

Kimi Delta Attention — a per-channel gated delta-rule linear attention, in Triton. A guided tour of what goes in, what comes out, and every step in between, with worked numeric examples. Grounded line-by-line in vLLM source.

Want the actual Triton, annotated? This page is the what & why. Its companion KDA_CODE.html is a line-by-line walk-through of three real kernels — start there if you're learning to read kernel code.
TL;DR KDA is linear attention: instead of an O(T²) softmax over all past tokens, each head keeps a fixed-size state matrix S ∈ ℝ^{V×K} (128×128) that is updated one token at a time. The update is the delta rule (write the prediction error, not the raw value) plus a per-channel forget gate (each of the 128 key-dimensions decays independently). Two implementations compute the same math: a token-by-token recurrent kernel for decode, and a chunked kernel that batches 64 tokens into matmuls for prefill. The gate is what makes it "Kimi": it is a fine-grained vector, not the scalar of Gated DeltaNet / Mamba-2.

1. Where KDA sits

Kimi-Linear interleaves two kinds of token-mixing layers. ~3 out of every 4 layers are KDA (linear, constant-memory); the rest are ordinary full softmax attention (MLA). In the released Kimi-Linear-48B-A3B there are 27 layers: 20 KDA, 7 full-attention.

Softmax attention (the other 7 layers)

  • Keeps every past K/V in the cache.
  • Cost grows with sequence length: O(T²) compute, O(T) memory.
  • Token t can look back at all tokens exactly.

KDA linear attention (the 20 layers)

  • Compresses the past into a fixed state S: 128×128 per head.
  • Cost is O(T) compute, O(1) memory in sequence length.
  • The state is a lossy, gated running summary — old info decays.
The family tree Linear attention → + delta rule (DeltaNet) → + a forget gate (Gated DeltaNet, used by Qwen3-Next/Mamba-2) → make the gate per-channel instead of scalar = KDA. The vLLM kernel literally reuses the Gated-DeltaNet recurrent kernel and flips an IS_KDA flag to switch the gate from a scalar multiply to a per-K-dimension vector multiply.

Two side-projections feed the kernel: the decay gate g1 (low-rank f_a_proj→f_b_proj) and beta (b_proj, then sigmoid). A separate output gate g2 (g_a_proj→g_b_proj) is applied after the kernel.

Two L2-norm + scale details that live inside the kernel use_qk_l2norm_in_kernel=True means q and k are L2-normalized per head right before use, and q is multiplied by scale = 1/√128. You will not see a separate norm op — it is fused into the first few lines of every KDA kernel.

The state is the whole point Softmax attention stores a growing KV cache. KDA stores exactly 32 × 128 × 128 floats per sequence regardless of how long the sequence is. That one matrix per head is the compressed memory of everything seen so far.

4. The core recurrence (the 4 lines)

Every KDA kernel — recurrent or chunked — computes this exact per-token update. S is the [V, K] state for one head. Read ⊙ as elementwise, ⊗ as outer product.

(a) decay     S ← S ⊙ exp(g)broadcast over V    # each K-column scaled by its own gate
(b) delta     u = β · ( v − S·k )         # prediction error, scaled by write strength
(c) write     S ← S + u ⊗ k           # rank-1 update: store the corrected value at key k
(d) read      o = S·q                # query the updated state

Why "delta rule"?

Naïve linear attention writes S += v ⊗ k — it appends every value. The delta rule first reads what it would predict for key k (that's S·k), then writes only the error v − S·k. This makes the state overwrite stale associations for a key rather than pile up — like one step of online gradient descent on a key→value memory.

Why the gate?

Step (a) multiplies the state by exp(g) with g ≤ 0, so exp(g) ∈ (0,1]: a forget factor. In KDA g is a vector of length K=128 — every key-dimension forgets at its own learned, input-dependent rate. That is the KDA difference; Gated DeltaNet uses one scalar per head.

These 4 lines, verbatim, in the decode kernel fused_recurrent.py:135–148 
b_h *= exp(b_gk[None, :])        # (a) per-K-channel decay   b_h is [BV, BK] = [V, K]
b_v -= tl.sum(b_h * b_k[None, :], 1) # (b) v - S·k   (sum over K)
b_v *= b_beta                       # (b) × β
b_h += b_v[:, None] * b_k[None, :]  # (c) S += u ⊗ k
b_o  = tl.sum(b_h * b_q[None, :], 1) # (d) o = S·q

5. Worked decode example

Let's shrink to K = V = 2 and walk two tokens through the recurrence by hand. Start with empty state S = 0, scale = 1 (so we can see the arithmetic).

Token 1  k=[1, 0], v=[2, 3], β=0.5, exp(g)=[0.9, 0.8], q=[1, 0]

  1. decay: S is still 0, so 0 ⊙ [0.9, 0.8] = 0. Nothing to forget yet.
  2. delta: S·k = 0, so the error is v − 0 = [2, 3]. Scale by β: u = 0.5·[2,3] = [1, 1.5].
  3. write: u ⊗ k = [1,1.5] ⊗ [1,0] writes u into the column for key-dim 0:
    S =
    10 1.50
  4. read: o = S·q = S·[1,0] picks column 0 → o = [1, 1.5].

Token 2  k=[0, 1], v=[1, 1], β=1.0, exp(g)=[0.9, 0.8], q=[0, 1]

  1. decay: scale each column by its gate (col0×0.9, col1×0.8): 0.90 1.350 The memory written for key-dim 0 has faded ~10%.
  2. delta: S·k with k=[0,1] reads column 1 → [0, 0]. Error = v − [0,0] = [1,1], β=1 → u = [1, 1].
  3. write: u ⊗ k writes into column 1: 0.91 1.351
  4. read: o = S·q = S·[0,1] picks column 1 → o = [1, 1].
What you just saw Each distinct key direction owns a "column" of the state. Writing at a new key fills its column; the gate slowly fades older columns; reading with a query is a dot-product lookup across columns. The delta rule (step b) means re-writing the same key replaces rather than accumulates — try repeating a key and watch S·k become non-zero, shrinking the error.

6. How the gate is built

The kernel consumes g as a finished log-decay, but it is assembled from three learned pieces. This is the Mamba-style discretization, done per channel (kda.py kda_gate_fwd_kernel / fused_kda_gate):

g_raw = f_b_proj( f_a_proj(h) ) + dt_bias    # [T, 32, 128] low-rank logits + per-channel bias
g = −exp(A_log) · softplus(g_raw)        # A_log per head, softplus per channel

−exp(A_log)

A = −exp(A_log) is guaranteed negative. It is the per-head base decay rate, learned. Negative ⇒ g ≤ 0 ⇒ exp(g) ∈ (0,1] is a valid forget factor.

softplus(g_raw)

softplus(x)=log(1+eˣ) ≥ 0 is the per-token, per-channel "Δt". Large input ⇒ big timestep ⇒ strong forget. The kernel uses the linear approximation softplus(x)≈x when x > 20 to avoid overflow.

per-channel

Because g_raw has shape […, 128], every key-dimension gets its own decay. That vector g is exactly the exp(g) applied column-wise in step (a).

The corresponding source (linear region picked when β·x > threshold): 

# kda.py — fused_kda_gate / kda_gate_fwd_kernel
b_a = -tl.exp(A_log)                                 # per-head, negative
g_scaled = b_g * beta
sp = tl.where(g_scaled > threshold, b_g,             # softplus, with linear tail
              (1.0 / beta) * log(1.0 + tl.exp(g_scaled)))
b_y = b_a * sp                                       # = g, the per-channel log-decay
Two different "gates", don't confuse them g / g1 here is the decay gate inside the recurrence (step a). The separate g2 from g_a_proj/g_b_proj is the output gate, applied after the kernel by o_norm as RMSNorm(o) × sigmoid(g2). And β (from b_proj) is yet another scalar — the delta-rule write strength in step (b).

7. Decode path: fused_recurrent_kda

DECODE Used when every sequence contributes just 1 (or a few) new tokens. One Triton program per (sequence, head), looping over the sequence's tokens and doing the 4-line update each step. fused_recurrent.py:27, dispatched with IS_KDA=True.

grid = (NK=1, NV, N·HV) — one program owns a [BV, K] slice of the state
state S [V,K] BV=8 rows 8 8 … 8 V=128 split into NV=16 row-blocks; K=128 kept whole (BK=128)
Each program holds b_h of shape [BV=8, BK=128] in registers and marches through time. BV is capped at 8 (vs 32 for plain GDN) to keep the per-channel gate's register pressure down — see fused_recurrent_kda_fwd line BV = min(npo2(V), 8).

The loop, annotated

b_h = load(initial_state[slot])          # [BV, K], the carried state for this seq+head
for t in range(T):                       # T tokens of this sequence
    b_q, b_k, b_v = load q[t], k[t], v[t]
    b_q = l2norm(b_q); b_k = l2norm(b_k)  # USE_QK_L2NORM_IN_KERNEL
    b_q = b_q * scale
    b_gk = load g[t]                      # [K]  per-channel decay (already = −exp(A_log)·softplus)
    b_h *= exp(b_gk[None, :])             # (a)
    b_v -= tl.sum(b_h * b_k[None, :], 1)  # (b)
    b_v *= b_beta                         # (b)
    b_h += b_v[:, None] * b_k[None, :]   # (c)
    b_o  = tl.sum(b_h * b_q[None, :], 1)  # (d)
    store o[t] = b_o
    store final_state[slot] = b_h         # write back every step (multi-token decode safe)
Paging & speculative decode ssm_state_indices maps each sequence to a cache slot; a slot ≤ 0 (NULL_BLOCK_ID) is skipped. With speculative decoding, the kernel seeds from the state after the last accepted token (num_accepted_tokens − 1) so rejected draft tokens never corrupt the carried state.

8. Prefill path: chunk_kda (5 sub-kernels)

PREFILL A long prompt has thousands of tokens; running them one-at-a-time wastes the GPU. The chunked path splits the sequence into chunks of BT=64 and expresses the work as matmuls: parallel within a chunk, sequential across chunks. It computes the identical result as the recurrence. Driver: _chunk_kda_fwd_with_cumulative_g (kda.py:1306).

Sequence → chunks: within-chunk parallel, cross-chunk recurrent
chunk 0
tok 0–63 →S→ chunk 1
tok 64–127 →S→ chunk 2
tok 128–191 →S→ …
Inside a chunk, the 64×64 token interactions are dense matmuls. Only the carry of the state matrix S from one chunk to the next is sequential (NT steps, not T).

The pipeline of 5 kernels

Each row is one Triton launch. Shapes assume one head; BT=64.

  1. Gate cumsum — fused_kda_gate_chunk_cumsum
    Builds g (the same −exp(A_log)·softplus gate as decode) and takes its cumulative sum along time within each chunk. So g[t] now holds Σs≤t decays. Then the decay from token j to token i in the same chunk is just exp(g[i] − g[j]). Scaled by RCP_LN2 (see §9).
    out: g [1, T, 32, 128]
  2. Intra-chunk Gram matrices — chunk_kda_scaled_dot_kkt_fwd → A, Aqk
    For each 64-token chunk, builds two 64×64 matrices, gate-weighted by exp2(gi − gj):

    A = β · (K Kᵀ)

    strictly lower-triangular key–key interactions; the system of within-chunk delta corrections.

    Aqk = scale · (Q Kᵀ)

    lower-triangular (incl. diagonal) query–key scores; the within-chunk part of the output.

  3. Invert the triangular system — solve_tril → A = (I + A)⁻¹
    The delta rule inside a chunk is a sequential dependency (token i's correction depends on earlier corrections). Inverting I + A resolves all of them at once — this is the "UT transform". A must be 16/32/64-wide; KDA uses 64.
  4. Resolve per-token updates — recompute_w_u_fwd → w, u, kg
    Applies the inverted matrix: u = A·(β v) (corrected values), w = A·(β k·exp2(g)) (key-side update weights), and kg = k·exp2(glast − g) (keys re-aligned to the chunk's end so they can update the carried state).
  5. Cross-chunk state recurrence + output — two kernels:
    • chunk_gated_delta_rule_fwd_h (chunk_delta_h.py) — the only sequential-over-chunks loop. For each chunk it: snapshots the incoming S into h; computes new values v_new = u − w·Sᵀ; decays S by the chunk's total per-channel gate exp2(glast); then S += kgᵀ·v_new.
    • chunk_gla_fwd_o_gk — forms the output by adding the intra-chunk term (Aqk masked)·v_new and the inter-chunk term (q·exp2(g))·hᵀ (query attends to the state carried into the chunk).

The one mental model for the chunk path Steps 2–4 turn "64 sequential delta-rule updates" into "a few 64×64 matmuls" using the triangular-inverse trick. Step 5a is the only part that still walks chunk-by-chunk, and it does so with the same decay→correct→write structure as the decode recurrence — just on whole chunks at once. Step 5b reads the state to produce outputs.

9. The exp2 / cumsum trick

You will see exp2(...) everywhere in the chunk kernels and a mysterious g = g * RCP_LN2 after the cumsum. Here is why.

Hardware likes exp2

GPUs have a fast native exp2 (base-2) instruction. To use it while still computing eˣ, rewrite eˣ = 2^(x / ln2) = exp2(x · RCP_LN2) where RCP_LN2 = 1/ln2 ≈ 1.4427.

Pre-scale once

So the driver scales the cumulative gate g ← g · RCP_LN2 once (kda.py:1400), and every downstream kernel can call exp2(g) directly and get exp(original g). The decode path doesn't cumsum, so it uses plain exp.

And why cumulative sums? Within a chunk the state decays multiplicatively across tokens. The product of per-step decays telescopes into a difference of cumulative log-decays:

∏j<s≤i exp(decays) = exp( Σs≤i decays − Σs≤j decays ) = exp( g[i] − g[j] ) = exp2( g[i] − g[j] )

That single subtraction is how every gate-weighting term in steps 2, 4, and 5 is computed — you'll spot exp2(b_g − b_gn) (decay to the chunk anchor) and exp2(b_gn − b_gk) patterns throughout.

10. Reading tips & file map

How to read these kernels

  • Find the state first. b_h shape [BV, BK] = [V, K] is S. Everything else updates it.
  • Map to the 4 lines (§4). Every kernel is decay → delta → write → read, possibly batched.
  • make_block_ptr(base, shape, strides, offset, block, order) is just "give me this tile"; the (H*K, 1) strides skip across the head dimension.
  • bos/eos come from cu_seqlens — they bound one sequence inside the packed T axis.
  • Ignore autotune configs on first read; they only pick BK/BV/num_warps.
  • i_t, i_v, i_bh = tl.program_id(...) tells you what slice this program owns.

File map

  • fla/ops/kda.py — entry points, gate, KKt, recompute_w_u, GLA output.
  • fla/ops/fused_recurrent.py — the decode recurrence (IS_KDA flag).
  • fla/ops/chunk_delta_h.py — cross-chunk state recurrence (step 5a).
  • fla/ops/cumsum.py, solve_tril.py, l2norm.py — the helpers.
  • mamba/gdn/kimi_gdn_linear_attn.py — the layer; prefill vs decode dispatch.
