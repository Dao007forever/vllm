Reading KDA Kernel Code

An annotated, line-by-line Triton walk-through. KDA.html explained what the kernel computes; this page teaches you to read the actual code — the idioms first, then three real kernels from easiest to hardest.

TL;DR A Triton @triton.jit function is the body of one program in a grid of thousands. You read it in four moves: (1) what does tl.program_id slice this program to? (2) what tile does each make_block_ptr / pointer load? (3) what's the math on those tiles? (4) where does it store? Master those and every kernel below reads the same way.

1. The Triton mental model

A CUDA kernel is written from the viewpoint of one thread. A Triton kernel is written from the viewpoint of one program (≈ one CUDA block) that operates on whole tiles (small arrays) at once. Triton handles the threads-within-a-block for you; you think in [BT, BK]-shaped blocks and write what looks like NumPy. The compiler turns tl.dot, tl.load etc. into tensor-core and vectorized memory ops.

The launch grid

The host launches kernel[grid](...) where grid is a tuple like (NT, NC*NC, B*H). Triton runs the function once for every point in that 3-D grid, in parallel. Inside, tl.program_id(0..2) tells a program which point it is — and therefore which slice of the tensors it owns.

constexpr vs runtime

Arguments typed tl.constexpr (e.g. BT, K, IS_VARLEN) are compile-time constants — Triton generates a specialized kernel per distinct value, so if IS_VARLEN: costs nothing at runtime. Plain args (pointers, T) are runtime.

Everything is a flat pointer + strides A tensor q: [B, T, H, K] arrives as a bare pointer to element 0. To read element (b,t,h,k) you compute the offset yourself: ((b*T + t)*H + h)*K + k. The "stride" between consecutive tokens is H*K, between heads is K, etc. Half of kernel-reading is just decoding these offset expressions back into indices.

3. Kernel 1 — the decode recurrence

fused_recurrent_gated_delta_rule_fwd_kernel  easy

The token-by-token recurrence. Best first kernel: raw pointers (no block_ptr), and it maps 1:1 to the 4-line math. fla/ops/fused_recurrent.py:27

grid(NK=1, NV, N·HV) — one program per (sequence, value-block, head) ownsa [BV=8, BK=128] tile of the state S loopsover the sequence's T tokens

3.1 — Figure out who you are

i_k, i_v, i_nh = tl.program_id(0), tl.program_id(1), tl.program_id(2)
i_n, i_hv = i_nh // HV, i_nh % HV       # sequence index, value-head index
i_h = i_hv // (HV // H)                 # map value-head → key-head (GQA). KDA: HV==H ⇒ i_h==i_hv
The 3rd grid axis packs both sequence and head into one number; these two lines unpack it. HV is the number of value heads, H the number of key/query heads — they differ only under grouped-query attention. KDA uses neither (HV = H = 32), so i_h == i_hv. 
if IS_VARLEN:
    bos, eos = tl.load(cu_seqlens + i_n), tl.load(cu_seqlens + i_n + 1)
    all = T
    T = eos - bos                       # T now means THIS sequence's length
else:
    bos, eos = i_n * T, i_n * T + T
    all = B * T
if T == 0: return                       # empty sequence ⇒ nothing to do
Standard continuous-batching preamble. After this, all token indexing is relative to bos. all (total tokens across the batch) is only used later to stride the output tensor.

3.2 — Build pointers to this program's data

o_k = i_k * BK + tl.arange(0, BK)     # [128] the K indices  (i_k=0, BK=128)
o_v = i_v * BV + tl.arange(0, BV)     # [8]   the V indices this program owns

p_q = q + (bos * H  + i_h ) * K + o_k    # → q[bos, i_h, :]   shape [BK]
p_k = k + (bos * H  + i_h ) * K + o_k
p_v = v + (bos * HV + i_hv) * V + o_v    # → v[bos, i_hv, :]  shape [BV]
p_beta = beta + bos * HV + i_hv          # scalar per (token, head) — KDA beta is [.,.,H]
p_gk = g + (bos * HV + i_hv) * K + o_k   # IS_KDA: per-CHANNEL gate, shape [BK]
p_o = o + ((i_k * all + bos) * HV + i_hv) * V + o_v

Each p_* points at the first token's slice for this head. Note the strides baked into the offsets: tokens are H*K apart for q/k, HV*V apart for v/o, HV*K for the per-channel gate. We'll advance by exactly those at the bottom of the loop.

The if not IS_KDA: p_g = g + bos*HV + i_hv branch (a scalar gate) is the Gated-DeltaNet sibling. IS_KDA=True takes the per-channel p_gk branch — that single flag is the entire KDA-vs-GDN difference in this kernel.

3.3 — Load the carried state

b_h = tl.zeros([BV, BK], dtype=tl.float32)   # the state tile S[v, k]
if USE_INITIAL_STATE:
    if IS_CONTINUOUS_BATCHING:
        i_t = (tl.load(num_accepted_tokens + i_n) - 1) if IS_SPEC_DECODING else 0
        state_idx = tl.load(ssm_state_indices + i_n * stride_indices_seq + i_t)
        if state_idx <= 0: return               # NULL_BLOCK_ID ⇒ no state slot
        p_h0 = h0 + state_idx * stride_init_state_token
    else:
        p_h0 = h0 + bos * HV * V * K
    p_h0 += i_hv * V * K + o_v[:, None] * K + o_k[None, :]   # [BV, BK] tile
    b_h += tl.load(p_h0, mask=mask_h, other=0).to(tl.float32)
The state lives in a paged cache: ssm_state_indices maps a sequence to a physical slot (like a KV-cache block table). state_idx ≤ 0 means "no slot" → bail. For speculative decoding, i_t = num_accepted_tokens − 1 seeds from the state after the last accepted token, so rejected draft tokens never poison the recurrence. The last line materializes the [BV, BK] tile via 2-D broadcast offsets.

3.4 — The recurrence (the 4 lines)

for i_t in range(0, T):
    b_q = tl.load(p_q, mask=mask_k, other=0).to(tl.float32)
    b_k = tl.load(p_k, mask=mask_k, other=0).to(tl.float32)
    b_v = tl.load(p_v, mask=mask_v, other=0).to(tl.float32)

    if USE_QK_L2NORM_IN_KERNEL:                       # fused L2 norm
        b_q = b_q / tl.sqrt(tl.sum(b_q * b_q) + 1e-6)
        b_k = b_k / tl.sqrt(tl.sum(b_k * b_k) + 1e-6)
    b_q = b_q * scale

    b_gk = tl.load(p_gk).to(tl.float32)              # [BK] per-channel decay
    b_h *= exp(b_gk[None, :])                       # (a) decay each K-column of S
    b_v -= tl.sum(b_h * b_k[None, :], 1)            # (b) v − S·k   (reduce over K)
    b_beta = tl.load(p_beta).to(tl.float32)
    b_v *= b_beta                                     # (b) × β
    b_h += b_v[:, None] * b_k[None, :]             # (c) S += u ⊗ k
    b_o = tl.sum(b_h * b_q[None, :], 1)             # (d) o = S·q
    tl.store(p_o, b_o.to(p_o.dtype.element_ty), mask=mask_v)
This is the heart, and it's exactly the math from KDA.html §4. Things to notice as a reader:
  • b_h is [BV, BK] = [V, K], so b_k[None,:] broadcasts along V and tl.sum(..., 1) reduces along K. Getting the axis right is the whole game.
  • All compute is float32 (.to(tl.float32) on every load) even though storage is bf16 — linear-attention states are numerically touchy.
  • The gate is applied to the state, not to k or v. That's the "gated" in gated delta rule.

3.5 — Write back & advance

    if INPLACE_FINAL_STATE:
        final_state_idx = tl.load(ssm_state_indices + i_n * stride_indices_seq + i_t)
        if final_state_idx > 0:
            p_ht = ht + final_state_idx * stride_final_state_token + i_hv*V*K + o_v[:,None]*K + o_k[None,:]
            tl.store(p_ht, b_h, mask=mask_h)            # persist S after THIS token

    p_q += H * K;  p_k += H * K          # advance one token (the strides from §3.2)
    p_v += HV * V; p_o += HV * V
    p_gk += HV * K
    p_beta += HV
The state is written every token (not just at the end) because a single decode step may carry several tokens (speculative decoding), and each needs its own checkpoint. Then the pointers step forward by exactly the per-token strides we identified in §3.2 — a tidy way to walk the sequence without recomputing offsets.
Check yourself: why is BV capped at 8 here but 32 in the generic GDN kernel? Each program holds b_h: [BV, BK] = [BV, 128] in registers plus the per-channel gate vector. The per-channel gate adds register pressure the scalar-gate GDN path doesn't have, so KDA shrinks the value tile to 8 rows (BV = min(next_power_of_2(V), 8) in fused_recurrent_kda_fwd) to avoid spilling. NV = 128/8 = 16 programs cover the V dimension per head.

4. Kernel 2 — the gate cumsum (a trick worth stealing)

kda_gate_cumsum_fwd_kernel  medium

Used only on the prefill path. Computes the per-channel decay and its within-chunk cumulative sum in one launch. Teaches two things: make_block_ptr tiles, and the "cumsum as a matmul" trick. fla/ops/kda.py:1182

grid(⌈D/BD⌉, NT, B·H) — one program per (channel-block, chunk, head) inraw gate g [B,T,H,D], per-head base A_log, g_bias outcumulative log-decay y [B,T,H,D], scaled by 1/ln2 
p_g = tl.make_block_ptr(g + (bos*H + i_h)*D, (T, D), (H*D, 1),
                       (i_t*BT, i_d*BD), (BT, BD), (1, 0))
b_g = tl.load(p_g, boundary_check=(0, 1)).to(tl.float32)    # [BT, BD] one chunk × channel-block
if HAS_BIAS:
    b_g += b_bias[None, :]                                     # per-channel bias, broadcast over time
First real make_block_ptr. Decode its 6 arguments:
  1. g + (bos*H + i_h)*D — base, advanced to this sequence+head.
  2. (T, D) — the logical 2-D shape this pointer indexes (time × channels).
  3. (H*D, 1) — strides: stepping one token skips H*D elements; one channel skips 1.
  4. (i_t*BT, i_d*BD) — the tile's top-left corner (chunk i_t, channel-block i_d).
  5. (BT, BD) — tile size: 64 tokens × BD channels.
  6. (1, 0) — memory order (last dim contiguous).
boundary_check=(0,1) zero-fills the ragged last chunk. So b_g is a clean [64, BD] tile regardless of where the sequence ends. 
b_a = -tl.exp(tl.load(A + i_h).to(tl.float32))       # per-head NEGATIVE base decay
b_g_scaled = b_g * beta
b_softplus = tl.where(b_g_scaled > threshold,        # softplus(g) with linear tail for stability
                      b_g,
                      (1.0 / beta) * log(1.0 + tl.exp(b_g_scaled)))
b_gate = b_a * b_softplus                          # [BT, BD] per-token, per-channel log-decay ≤ 0
The gate construction from KDA.html §6: g = −exp(A_log)·softplus(raw). −exp(...) guarantees a negative decay; softplus ≥ 0 is the per-channel timestep. The tl.where swaps in the linear approximation softplus(x)≈x once x > 20 to dodge exp overflow. Nothing is summed yet — b_gate is the raw per-token decay. 
o_t = tl.arange(0, BT)
m_cumsum = tl.where(o_t[:, None] >= o_t[None, :], 1.0, 0.0)   # lower-triangular ONES [BT, BT]
b_y = tl.dot(m_cumsum, b_gate, allow_tf32=False) * cumsum_scale  # cumulative sum!  × RCP_LN2
tl.store(p_y, b_y.to(p_y.dtype.element_ty), boundary_check=(0, 1))
The trick. A prefix-sum along time is a matrix multiply by a lower-triangular matrix of ones: row i of m_cumsum @ b_gate equals Σj≤i b_gate[j]. So one tl.dot on the tensor cores produces the within-chunk cumulative decay for all 64 tokens and all channels at once. The trailing * cumsum_scale (= RCP_LN2 ≈ 1.4427) bakes in the natural-log→log2 conversion so every downstream kernel can use the fast exp2 and still get exp. Reading kernels well = recognizing when a "non-matmul" op (cumsum) has been recast as a matmul to hit the tensor cores.

5. Kernel 3 — the K·Kᵀ Gram matrix (the hard one)

chunk_kda_scaled_dot_kkt_fwd_kernel_intra_sub_inter  hard

Builds the within-chunk key–key (A) and query–key (Aqk) matrices that feed the triangular solve. This is "real" Triton: 2-D tiling, transposed block pointers, and a clever gate factorization. fla/ops/kda.py:521

grid(NT, NC·NC, B·H) — one program per (chunk, sub-chunk pair, head) sizesBT=64 (chunk), BC=16 (sub-chunk), NC=4, BK=128 (K loop runs once) outtwo 64×64 matrices per chunk, gate-weighted, β-scaled

Why sub-chunks? A 64×64 matrix is split into a 4×4 grid of 16×16 blocks. This kernel fills the strictly-lower-triangular off-diagonal blocks (row sub-chunk i_i below col sub-chunk i_j); a sibling kernel (…_intra_sub_intra) fills the diagonal blocks. Splitting the work this way keeps each program's tiles tiny and lets the triangular structure skip half the blocks. 

i_t, i_c, i_bh = tl.program_id(0), tl.program_id(1), tl.program_id(2)
i_b, i_h = i_bh // H, i_bh % H
i_i, i_j = i_c // NC, i_c % NC          # row sub-chunk, col sub-chunk (each in 0..NC-1)
... varlen remap of i_t, bos, eos, T ...
if i_t * BT + i_i * BC >= T: return     # row sub-chunk past end of sequence
if i_i <= i_j: return                    # keep only STRICTLY-lower blocks (i_i > i_j)
The two returns are how Triton expresses "this program has no work": ragged-edge programs and upper-triangular blocks simply exit. The grid is launched dense (NC*NC pairs) and the kernel prunes itself — common pattern, and the first thing to check when a program seems to compute nothing. 
b_b = tl.load(p_b)          # [BC] beta for the 16 ROW tokens
b_A = tl.zeros([BC, BC], tl.float32)     # 16×16 key–key accumulator
b_Aqk = tl.zeros([BC, BC], tl.float32)   # 16×16 query–key accumulator
for i_k in range(tl.cdiv(K, BK)):       # tile over K (for K=128, BK=128 ⇒ one pass)
The accumulators are 16×16 (one off-diagonal block of the 64×64 matrix). The for i_k loop tiles the contraction (K) dimension — here it runs once, but the structure is what lets the same kernel handle K > 128. 
    p_q  = block_ptr → q[ROW tokens, K-tile]     # [BC, BK]
    p_k  = block_ptr → k[ROW tokens, K-tile]     # [BC, BK]
    p_g  = block_ptr → g[ROW tokens, K-tile]     # [BC, BK]  cumulative gate
    b_kt = block_ptr → k[COL tokens, K-tile]ᵀ    # [BK, BC]  note: shape (K, T), TRANSPOSED
    p_gk = block_ptr → g[COL tokens, K-tile]ᵀ    # [BK, BC]

    o_k = i_k*BK + tl.arange(0, BK); m_k = o_k < K
    b_gn = tl.load(g + (i_t*BT + i_i*BC)*H*K + o_k, mask=m_k)  # [BK] gate at ROW sub-chunk's FIRST token = "anchor"

Two pointer styles in play. p_q/p_k/p_g are normal [BC, BK] tiles for the row tokens. b_kt/p_gk use a transposed block pointer — created with shape (K, T) and strides (1, H*K) — so the column tokens land as [BK, BC], ready to be the right-hand side of a dot.

b_gn is the key idea: the cumulative gate at the first token of the row sub-chunk, used as a shared anchor. Hold that thought.

    b_g = tl.load(p_g)
    b_k = tl.load(p_k) * exp2(b_g - b_gn[None, :])     # row key × decay(anchor→row token)
    b_gk = tl.load(p_gk)
    b_kt = tl.load(b_kt)
    b_ktg = b_kt * exp2(b_gn[:, None] - b_gk)            # col key × decay(col token→anchor)
    b_A += tl.dot(b_k, b_ktg)                              # [BC,BC] gate-weighted Kᵢ·Kⱼᵀ

    b_q = tl.load(p_q)
    b_qg = b_q * exp2(b_g - b_gn[None, :]) * scale
    b_Aqk += tl.dot(b_qg, b_ktg)                            # [BC,BC] gate-weighted Qᵢ·Kⱼᵀ

The gate factorization — the cleverest line in the kernel. The decay weighting a pair (row token i, col token j) needs is exp2(g[i] − g[j]). Computing that as an [BC,BC] matrix would break the matmul. Instead they split it through the anchor:

exp2(g[i] − g[j]) = exp2(g[i] − ganchor) · exp2(ganchor − g[j])

The left factor depends only on the row token → folded into b_k. The right factor depends only on the col token → folded into b_ktg. Now an ordinary tl.dot(b_k, b_ktg) sums over K and the two factors multiply to reconstruct exp2(g[i]−g[j]) for every (i,j). The anchor also keeps both exponents near zero, which is numerically safer than exp2 of a large cumulative value. Aqk reuses the same b_ktg with the scaled query.

b_A *= b_b[:, None]                       # scale each ROW by its beta (delta-rule write strength)
tl.store(p_A,   b_A,   boundary_check=(0,1))  # write the 16×16 block into the 64×64 A
tl.store(p_Aqk, b_Aqk, boundary_check=(0,1))
Finally each row of the key–key block is scaled by β (queries are not β-scaled — only the "write" side is), and the two blocks are stored into their slots of the full [T, BT] matrices. Downstream, solve_tril inverts I + A and Aqk becomes the intra-chunk part of the output — see KDA.html §8.
Check yourself: where did the per-token decay exp2(g[i]−g[j]) actually get computed? Nowhere as an explicit matrix — that's the point. It's implicit in the dot product: the row factor exp2(g[i]−ganchor) rides on b_k, the column factor exp2(ganchor−g[j]) rides on b_ktg, and tl.dot's multiply-accumulate over K multiplies them together for each (i,j) pair. Spotting "the weighting is folded into the operands of a matmul" is a core kernel-reading skill.

6. How to attack an unfamiliar kernel

  1. Read the host wrapper first, not the @triton.jit. The Python function that launches it (e.g. chunk_kda_scaled_dot_kkt_fwd) shows the real tensor shapes, computes the block sizes, and defines grid=(...). That tuple is your map.
  2. Decode tl.program_id against the grid. Write down, in indices, exactly what slice this one program owns. Everything else hangs off that.
  3. Translate every make_block_ptr / pointer into "tile of tensor X at [rows, cols]". Ignore the stride arithmetic once you've confirmed it matches the tensor's logical shape.
  4. Find the accumulator and the tl.dots. The matmuls are the load-bearing math; the exp2/where/broadcast around them are weighting and masking.
  5. Mentally collapse constexpr branches. Pick your case (IS_VARLEN=True, IS_KDA=True, …) and read only that path.
  6. Check the returns and masks. They tell you which programs do nothing and how ragged edges are handled — easy to mistake for missing logic.
  7. Save autotune/heuristics for last. They only choose tile sizes and warp counts; they never change what's computed.
The recurring KDA idioms, in one place
  • State as a tile: b_h: [BV, BK]; broadcast a vector over one axis, tl.sum over the other.
  • Gate via cumsum + difference: decay from j→i is exp2(g[i]−g[j]), and that difference is often factored through an anchor so it rides on the operands of a dot.
  • exp2 everywhere on the chunk path because gates were pre-scaled by RCP_LN2; plain exp on the decode path because it doesn't cumsum.
  • Cumsum / prefix ops = multiply by a triangular-ones matrix to use tensor cores.
  • One constexpr flag (IS_KDA) switches the scalar GDN gate to KDA's per-channel gate.
