MiniMax M3 Hand-Written Kernels

The hand-written CUDA kernels in the MiniMax M3 implementation — read line-by-line. These are hand-written fusions: M3 runs under breakable CUDA graph (no inductor), so the compile-time fusion passes never fire and every fusion has to be a kernel someone wrote. Companion to FUSIONS.html (the theory) — this is the practice.

TL;DR A handful of bespoke CUDA kernels live under csrc/libtorch_stable/. Two are the stars: (1) fused_minimax_m3_qknorm_rope_kv_insert — the whole attention prologue (four Gemma RMSNorms + partial-NeoX RoPE + KV/index-cache writes) collapsed into one kernel where each warp owns one (token, head); and (2) minimax_allreduce_rms_qk — a cross-GPU Lamport one-shot all-reduce that reduces the QK variance across tensor-parallel ranks and normalizes in the same launch. Around them sit MoE helpers: a fused silu_and_mul_per_block_quant, a tiny fp32_router_gemm, and MXFP8 grouped matmuls. The recurring toolkit — warp-per-head, shuffle reductions, vectorized bf16 I/O, compile-time template gating, and PDL — is the same kit you'd reach for writing your own.

1. How these were found

The hand-written M3 kernels all live under csrc/libtorch_stable/ (the torch stable-ABI tree). To list M3's device code and trace where each piece came from: 

# list the M3 / MoE device sources
git ls-files 'csrc/libtorch_stable/**/*.cu' | grep -E 'minimax|fp32_router|mxfp8|silu_mul_block'
# and the commit that introduced (or last touched) one
git log --follow --oneline -- csrc/libtorch_stable/fused_minimax_m3_qknorm_rope_kv_insert_kernel.cu

Several files arrived via a mechanical stable-ABI migration (which only moves files, not compute). Filtering that out, the genuinely hand-written compute kernels are:

Why hand-written at all? M3 auto-enables breakable CUDA graph (VLLM_USE_BREAKABLE_CUDAGRAPH=1), which sets the compilation mode to NONE — no dynamo, no inductor, so none of vLLM's pattern-matcher fusion passes run. Any fusion M3 wants, it must ship as an explicit kernel. See FUSIONS.html §10 for the why; this page is the how.

3. Sparse attention + the lightning indexer

§2 drew the attention block as a single box. This section opens it — and answers "what actually happens for a prompt of N tokens." Unlike §4–§5, the code here is Triton (with an SM100 MSA fast path, M3_MSA.html), not hand-written CUDA; it is on this page because it is the consumer that kernel ① (§4) feeds.

By the time control reaches here, kernel ① has already run for the current tokens: it normalized + RoPE'd the main q and the index index_q, scatter-inserted k/v into the main GQA cache, and wrote one index_k vector per token into the indexer side-cache. Two steps remain, both behind a single eager_break:

indexer scores every visible block→ top-k = 16 blocks per (head, token)→ topk_indices_buffer→ attend only those blocks

3a. Two caches, two head counts

M3 attention keeps two paged caches, both filled by kernel ①:

CacheShape (per block)Holds
main GQA KV[blocks, 2, 128, 4, 128] K&V (the 2), 128 tokens/block, 4 KV heads × 128 head_dim
indexer side-cache[blocks, 128, 128] one 128-dim index key per token — no head dim (index_k is replicated, not sharded)

The head bookkeeping is the crux. M3 has 64 query heads, 4 KV heads (GQA group of 16) and 4 index-query heads — exactly the KV-head count (num_index_heads == num_kv_heads). Each index head scores against the one shared index key per token and selects blocks for its own KV head; the 16 query heads in that group then attend to those blocks. Clean map: index head h → KV head h → 16 query heads. So the 4 KV heads can each pick a different set of 16 blocks.

Only 57 of 60 layers do this sparse_attention_freq = [0]*3 + [1]*57: layers 0–2 are ordinary dense GQA (full causal attention, no indexer); layers 3–59 are the sparse layers described here.

3b. The indexer: score → top-k blocks

Score. A "block" is one 128-token page. For query token t and index head h, the block's score is the max over its 128 positions of the index dot product (sparse_score_type = "max"), causal-masked so a token never scores positions after itself: 

score[h, t, blk] = maxpos ∈ blk, pos ≤ t  ( index_q[t, h] · index_k[pos] )

— a block scores high if any token in it is a strong match. The dot is a small per-page q @ kᵀ fused with the running per-block max.

Top-k. Per (head, token), keep the 16 highest-scoring blocks (sparse_topk_blocks = 16). Two ranges are force-kept regardless of score: sparse_init_block = 0 (none) and sparse_local_block = 1 — the query's own current block always stays in. Output: topk_idx[4, total_q, 16] of 0-indexed block ids, −1 padding any unused slot.

The shared buffer + the eager break The indexer writes topk_idx into a stable-address topk_indices_buffer (decode at [:, :nd], prefill at [:, nd:]) and the attend reads it back. Handing it across the eager break as a fresh Python tensor would freeze the indices into a captured graph; the fixed-address buffer keeps the pointer stable for cudagraph replay while the values change each step.

3c. The sparse attend

The real attention. Each query head reads its KV head's 16 block ids, gathers those pages from the main KV cache via the request's block_table, and runs a flash-style online softmax over them — at most 16 × 128 = 2048 keys, independent of context length. That fixed ceiling is the whole point: per-token attention cost stops growing once the context passes 2048 tokens.

3d. Prefill vs decode — same pick, different kernels

The selection rule is identical; the kernels differ because the batch shapes do.

PrefillDecode
query tokensall prompt tokens at once (varlen)1 per request (1+N with MTP), flattened request-major
scoreminimax_m3_index_scoreminimax_m3_index_decode — split-K, fixed chunk count

Decode pays the split-K complexity for one reason: a cudagraph needs a fixed grid, so chunk counts come from shape constants (not seq_len) and the partials are combined in a merge pass. Prefill is variable-length and runs eager, so it can size its grid to the real sequence.

3e. Worked example: a 4000-token prompt (and why 1000 is dense)

A 1000-token prompt runs fully dense. 1000 tokens at block_size = 128 is ⌈1000/128⌉ = 8 blocks ≤ 16 top-k, so every token's "top-16" is simply all of its visible blocks. The indexer still scores, the buffer is still written, the attend still runs — but nothing is pruned, and the result is numerically identical to dense causal GQA. Sparsity only engages once a token sees more than 16 blocks, i.e. once the context passes topk × block_size = 16 × 128 = 2048 tokens.

4000 tokens is ⌈4000/128⌉ = 32 blocks (block 31 holds the last 32 tokens), so pruning is real — but it kicks in part-way through, because the threshold is per token position, not per prompt:

prefillall 4000 tokens in one pass · token at position p sees ⌈(p+1)/128⌉ blocks
→
p ≤ 2047≤ 16 blocks → top-k is the identity (dense)
→
p ≥ 204817–32 blocks → indexer keeps 16 (1 local + 15 by score) (sparse)

So a single prefill pass is mixed: the first 2048 query tokens attend densely, the back ~half are pruned to 16 blocks each. Then every decode step (generating tokens 4001, 4002, …) is fully sparse — at running length L a token sees ⌈L/128⌉ ≥ 32 blocks and keeps 16:

phaseposition / lengthblocks ⌈·/128⌉keptregime
prefillp = 2047 (last dense token)1616dense
prefillp = 2048 (first pruned)1716sparse
prefillp = 3999 (end of prompt)3216sparse (½ kept)
decodeL = 40003216sparse (½ kept)
decodeL = 81926416sparse (¼ kept)
decodeL = 131072 (128k)102416sparse (~1.5% kept)

Zoom in on the boundary block. Block 16 (positions 2048…2175) is exactly where pruning first appears. Every token in it sees blocks 0…16 = 17 blocks, so one must go. Block 16 is the query's own block and is force-kept as the local block, which leaves 15 slots for the 16 earlier blocks 0…15: each token keeps 1 local block + the top-15 of its 16 previous blocks, dropping exactly the single lowest-scoring earlier block. The pick is per token and per KV head, so neighbouring tokens (and the 4 KV heads) may each drop a different one.

How the local (diagonal) block's K/V is used when it isn't "full" In this prefill, block 16 is physically full — the whole 4000-token prompt was scatter-inserted into the cache up front by kernel ①. What makes the block "partial" for a given query is causality. The selected page is always loaded as a full 128-slot tile; two masks in the attend kernel then decide which of its keys count, so there is no special partial-block code path:
  • Causal mask — qk += where(q_pos − n ≥ blk·128, 0, −∞): a key at position blk·128 + n survives only if it is ≤ the query's own position. This bites only on the diagonal (local) block. For the first token of block 16, p = 2048, only position 2048 survives, so the local block contributes 1 key; by p = 2175 all 128 do.
  • Seq-len mask — pos < seq_len: positions past the end of the sequence (unwritten cache slots) drop out. This is the mask that handles a physically partial page: the last block of the prompt (block 31 holds only 32 real tokens), and, in decode, the current block as it fills one token per step.
Masked lanes load as 0 and take −∞ in qk → exp2(−∞) = 0, so they never enter the online-softmax numerator or denominator — the result is exact regardless of how many real keys the page holds. The 15 earlier blocks need no causal mask (every one of their 128 keys precedes the query) and contribute in full.
query position in block 16local-block keys (after causal mask)earlier blocks attended
2048 (first)115 of 16
2049215 of 16
………
2175 (last)12815 of 16

So the KV each head actually attends = the 15 full earlier blocks (15 × 128 = 1920 keys, unmasked) + the causal current block (1…128 keys). For a token in block 16 that is 1920 + (1…128) = 1921…2048 effective keys — 1921 at p = 2048, the full 2048 at p = 2175. All 64 query heads do this; the 16 heads in a GQA group reuse their shared KV head's same 16 blocks (and the 4 KV heads may hold different sets). Note the kernel still streams the full 16 × 128 = 2048 QK tiles and masks out-of-range lanes to −∞, so the compute is a flat ~2048 scores/head — the mask only zeroes a lane's softmax contribution; it doesn't skip the dot. This "15 full blocks + 1 causal block" shape holds for every token once the context passes 16 blocks.

The win is asymptotic: the per-token attend is pinned at 16 blocks while the context grows without bound. A 1000-token prompt never reaches the threshold, so it pays the indexer's bookkeeping for no pruning; a 4000-token prompt prunes its back half during prefill and every token during decode.

4. Kernel ①: fused_minimax_m3_qknorm_rope_kv_insert

This is the headline. It replaces the per-token Python sequence in MiniMaxM3SparseAttention.forward: 

# what the kernel replaces (5 norms/ropes + 2 cache writes, ~8 launches)
q  = q_norm(q);  k = k_norm(k);  q, k = rotary_emb(pos, q, k)
index_q = index_q_norm(index_q);  index_k = index_k_norm(index_k)
index_q, index_k = rotary_emb(pos, index_q, index_k)
_insert_kv(k, v, index_k)

Everything arrives in one packed qkv tensor — the fused projection emits, per token, [ q | k | v | index_q | index_k ] (the "5 results"; the dense layer emits just [ q | k | v ]). The kernel reads each sub-block straight out of that row, normalizes and rotates it, and scatters K/V/index into their paged caches — all in registers, no intermediates to HBM.

read packed qkv→ RMSNorm (q,k,iq,ik)→ partial-NeoX RoPE→ store in place / gather→ scatter K,V,index → caches ~8 launches → 1

4a. The grid: one warp owns one (token, head-slot)

The decomposition is the key idea. head_dim = 128. A warp is 32 lanes; give each lane 4 contiguous dims (32 × 4 = 128). So one warp handles exactly one head of one token, and the whole RMSNorm reduction is a single warp-shuffle — no shared memory, no block sync.

A warp covers head_dim=128 as 32 lanes × 4 dims

Each lane loads its 4 dims as one 8-byte (uint2) read. blue = first RoPE half, amber = second half (see 4c).

The grid is 1-D: num_tokens × slots_per_token warps, packed into 256-thread blocks (8 warps each). Slots are enumerated per token:

slots → Q×nq K×nkv V×nkv IQ×niq IK×1 
// each warp decodes its job from one flat index
int globalWarpIdx = blockIdx.x * warpsPerBlock + threadIdx.x / 32;
int slots_per_token = nq + nkv + v_slots + idx_slots;   // gated by template bools
int tokenIdx = globalWarpIdx / slots_per_token;
int slot     = globalWarpIdx % slots_per_token;         // → isQ / isK / isV / isIQ / isIK
Why warp-per-head is the right granularity here The norm reduces over the 128-dim head. Mapping a head to a warp means the reduction is __shfl_xor_sync only — the fastest possible reduce, entirely in registers. Mapping a head to a block would force shared-memory + __syncthreads; mapping to a lane would serialize 128 dims. The 32×4 split is exact for head_dim 128.

4b. Gemma RMSNorm by warp-shuffle

Each lane squares its 4 dims, then a butterfly __shfl_xor sums across the 32 lanes so every lane ends up with the full head's sum-of-squares. Gemma form: scale by (1 + w). 

__device__ float warpReduceSum(float v) {
  for (int mask = 16; mask > 0; mask >>= 1)
    v += __shfl_xor_sync(FINAL_MASK, v, mask, 32);   // 5 steps → all-lane sum
  return v;
}
// ── Gemma RMSNorm over the full head ──
float sumsq = 0;
for (int i=0;i<4;i++) sumsq += elems[i]*elems[i];
sumsq = warpReduceSum(sumsq);
float rms_rcp = rsqrtf(sumsq / 128.0f + eps);
for (int i=0;i<4;i++) {
  float w = 1.0f + (float)weight[laneId*4 + i];          // Gemma (1+w)
  elems[i] = elems[i] * rms_rcp * w;
}

The same normAndRope helper serves all four norms — pass q_norm_w, k_norm_w, iq_norm_w, or ik_norm_w; pass nullptr for V (which skips both norm and RoPE).

4c. Partial-NeoX RoPE by XOR-shuffle

RoPE rotates pairs (i, i+half) where half = rotary_dim/2. In NeoX layout the partner of dim i lives half dims away — i.e. in another lane. The kernel fetches the partner's 4 values with a single __shfl_xor across a fixed lane-distance, so no element ever leaves registers.

Partner lane = current lane XOR (half / 4)
L0 → dims 0–3 ⟷ shfl_xor(half/4) L(half/4) → dims half..half+3
Because half is a multiple of 4, a lane sits wholly in the first half (owns x[i]) or the second (owns x[i+half]); its partner is always half/4 lanes away. 
int half = rotary_dim / 2;
int lane_xor = half / 4;                  // partner-lane distance
float partner[4];
for (int i=0;i<4;i++)
  partner[i] = __shfl_xor_sync(FINAL_MASK, elems[i], lane_xor, 32);
if (in_rope) {
  bool first_half = dim0 < half;
  for (int i=0;i<4;i++) {
    float c = cos_ptr[i_base+i], s = sin_ptr[i_base+i];
    elems[i] = first_half ? elems[i]*c - partner[i]*s     // x·cos − x_partner·sin
                          : elems[i]*c + partner[i]*s;     // x·cos + x_partner·sin
  }
}
"Partial" RoPE Only the leading rotary_dim dims rotate; trailing dims pass through untouched (in_rope = dim0 < rotary_dim). All branches — q, k, index_q, index_k — share the same cos_sin_cache table, so RoPE is one code path parameterized by the slot.

4d. Vectorized bf16 I/O

Each lane moves its 4 bf16 values as a single 8-byte uint2 load/store, converting bf16↔fp32 in registers. One memory transaction per lane, fully coalesced across the warp (128 contiguous bytes). 

// 4 contiguous bf16 → 4 fp32 registers (one uint2 = 8 bytes)
uint2 v = *reinterpret_cast<uint2 const*>(src);

The in-place + gather trick Q/K/index are rewritten in place inside qkv. But Q and index_q can optionally be written to dedicated contiguous buffers (q_out, index_q_out) instead — so the downstream SM100 sparse kernel's flat TMA descriptor can address them as [tokens·heads, head_dim] without a separate q.contiguous() copy. The de-interleaving folds into the store the kernel was doing anyway.

4e. Compile-time gating & PDL

Dense vs sparse, and whether to write caches, are template bools — so the index slots, V slots, and cache writes compile away on paths that don't use them. Three instantiations:

<kIsSparse, kInsertKV> <false,false> dense <true,false> sparse-profiling <true,true> sparse-serving 
template <typename scalar_t, bool kIsSparse, bool kInsertKV>
__global__ void fusedMiniMaxM3QNormRopeKVInsertKernel(...);
// host picks one:
has_index ? (insert_kv ? LAUNCH(true,true) : LAUNCH(true,false))
          : LAUNCH(false,false);            // dense: generic Attention owns the KV write

The paged KV cache is addressed by strides read off the tensor (block / kv / token / head), so the same kernel honors whichever physical layout the attention backend allocated — NHD (s_token < s_head) or HND (swapped) — with no extra argument. 

int64_t off = b*kv_s_block + kv*kv_s_kv + t*kv_s_token + head*kv_s_head;
storeElems(kv_cache + off + dim_base, elems);   // head_dim always stride-1

Finally, PDL (Programmatic Dependent Launch) overlaps this kernel with its neighbors. It waits for the producing GEMM, then signals completion so a dependent successor can start early — pure launch-latency hiding, a no-op on pre-Hopper. 

#if __CUDA_ARCH__ >= 900
  cudaGridDependencySynchronize();          // wait for the qkv-projection GEMM
#endif
  // ... load → norm → rope → store → cache insert ...
#if __CUDA_ARCH__ >= 900
  cudaTriggerProgrammaticLaunchCompletion();  // let the successor launch early
#endif

5. Kernel ②: minimax_allreduce_rms_qk (cross-TP, Lamport)

Kernel ① assumes the whole 128-dim head lives on one GPU. When a head is split across tensor-parallel ranks, the RMSNorm variance is a sum over dims that live on different GPUs — so before you can normalize, you must all-reduce the per-token sum-of-squares. This kernel fuses that cross-GPU reduction and the normalization into a single launch, using a Lamport one-shot all-reduce (adapted from TensorRT-LLM).

local Σx² (this rank's shard)→ Lamport one-shot exchange→ Σ across ranks → rsqrt→ normed q, k

The Lamport one-shot, and the negative-zero sentinel

A normal all-reduce needs a separate "data ready" flag per peer. The Lamport trick removes the flag by making the data itself the flag: peer buffers are pre-filled with −0.0 (a value the real data never legitimately takes), each rank writes its partial variance into every peer's buffer, and consumers spin until the slot is no longer −0.0. No fences, no atomics on the hot path — just polling a sentinel. 

// each rank publishes its partial variance to every peer's buffer...
if (threadIdx.x == 0)
  for (int r=0; r<NRanks; ++r)
    ((float*)comm.data_bufs[r])[rank*tot_tokens + token_id] = sum_variance;

// ...then spins reading until ALL ranks' slots are filled (not −0.0)
bool done = false;
while (!done) {
  done = true;
  for (int r=0; r<NRanks; ++r) {
    vars[r] = ld_global_volatile(&((float*)comm.data_bufs[rank])[r*tot_tokens + token_id]);
    done &= !is_neg_zero(vars[r]);           // −0.0 == "not ready yet"
  }
}
for (int r=0;r<NRanks;++r) sum_variance += vars[r];   // full-head Σx²
// normalize with the full mean across all ranks
vals[i] = vals[i] * rsqrtf(sum_variance/hidden_dim/NRanks + eps) * weight[i];
Triple-buffered, self-clearing The workspace rotates between 3 buffers (flag_value % 3) so consecutive all-reduces never alias, and each call clears the buffer it'll reuse two rounds later back to −0.0 — the sentinel maintains itself, so there's no separate reset kernel.

The QK float4 variant

The serving path uses minimax_reduce_qk_rms_kernel_lamport_float4: it reduces variance for Q and K together in one launch (doubled comm buffer), processes 4 rows at a time as float4 for coalescing, and on Hopper uses cluster groups + the griddepcontrol PTX for PDL. Per-rank dimensions (OriginQDim/NRanks) are template constants, so the warp counts are fixed at compile time.

This is the path behind the Python minimax_qk_norm_fusion The MiniMaxText01RMSNormTP.forward_qkv wrapper dispatches here when tp_world > 1, the Lamport workspace exists (needs P2P/IPC), and tokens ≤ 2048; otherwise it falls back to eager all-reduce + RMSNorm. See FUSIONS.html §11 for the dispatch logic.

6. The MoE kernels

The FFN is a 256-expert MoE; three more hand-written kernels feed it.

silu_and_mul_per_block_quant

SwiGLU activation silu(gate)·up fused with per-block fp8 quant for the expert GEMM. Computes SiLU in fp32, finds the block abs-max with fmaxf, and quantizes — using the same warp-shuffle reduction + vectorized I/O pattern as kernel ①.

quantization/fused_kernels/fused_silu_mul_block_quant.cu · #44173

fp32_router_gemm

The router is a tiny, latency-critical GEMV: act × W_router(fp32) → fp32 with H=3072, E=256, M≤32. Hand-tuned for the skinny-M decode shape (bf16 or fp32 activation; weight always fp32 for routing stability). Adapted from DeepSeek-V3's router GEMM.

fp32_router_gemm.cu, fp32_router_gemm_entry.cu · #34

cutlass_mxfp8_grouped_mm

Grouped (per-expert) matmul in MXFP8 for the expert up/down projections, built on CUTLASS, plus mxfp8_experts_quant to quantize activations into the MXFP8 block format the grouped MM expects.

How do I verify a hand-written kernel matches the unfused model?

Each kernel has a companion pytest (e.g. test_fused_minimax_m3_qknorm_rope_kv_insert.py) that builds the same inputs, runs the eager norm → rope → insert sequence as the reference, and asserts allclose at the dtype tolerance. The kernels deliberately do all reductions in fp32 and round to scalar_t only on store, matching the materialized unfused output — that's what keeps the diff within tolerance.

Why "libtorch_stable"? Does it change the kernels?

It changes the wrapper, not the compute. The migration moves the torch op boundary to the stable ABI (torch::stable::Tensor, STD_TORCH_CHECK, VLLM_STABLE_DISPATCH_*) so the extension binary keeps working across PyTorch versions without recompiling. The __global__ kernel bodies are unchanged.

The throughline to FUSIONS.html Kernel ① is exactly the "hand-written route" from §10: an explicit fused op called from forward, with a capability gate (template instantiation), an fp32 reference contract, and graceful fallbacks (dense path, pre-Hopper no-PDL). Kernel ② is the M3 realization of the AllReduce+Norm fusion the compile passes would otherwise do (§6) — but written by hand because, under breakable CG, there is no pass.
