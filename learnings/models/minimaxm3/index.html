What M3 is. MiniMax M3 is a DeepSeek-V3.2-style sparse-attention MoE model — not the linear-attention MiniMax-Text-01 / M2 family. Each long-context attention layer runs a tiny "lightning indexer" that scores 128-token KV blocks, keeps the top-16, and attends only to those. The vLLM implementation lives under vllm/models/minimax_m3/, split into a hardware-neutral common/ and per-vendor nvidia/ + amd/ branches (mirrors deepseek_v4). Everything below describes the NVIDIA / Blackwell branch.

1Model architecture

The registry entry is MiniMaxM3SparseForCausalLM (text) wrapped by MiniMaxM3SparseForConditionalGeneration for the VL variant model.py:935. The backbone is 60 decoder layers over a 6144-d residual stream:

Two independent config-driven splits decide each layer's behaviour minimax_m3.py:85:

  • moe_layer_freq = [0]*3 + [1]*57 → layers 0–2 use a dense MLP, layers 3–59 use the MoE block.
  • sparse_attention_freq = [0]*3 + [1]*57 → layers 0–2 use full attention, layers 3–59 use the lightning-indexer block-sparse path.

For the default checkpoint the two splits coincide, but the code keeps them separate — don't assume "MoE layer" ⇒ "sparse-attention layer".

2The decoder layer

One class MiniMaxM3DecoderLayer serves every layer and branches internally on its index model.py:658. All norms are Gemma-style RMSNorm (use_gemma_norm=True, the (1+w) scaling baked into FlashInfer kernels).

residual ← hidden_states  (bf16 stream)
↓
input_layernorm — Gemma RMSNorm (fused with the previous layer's deferred all-reduce when fuse_input_allreduce)
↓
self_attn — MiniMaxM3Attention (dense) or MiniMaxM3SparseAttention (indexer + block-sparse). o_proj with reduce_results=False
↓  + residual
post_attention_layernorm — fused_allreduce_gemma_rms_norm (completes the attention all-reduce)
↓
block_sparse_moe (sparse layers) or mlp (dense layers 0–2)
↓  + residual → next layer

Both the attention o_proj and the dense-MLP down_proj set reduce_results=False; the tensor-parallel all-reduce is deferred and fused into the next Gemma RMSNorm via fused_allreduce_gemma_rms_norm (under PP=1) model.py:760.

3Attention & the lightning indexer

Dense layers (0–2) — full attention

  • Merged QKVParallelLinear (no bias), 64 q / 4 kv heads × 128.
  • Per-head Gemma QK-norm + partial NeoX RoPE (first 64 of 128 dims rotated).
  • No output gate (attention_output_gate=False).
  • QK-norm + RoPE + KV-insert collapse into one C++ op fused_minimax_m3_qknorm_rope_kv_insert model.py:368.
  • Runs the generic Attention layer → FlashInfer trtllm-gen on Blackwell.

Sparse layers (3–59) — indexer + block-sparse

  • One fused GEMM MinimaxM3QKVParallelLinearWithIndexer emits [q | k | v | index_q | index_k] model.py:437.
  • Index query: 4 heads × 128; index key: a single 128-d vector per token.
  • Indexer scores each 128-token block (max over the block), keeps top-16.
  • Main attention then runs only over the selected blocks.
  • Index value/output projections are always disabled in M3.

Impl is chosen at runtime by select_main_impl_cls sparse_attention.py:376: on Blackwell (capability-family 100) with a bf16 KV cache and topk_blocks ∈ {4,8,16,32}, prefill uses the SM100 MSA FMHA (fmha_sm100); otherwise (fp8 KV, non-Blackwell) it falls back to Triton. Decode always uses Triton split-K — there is no MSA decode kernel yet sparse_attention_msa.py:51.

4MoE & dense MLP

MoE block (layers 3–59)

  • Router GateLinear: weights & logits in fp32, sigmoid scoring + per-expert correction bias model.py:226.
  • 128 experts, top-4, renormalize, routed_scaling_factor=2.0.
  • 1 shared expert (always-on MLP, fused into the FusedMoE call).
  • Activation swigluoai_uninterleave folded into the expert kernel.

Dense MLP (layers 0–2)

  • MergedColumnParallelLinear gate_up → RowParallelLinear down, no bias.
  • dense_intermediate_size = 12288.
  • SwiGLU-OAI via the silu_and_mul_with_clamp C++ kernel: clamp(gate)·σ(α·gate)·(clamp(up)+β) with α=1.702, limit=7.0.
  • down_proj all-reduce deferred into the next layer's input norm.

7KV cache size per token

Each sparse attention layer registers two caches; each dense layer registers only the main one. All figures are per tensor-parallel rank at TP=1 (they scale ~1/tp down to a 1-head floor).

① Main GQA K/V cache — every layer

FullAttentionSpec · shape (blocks, 2, 128, kv_heads, head)

Per token = 2 (K,V) × 4 kv_heads × 128 × dtype

bf16 = 2048 B  ·  fp8 = 1024 B model.py:537

② Indexer key cache — sparse layers only

MLAAttentionSpec · shape (blocks, 128, head) — key-only, 1 head

Per token = 1 × 128 × 2 (always bf16)

bf16 = 256 B  (does not shrink under fp8 KV) indexer.py:142

The sparse layers keep the full K/V cache — sparsity is applied only at attention time by selecting blocks, so nothing is dropped from storage. The indexer key cache is purely additive.

With MTP / spec decode enabled, the single MTP layer is a full sparse layer and adds one more layer's worth: +2,304 B (bf16) or +1,280 B (fp8) per token — grand totals ≈136.5 KiB (bf16) / ≈75.5 KiB (fp8). It is only allocated when a speculative/MTP config is active mtp.py:54.
Asymmetry to remember: fp8 halves the main K/V cache but leaves the indexer cache at bf16 (it's hard-coded to bf16 today), so the indexer's share of the footprint nearly doubles in relative terms (10.6% → 19.2%) when you switch to fp8 KV.
