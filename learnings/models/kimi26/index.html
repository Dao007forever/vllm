Kimi-K2.6 in vLLM

A DeepSeek-V3-architecture text MoE model: MLA attention, fine-grained fp8 MoE, the kernels it runs on NVIDIA Blackwell (sm_100), tensor dtypes, and the (tiny) MLA KV-cache per token.

TL;DR Kimi-K2.6 has no dedicated model file in vLLM. It is a DeepSeek-V3-architecture text MoE model (Multi-head Latent Attention + sigmoid/noaux_tc MoE) served through DeepseekV2ForCausalLM (deepseek_v2.py). ~64 decoder layers, 384 routed experts (top-8) + 1 shared, run at TP=8. On Blackwell the default attention is FlashInfer MLA (trtllm-gen) with optional fp8 KV; experts are block-wise fp8 (DeepGEMM grouped GEMM, UE8M0 packed scales). MLA caches a single 576-dim latent per token per layer → ≈72 KiB/token (bf16) or ≈36 KiB/token (fp8) over 64 layers — roughly 71× smaller than the equivalent multi-head cache.
Scoping note — "Kimi-K2.6" is not a distinct module here. There is no kimi_k26.py. The latest dedicated Kimi model in this tree is the VL model kimi_k25.py, whose text backbone is itself DeepseekV2ForCausalLM. Kimi-K2.6 is referenced only as a deep-MoE/TP workload: env_override.py:774 ("Kimi-K2.6 at TP=8 … hundreds of layers deep") and test_inductor_fallback_allow_list_patch.py:210 ("Kimi-K2.6 has ~64 decoder layers per replica"). Everything below documents the DeepSeek-V3 MLA+MoE backbone it runs on; concrete hyper-parameters are checkpoint-driven and cited from in-repo tests. Re-verify against the checkpoint config before relying on exact counts.

2 · MLA attention (latent KV)

MLA is the whole point of the architecture: instead of caching K and V per head, it caches a single compressed latent per token and reconstructs K/V on the fly by absorbing the up-projection weights into the attention kernel. Kimi-K2.6 uses the dense MLA path — DeepseekV2MLAAttention → MultiHeadLatentAttentionWrapper → the MLAAttention backend (is_sparse=False, no DeepSeek-V3.2 indexer) deepseek_v2.py:878.

The KV cache holds only the normalized latent kv_c (512) plus the rope key k_pe (64) — one 576-dim vector per token, num_kv_heads=1. kv_b_proj (which would expand it to 128 heads of K/V) is handed to the backend and folded into the attention math via bf16 BMMs.

Projections

  • fused_qkv_a_proj — one GEMM emits [q_lora 1536 | kv_lora+rope 576] (the [2112, 7168] shape has a bf16 fast-path on SM90/SM100) :928
  • q_a_layernorm (RMSNorm 1536) → q_b_proj → 128 × (128+64)
  • kv_a_layernorm (RMSNorm 512); kv_b_proj 512 → 128 × (128+128) is absorbed
  • o_proj: 128 × 128 → 7168 (RowParallel)

Weight absorption

  • kv_b_proj is split into W_UK / W_UV and stored as bf16 copies mla_attention.py:839
  • Decode: q_nope·W_UK_T maps the query into the 512-dim latent space, then attend against the cached latent (a single 576-dim "head")
  • Output is mapped back with ·W_UV
  • Both BMMs run in bf16 — "no quantized bmm's for W_UV/W_UK_T" :836

3 · MoE & dense MLP

MoE block (DeepseekV2MoE)

  • Router GateLinear: sigmoid scoring, noaux_tc with an fp32 e_score_correction_bias :274
  • Grouped top-k: use_grouped_topk, n_group=1, topk_group=1; renormalize=False (DSV3 doesn't renormalize)
  • 384 routed experts, top-8, routed_scaling_factor=2.827 applied to the output
  • 1 shared expert (always-on MLP, reduce_results=False, fused into the FusedMoE call)
  • Activation SiluAndMul

Dense MLP (first layers)

  • MergedColumnParallelLinear gate_up → RowParallelLinear down, no bias :199
  • intermediate_size (config), SiluAndMul
  • Used for the first first_k_dense_replace layers; the rest are MoE
  • These layers still run full MLA attention — only the FFN is dense

Backend selection on sm100. For dense MLA the decode priority is FLASHINFER_MLA → TOKENSPEED_MLA → CUTLASS_MLA → FLASH_ATTN_MLA → FLASHMLA → TRITON_MLA cuda.py:111. FlashInfer MLA (trtllm-gen) is the default; TOKENSPEED_MLA (fp8-Q + fp8-KV, R1 dims only) overtakes it past batch≈8.
Not on the Blackwell path. The fuse_mla_dual_rms_norm pass (which would fuse the q_a_layernorm + kv_a_layernorm pair) is ROCm/AITER-only and is force-disabled on non-ROCm compilation.py:277. On Blackwell those two RMSNorms run as separate kernels; the Blackwell-relevant fusion is AllReduce+RMSNorm for the TP=8 collectives.

6 · MLA KV cache per token

MLA registers an MLAAttentionSpec with num_kv_heads=1 and head_size = kv_lora_rank + qk_rope_head_dim — "only one vector instead of K + V" mla_attention.py:960. There is no 2× (K+V) factor and no per-head multiplication, unlike a standard attention spec kv_cache_interface.py:369.

head_size = kv_lora_rank + qk_rope_head_dim = 512 + 64 = 576
bytes / token / layer = num_kv_heads(1) × head_size(576) × dtype_bytes
  • bf16 → 576 × 2 = 1152 B   • fp8 → 576 × 1 = 576 B

total (64 layers) → bf16 = 64 × 1152 = 73,728 B ≈ 72 KiB
                  fp8  = 64 × 576  = 36,864 B ≈ 36 KiB

Every layer contributes the MLA cache — the leading dense-FFN layers still run MLA attention deepseek_v2.py:1085. (DeepSeek-V3's 61-layer default gives 70,272 B bf16 / 35,136 B fp8.)

Storing one 576-dim latent instead of 128 heads of K and V yields a ≈71× per-layer compression — the reason MLA models hold enormous contexts on modest KV budgets.
TP=8 duplication & DCP. Because MLA has num_kv_heads=1, the single latent can't be sharded across attention heads — every tensor-parallel rank duplicates the full per-token cache (8× at TP=8). Adding -dcp 8 shards along the token dimension instead, removing the duplication without extra GPUs context_parallel_deployment.md:37.
Versus MiniMax M3. M3's lightning-indexer sparse attention costs ≈134 KiB/token (bf16) over 60 layers because it keeps a full GQA K/V cache plus an indexer cache. Kimi-K2.6's MLA holds only a 576-dim latent, landing at ≈72 KiB/token (bf16) over more layers — about half the footprint. See the M3 walkthrough for the contrast.
