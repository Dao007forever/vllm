Why Mamba Is a Different Cache Problem

Sibling document to LINEAR_ATTENTION.html: one layer down, what makes Mamba specifically awkward for the cache subsystem — selectivity, two state tensors, two kernels, a block-size constraint that doesn't come from the math.

TL;DR

Mamba is a selective state-space model (state-update weights are a function of the input) with two state tensors per layer, two kernels (parallel scan for prefill, step recurrence for decode), and a block-size constraint imposed by vLLM's allocator rather than the math. The conv state forces a special align cache mode; selectivity blocks fast-forward replay; the sequential recurrence blocks DCP/PCP. Each row of the summary table below is a place the cache infrastructure grew a Mamba-specific affordance.

1. Mamba's "state" is two tensors, not one

MambaSpec.shapes is tuple[tuple[int, ...], ...] (kv_cache_interface.py:532) — plural. Every Mamba layer keeps two buffers per request:

SSM state d_inner × d_state

The long-range recurrent summary. This is the "linear-attention state" from LINEAR_ATTENTION.html.

Conv state d_inner × (d_conv − 1)

A tiny rolling window of recent activations feeding the depthwise 1D convolution between input projection and the selective scan (d_conv = 4 typically).

Both have to be persisted, restored, and shipped together. The cache infrastructure treats them as a single "page" — page_size_bytes sums both — but the kernel addresses them separately.

page_size_bytes = sizeof(SSM) + sizeof(conv) + padding. Both tensors must be saved/restored atomically — the cache "page" is the bundle, but two distinct kernels touch the two halves.
Why page_size_padded exists The two tensors' raw byte sum may not match attention's page_size, so vLLM pads the Mamba page up to the uniform invariant from KV_CACHE_ALLOCATION.html. Padding bytes are unused but make the column-packed tensor layout work.
What this changes vs "generic linear attention" A state hand-off between layers, steps, or machines must move both tensors atomically. Restoring only the SSM state and zeroing the conv state would inject a 3-step transient where the convolution sees fresh-start input — garbage tokens until the conv buffer fills. The align cache mode exists for exactly this conv-state subtlety.

2. Selectivity: state-update weights depend on the input

Original S4 had fixed (A, B, C) matrices. Mamba's selective SSM makes A_t, B_t, C_t functions of the input token at step t:

A_t, B_t, C_t = projection(x_t)
S_t = A_t · S_{t-1} + B_t · x_t
y_t = C_t · S_t

This is the architectural innovation that earned the "Mamba" name over S4.

S4 (fixed gates): A, B, C are model parameters — same for every step. Mamba (selective): they are functions of the input. To replay step 6, you must re-run the input projection on "Mamba" — you can't shortcut by multiplying by precomputed Ak. That's why a lost prefix can't be cheaply recovered.
For the cache, this means You cannot cheaply replay a missed prefix. With a fixed-A SSM, you could in principle precompute powers of A and fast-forward. With selectivity, the weights are determined by the actual tokens, so replay requires running the full input projection on every missed token. Replay costs the same as the original compute.

This is the load-bearing reason MambaManager does not implement "replay from the latest checkpoint" as a recovery path during eviction or preemption. If the state is gone, the only way back is to re-run the full prefix — same as a cold prefill.

3. Two kernels: parallel scan (prefill) vs step recurrence (decode)

Softmax attention uses one kernel for prefill and decode (FlashAttention varlen handles both). Mamba2 uses two (mamba2_attn.py:107-167):

Prefill: chunk-wise parallel scan

Processes chunk_size tokens at once. Produces both per-token outputs and the final state. chunk_size is configured via model_config.get_mamba_chunk_size() — a per-model constant, typically 256.

Decode: step recurrence

One token at a time: S_t = A_t · S_{t-1} + B_t · x_t. Different metadata shape from prefill.

Two kernels, two metadata shapes. The chunk scan returns both per-token outputs and the chunk-tail state — that tail becomes the input to the next chunk (or the first decode step). last_state_block_idx is what tracks which physical block holds that hand-off state.
What this changes for the scheduler Chunked prefill must hand off the SSM state from chunk N's tail to chunk N+1's head. For attention, this is implicit in the block table (just append). For Mamba, the state at the chunk boundary must explicitly be the input to the next forward. This is what last_state_block_idx (single_type_kv_cache_manager.py:805) tracks.

4. chunk_size is not block_size

Mamba2 has three sizes that get confused:

NameSourceWhat it controls
block_sizecache_config.block_size (or aligned)Logical block size for BlockPool — checkpoint cadence in all mode.
chunk_sizemodel_config.get_mamba_chunk_size()Parallel-scan kernel tile. Architectural constant.
d_convModel architecture (typically 4)Conv state lookback length.
The inverse of attention's situation For attention, block_size is constrained by the kernel (see KERNEL_BLOCK_ALIGNMENT.html). For Mamba, block_size is constrained by no kernel — it's a pure allocation knob — but it must satisfy the uniform-page-size invariant across groups when paired with attention. The constraint comes from the allocator, not the math.

5. Why mamba_cache_mode="align" exists (it's the conv state)

LINEAR_ATTENTION.html "Three cache modes" introduced the three modes. The align mode is Mamba-specific, and the reason is the conv state.

The depthwise convolution of width d_conv reads d_conv − 1 past tokens' projected activations. At a block boundary, the next block's first token needs the previous block's last d_conv − 1 tokens to seed the conv — otherwise the convolution attends to zeros, producing wrong outputs.

Code: two blocks live per request in align mode 
# kv_cache_interface.py:555
return self.page_size_bytes * (2 + self.num_speculative_blocks)

The "previous block" holds the conv-state lookback for the start of the "current block." When the current block fills, the previous can be freed; the current becomes the new previous.

Code: last_state_block_idx tracking

Without align, a hybrid model's Mamba layers would diverge by a few tokens at every attention block boundary because the conv state at the boundary was zero-padded.

Hence the "2" in page_size_bytes * (2 + num_speculative_blocks) — one current, one prev. last_state_block_idx is the pointer that says which block plays the "prev" role; when the current fills, the old prev is freed and current becomes prev.

A pure-Mamba (non-hybrid) model could run with mode="none" because there are no attention boundaries to align to — but the conv-state issue still exists if you ever evict and resume mid-sequence. none is fine for monolithic in-place inference; align is required for chunked prefill, hybrid models, or anything that crosses a block boundary.

6. num_speculative_blocks: rollback budget for spec decode

Speculative decoding generates K draft tokens, runs them through the target model, and verifies acceptance. Rejected drafts must be rolled back. For attention, rollback is trivial — don't advance the block table cursor; the unused K/V at the tail is just unread.

For Mamba, rollback is hard The state at draft step K is already overwritten by the draft tokens (the recurrence advanced). To roll back to "only the first 2 of 5 drafts accepted," you need the state as of step 2, not the current state.

num_speculative_blocks (kv_cache_interface.py:537) reserves extra blocks per request so the pre-draft state is checkpointed and the draft runs forward into a separate block. After verification, discard the unaccepted-draft block and the pre-draft block becomes the new running state.

Code: the conservative subtraction in remove_skipped_blocks 
# single_type_kv_cache_manager.py:865
# NOTE: with async scheduling, num_computed_tokens can contain
# draft tokens from the previous step that may or may not be rejected.
# Assume all tokens are rejected so we don't free blocks we might need.
num_computed_tokens = max(0, num_computed_tokens - self.num_speculative_blocks)
Why "assume all rejected" Free the "draft" block optimistically and you've lost the only copy of valid state. Attention can afford to be optimistic because rejected K/V is harmless dead bytes; Mamba cannot because state is overwritten in place.

7. DCP and PCP are not supported

Code: hard asserts 
# MambaManager.find_longest_cache_hit
# single_type_kv_cache_manager.py:825-826
assert dcp_world_size == 1, "DCP not support mamba now."
assert pcp_world_size == 1, "PCP not support mamba now."

DCP (decode context parallelism) and PCP (prefill context parallelism) shard a request's tokens across ranks for attention computation. For softmax attention this works because each token's K/V is independent — rank 0 computes K/V for tokens 0..N/2, rank 1 for N/2..N, then they exchange.

For Mamba, the recurrence is sequential: S_t depends on S_{t-1}. Sharding tokens across ranks breaks the chain. There are parallel-scan algorithms that reduce sequential dependency (the chunk-wise scan already uses one within a chunk) but they don't trivially extend to cross-rank parallelism without communication on every step.

Fundamental architectural mismatch, not an unimplemented feature Any cluster-scale plan for Mamba inference has to either replicate (TP for parameters but not for sequence dimension) or pipeline-parallelize (PP splits layers, sequence stays whole on each stage). Sequence-dimension sharding is off the table.

8. Hybrid layouts (Jamba, Granite-Hybrid)

Jamba, Granite-Hybrid, and similar architectures interleave Mamba layers with attention layers in a fixed ratio. The design rationale matters for the cache:

Mamba layers: cheap context

Carry long-context "feel" cheaply. State is O(1), so a 256k-context Mamba layer holds 256k tokens of context for the same RAM cost as a 4k-context one. This is where the long-context budget goes.

Attention layers: exact retrieval

Some queries genuinely need to attend back to a specific token — a cited fact, a code symbol, a named entity. Attention layers are kept (at much lower count) to preserve this capability.

Cache implications
  • Memory profile is dominated by attention layers (linear in N) even though there are fewer of them.
  • Prefix-caching benefit is dominated by attention.
  • The align cache mode is not optional — without it, Mamba layers desynchronize from attention at block boundaries.

Grouping consequence: Jamba-class models always have at least two groups (one Mamba, one attention), each with its own block table, eviction rule, and SingleTypeKVCacheManager. The hybrid coordinator fans find_longest_cache_hit across them with the alignment_tokens discipline so neither group hits a prefix the other doesn't.

9. What "block size" even means for Mamba

For attention, "block of size 16" means: K/V for 16 consecutive token positions, addressable by slot = block_id * 16 + offset. The 16 is a real, semantic unit.

For Mamba, "block of size 16" means: a snapshot of state taken once per 16 input tokens. The block contains one (SSM_state, conv_state) pair, not 16 of them.

The "16" is the interval between snapshots, not the content multiplicity The slot mapping math (slot = block_id * block_size + offset) doesn't even apply to reading — Mamba kernels read the state at the block's snapshot point, not at specific offsets within the block.

It also explains why Mamba's block_size is often much larger (64, 128, 256, ...) — there's no kernel cost to a larger block, just a checkpoint-frequency / memory tradeoff.

11. The general lesson

The cache subsystem was originally designed for one shape of compute (softmax attention). Adding Mamba required:

  • A new SingleTypeKVCacheManager subclass with its own scan direction and hit semantics.
  • A new KVCacheSpec subclass (MambaSpec) with multiple state shapes and a padding field.
  • New scheduler bookkeeping (last_state_block_idx, num_speculative_blocks).
  • A new metadata path in attention backends (Mamba2AttentionMetadata).
  • A new alignment discipline (alignment_tokens / mamba_cache_mode="align").
  • Explicit guards excluding it from features that don't compose (assert dcp_world_size == 1).
Each is small and well-localized Together they make the point that "the KV cache" is not a single abstraction with one extension point — it's a stack of conventions about what a block contains, when it gets written, when it can be reused, and what the kernel expects to find there. Mamba reuses the bottom of the stack (BlockPool, hashing) and diverges at the top (everything semantic).

When future architectures land — linear-attention variants, hybrid retrieval modules, learned-cache schemes — the same pattern will repeat: keep the allocator, replace the semantics, add a manager subclass, document the new constraints.

See also: LINEAR_ATTENTION.html for the conceptual frame this builds on, and KV_CACHE_ALLOCATION.html for the column-packing layout that hosts both Mamba and attention groups.
