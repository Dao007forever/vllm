P/D Disaggregation — The Protocol

Why split prefill and decode in the first place, the five-method connector contract every transport implements, and where the cache transfer plane sits relative to vLLM's scheduler and worker.

TL;DR

Prefill and decode have different hardware profiles — prefill is compute-bound and bursty, decode is memory-bandwidth-bound and steady. P/D disaggregation puts them on different worker pools and moves KV cache between them. Every transport (NIXL, Mooncake pipe, Mooncake store) implements the same five-method KVConnectorBase_V1 contract; the differences live in who initiates the transfer, through what, and when the bytes land.

1. Why disaggregate at all

Prefill processes a long prompt in one large matmul-heavy forward, saturating tensor cores. Decode generates one token at a time, each step memory-bound by the KV cache size. On the same GPU pool, the two collide:

  • A long prefill stalls all in-flight decodes behind it.
  • Decode batching wants many tokens-in-flight; prefill wants few large requests-in-flight. The optimal batch composition is the opposite.
  • Hardware is wasted: an H100 doing decode runs at ~10% of its flop ceiling. An A100 doing prefill is fine.

The fix is to put prefill on one cluster (P pool) and decode on another (D pool), then ship the KV cache between them. P frees its KV the moment D has it; D doesn't even have a model-execute step for the prompt — just loads the KV and starts generating.

Why split the model in two
Colocated
  • Prefill blocks decode queue.
  • Same hardware does both badly.
  • One bad batch composition tanks throughput.
vs
Disaggregated
  • P and D scale independently.
  • Each pool tuned for its workload.
  • KV transfer becomes the new bottleneck — that's what these docs are about.

2. The connector contract

Every KV transfer transport in vLLM is a KVConnectorBase_V1 subclass living at vllm/distributed/kv_transfer/kv_connector/v1/base.py. The class has two halves: scheduler-side methods called inside the engine loop, and worker-side methods called on each model-execute worker.

Scheduler-side

  • get_num_new_matched_tokens(request) — "how many tokens of this request's prefix can you supply?" Returns a token count and an async flag. Local prefix cache hits are subtracted before the connector is asked.
  • update_state_after_alloc(...) — called once the scheduler has actually allocated GPU blocks to receive the load. The connector flips its per-request can_load flag here.
  • build_connector_meta(scheduler_output) — builds a KVConnectorMetadata describing what the worker must do this step. The struct rides on the ExecuteModelRequest down to the worker.
  • request_finished(request, ...) — returns (should_defer_free, kv_transfer_params). If the connector still has pending saves for the request, it returns True so the scheduler doesn't free the blocks yet.

Worker-side

  • start_load_kv(forward_context, ...) — called before the forward pass. Connectors that need KV loaded for this step initiate the transfer here.
  • wait_for_layer_load(layer_name) / save_kv_layer(layer_name, kv_layer, ...) — per-layer hooks for connectors that want to overlap transfer with the forward pass on a layer-by-layer basis. Both NIXL and Mooncake leave these as no-ops — they batch instead.
  • wait_for_save() — called after the forward pass before reporting the step done. Synchronizes any saves the worker scheduled this step.
  • get_finished(finished_req_ids) — called each step. Returns (done_sending, done_recving): which requests have completed their async save/load. The scheduler uses these to release blocks and unblock waiting requests.
What the contract does not mandate

It says nothing about who initiates the wire transfer, which direction it flows, or whether there's a store in the middle. Push vs pull is entirely the transport's choice. The contract just demands that get_finished() eventually reports completion.

3. Request lifecycle across P and D

A decode request on D arrives with a kv_transfer_params dict identifying its remote prefill (engine id, request id, host, port, etc.). It then flows through:

  1. D scheduler: ask the connector for matched tokens. get_num_new_matched_tokens may return immediately (Mooncake store: lookup keys in the store) or asynchronously (NIXL: kick off a handshake, return 0 + async-flag).
  2. D scheduler: allocate blocks for the load. The block manager reserves GPU paged-cache blocks based on the matched-token count. The connector is notified via update_state_after_alloc with the allocated block IDs.
  3. D scheduler: build metadata. build_connector_meta emits a per-step metadata struct listing every request that needs send or receive this step.
  4. D worker: start_load_kv. The worker reads the metadata, initiates async transfers (the transport-specific part), and returns immediately.
  5. D worker: forward pass. The forward runs before the transfer completes in some transports (relying on per-layer wait), after in others (NIXL waits on a host-side check before the prompt's first decode).
  6. D worker: get_finished. Each step the worker polls in-flight transfers. When all of a request's transfers are done, it appears in done_recving.
  7. P side, in parallel: P has been holding the request's blocks pinned (because request_finished said "defer free"). The completion signal — either a NIXL notif or a Mooncake response — flips the request to done_sending and lets P reclaim the blocks.

4. Push vs pull as a design axis

The single biggest architectural divergence between the three transports is who initiates the wire transfer:

Three initiation models
InitiatorMechanismCode reference
NIXL (pull) D worker NIXL READ xfer over RDMA nixl/worker.py:2220 make_prepped_xfer("READ", ...)
Mooncake pipe (push) P worker (after D requests) Mooncake batch_transfer_sync_write mooncake_connector.py:1365 self.engine.batch_transfer_sync_write(remote_session, src_ptrs, dst_ptrs, lengths)
Mooncake store (decoupled) P puts, D gets — via store batch_put_from_multi_buffers / batch_get_into_multi_buffers mooncake/store/worker.py:578, 730

Why does it matter? Two things change with the choice:

  • Who owns memory addressing. In a pull, D knows its own destination addresses and just needs P's source addresses. In a push, D must hand P its destination addresses upfront, before P has issued the transfer. The push case requires a request/response hop in the handshake; the pull case can issue READs as soon as the remote agent is registered.
  • Where the latency cost lands. A pull adds one round-trip of "ask P, then read P." A push has the request round-trip but then transfers as fast as the wire allows. With a store, you pay the wire latency twice (P→store, store→D) but get cross-run caching back.

5. Direct pipe vs shared store

The second axis: whether KV passes directly from P to D, or sits in a shared store that both consult.

Direct pipe (NIXL, Mooncake pipe)
  • Lowest latency.
  • P blocks pinned until D acknowledges.
  • No reuse across engine runs — if the same prompt comes back later, P re-prefills.
  • Requires P and D to be alive simultaneously and reachable to each other.
vs
Shared store (Mooncake store)
  • Cross-run reuse: a popular system prompt is put once, fetched many times.
  • Decouples P and D in time and space: P can finish and exit; D can join later.
  • Two wire hops instead of one.
  • Store cluster is now a new failure domain (and an ops burden — replication, eviction, capacity).

The Mooncake store variant is also the only one that's content-addressed: keys are derived from vLLM's prefix-cache block hashes. That's what makes cross-run reuse work — two different engine processes can independently hash the same prompt and arrive at the same key.

6. kv_producer, kv_consumer, kv_both

Every worker is configured at startup with a kv_role string. The connector branches on this to know which side of the transfer plane it's on:

  • kv_producer: prefill side. Saves, never loads.
  • kv_consumer: decode side. Loads, never saves.
  • kv_both: hybrid — some clusters run a unified pool that does both. Common in the Mooncake store deployment because cache hits short-circuit prefill anyway.
The store's force_skip_save rule

In MOONCAKE_STORE.html, the scheduler sets force_skip_save = (kv_role == "kv_consumer"). A pure consumer never writes back — even if it had a partial KV state to contribute. This is a deliberate design choice: the store treats prefill workers as the only source of truth. Mixed kv_both workers are the only ones that both feed and drain the store.

7. Composition with local prefix cache

Both vLLM workers already have a local prefix cache (the in-GPU block-hash cache covered in KV_CACHE.html). The connector and the local cache compose:

  1. Scheduler queries local prefix cache first. Matched tokens are subtracted.
  2. Connector is asked about the remaining tokens via get_num_new_matched_tokens. It returns how many it can supply from remote.
  3. The Mooncake store variant uses this to mask out tokens already covered locally: the recv thread won't redownload them. See connector/MOONCAKE.md for the vllm_cached_tokens mask field on LoadSpec.

This is why a hot system prompt costs essentially zero KV transfer: after the first call, every D worker has it in its local prefix cache, and the connector is never asked.
