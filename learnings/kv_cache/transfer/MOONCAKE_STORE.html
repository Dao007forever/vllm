Mooncake Store — KV Through a Content-Addressed Cache

P puts KV into a distributed store keyed by content hash. D gets KV from the same store. Workers never directly talk to each other. Cache hits cross engine runs.

TL;DR

The Mooncake store connector turns P/D disaggregation into a producer/consumer system with an intermediary. P calls batch_put_from_multi_buffers with content-addressed keys; D calls batch_get_into_multi_buffers with the same keys. Two background threads on each worker (KVCacheStoreSendingThread and KVCacheStoreRecvingThread) handle the actual transfers asynchronously. Because keys are content-addressed, a hot system prompt cached once is reused by every D worker, every engine run, forever — or until evicted.

1. Why a store, not a pipe

Direct P→D connectors (NIXL, Mooncake pipe) cache nothing across runs. If two API calls come in with the same long system prompt five minutes apart, P does the prefill twice. That's wasteful when prompts are reused — which is the case for most production deployments (system prompts, RAG contexts, agent scratchpads).

Mooncake Store inserts a content-addressed cache between P and D. The first call pays the prefill + store-put cost. Every subsequent call hits the cache and pays only the store-get cost. The same prompt across different engine processes deduplicates automatically because keys are derived from vLLM's prefix-cache block hashes.

2. Content addressing & the key format

Every block stored gets a key encoding its identity, derived from PoolKey.to_string() (mooncake/store/data.py:58-67): 

def to_string(self) -> str:
    return (
        f"{self.key_metadata.model_name}"
        f"@tp_rank:{self.key_metadata.tp_rank}"
        f"@pcp{self.key_metadata.pcp_rank}"
        f"@dcp{self.key_metadata.dcp_rank}"
        f"@pp_rank:{self.key_metadata.pp_rank}"
        f"@group:{self.key_metadata.group_id}"
        f"@{self.chunk_hash}"      # vLLM's prefix-cache block hash
    )

A worked example for a Llama-3-8B on TP=2, PP=1: 

llama-3-8b@tp_rank:0@pcp0@dcp0@pp_rank:0@group:0@a3f2c1d4...
llama-3-8b@tp_rank:1@pcp0@dcp0@pp_rank:0@group:0@a3f2c1d4...

Both keys carry the same chunk_hash but differ in tp_rank — one for each shard of the KV. Any D worker that knows the model name and prompt can independently compute these keys and ask the store "do you have these?" Cross-process dedup falls out of the design.

Why the prefix-cache hash, not a new hash?

vLLM already content-addresses blocks for the local prefix cache. The store connector reuses that hash so a local prefix-cache hit and a store hit refer to the same logical block. See KV_CACHE.html for the receptive-field argument behind that hash and why it's sound.

4. The two background threads

Unlike NIXL (which polls in the scheduler loop) and the Mooncake pipe (which uses an asyncio event loop), the Mooncake store uses classic worker threads with queues.

4.1 Sending thread (P or kv_both only)

KVCacheStoreSendingThread (mooncake/store/worker.py:464-623) is the producer. The scheduler enqueues a ReqMeta with can_save=True each step, and the thread:

  1. Floors token_len_chunk to the LCM block size across KV cache groups.
  2. Consults coord.store_mask(token_len) to skip chunks that shouldn't be stored (SWA chunks outside the window, Mamba's recurrent tail, etc.). The mask comes from the coordinator — see next section.
  3. Applies a TP striding (sl = slice(self.tp_rank % self.put_step, None, self.put_step)), which spreads put load across TP ranks when the store can serve a block from any of them.
  4. Calls batch_is_exist(keys) to dedup — if another process or run already put these blocks, skip the put.
  5. Calls batch_put_from_multi_buffers(keys, addrs, sizes, replicate_config) for the survivors.
  6. Handles failures: a NO_AVAILABLE_HANDLE (-200) return code triggers the store-pressure path.

4.2 Receiving thread (D or kv_both only)

KVCacheStoreRecvingThread (mooncake/store/worker.py:657-759) is the consumer. The scheduler enqueues a ReqMeta with load_spec.can_load=True, and the thread:

  1. Uses load_spec.token_len as the load range and load_spec.vllm_cached_tokens as a mask offset — no need to re-fetch what's already in the local prefix cache.
  2. For each chunk hash, builds keys with group:g + this worker's (tp_rank, pp_rank).
  3. Calls batch_get_into_multi_buffers(keys, addrs, sizes). The store does an RDMA write straight into D's KV cache memory.
  4. If disk-offload buffer budget is configured, splits the get into sub-batches that fit (_split_disk_offload_load_batches, store/worker.py:185-191, 693-721). Single oversize keys are skipped entirely with a warning.
One thread, one queue, no fan-out

There's a single send thread and a single recv thread per worker — not a thread pool. Concurrency comes from the store itself, which batches RDMA writes internally. From vLLM's side, the bottleneck is "how fast can I assemble batches", not "how many transfers can I parallelize."

5. The coordinator and load/store masks

Not every chunk should be stored, and not every chunk that's in the store should be loaded. Sliding-window attention only needs the last K chunks; Mamba's recurrence collapses earlier state into the current one. The store connector handles this with a MooncakeStoreCoordinator (mooncake/store/coordinator.py:55-182) that mirrors vLLM's hybrid KV cache coordinator: 

def load_mask(self, block_hashes, token_len):
    """Per-group load masks: mask[g][i] is True iff group g's
    spec would populate chunk i locally (e.g., SWA / Mamba tail-only)."""
    masks, _ = self.find_longest_cache_hit(
        block_hashes, token_len, ExternalCachedBlockPool()
    )
    return masks

def store_mask(self, aligned_token_len):
    """Per-group store masks: mask[g][i] is True iff chunk i would
    be populated by some future cache hit."""

Mechanically: the producer thread uses store_mask to decide what's worth storing; the consumer thread uses load_mask to decide what's worth fetching. Together they ensure hybrid models (full attention + SWA, or attention + Mamba) put exactly the chunks that future fetches will want and skip chunks that will be re-derived locally on load.

The hybrid memory allocator (HMA) wrinkle

For models with multiple KV cache groups (different block sizes per group), the connector stretches its effective block size to the LCM across groups. Token ranges are floored to this LCM in _handle_request. Companion read: KV_CACHE_ALLOCATION.html explains why groups exist in the first place.

6. TP/PP via candidate keys + dedup

Because keys carry tp_rank and pp_rank, handling TPP ≠ TPD doesn't need a TP mapping module — the consumer just asks the store about all relevant ranks. Lookup logic (mooncake/store/worker.py:1210-1236): 

tp_count = min(self.tp_size, self.num_kv_head)
for g_idx, db in enumerate(self.token_dbs):
    for chunk_id, h in enumerate(group_hashes):
        for tp in range(tp_count):
            for pp in range(self.pp_size):
                md = dataclasses.replace(db.metadata, tp_rank=tp, pp_rank=pp)
                candidate_keys.append(PoolKey(md, h.hex()).to_string())

res = self.store.batch_is_exist(candidate_keys)

A chunk hash is considered "present" only when all expected (tp, pp) shards are in the store (mooncake/store/worker.py:1225-1231): 

expected_per_key = max(1, tp_count * self.pp_size)
present_count: dict[tuple[int, bytes], int] = {}
for gh, exists in zip(candidate_meta, res, strict=True):
    if exists == 1:
        present_count[gh] = present_count.get(gh, 0) + 1
exists_set = {gh for gh, c in present_count.items() if c >= expected_per_key}

This is the magic that absorbs heterogeneous TP between P and D: Mooncake store doesn't care if P was TP=4 and D is TP=2. The keys are just shard-labeled. D fetches the shards it needs by iterating over the appropriate TP indices. See PARALLELISM.html for the cross-connector comparison.

7. Store pressure: skip-and-clear

A real-world store cluster can saturate — bandwidth saturated, backing disk full, replication queue full. Mooncake signals this by returning NO_AVAILABLE_HANDLE from batch_put_from_multi_buffers. The connector handles it with a tiny state machine on the send thread (mooncake/store/worker.py:445-462): 

def _mark_request_skipped_for_pressure(self, req_id: str) -> bool:
    with self.done_task_lock:
        already_skipped = req_id in self._skip_store_requests
        self._store_pressure_active = True
        self._skip_store_requests.add(req_id)
    return already_skipped

def _clear_store_pressure(self) -> bool:
    with self.done_task_lock:
        if not self._store_pressure_active and not self._skip_store_requests:
            return False
        self._store_pressure_active = False
        self._skip_store_requests.clear()
    return True

Behavior:

  • First NO_AVAILABLE_HANDLE for a request: mark it skipped, log a warning, return without retrying. Other requests on the same thread are still tried.
  • Future batches for this request are silently dropped while the pressure flag is set (_should_skip_request check at the top of _handle_request).
  • First successful put after pressure: clear the flag, log recovery, resume normal behavior.
Skipping is not a write-through-disk fallback

Under pressure, the connector simply doesn't store. There's no retry queue, no spill-to-disk-locally. The reasoning: if the store can't take it, putting it in a local buffer doesn't help — D was going to fetch it from the store, not from this P. The next time the same prompt arrives, prefill recomputes. This keeps the connector simple and forces cluster scaling decisions to live where they belong (in the store layer).

8. Failure modes

Store rejects put (code < 0)

Each failed key is logged with its code. If the code is NO_AVAILABLE_HANDLE, the pressure flag fires. Other negative codes are logged but otherwise tolerated. No exception is raised — the engine continues.

D's get fails

Same shape on the recv thread: failed keys are logged, the request is marked finished without all blocks loaded, and the engine recomputes the missing tokens locally. Strictly best-effort on the consumer side.

Oversize key

With disk offload enabled, a single key that exceeds the worker's staging budget is skipped entirely with a warning (mooncake/store/worker.py:693-721). This shouldn't happen with vanilla block sizes, but big-block deployments need to size their staging budget accordingly.

kv_consumer accidentally writes

Can't happen by construction. The scheduler sets force_skip_save = (kv_role == "kv_consumer") at ReqMeta construction time (mooncake/store/scheduler.py:167), and the send thread isn't even started for pure consumers (mooncake/store/worker.py:1121-1140).

9. How this differs from the pipe

AspectMooncake pipeMooncake store
Connection topology P↔D direct P→Store→D
Cross-run reuse No Yes (content-addressed)
P/D simultaneity required Yes — both must be alive No — P can exit, D joins later
Wire hops One (P→D) Two (P→Store, Store→D)
TP handling Pre-planned via _get_tp_ratio Candidate-key lookup + dedup
Threading asyncio + ThreadPoolExecutor Two dedicated background threads
Failure surface Per-request timeout / ZMQ response error Store pressure + per-batch skip
Best for Low-latency P/D in same DC, no cross-call caching Heavy prompt reuse, multi-tenant clusters, P fleet churn
