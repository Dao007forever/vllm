A Mathematical Abstraction for Heterogeneous-Parallel KV Transfer

The NIXL connector reconciles heterogeneous PP and TP between prefill (P) and decode (D) with two hand-rolled, axis-specific mappings composed in a nested loop. This note names the mathematical objects that structure is an instance of — product index spaces, partition lattices, and affine layouts — so a future abstraction can add a new parallel dimension by adding a factor, not by writing new mapping code.

TL;DR

The global KV cache is a product index space Layer × Head × Block × …. A parallel layout partitions each axis independently and hands the pieces to workers arranged in a mesh. Because the axes are orthogonal, the set of P→D transfers is the common refinement (meet) of P's and D's partitions, and that meet factorizes:

P ∧ D  =  (PLayer ∧ DLayer)  ×  (PHead ∧ DHead)  ×  (PBlock ∧ DBlock)

Compute one tiny meet per axis, take the product — that is the transfer plan. The within-cell address math is an affine map on an integer lattice (a CuTe-style layout). These are exactly the objects behind PyTorch DTensor (DeviceMesh + Placement), JAX/GSPMD (Mesh + PartitionSpec), and CUTLASS/CuTe Layouts.

1. The KV cache is a product index space

Forget sharding for a moment. The logical KV state of a model is a single tensor addressed by a tuple of independent coordinates:

G  =  Layer  ×  Head  ×  Block  ×  {K, V}  ×  intra-block-slot

This is a Cartesian product: a global address is a tuple (ℓ, h, b, …), and the axes are orthogonal — the set of valid layer indices doesn't depend on which head you picked, and vice versa. That orthogonality is not a coincidence of the implementation; it is a property of the model's KV state, and it is the entire reason “one mapping after another” works.

A parallel layout is then a function

place: G  →  (worker, local_offset)

and each parallelism strategy partitions one axis:

The claim of this note. Because place factorizes over orthogonal axes, every question about heterogeneous transfer decomposes per-axis and recombines by a product. PP and TP are not special; they are two factors in the same construction.

2. What “partition” means (with examples)

A partition of a set S is a way of cutting S into pieces (cells) such that:

  1. Every element is covered — the cells' union is all of S.
  2. Cells don't overlap — they are pairwise disjoint.
  3. Cells are non-empty.

Equivalently: every element of S belongs to exactly one cell. “Sharding axis X across N workers” is a partition of axis X into N cells, where cell i = the coordinates owned by worker i.

Example — the Layer axis under PP. Take a model with 8 hidden layers, S = {0,1,2,3,4,5,6,7}.
P side, PP=2: pp0 = {0,1,2,3} pp1 = {4,5,6,7}
D side, PP=4: pp0 = {0,1} pp1 = {2,3} pp2 = {4,5} pp3 = {6,7}

Both are valid partitions of the same 8-element set: disjoint, covering, non-empty. They are different partitions — that difference is exactly “heterogeneous PP.” vLLM enforces partition-hood of the layer axis in pp_layer_map.py: the PPLayerMap raises unless the P-side boundaries are contiguous (no overlap) and cover all hidden layers (full coverage).

Partitions of a fixed set are themselves ordered: partition A is finer than B (and B is coarser) if every cell of A sits inside some cell of B. Above, D's PP=4 partition is finer than P's PP=2 partition. Under this “finer-than” order, the partitions of a set form a lattice — which is what makes the next section's “meet” well-defined.

3. The transfer plan is a common refinement (meet)

P and D partition the same global set G, just differently. A single NIXL read moves a chunk of KV from one P worker to one D worker. So a valid chunk is a set of coordinates that lives entirely inside one P cell (so it has a single source) and entirely inside one D cell (so it has a single destination). The largest such chunks are exactly the non-empty intersections

P ∧ D  =  {  p ∩ d  :  p ∈ P,   d ∈ D,   p ∩ d ≠ ∅  }

This is the common refinement, or meet (∧), of the two partitions: the coarsest partition that is finer than both. The set of transfers you must perform is the set of cells of P ∧ D. Each cell carries a unique (source worker, destination worker) pair — that pair is the transfer.

Worked example — one axis (Layer), uneven on both sides. 8 layers. Suppose (artificially uneven, to show the general case):
P: P0 = {0,1,2,3,4} P1 = {5,6,7}
D: D0 = {0,1,2} D1 = {3,4,5} D2 = {6,7}
P∧D: {0,1,2}  P0→D0 {3,4}  P0→D1 {5}  P1→D1 {6,7}  P1→D2

Four cells → four transfers. Notice D1 = {3,4,5} had to be filled from two sources (P0 gives layers 3–4, P1 gives layer 5) — the meet discovered that automatically. This is the layer-axis analogue of NIXL's “a D worker reads from multiple P ranks.”

Why this is the right object. “Homogeneous” is just the special case where one partition refines the other (e.g. P=D, so P∧D = P = D, one transfer per worker). The meet handles homogeneous, “D finer than P,” “P finer than D,” and fully-misaligned partitions with one definition — no case-split. The current code's three TP branches (D_TP ≥ P_TP, P_TP > D_TP, MLA) are re-derivations of the meet for specific partition shapes — traced out case by case in §10.

4. What P ∧ D = (PA ∧ DA) × (PB ∧ DB) means

Now two axes at once — A = Layer, B = Head — so G = A × B. A worker that is sharded on both axes owns a rectangle: a Layer-cell crossed with a Head-cell. So P's partition of G is itself a product, P = PA × PB, and likewise D = DA × DB.

The theorem (a one-line set-algebra fact) is that the meet of two product partitions is the product of the per-axis meets:

(aP×bP) ∩ (aD×bD)  =  (aP ∩ aD) × (bP ∩ bD)   ⇒   P ∧ D  =  (PA ∧ DA) × (PB ∧ DB)

In words: a rectangle-vs-rectangle intersection is just the intersection of the row-ranges crossed with the intersection of the column-ranges. So to enumerate every transfer you do not walk the giant global product. You compute the Layer meet (small), compute the Head meet (small), and take their Cartesian product. That product is the nested loop.

Worked example — two axes. 4 layers {0,1,2,3}, 4 KV heads {0,1,2,3}.
Layer: P has PP=2, D has PP=1 PL: {0,1} | {2,3} DL: {0,1,2,3} meetL = {0,1} | {2,3} (2 cells)
Head: P has TP=2, D has TP=4 PH: {0,1} | {2,3} DH: {0}|{1}|{2}|{3} meetH = {0}|{1}|{2}|{3} (4 cells)

Global space has 4×4 = 16 coordinates. P shards it into 2×2 = 4 workers; D into 1×4 = 4 workers. The transfer plan is meetL × meetH = 2 × 4 = 8 cells — computed from a 2-cell meet and a 4-cell meet, never by scanning all 16 coordinates.

This is the formal content of your intuition. “Each dimension is orthogonal and we can do one mapping after another” is precisely the statement that the meet distributes over the product. Add a third orthogonal axis (say context parallel on the Block axis) and the plan becomes meetL × meetH × meetB — one more nested loop, zero new mapping logic.

5. Inside a cell: affine layouts

The meet tells you which P worker talks to which D worker. It does not tell you the byte offsets. For that, each worker's local buffer is described by a layout: a map from a logical coordinate to a flat offset, built from a shape and a stride per axis — exactly the CUTLASS / CuTe Layout object, whose defining operation is composition.

offset(ℓ, h, b, s)  =  strideL·(ℓ − startL) + strideH·(h − startH) + strideB·b + s

Each side's local buffer is an affine sub-map of the global layout (a base offset plus per-axis strides), and a transfer reindexes one into the other: D_local = D_global⁻¹ ∘ P_global ∘ P_local. Affine maps on integer lattices compose and invert cleanly — they form a groupoid — which is why the messy-looking transforms in the connector are actually closed under composition:

  • Block-size ratio (get_mapped_blocks): a stride/scale on the Block axis — one logical block becomes ratio contiguous kernel blocks.
  • Logical→physical kernel blocks (_logical_to_remote_kernel_block_ids): another Block-axis sub-lattice map, with a possibly different ratio per remote PP stage.
  • Head-split offset (rank_offset_factor, head slots): the base offset on the Head axis selecting which sub-range of heads this D rank pulls.
  • HND↔NHD permute (enable_permute_local_kv): a permutation of the stride vector — a relabeling layout, composable as a post-process.
Why it matters for the abstraction. If a cell's address math is “an affine layout,” then the per-axis meet only needs to emit, for each cell, a tiny (shape, stride, base) triple on each side. The transfer engine composes them generically — it never needs to know that one axis was “PP” and another was “TP.”

6. Where partitions break: replication (MLA / GQA)

A partition assumes each coordinate has exactly one owner. Two important cases violate that:

  • MLA replicates the (latent) KV cache across all TP ranks — every rank owns the whole Head axis.
  • GQA with num_kv_heads < tp_size replicates each KV head across several ranks.

A replicated axis is a covering, not a partition (cells overlap). The fix is the same one DTensor uses: enrich the per-axis placement vocabulary from “always Shard” to

Placement(axis)  =  Shard(into N)  |  Replicate

When an axis is Replicate on the source, the planner picks a canonical owner (e.g. the first rank holding each head) so the covering collapses back to a partition and the meet is well-defined again. That canonicalization is exactly what the code does today: np.unique(…, return_index=True) for GQA dedup, and the single-read + notify-others path for MLA in _read_blocks_for_req. So the minimal extension to “just partitions” is Shard | Replicate per axis — nothing more exotic.

Replicate is directional. The phrase “on the source” above is load-bearing: a Replicate axis means opposite things depending on which side of the transfer it sits on.
  • Source Replicate → dedup (pick one). Every P rank holds an identical copy, so the meet's redundant owners collapse to a single canonical holder — fewer reads. This is the MLA single-read and the GQA np.unique.
  • Destination Replicate → fan-out (give to all). Every D rank needs its own full copy, so the one meet-cell must be paired with every reader — more reads. This is why the connector computes compute_tp_mapping per local rank: each D rank independently re-issues the same read.
It is the same split DTensor draws between Replicate→Shard (a local slice, no comms) and Shard→Replicate (an all-gather/broadcast). In the owner-sequence model (§10) a Replicate placement is just n identical sequences, and the planner reads them directionally: 
Replicate(size, n) = (range(size),) * n        # every rank holds the whole axis

# axis_edges (§10) reads that directionally:
#   SOURCE side: identical seqs -> keep ONE canonical rank   (dedup,   fewer reads)
#   DEST   side: pair the data with EVERY rank that needs it (fan-out, more reads)
So a replicated axis needs no special case in the overlap itself — the dedup-vs-fan-out asymmetry lives entirely in how axis_edges enumerates source and destination ranks.
Hybrid models = a sum of products. Mamba/SSM groups index the Block axis as whole state, not token-blocks, so their refinement rule differs (you can't trim a partial prefix). The honest type is a coproduct over kv-cache groups, G = Σg Gg, each group its own product space with its own per-axis placements. The connector's source_ranks_per_group and _group_spec_types are this sum already.

7. The rank side is also a product: the mesh

The workers aren't a flat list; they're a grid. With PP and TP, worker identity is the pair (pp_rank, tp_rank) — a coordinate in a PP × TP device mesh, flattened to a linear rank by mixed-radix encoding (rank = pp_rank·TP + tp_rank, or whatever the world ordering is). Heterogeneity is simply that the P-mesh and D-mesh have different extents:

meshP : 2 × 2  (PP=2, TP=2)     meshD : 1 × 4  (PP=1, TP=4)

The per-axis owner map answers “given this meet-cell on axis X, which P-mesh coordinate (along X) holds it?” Cross the per-axis answers and you get the full source/destination mesh coordinates — the colored labels in the grid above. This is precisely DeviceMesh (PyTorch) / Mesh (JAX): named axes with extents, plus a rule to map mesh coordinates to ranks.

Mapping the abstraction onto torch.DeviceMesh / DTensor

The pieces line up one-to-one with PyTorch DTensor, which is worth making explicit because it pins down where heterogeneity actually lives. A DeviceMesh is the n-dimensional grid of ranks; its mesh_dim_names are the parallel axes. A DTensor's placement list has one entry per mesh dim (this is a hard DTensor rule: len(placements) == device_mesh.ndim), and each entry — Shard(tensor_dim) or Replicate() — says how the tensor is laid out along that mesh dim.

The running example as two meshes + placements. 
from torch.distributed.device_mesh import init_device_mesh
from torch.distributed.tensor import DTensor, Shard, Replicate

# Rank grid: one mesh dim per parallel axis.
mesh_P = init_device_mesh("cuda", (2, 2), mesh_dim_names=("pp", "tp"))   # 2×2 = 4 P ranks
mesh_D = init_device_mesh("cuda", (1, 4), mesh_dim_names=("pp", "tp"))   # 1×4 = 4 D ranks

# KV-cache logical tensor dims:  0 = layer,  1 = head,  2 = block, ...
# One placement per MESH dim, in mesh-dim order (pp, tp):
P_placements = [Shard(0), Shard(1)]      # pp-dim shards layers; tp-dim shards heads
D_placements = [Shard(0), Shard(1)]      # SAME spec — only the mesh SHAPE differs
D_placements_mla = [Shard(0), Replicate()]   # MLA: tp-dim replicates the head axis

# The P→D KV transfer is conceptually a redistribute across the two meshes:
kv = DTensor.from_local(local_kv, mesh_P, P_placements)   # what P holds
kv.redistribute(mesh_D, D_placements)                     # what D needs   ← the NIXL reads
Where heterogeneity lives. When P and D use the same strategy (shard layers by PP, heads by TP), the placement lists are byte-for-byte identical — [Shard(0), Shard(1)] on both sides. The only difference is the mesh extents: (2,2) vs (1,4). So “heterogeneous PP/TP” is precisely mesh_P.shape ≠ mesh_D.shape, and the transfer plan is whatever redistribute would compute — which, for orthogonal mesh dims, is exactly the per-axis meet-product of §4.

Where the analogy stops (and why NIXL exists). DTensor.redistribute operates within a single DeviceMesh — one process group, collectives over NCCL, and the source and target placements share the same ranks. P/D disaggregation has two separate worlds (the prefill cluster and the decode cluster) with disjoint rank spaces, bridged by NIXL RDMA rather than NCCL. So you can't literally call .redistribute() across them; NIXL is the cross-mesh transport DTensor lacks. What transfers cleanly is the vocabulary and the algebra: describe each side as (mesh, placements), and the read plan is the cross-mesh generalization of redistribute — the per-axis meet-product. (Caveat: DTensor's Shard uses torch.chunk, so an indivisible axis gets an uneven last shard; the connector's contiguous-range splits assume divisibility, which vLLM's configs enforce.)

8. Prior art: the objects already have implementations

PyTorch DTensor

DeviceMesh (named axes + extents) and a per-mesh-dim Placement list of Shard(dim) / Replicate(). redistribute(placement→placement) is exactly a heterogeneous reshard. Closest match to the Shard | Replicate vocabulary you need for MLA/GQA — see §7's concrete mapping for the full correspondence.

JAX / GSPMD

Mesh + PartitionSpec (which mesh axis each tensor dim maps to). The SPMD partitioner solves general resharding and decomposes it per mesh axis — the production-grade version of “the meet distributes over the product.”

CUTLASS / CuTe

Layout = shape × stride, hierarchical and composable; composition is the primitive operation. This is the right object for the within-cell affine address math (block-size ratio, head-split offset, HND/NHD permute) — “do one mapping after another” as literal function composition.

10. Proposed abstraction: ordered-sequence Placement + plan_transfer

The objects above suggest one interface, and §5 pins the representation: describe each side's placement on an axis as each rank's owner-sequence, in local-buffer order. A global index's membership in a sequence is the partition (who holds it); its position is the local slot (the layout). One object answers both — so there is no separate owner/local_layout method, no interval arithmetic, and no per-axis LayerAxis/HeadAxis class. An Axis is just metadata; the planner is dimension-agnostic.

The whole interface. 
# Placement = each rank's owner-sequence, in LOCAL BUFFER ORDER.
#   membership = the partition        |        position = the local slot
Placement = tuple[Sequence[int], ...]            # owners[r] = global indices rank r holds

def Shard(size, n):     return tuple(range(i*size//n, (i+1)*size//n) for i in range(n))  # range = compact
def Replicate(size, n): return (range(size),) * n                                         # n identical seqs
#   blocks / permuted / strided: pass explicit lists instead of ranges — same interface.

@dataclass
class Axis:                  # just metadata now — no per-axis cells/owner/layout
    name: str                # "layer" | "head" | "block" | ...
    size: int

@dataclass
class Side:                  # one engine: mesh extents + an owner-sequence Placement per axis
    mesh:  tuple[int, ...]   # extents per axis, in AXES order   (pp, tp)
    place: dict[str, Placement]

def meet_edge(src_seq, dst_seq):        # the ONLY core op: membership + slot in one pass
    dst_slot = {e: j for j, e in enumerate(dst_seq)}
    pairs    = [(i, dst_slot[e]) for i, e in enumerate(src_seq) if e in dst_slot]
    return runs(pairs)                  # maximal (src_slot, dst_slot, len) where BOTH step by 1

def axis_edges(P_place, D_place):       # per axis: (src_rank, dst_rank, runs) for overlapping pairs
    seen, edges = {}, []
    for dr, d in enumerate(D_place):    # destination fan-out: every reader that needs data
        for sr, s in enumerate(P_place):
            r = meet_edge(s, d)
            if not r: continue
            if (dr, tuple(s)) in seen: continue   # source dedup: identical replica already covers dr
            seen[(dr, tuple(s))] = sr
            edges.append((sr, dr, r))
    return edges

def plan_transfer(axes, P: Side, D: Side):
    per_axis = [axis_edges(P.place[ax.name], D.place[ax.name]) for ax in axes]
    for combo in itertools.product(*per_axis):           # one edge per axis == the nested loop
        src = mesh_rank(P, [e[0] for e in combo])         # e = (src_rank, dst_rank, runs)
        dst = mesh_rank(D, [e[1] for e in combo])
        yield Read(src, dst, region=[e[2] for e in combo])  # product of per-axis runs = a descriptor box

axis_edges bakes in the directional Replicate of §6 for free: the outer loop over destination ranks fans out, and the seen guard dedups identical source replicas to one canonical holder. (runs and mesh_rank are spelled out under “Building the plan” below.)

The mechanical equivalences to today's code:

  • layer owner-sequences (a range per PP rank) → replace PPLayerMap + the layer-name matching (which is list.index(name)).
  • head owner-sequences (Shard or Replicate) → replace all three branches of compute_tp_mapping plus the GQA/MLA dedup.
  • block-id sequences (explicit lists) → absorb block_size_ratio and physical_blocks_per_logical as the list contents.
  • plan_transfer's itertools.product → is the nested loop in _read_blocks_for_req; each meet_edge run → one NIXL descriptor.

Concrete instantiation — PP and TP on the running example

Same numbers as the grid in §4: 4 layers, 4 KV heads; P is PP=2, TP=2 (mesh 2×2), D is PP=1, TP=4 (mesh 1×4). There are no axis classes to write — the axes are metadata, and the placements are owner-sequences the Shard constructor builds:

The two axes and both sides' placements. 
AXES = [Axis("layer", 4), Axis("head", 4)]

P.place["layer"] = Shard(4, 2)             P.place["head"] = Shard(4, 2)
   = (range(0,2), range(2,4))                 = (range(0,2), range(2,4))
     pp0:{0,1}   pp1:{2,3}                       tp0:{0,1}   tp1:{2,3}

D.place["layer"] = Shard(4, 1)             D.place["head"] = Shard(4, 4)
   = (range(0,4),)                            = (range(0,1),range(1,2),range(2,3),range(3,4))
     pp0:{0,1,2,3}                              tp0:{0} tp1:{1} tp2:{2} tp3:{3}

Reading a slot off a sequence is the whole layout story — no base/stride object. On the head axis, P's tp0 holds range(0,2)=[0,1]; the slot of head 1 is [0,1].index(1)=1, and that “1” is exactly NIXL's rank_offset_factor. On the layer axis, D's pp0 holds range(0,4)=[0,1,2,3]; the slot of layer 2 is index(2)=2, exactly the layer-region base _local_region_indices_for_layer_names computes. Because a cell is a range, .index(e) == e − start — O(1), no materialization.

The three TP branches are one head placement

The claim from §3 — that compute_tp_mapping's three branches (D_TP ≥ P_TP, P_TP > D_TP, MLA) are just the Head-axis edges for three placement shapes — is worth seeing concretely. Hold PP fixed (it's orthogonal) and run axis_edges on the head placements for 8 KV heads (to make the splits visible), shown here as explicit owner-lists. The code is identical every time; only the two head Placements change.

Reading the tables. Each row is one (P rank → D rank) edge — the output of meet_edge(src_seq, dst_seq) for that pair:
  • shared — the global heads both ranks hold (what moves): set(src_seq) ∩ set(dst_seq).
  • src slots — where those heads sit in the source rank's local buffer: src_seq.index(e).
  • dst slots — where they sit in the destination rank's buffer: dst_seq.index(e).
  • run — those slot pairs collapsed into one contiguous descriptor (src_slot→dst_slot ×len): copy len contiguous slots from src_slot on the source into dst_slot on the destination.
Example — Case 1's P0 → D1, with P0=[0,1,2,3], D1=[2,3]: shared {2,3}; src slots [0,1,2,3].index(2,3)=2,3; dst slots [2,3].index(2,3)=0,1; pairs (2,0),(3,1) step together → one run (2→0 ×2) = “send P0's slots 2–3 into D1's slots 0–1 as one 2-element copy.”

Building the plan: from Placement + Axis to reads

Now the glue. Each engine is described by a spec: its mesh extents plus an owner-sequence Placement per axis. That spec is the entire input — building the plan is constructing the two Sides and calling plan_transfer. Here are its two remaining helpers (runs, mesh_rank) and the example's specs, with every intermediate value the call produces:

Constructing the specs and calling the driver. 
def runs(pairs):                     # collapse (src_slot,dst_slot) into contiguous runs
    out = []
    for s, d in pairs:
        last = out[-1] if out else None
        if last and s == last[0]+last[2] and d == last[1]+last[2]:
            out[-1] = (last[0], last[1], last[2]+1)              # extend the run by 1
        else:
            out.append((s, d, 1))                                # start a new run
    return out

def mesh_rank(side, coords):         # mixed-radix flatten of per-axis coords -> global rank
    r = 0
    for extent, c in zip(side.mesh, coords):
        r = r * extent + c
    return r

# 1. Axes are metadata; each side = mesh extents + an owner-sequence per axis.
AXES = [Axis("layer", 4), Axis("head", 4)]
P = Side(mesh=(2, 2), place={"layer": Shard(4,2), "head": Shard(4,2)})   # PP=2, TP=2
D = Side(mesh=(1, 4), place={"layer": Shard(4,1), "head": Shard(4,4)})   # PP=1, TP=4

# 2. That is the entire input. Build the plan:
reads = list(plan_transfer(AXES, P, D))        # -> 8 Reads

# Inside, traced for read #6 (layer edge Lb, head edge H1):
#   axis_edges(layer): (pp0,pp0,[(0,0,2)]) , (pp1,pp0,[(0,2,2)])             <- meet_edge per pair
#   axis_edges(head) : (tp0,tp0,[(0,0,1)]),(tp0,tp1,[(1,0,1)]),(tp1,tp2,[(0,0,1)]),(tp1,tp3,[(1,0,1)])
#   product          : 2 × 4 = 8 reads                                       <- itertools.product
#   #6 = ( layer (pp1,pp0,[(0,2,2)]) , head (tp0,tp1,[(1,0,1)]) )
#   src = mesh_rank(P,[1,0]) = 1*2+0 = 2     dst = mesh_rank(D,[0,1]) = 0*4+1 = 1
#   region = [ layer (0->2 ×2) , head (1->0 ×1) ]    # a 2-layer × 1-head box
  1. Per-axis edges. axis_edges overlays the two sides' owner-sequences, emitting (src_rank, dst_rank, runs) for every overlapping pair. This is the only place sequences are compared.
  2. Product = reads. itertools.product over the per-axis edge lists picks one edge per axis — 2×4 = 8 reads, the §4 factorization mechanized.
  3. Ranks. mesh_rank mixed-radix-flattens the per-axis src_ranks (and dst_ranks) into a global source and destination rank on each mesh.
  4. Region. The read's region is the product of the per-axis runs — a multi-dim contiguous box; each run's len is a descriptor extent the transport composes (§5).

Each run is (src_slot → dst_slot × len) = one contiguous descriptor. Two patterns: the head src_slot alternates 0, 1 down each colour-pair — the TP head-split (rank_offset_factor) picking which of the P rank's two heads to read. The layer dst_slot jumps to 2 for the bottom four rows — the PP collapse: D's single pp0 stores all 4 layers, so layers {2,3} land at dst slot 2. MLA variant in one line. Swap both head placements to Replicate(4, ·): every head sequence becomes range(0,4), so each head edge is one whole-head run (0→0 ×4) — no head-split. axis_edges dedups the redundant P replica (source) and fans out to every D rank (destination, §6), reproducing NIXL's “single read, notify the rest” MLA path with no new branch — the 8 reads stay, but each is one contiguous whole-latent copy instead of a head-slice.

Representation note: range for the regular case, list for the rest

Owner-sequences carried the whole section; the one per-axis decision left is whether a rank's sequence is a range or an explicit list. They share the sequence interface (in, index, len, slicing), so meet_edge / axis_edges are written once. range just makes the slot lookup O(1) — range(start,…).index(e) == e − start, which is the rank_offset_factor the interval code computes by hand:

Axisowner-sequence representation.index(e)covers
Head, contiguous Layer shardsrange(start, stop) — O(1)e − start (= rank_offset_factor)the regular PP/TP case
Block ids, permuted / strided, GQA-replicatedexplicit listinverse-dict lookupblocks, HND↔NHD permute, replication
Why a sequence — not a set, not an interval. An interval expresses only contiguous shards. A set gives membership but drops order, forcing a separate layout to recover offsets. A sequence keeps order, so one object is both the partition (membership) and the layout (position) — and degrades gracefully to an explicit list for blocks, permutations, and striding that intervals can't express. It's also what the connector already stores: local_block_ids/remote_block_ids are lists zipped positionally, and _local_region_indices_for_layer_names is literally list.index(name) — the interval base/stride is just the O(1) range encoding of those lists.
Litmus test for the abstraction. Adding context/sequence parallel should be: supply that axis's owner-sequences (a range for a contiguous shard, an explicit list otherwise) and register the axis. Zero changes to plan_transfer, meet_edge, or the transport — no new owner/layout method, no Shard | Replicate branch. If a new parallel dimension forces an edit to the planner, the axis abstraction has leaked — the per-axis meet + product structure is the invariant to protect.
