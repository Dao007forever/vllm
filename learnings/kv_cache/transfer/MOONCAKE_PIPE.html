Mooncake Direct Pipe — P Pushes KV to D

The prefill worker invokes Mooncake's batch_transfer_sync_write to write KV cache directly into D's pre-registered GPU memory. FastAPI bootstrap server publishes P workers; D queries, requests, P pushes.

TL;DR

The direct Mooncake connector is the push counterpart to NIXL. D first looks up P workers via an HTTP/JSON bootstrap server hosted on P's global rank 0, then opens a ZMQ socket to a specific P worker and sends a MooncakeXferMetadata containing D's hostname, port, TP topology, and the destination block IDs. P validates, plans, and writes KV directly into D's memory via Mooncake RDMA. The transfer is a one-shot push from P to D; D blocks waiting for the response and learns of completion via that same ZMQ socket.

1. Why push, not pull

There's nothing wrong with push: it's how MPI-style accelerator networks have shipped data for decades. The argument for push over pull is that P knows when its KV is ready. With a pull model, D has to either wait for an explicit "ready" signal from P before issuing the READ, or speculatively READ and risk getting half-written data. Push collapses those into one round-trip: D says "here's where to put it", P writes when ready, P responds when done.

The wire call lives at vllm/distributed/kv_transfer/kv_connector/v1/mooncake/mooncake_connector.py:1365: 

ret_value = self.engine.batch_transfer_sync_write(
    remote_session, src_ptrs, dst_ptrs, lengths
)

remote_session is "{D_hostname}:{D_port}", src_ptrs are P's GPU addresses, dst_ptrs are D's GPU addresses. That's a clean RDMA write.

3. The FastAPI bootstrap server

The direct Mooncake connector splits discovery from data transfer. Discovery is a small FastAPI service that runs on P's global rank 0 (mooncake_utils.py:44-81): 

class MooncakeBootstrapServer:
    """
    A centralized server running on the global rank 0 prefiller worker.
    Prefiller workers register their connection info (IP, port, ranks) here.
    """

    def _register_routes(self):
        self.app.post("/register")(self.register_worker)
        self.app.get("/query", response_model=dict[int, EngineEntry])(self.query)

Each P worker, on startup, POSTs to /register with its own engine ID, DP/TP/PP ranks, and the ZMQ socket it bound for data transfer (mooncake_connector.py:956: await self.register_worker_with_bootstrap()): 

class RegisterWorkerPayload(BaseModel):
    engine_id: EngineId
    dp_rank: int
    tp_rank: int
    pp_rank: int
    addr: WorkerAddr             # e.g., "tcp://10.0.0.5:51234"

D workers GET /query to receive the full registry: dict[dp_rank, EngineEntry] where EngineEntry.worker_addr is dict[tp_rank, dict[pp_rank, addr]]. Given a target (P engine, TP rank, PP rank), D extracts the ZMQ address and goes direct.

Why a registry server, not ZMQ broadcast?

Two reasons. (1) DP — the registry handles many P DP groups sharing a single bootstrap endpoint. (2) D may scale up after P has started; D needs a way to discover the current P population, not just receive notifications about it. A HTTP/JSON registry is the lowest-common-denominator way to do that.

4. The ZMQ per-worker handshake

Each P worker (not just rank 0) opens its own ZMQ ROUTER socket and registers that with the bootstrap server. The data plane handshake is per worker pair: D connects directly to the P worker it wants to talk to.

The sender listener (mooncake_connector.py:942-979) runs in an asyncio event loop on a dedicated thread: 

sock = self.async_zmq_ctx.socket(zmq.ROUTER)
self.side_channel_port = sock.bind_to_random_port(f"tcp://{self.hostname}")
await self.register_worker_with_bootstrap()

sender_tasks = [
    asyncio.create_task(self._sender_worker(sock))
    for _ in range(self.num_sender_tasks)
]

while True:
    identity, metadata_bytes = await sock.recv_multipart()
    await self.sender_worker_queue.put((identity, metadata_bytes))

Multiple sender tasks drain the queue concurrently and call send_kv_to_decode, which is where the rest of the handshake (validation, wait-for-ready, transfer) happens.

The metadata D sends is the MooncakeXferMetadata msgspec struct (mooncake_connector.py:254-264): 

class MooncakeXferMetadata(msgspec.Struct, omit_defaults=True):
    remote_hostname: str       # D's own hostname
    remote_port: int           # D's Mooncake RPC port
    remote_tp_size: int
    remote_tp_rank: int
    req_blocks: dict[ReqId, tuple[TransferId, list[list[int]]]]
    kv_caches_base_addr: list[int]   # D's base addresses (destination)
    block_lens: list[int]            # D's per-region block sizes

The notable line is kv_caches_base_addr: D ships its own memory pointers to P. Combined with the per-request block_ids, P has everything it needs to address D's GPU memory. (This is also why D's memory must already be registered with Mooncake on D's side — see next section.)

5. Memory registration

Both P and D call register_kv_caches (mooncake_connector.py:1388-1448) at worker startup: 

for layer_name, cache_or_caches in kv_caches.items():
    for cache in cache_list:
        base_addr = cache.data_ptr()
        if base_addr in seen_base_addresses:
            continue
        seen_base_addresses.append(base_addr)
        block_len = cache.stride(0) * cache.element_size()
        kv_data_ptrs.append(base_addr)
        kv_data_lens.append(self.num_blocks * block_len)

ret_value = self.engine.batch_register_memory(kv_data_ptrs, kv_data_lens)

Two subtleties worth flagging:

  • Stride-based block length. The connector uses cache.stride(0) * cache.element_size() rather than the shape-based size. That's so RDMA reaches the last block's padding region (relevant for DeepseekV4 MLA alignment). It always matches or exceeds the logical size.
  • One registration covers both directions. A worker doesn't register "send memory" and "receive memory" separately — the same registration is used as src_ptrs when this is P or as dst_ptrs when this is D. kv_both workers therefore have everything they need after a single call.

6. The push transfer

When P receives MooncakeXferMetadata from D, it doesn't transfer immediately. The KV blocks may not be ready yet — D's worker-level handshake can reach P before P's scheduler has finished writing the prefill KV and recorded the local block ids. So P blocks on a per-request ready event rather than transferring on arrival. send_kv_to_decode (mooncake_connector.py:1003-1173) couples two state machines:

"Isn't prefill already done before D is contacted?"

In the reference proxy (examples/disaggregated/disaggregated_serving/disagg_proxy_demo.py:250-278) it is: create_completion fully drains the max_tokens=1 prefill response before it ever forwards to a decode instance, so at the HTTP level P has finished prefill by the time D's request exists, and this ready wait is effectively a no-op. But the connector does not assume that serialization. Two things make the wait necessary in general: (1) "prefill HTTP response returned" is a different event from "P's worker ran request_finished(), knows the block ids, and confirmed the KV writes landed" — the worker-level ZMQ handshake can race ahead of P's scheduler (see the two-path record_send_reqs at mooncake_connector.py:1698-1724, whose empty-block_ids branch handles exactly the case where D's MooncakeXferMetadata arrived first); and (2) a production proxy may dispatch to P and D concurrently to overlap D's block allocation and handshake with P's prefill compute and cut TTFT, in which case D's request genuinely arrives mid-prefill.

  1. For each D-requested transfer id, find the corresponding SendBlockMeta in self.reqs_need_send. That dict is populated by P's scheduler each step with the local block ids for the prompt being prefilled. A SendBlockMeta has an asyncio.Event called ready that the scheduler sets when prefill writes finish.
  2. Await all ready events. As each fires, P builds the transfer parameters (_build_transfer_params), offloads _send_blocks to a ThreadPoolExecutor (CUDA device-binding required, see _bind_sender_thread_device), and pushes the response back to D.

The push itself is offloaded to a thread pool because batch_transfer_sync_write is blocking (mooncake_connector.py:1131-1138): 

ret_value = await self.sender_loop.run_in_executor(
    self._sender_executor,
    self._send_blocks,           # the blocking RDMA call
    remote_session,
    src_ptrs,
    dst_ptrs,
    lengths,
)

Each pool thread is initialized with _bind_sender_thread_device (mooncake_connector.py:1351-1355) which sets the CUDA device per thread — CUDA device selection is thread-local, and without binding, NVLink transfers fail for TP ranks > 0.

7. Completion via ZMQ response

After each batch of ready requests is sent, P pushes a MooncakeXferResponse back over the same ZMQ multipart socket (mooncake_connector.py:1166-1172): 

response = MooncakeXferResponse(
    status=response_status,    # CONTINUE or FINISH
    ok_reqs=[...],             # transferred successfully this batch
    err_reqs=err_reqs or None,
)
await sock.send_multipart((identity, self._encoder.encode(response)))

Status semantics:

  • CONTINUE: P is still waiting on other requests in the D-batch. Expect another response later.
  • FINISH: All originally-requested D requests have been handled (success or error). Stop expecting messages.
  • ERROR: A fatal error happened that affected the whole batch (e.g., D's TP rank doesn't pair with this P).

On D's side, the receiver thread (mooncake_connector.py:1687-1729) loops on its ZMQ socket, decodes each response, and marks request IDs as finished in done_recving.

8. Timeouts & failure modes

P waits too long for ready event

If a prefill stalls and the SendBlockMeta.ready event never fires, the asyncio.wait(...) hits VLLM_MOONCAKE_ABORT_REQUEST_TIMEOUT and P returns a FINISH response with all pending IDs in err_reqs and a "Timeout waiting for P side ready" message (mooncake_connector.py:1068-1085). D treats those as failed and recomputes.

D's TP rank doesn't pair with this P

The TP mapping is consulted: P checks self.transfer_topo.handshake_target_ranks(meta.remote_tp_size). If D's rank isn't in the expected set, P immediately responds with ERROR (mooncake_connector.py:1007-1021). D logs and skips.

RDMA write fails

batch_transfer_sync_write returns non-zero. P records a failed-transfer stat (mooncake_connector.py:1376-1385) and includes the request id in err_reqs in the response. The error case is per-request, so other requests in the same batch can still succeed.

Region length mismatch

Validation happens before any transfer (_validate_asymmetric_region_lengths, mooncake_connector.py:193-244). If the producer's region sizes don't match the consumer's under the TP ratio, P refuses the whole batch with an ERROR response. This catches misconfigured P/D cluster pairs at handshake time rather than corrupting data.

9. How this differs from NIXL

AspectNIXLMooncake pipe
Initiator of wire transfer D (READ) P (WRITE)
Discovery ZMQ ROUTER on a fixed side-channel port FastAPI bootstrap server on rank 0 + per-worker ZMQ sockets
What D ships to P (GET_META_MSG, target_tp_rank) — just queries metadata Full MooncakeXferMetadata including D's memory pointers + block ids
Knows where to write D, by reading from P P, by writing into D's pointers
What handles ready-coordination Lease + heartbeat; P waits for D's notif P waits on asyncio.Event; aborts via timeout
Per-layer hooks used? No (all-at-once) No (all-at-once)
Threading Polling in scheduler loop + ThreadPoolExecutor for handshake asyncio event loop on dedicated thread + ThreadPoolExecutor for the blocking RDMA call
Failure surface Lease expiry (P), check_xfer_state error (D) Timeout (P), error in MooncakeXferResponse (D)
Best-suited for Many small fan-outs to a stable P pool (heartbeat keeps things fresh) Bursty P→D batches where P knows exactly when KV is ready
The "P knows ready" advantage in code

Look at send_kv_to_decode (mooncake_connector.py:1054-1108): P uses asyncio.FIRST_COMPLETED to send each ready request as it becomes available, while continuing to wait on the rest. That granularity is harder to express in a pull model — D would need P to signal each request individually as it becomes ready, then issue READs piecemeal. Push collapses it into "P sends as soon as it can, all in one connection."
