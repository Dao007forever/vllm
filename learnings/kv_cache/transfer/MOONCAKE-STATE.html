Mooncake State Transitions — The CQE-Orphan Wedge

Post-mortem of the 2026-06-10 decode-worker wedge (minimax-m3-vllmdecodeworker-…-5r4qw, dynamo-prod), traced to QP/slice/task state-machine holes in Mooncake. Verified against the deployed fork commit 25e44ac (log line numbers multi_transport.cpp:211, client_service.cpp:1145, worker_pool.cpp:307 all match).

TL;DR

A rolling update (maxUnavailable: 3, maxSurge: 0) killed decode-worker pods mid-transfer — a routine event for a distributed KV cache. But surviving workers wedged for 40+ minutes (KV pinned 85–98%, Running→0, Waiting 17→106) instead of recovering in seconds. The chain: handshake/reconnect logic calls resetConnection() which moves QPs to RESET — silently discarding outstanding WRs (only ERR flushes them as CQEs). The orphaned slices stay POSTED forever. The 5s slice watchdog flags them TIMEOUT, but that status is an advisory observation no layer acts on. The only honored timer is the store’s 60s deadline — so every poisoned load burns 60s in a single serial recv thread (drain 1/min vs arrival 3/min), deferred requests pin KV blocks, and the scheduler starves. Not a fabric fault; not a vLLM fault; an ownership hole in the QP-teardown path.

2. QP state machine: two disposal semantics

An RC queue pair is hardware state on each NIC with no liveness protocol between peers — no FIN/RST equivalent. When a peer pod dies, the surviving QP receives no signal; an idle QP to a corpse looks healthy forever. Failure is only inferred by sending work and timing out. That makes the state-transition choices below load-bearing: they decide whether in-flight work is returned with proof or silently destroyed.

ibv_qp state transitions as Mooncake uses them
The two disposal semantics: ERR flushes every outstanding WR to the CQ as an error completion (the contract that returns buffer ownership); RESET discards them with no completion. disconnectUnlocked() picks RESET. The later EINVAL from beginDestroy() is a tombstone, not the cause — by then the WRs are already gone.
The code, condensed rdma_endpoint.cpp — disconnectUnlocked() 
attr.qp_state = IBV_QPS_RESET;                // RTS → RESET, directly
ibv_modify_qp(qp_list_[i], &attr, IBV_QP_STATE);
// After resetting QP, the wr_depth_list_ won't change
if (wr_depth_list_[i] != 0) {
    LOG(WARNING) << "Outstanding work requests found, "
                    "CQ will not be generated";   // ← it KNOWS
    __sync_fetch_and_sub(cq_outstanding_, wr_depth_list_[i]);
    wr_depth_list_[i] = 0;        // fixes the COUNTERS, abandons the SLICES
}
The endpoint repairs its own books and moves on. It cannot repair the slices even if it wanted to: at this layer there is no list of posted slices — only a count. The Slice* pointers live solely in the hardware send queue’s wr_ids, which the RESET just erased.
Correction (verified post-publication): the build never runs this path via resetConnection

Upstream PR #323 (May 2025, a cmake dedup refactor) hoisted add_definitions(-DCONFIG_ERDMA) — an Alibaba-eRDMA-specific define — unconditionally into mooncake-common/common.cmake:47. No option() guard exists; verified in compile_commands.json and behaviorally (a passive re-establish produced 0 ERR / 11 RESET transitions). So in every build of this tree, including the production binary, resetConnection() takes the #ifdef CONFIG_ERDMA branch: reconstruct() = deconstructLocked() (ibv_destroy_qp with WRs outstanding) + fresh construct().

The disposal semantics are unchanged — destroying a QP discards outstanding WRs without CQEs, exactly like RESET — but the orphan instruction is QP destroy in deconstructLocked, whose “Outstanding work requests found, CQ will not be generated” warnings (9× on the wedged pod) are the orphaning moments. The RESET→ERR EINVAL tombstones come from beginDestroy hitting the freshly created QPs, which are born in RESET. disconnectUnlocked still runs only via its 4 direct call sites (handshake validation failures). Fix implications: synthesizing FAILED for tracked slices after the destroy (quiesce-then-synthesize) is the primary, build-flag-independent fix; an ERR-not-RESET patch to disconnectUnlocked alone is dead code for the dominant paths; and CONFIG_ERDMA should be an opt-in cmake option (upstream issue material). Reconstruct-per-reset also mints new qp_nums every reset, forcing full re-handshakes during exactly the churn windows that trigger resets.

4. Task status enum: facts vs observations

TransferStatusEnum mixes two kinds of values in one type: facts about buffer ownership (COMPLETED, FAILED: hardware provably done, buffers returned) and advisory observations (WAITING, TIMEOUT: how long things are taking). Each consumer then guessed differently:

Metrics code — semantically loose

transfer_engine_impl.h:254-258 counts TIMEOUT (and CANCELED — which the RDMA transport never produces; only the nvmeof transport does) as terminal. Convenient for accounting; wrong about ownership.

vs
Store check_task_status — correct, but inert

transfer_task.cpp:725-742 routes TIMEOUT to default: = “still in progress”. This is the defensible behavior: it protects freeBatchID and buffer ownership. But since nothing acts on the observation, it decays into the 60s purgatory.

The actual bug TIMEOUT is an observation that demands an action — probe the peer, cancel via QP→ERR, flush — and no layer ever acts on it. There is no cancel API on the engine; nothing wires the watchdog to beginDestroy or probePeerAliveByID (which exists, public, unused in this flow). The observation is computed at 5s and discarded.

5. The timeout ladder (why exactly 60s)

Four timers notice the problem before 60s; the 60s one is the first that both notices and is allowed to act:

Why you can’t just shrink the 60s The deadline-return is ownership abandonment: it sets TRANSFER_FAIL while WRs may still be outstanding, then the destructor frees the batch. It is accidentally safe only because of an unwritten ordering: 5s observe < 30s force-destroy (HW quiesced) < 60s abandon. By 60s every involved QP has completed, flushed, or been destroyed — no late DMA is possible. Cut the deadline below ~35–40s and a QP that failed its ERR transition can still be in RTS retrying, with a late RDMA-READ response able to DMA into memory vLLM already reused: you trade a visible wedge for rare, unattributable KV corruption. And the safe end (40s → drain 1.5/min) still loses to the ~2.4–3/min arrival rate. The helpful range isn’t safe; the safe range doesn’t help.

Side effect of the gap: wait_for_completion’s polling path (build has USE_EVENT_DRIVEN_COMPLETION=OFF) is a while(true) busy-spin with no sleep, re-logging “Slice timeout detected” per stuck task per iteration — 4–16M log lines/min (×4 TP ranks), one core burned per stuck batch. The 26.7M-line log hour was this loop, not traffic.

All nine call sites live inside the endpoint’s own handshake machinery (setupConnectionsByActive / setupConnectionsByPassive). Every one of them fires during a peer rollout:

7. The ownership inversion

  1. resetConnection has one legitimate job: return QPs to a state from which a fresh INIT→RTR→RTS can be built. IB requires passing through RESET for that — the connection state machine part is correct.
  2. But a QP is two things at once: connection state, and a container of posted work. Each WR’s wr_id is a raw Slice* borrowed from a task, which belongs to a batch, which a TransferFuture waits on, which a Python recv thread is blocked inside. The only protocol for returning the borrowed slice is a CQE.
  3. RESET burns the loan record without returning the collateral. The code fixes the only thing visible at its layer (the counters) and logs a warning. Upstream, every waiter is left waiting for a completion that can never come, until the 60s deadline expires the wait.
  4. Blast-radius mismatch: the decision to reset is control-plane (stuck handshake, failed RPC, restarted peer); the cost lands on the data plane (whatever transfers were in flight). During a peer death these are not independent: the same condition that makes handshakes fail (dead peer) guarantees in-flight WRs exist (the peer died mid-service). The orphaning is the common case during a rollout, not a rare race.
  5. Fairness note: discarding those WRs is defensible in intent — any WR outstanding at reconnect time targets a connection that no longer exists; it’s doomed regardless. The bug is choosing silent discard (RESET) over flush-with-proof (ERR) for disposing of doomed work, in the codepath that runs precisely when the layers above have the most work in flight.

8. Queueing collapse arithmetic

The rates

  • Arrival: ~2.4–3 req/min (frontend has no readiness signal to stop routing; measured from Deferred slope 1→13 in 4 min and Waiting slope 26→106 in 34 min)
  • Drain: 1 poisoned req/min — each burns the full 60s, and KVCacheStoreRecvingThread is one thread, serial queue (worker.py:388-399: while True: queue.get(); _handle_request()). No pool, no executor.
  • Cost per stuck request: long-context loads hold ~4–15% of KV each (sample request: 5,752 keys ≈ 39GB) the entire time they wait.

Where the parallelism is (and isn’t)

  • Within one batch call: all keys submitted concurrently, futures waited with overlapping deadlines → 60s per call, not per key ✅
  • Across TP ranks: 4 recv threads, but loading shards of the same requests in lockstep — no request-level speedup (just ×4 log flood)
  • Across requests: none. load_async decouples scheduler↔loader, not load↔load. Async ≠ concurrent.

Slow drain → more deferred → more pinned KV → slower drain. Compounded by: failed loads convert to recomputes that re-enter the starved queue; kv_both means recomputed KV then queues a put through the equally serial, equally poisoned send thread; and the rollout re-poisons every ~10 min.
Counterfactuals With the 5s fail-fast honored: drain ~12/min » arrival → each kill wave is a 5–10s blip, Deferred never exceeds 1–2, KV never pins. With only a parallel recv pool (N≥4): steady-state pinned KV ≈ 3/min × 1min × ~5% ≈ 15% — survivable, but every affected request still eats a 60s TTFT penalty. The 40-minute wedge is the gap between 5s and 60s passed through two amplifiers: a serial queue, then KV-block exhaustion.

Rollout strategy maxSurge: 0 (GPUs can’t surge), maxUnavailable: 3 Old pods torn down before replacements exist; each batch kill re-poisons the survivors; fresh pods come up into the storm and re-wedge (t8nhp: 154M log lines, served ~1 min)

Recovery only ever happened when a pod process restarted into a calm window between rollout batches. The “fabric flapping” through 08:50 was the rollout itself progressing.

10. Fixes — safe, rejected, and why

Rejected

CandidateWhy rejected
One-line: add case TIMEOUT: to the failed cases in check_task_status() Unsafe. RDMA has no per-WR cancel; a TIMEOUT slice may still be owned by the HCA, and a late RDMA-READ response can still DMA into the destination buffer. Declaring it terminal lets freeBatchID free bookkeeping under live hardware: use-after-free on the slice (late CQE carries a dangling wr_id) + silent corruption of reused buffers. The store’s conservatism is the correct behavior.
Reduce the 60s deadline Same abandonment, more often. Safe only above the 30s force-destroy quiesce point (§5); the safe range (≥~40s → 1.5/min) still drains slower than arrival. Wrong knob.
Lower the HW retry budget (RETRY_CNT=7 → 1) ~0.5s detection was never the problem (Fate A worked); a budget of 1 converts transient RoCE congestion into hard failures and spurious recomputes.

The fix stack (in leverage order)

  1. Flush, don’t discard. Change disconnectUnlocked()’s target state from RESET to ERR and delete the counter-zeroing block. Flush CQEs arrive in microseconds; the worker pool’s normal completion path fails each slice; counters decrement organically; tasks terminate in ms instead of 60s. Natural because status_ = UNCONNECTED already blocks new posts, and the next doSetupConnection already starts with “any state → RESET” (ERR→RESET is valid) — deferring the RESET to reconnect time costs nothing.
  2. Act on the observation. Wire the 5s watchdog to the QP owner: slice timeout → probePeerAliveByID → if unresponsive, beginDestroy() the endpoints to that peer → flush → organic FAILED in ~5–7s with ownership provably returned. (The collateral — co-resident batches on those QPs flush too — is correct: if the peer is dead, every transfer on the endpoint is doomed.)
  3. Synthesize completions where quiescence is guaranteed. Track posted slices per QP (append on post, remove on CQE — today only a count exists). After modify(RESET) or ibv_destroy_qp succeeds, the HCA is quiesced — then and only then, walk the tracked orphans and mark them FAILED in software. Closes the hole no matter which path strands them.
  4. State-aware beginDestroy. On modify(ERR) EINVAL, ibv_query_qp: already-ERR (HW auto-transitioned) → harmless, flushes coming; RESET → invoke fix-3 synthesis immediately instead of the 30s abandon.
  5. Handshake deadline. Non-blocking connect() + poll with a few-second timeout in SocketHandShakePlugin; fail pre-post slices (ts==0, no HW state — software-cancel is safe here) on the same deadline.
  6. Bounded recv concurrency in the vLLM connector so one poisoned load can’t head-of-line block the queue; plus sleep/backoff (or USE_EVENT_DRIVEN_COMPLETION=ON) for the poll spin.
  7. Defense in depth (k8s): KV-transfer-aware readiness probe (auto-restarts wedged pods and halts a rollout marching into a broken state); gate worker rollouts on the previous batch actually serving.
