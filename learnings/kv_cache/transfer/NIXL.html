KV Cache · Transfer · NIXL

NIXL — D Pulls KV From P

The decode worker initiates a NIXL READ directly against the prefill worker's registered KV memory. ZMQ-based side-channel handshake, lazy per-engine agent registration, lease-with-heartbeat so P knows when to free.

TL;DR

NIXL is the pull-model transport. D's worker registers all local KV memory with NIXL eagerly at startup, then on each new remote engine it (1) ZMQ-handshakes to fetch P's agent metadata, (2) registers P as a remote NIXL agent, and (3) issues a single async READ xfer that copies P's KV blocks directly into D's blocks. P doesn't push — P just keeps the blocks alive on a 30-second lease that D's heartbeat refreshes. When D's transfer completes, NIXL delivers a notification to P which lets P free the lease early.

1. Why pull, not push

NIXL was designed to look like RDMA READ: the initiator knows both ends of the copy and just performs it. That's a tidy fit for P/D disaggregation because D is the one being unblocked — D wants the KV, D knows where to put it, so D is the natural caller.

Concretely, the only entrypoint into the wire transfer is in vllm/distributed/kv_transfer/kv_connector/v1/nixl/worker.py:2220: 

handle = self.nixl_wrapper.make_prepped_xfer(
    "READ",
    local_xfer_side_handle,        # D's destination descriptors
    local_block_descs_ids,
    remote_xfer_side_handle,       # P's source descriptors
    remote_block_descs_ids,
    notif_msg=notif_id,            # piggy-back notif to P on completion
)
self.nixl_wrapper.transfer(handle)

The "READ" string is the entire commitment: P never calls transfer. P just holds memory and answers ZMQ handshakes.

2. The wire-level flow

NIXL end-to-end — sequence
intra-engine call control plane (ZMQ side channel) data plane (NIXL RDMA) completion notif
Setup is symmetric and eager (both workers pin all KV with NIXL; P also binds its ZMQ listener). Per request, D drives everything: a one-time control-plane handshake (②③) bootstraps the remote agent, then a single data-plane READ (④) pulls the blocks. P never initiates — it only answers ② and reacts to ⑤.
  1. Request lands on D. D's scheduler hands the worker a request carrying kv_transfer_params — remote_engine_id, remote_host/port, remote_request_id, and remote_block_ids. This is the entry to start_load_kv (nixl/worker.py:1956-1968).
  2. Handshake REQ (first contact only). If the engine_id is new, D opens a ZMQ REQ socket to P's listener and sends (GET_META_MSG, tp_rank) for each P rank it needs. Already-seen engines skip straight to step 4 (nixl/worker.py:720-742).
  3. Metadata reply. P's ROUTER answers with the encoded NixlAgentMetadata for that rank — base addresses, block lengths, KV layout, and the compatibility hash D validates before trusting the layout (nixl/scheduler.py:303-321). D then calls add_remote_agent + prep_xfer_dlist and caches the result per (engine_id, rank) (nixl/worker.py:1309-1312).
  4. The READ. _read_blocks_for_req issues one async make_prepped_xfer("READ", …) per source rank. Data moves P→D over RDMA in a single all-at-once transfer; the call returns immediately (nixl/worker.py:2021-2095, 2220).
  5. Completion notif. The READ piggy-backs notif_msg="{req_id}:{D_world_size}", which NIXL delivers to P once the transfer lands (nixl/worker.py:1792-1845).
  6. Both sides settle. D polls check_xfer_state; once every handle is DONE the request enters done_recving (nixl/worker.py:1868-1913). P counts notifs and, once consumers_per_producer is reached, drops the request from _reqs_to_send and releases the lease.

3. The ZMQ side-channel handshake

The handshake exists out-of-band on a ZMQ socket per P engine. Its job: tell D enough about P that NIXL can READ from P.

3.1 The listener (P side)

Each P engine starts a daemon thread that binds a ZMQ ROUTER on tcp://VLLM_NIXL_SIDE_CHANNEL_HOST:VLLM_NIXL_SIDE_CHANNEL_PORT (nixl/scheduler.py:271-286): 

self._nixl_handshake_listener_t = threading.Thread(
    target=self._nixl_handshake_listener,
    args=(encoded_data, ready_event, self._stop_event,
          self.side_channel_host, self.side_channel_port),
    daemon=True,
)
self._nixl_handshake_listener_t.start()

The listener answers (GET_META_MSG, target_tp_rank) requests with the encoded NixlHandshakePayload corresponding to that rank (nixl/scheduler.py:303-321). One rank per handshake response — D asks repeatedly for each P rank it needs.

3.2 The initiator (D side)

D doesn't handshake at startup; it handshakes per remote engine, on first contact. When start_load_kv sees a request referring to an engine_id it hasn't seen before (nixl/worker.py:1956-1968): 

if remote_engine_id not in self._remote_agents:
    with self._handshake_lock:
        if remote_engine_id not in self._remote_agents:
            self._background_nixl_handshake(req_id, remote_engine_id, meta)
            continue
self._read_blocks_for_req(req_id, meta)

The background handshake is just a job on a ThreadPoolExecutor (nixl/worker.py:720-742). It opens a ZMQ REQ socket to P's listener, queries every P rank it needs based on the TP mapping, validates the compatibility_hash, and adds each P rank as a NIXL remote agent. Once done, queued requests are released to _read_blocks_for_req.

3.3 What's exchanged

The NixlAgentMetadata dataclass (nixl/metadata.py:42-54) is the contract: 

@dataclass
class NixlAgentMetadata:
    engine_id: str
    agent_metadata: bytes               # opaque NIXL agent state
    kv_caches_base_addr: list[int]      # P's GPU memory base pointers
    device_id: int
    num_blocks: int
    block_lens: list[int]
    kv_cache_layout: str                # "HND" or "NHD"
    block_size: int
    ssm_sizes: tuple[int, int]
    attn_backend_name: str
    physical_blocks_per_logical_kv_block: int

Two things to notice: (a) the metadata carries enough to validate layout compatibility — backend name, block layout, KV layout — so D fails fast on mismatch instead of silently corrupting; (b) the actual agent_metadata: bytes blob is opaque to vLLM and just passed verbatim to nixl_wrapper.add_remote_agent().

Example: Llama 3.1 8B, FlashAttention, TP = 1

A prefill worker serving Llama-3.1-8B (32 layers, 8 KV heads × 128 head-dim, bf16, block_size = 16) answers a GET_META_MSG with roughly: 

NixlAgentMetadata(
    engine_id   = "prefill-3f9c…",        # this P engine's id
    agent_metadata = b"\x00\x12…",        # opaque UCX agent state (~few KB)
    kv_caches_base_addr = [0x7f841a…, …],  # 64 ptrs = 32 layers × (K, V)
    device_id   = 0,
    num_blocks  = 28000,                  # whatever fits free HBM (illustrative)
    block_lens  = [32768, 32768, …],     # 64 × 32 KiB = 16 tok × 1024 × 2 B
    kv_cache_layout = "HND",             # NIXL PD forces HND (faster head-split)
    block_size  = 16,
    ssm_sizes   = (0, 0),                 # no Mamba
    attn_backend_name = "FLASH_ATTN",
    physical_blocks_per_logical_kv_block = 1,  # kernel block == logical
)

Where each derived number comes from:

  • 64 base addrs & block_lens = 32 layers × 2, because FlashAttention keeps K and V in separate regions (4.2). Every entry is equal — block lengths vary across layers only for MLA.
  • 32768 = 16 (tokens) × (8 KV heads × 128) × 2 (bf16 bytes). At TP = 2 each rank would instead publish 16 KiB regions (4 KV heads/rank) and a different kv_caches_base_addr — metadata is per worker (4.3).
  • ssm_sizes = (0, 0) and physical_blocks_per_logical = 1: pure attention with kernel block == logical block, so the SSM descriptor section is empty and FA strides over num_blocks (4.3, 4.5).
  • kv_cache_layout = "HND": NIXL overrides the layout to HND for P/D so heads are the outer axis and a TP rank's head-slice is contiguous.

D checks this against its own config: same attn_backend_name, kv_cache_layout, and block_size, and each block_lens[i] scaled by tp_ratio (the gate in 4.7's handshake). A mismatch — say D runs FlashInfer, or a different block size — is a RuntimeError, not silent corruption (9).

Why a side channel?

NIXL itself has a notification primitive (send_notif / get_new_notifs), but you can't send a notif to an agent you haven't registered — and to register an agent you need its metadata. So bootstrap has to ride on something else. ZMQ is the cheapest "control plane that's already there" option.

4. Memory registration: eager local, lazy remote

Two separate registrations, and they happen at very different times.

4.1 Local (eager, at worker startup)

Every worker, on both P and D, registers all its KV cache memory with NIXL once at startup (nixl/worker.py:960-964): 

descs = self.nixl_wrapper.get_reg_descs(caches_data, self.nixl_memory_type)
self.nixl_wrapper.register_memory(descs, backends=self.nixl_backends)
self._registered_descs.append(descs)

This is what makes NIXL fast: by transfer time, the GPU memory is already pinned and known to the NIXL backends. Registration is a one-time cost; per-request transfer is just descriptor lookup + DMA. The same memory is used for sending (when this rank is P) and receiving (when this rank is D) — the registration is symmetric.

4.2 What's a "region"? K/V split, per layer, per model

That one register_memory call hides structure. The caches_data list it registers is a sequence of regions — each region is one (base_addr, size, device_id) tuple, i.e. one contiguous KV tensor. The loop in register_kv_caches (nixl/worker.py:847-936) walks the layers and appends a region per distinct base address; the count lands in num_regions = len(caches_data) and the per-region stride in block_len_per_layer (nixl/worker.py:943-944).

How many regions a layer contributes is decided by the attention architecture, not the model brand. The code comment is explicit (nixl/worker.py:834-841): 

# K and V are now in different regions. Advantage is that we can
# elegantly support MLA and any cases where the K and V tensors
# are non-contiguous ... Disadvantage is that the encoded
# NixlAgentMetadata is now larger (roughly 8KB vs 5KB).
  • Standard attention (MHA / GQA, FlashAttention) → kv-first layout [2, num_blocks, H, N, D] (the 2 is the outer axis), so K and V are two distinct tensors ⇒ 2 base-address regions per layer.
  • MLA → a single compressed latent cache per layer ⇒ 1 region per layer (no K/V to split).
  • FlashInfer → blocks-first layout [num_blocks, 2, H, N, D], so K and V are interleaved inside one tensor ⇒ 1 base-address region, then num_regions *= 2 to index K/V virtually (nixl/worker.py:946-955). The chooser is split_k_and_v (kv_connector/utils.py:496-501); the joint case is detailed at the end of this section.

At block_size = 16 tokens in bf16, each Llama region is a 16 × 1024 × 2 = 32 KiB block stride (one entry in block_len_per_layer for K, one for V), while each MLA region is 16 × 576 × 2 = 18 KiB. Two consequences flow from the region count:

  • Metadata + descriptor size. The exchanged NixlAgentMetadata.kv_caches_base_addr / block_lens lists carry one entry per region, and the total descriptor space is num_regions × num_blocks. Llama's K/V split doubles both versus an MLA model of the same depth — the "~8KB vs ~5KB" the comment warns about.
  • It's still one handle. No matter how many regions, there is one register_memory and one prepped xfer handle per remote rank. A single READ addresses every layer’s K and V via descriptor ids (region_id × num_blocks + block_id, nixl/worker.py:113-115) — which is exactly why the transfer in section 6 is all-at-once with no per-layer batching.

Joint-KV (FlashInfer): how K and V share one region

The blocks-first case is the interesting one: only one base address per layer, yet K and V still need independent addressing. They're separated by byte offset inside each block's page — K is the first half, V the second — and exposed as two virtual regions so the rest of the connector's math is untouched.

One block's page (joint KV) → two virtual regions
block
page K halfoffset 0 · block_len / 2 V halfoffset block_len/2 · block_len / 2
get_backend_aware_kv_block_len returns block_len // 2 per stream (nixl/worker.py:2419-2423); _build_fa_local emits the K descriptor at addr and the V descriptor at addr + block_len/2 (nixl/worker.py:1116-1132).

Descriptors are emitted K-then-V per layer, so with num_regions doubled the region_id × num_blocks + block formula resolves cleanly:

Virtual regionMeaningDescriptor-id band
2i layer i — K (first half) [2i · num_blocks, (2i+1) · num_blocks)
2i + 1 layer i — V (second half) [(2i+1) · num_blocks, (2i+2) · num_blocks)

So "which half of the page" reduces to "which virtual-region band the descriptor id lands in" — the same arithmetic separate-region backends use, which is the whole point: a blocks-first FlashInfer layout looks identical to FA's two regions everywhere downstream. It also composes with HND (4.6): within each half a TP rank's heads are contiguous, so a head-split read takes the K half at rank_offset × per_head_len and the V half at block_len/2 + rank_offset × per_head_len — two clean descriptors a single whole-page copy could never carve.

Not for cross-layer blocks.

The half-split is disabled when cross-layer blocks are used, because those interleave per layer (L0_K, L0_V, L1_K, L1_V, …), so a single block_len/2 cut wouldn't separate K from V (kv_connector/utils.py:488-494).

Column-packed (HMA) hybrid models

Hybrid models that interleave attention types — Gemma-3's 1 full : 5 sliding-window pattern, say — share tensors a third way. With G groups of group_size layers each, the allocator lays them out column-packed: it allocates group_size "column" tensors, and each column is shared by one layer from every group (the layers at the same slot within their group). The sharing axis is across groups, kept collision-free by equal group sizes and a global block-ID pool (any block id is owned by at most one group at a time).

Column-packed: 2 groups × 2 slots → 2 column tensors
Group 0 · FULL L0 L2
Group 1 · SWA L1 L3
→ NIXL region Tensor 0L0, L1 Tensor 1L2, L3
Each column is one physical allocation shared by one layer per group. NIXL registers each column once — 2 regions for 4 layers — and reads it with both groups' (disjoint) block ids flattened together.

NIXL calls this HMA (hybrid memory allocator): _is_hma_required trips when the hybrid manager is on and some group isn't full-attention (nixl/worker.py:225-231), and hma_group_size = len(kv_cache_tensors) is the column count (nixl/worker.py:238). Two things follow:

  • One region per column, not per layer. The registration loop walks every layer, but layers aliasing an already-seen column tensor share a data_ptr() and are skipped (nixl/worker.py:888-896): "HMA employs memory pooling to share tensors across groups … skipping all tensors but the ones pointed to by group0 … fewer regions." So the region count collapses to group_size — a Gemma-3 with group_size = 10 registers ~10 column regions, not 60 layer regions.
  • Per-group block ids are flattened at transfer. A request's blocks arrive per group (e.g. [[10,11], [20,21]]). Because the global pool hands out disjoint ids, inside one column group 0's layer sits at slots 10–11 and group 1's at 20–21 — no overlap — so _compute_desc_ids flattens across groups and runs region_id × num_blocks + block over every column at once (nixl/worker.py:106-115). One READ per column, addressed by the union of all groups' block ids, reconstructs every layer's KV.
Column-packing (HMA) ≠ cross-layer blocks.

Two tensor-sharing schemes that are easy to conflate: HMA / column-packed is multi-group — group_size separate column tensors, each shared by one layer per group (this section); the registration dedups to group_size regions and block ids are flattened across groups. Cross-layer blocks (_cross_layers_blocks, behind use_uniform_kv_cache) is single-group — one tensor with an extra num_layers axis, so the page becomes page_size × num_layers and a single block carries every layer at once (nixl/worker.py:870-874). It requires exactly one uniform group, so it does not apply to hybrid models like Gemma-3. Both share tensors, but along different axes and for different model shapes.

Cross-layer blocks (use_uniform_kv_cache)

The other tensor-sharing scheme is an opt-in NIXL optimization for single-group (uniform-attention) models — all-full Llama, say. Instead of one region per layer, all the group's layers are packed into one tensor with an extra num_layers axis, so a single block carries every layer's KV. It's gated tightly — prefer_cross_layer_blocks requires no Mamba, an FA/FlashInfer/Triton backend, HND layout, and the explicit enable_cross_layers_blocks=true flag (nixl/connector.py:59-86) — then use_uniform_kv_cache additionally demands a single uniform group and a backend whose layers axis isn't outermost (kv_connector_model_runner_mixin.py:115-183).

allocate_uniform_kv_caches allocates one contiguous buffer of tensor_size × num_layers, prepends a num_layers dim, and permutes by the backend's stride order (kv_connector_model_runner_mixin.py:209-276). For FlashAttention under HND that order is (1, 4, 0, 2, 3, 5) (flash_attn.py:163-165), giving physical memory: 

(num_blocks, num_kv_heads, num_layers, 2, block_size, head_size)

The num_layers axis sits inner (after blocks and heads), so for a given (block, head) all layers are contiguous — that's the "column-packing inside one tensor." Putting num_kv_heads ahead of num_layers keeps a TP rank's head-slice contiguous, so head-split (4.6) still works. The stride_order[0] != 0 gate just enforces that layers aren't outermost — if they were, each layer would be a separate slab and there'd be no benefit.

One block (one head shown) in a cross-layer tensor
1 block
1 region L0 KL0 V L1 KL1 V L2 KL2 V … Ln KLn V
The page is page_size × num_layers (nixl/worker.py:870-874), so one descriptor per block moves every layer's KV at once.

NIXL registers it via register_cross_layers_kv_caches (nixl/worker.py:777-786): one base address ⇒ ~1 region with num_blocks descriptors, each moving all layers. (virtually_split is off here — the per-layer L0_K, L0_V, L1_K, … interleave means a block_len/2 cut wouldn't isolate K from V.) This is the all-at-once transfer of section 6 taken to its extreme: fewest, largest descriptors, smallest NixlAgentMetadata — at the cost of an opt-in flag and the single-uniform-group requirement.

Cross-layer: one buffer → per-layer views

The allocation is a single buffer plus a chain of zero-copy views — no per-layer tensors are ever allocated (kv_connector_model_runner_mixin.py:257-274):

1 · ALLOCATE torch.zeros(total_size, int8) total = tensor_size × num_layers
→
2 · REINTERPRET .view(dtype).view(phys_shape) = cross_layers_kv_cache (what NIXL registers)
→
3 · UN-PERMUTE .permute(inv_order) = permuted_kv_cache, logical order, strided
→
4 · SLICE permuted[i] per layer → kv_caches[name]

Step 1 grabs total_size = tensor_size × num_layers bytes in one shot. Step 2 reinterprets the bytes as the KV dtype and views them in physical shape — (num_blocks, num_kv_heads, num_layers, 2, block_size, head_size) for FA+HND. That object, cross_layers_kv_cache, is the single region NIXL registers. Step 3 applies the inverse stride permutation to get permuted_kv_cache back in logical order (num_layers, num_blocks, 2, block_size, num_kv_heads, head_size) — a strided view over the same bytes, no copy. Step 4 indexes permuted_kv_cache[i] to drop the layer axis, yielding the ordinary per-layer KV tensor (num_blocks, 2, block_size, num_kv_heads, head_size) that the attention kernel expects; every layer name in that tensor's shared_by list maps to this same view.

The result is two aliases of one allocation, read along different axes:

One buffer, two readings (num_layers = 3, num_blocks = 4)
block 0 L0L1L2
block 1 L0L1L2
block 2 L0L1L2
block 3 L0L1L2
kv_caches[L0] = the blue column kv_caches[L1] kv_caches[L2]
Memory runs left-to-right, top-to-bottom (num_blocks outermost). A per-layer view reads down a column — kv_caches[L0] is every blue cell, a strided view with block-stride = 3 cells. A NIXL block descriptor reads across a row — block b's whole [L0 L1 L2] is contiguous (page_size×num_layers), so one descriptor moves all 3 layers.

The two axes never fight: the attention kernel sees only its per-layer view (PyTorch strides route each slot_mapping write to the right interleaved offset — the packing is invisible to it), while NIXL sees only the flat buffer where each block is one contiguous all-layers page. Same bytes, two stride patterns — which is exactly why the num_layers-not-outermost rule (above) matters: it's what makes the row contiguous and the column merely strided, rather than the reverse.

4.3 Two descriptor families: full-attention vs SSM

That flat descriptor formula — region_id × num_blocks + block_id — quietly assumes every region has the same stride (num_blocks). For pure-attention models that holds. But vLLM KV caches come in exactly two physical layouts, and they stride differently — the one thing a single flat index space cannot paper over. So _compute_desc_ids has two branches (nixl/worker.py:89-150), gated by a spec classifier that recognizes only these two and raises on anything else (nixl/tp_mapping.py:31-36): 

def _is_attention_spec(t):  return issubclass(t, AttentionSpec)  # Full / MLA / SWA / indexer
def _is_ssm_spec(t):        return issubclass(t, MambaSpec)      # conv + ssm state
# else: raise ValueError(f"Unknown spec type {t}")   # worker.py:146

Why the families must be kept apart:

Attention (FA) — block-indexed

  • AttentionSpec: Full, MLA, sliding-window, the M3 lightning-indexer — all attention.
  • One region per layer (K+V interleaved), virtually split into K and V streams — see 4.2.
  • Desc stride = num_blocks (physical / kernel granularity).
  • Head-shardable across TP — a rank can read a head-slice of a block.

SSM (Mamba) — state-indexed

  • MambaSpec: a convolution state + an SSM temporal state, one set per layer.
  • 4 sub-regions per layer (num_ssm_regions = len(block_len_per_layer) × 4) for the 3-read DS-conv transfer (nixl/worker.py:98, ssm_conv_transfer_utils.py).
  • Desc stride = logical_blocks = num_blocks // physical_blocks_per_logical — no kernel splitting; state is one entry per logical block.
  • Not head-split: the state is whole-per-layer, read intact.

A hybrid model (e.g. a Mamba/attention stack, scheduled by the hybrid memory allocator) carries both. The descriptor id space is then laid out as two concatenated sections — all FA descriptors first, then all SSM descriptors — with the SSM half shifted past the FA half:

Flat descriptor-id space (HMA hybrid)
FA section — [0, num_fa_descs)stride num_blocks · head-shardable SSM section — [num_fa_descs, …)stride logical_blocks · whole-state
num_fa_descs = num_fa_regions × num_blocks is the boundary — and it is recomputed per engine, because P and D can have different num_blocks and therefore different FA desc counts (nixl/worker.py:103, 134-136). 
# FA descriptors (per group):
desc_id = fa_region_id  * num_blocks     + block_id
# SSM descriptors (per group), shifted past the FA section:
desc_id = ssm_region_id * logical_blocks + block_id + num_fa_descs

When there is no Mamba, num_ssm_regions == 0 and the function takes an all-attention fast path — a single vectorized broadcast region_ids * num_blocks + block_arr over every region at once (nixl/worker.py:106-115). Llama, DeepSeek-V3, and Kimi K2 are all pure-attention, so they take that fast path and the SSM section is empty — the FA-vs-SSM split only ever materializes for hybrid models.

SPLIT vs REPLICATE lives inside the FA section.

A subtler distinction (used by mixed models like MiniMax-M3, which keep full-attention and an MLA lightning-indexer in one KV group) is whether an FA region is head-split across TP (GQA full-attention) or replicated identically on every rank (the key-only, num_kv_heads=1 indexer). Both are AttentionSpec, both sit in the FA section with the same num_blocks stride — so the descriptor math is unchanged; only the per-region transfer offset/length differs. That's a TP-mapping concern, not a descriptor-family one — which is exactly why FA-vs-SSM is the cut that _compute_desc_ids cares about, and head-split-vs- replicate is not.

Where a replicated FA region comes from: GQA with tp_size > num_kv_heads.

Replication isn't unique to the M3 indexer — any GQA model run with more TP ranks than KV heads produces it. QKVParallelLinear decides per layer (linear.py:1026-1031): 

if tp_size >= total_num_kv_heads:
    num_kv_heads = 1
    num_kv_head_replicas = divide(tp_size, total_num_kv_heads)   # ranks sharing a head
else:
    num_kv_heads = divide(total_num_kv_heads, tp_size)
    num_kv_head_replicas = 1

When tp_size > num_kv_heads each rank gets a single KV head and num_kv_head_replicas ranks hold a byte-identical copy of it — that duplicated KV is what an FA region marks REPLICATE; the indexer (num_kv_heads=1) is just the extreme where every rank replicates. It is legal only when the counts divide (divide() asserts): num_kv_heads | tp_size or tp_size | num_kv_heads. So for 8 KV heads, TP 4/8/16 work (TP 16 ⇒ 2× replicas); TP 6 or 12 raise. Past tp = num_kv_heads the KV cache stops shrinking — the extra ranks store redundant copies, which is the memory cost REPLICATE reflects.

4.4 Inside an SSM layer: the 4 sub-regions

An SSM layer caches no K and V. Its per-block page holds two tensors: a short convolution state (the causal-conv window, conv_kernel−1 rows — a kernel of width K needs the current input plus the K−1 previous ones, and the current input is computed live each step, so only those K−1 priors are cached) and an SSM temporal state (the recurrent hidden state). For NIXL transfer the connector carves each page into 4 sub-regions — the docstring of _build_mamba_local is literally "Build 4 desc regions (x, B, C, ssm) per layer" (nixl/worker.py:1014-1052).

The trick is the DS (dim-first) layout (conv_dim, conv_rows), which lays the conv state's three sub-projections out contiguously (ssm_conv_transfer_utils.py):

Read the shape as columns × window: conv_dim is the channel axis (the depthwise conv runs per channel — the code calls these positions "columns") and conv_rows = conv_kernel−1 is the window length. The three sub-projections partition conv_dim, so each is (its_columns, conv_rows) and they sum back to the whole — for Mamba2 conv_dim = x_cols + B_cols + C_cols (that's local_proj_dims). The conv_rows axis is innermost, so each channel's window is contiguous and therefore a contiguous range of channels — one sub-projection — is a contiguous byte range cols × conv_rows × dtype. That contiguity is what makes the 3-read possible. (So (conv_dim, conv_rows) is the whole conv state; (x_cols, conv_rows) etc. are its column-slices — same axis, named conv_dim whole and "columns" when subdivided.)

One SSM block's page → 4 sub-regions
Mamba2
page xproj0 Bproj1 Cproj2 ssm temporal stateregion 4
GDN
page Qproj0 Kproj1 Vproj2 ssm temporal stateregion 4
conv state — 3 contiguous sub-projections (the "3-read") SSM temporal state — the 4th region
Mamba2 splits conv into [x, B, C] (B and C equal-sized); GDN splits it into [Q, K, V] (dim(Q)=dim(K)). Both sit ahead of the ssm temporal state in one page.

Why 4 separate descriptors rather than one whole-page copy? Because the conv state is TP-sharded along the column (dim) axis, sub-projection by sub-projection — and that sharding does not line up with the page's byte order. A rank's local page is [x-slice | B-slice | C-slice], but in the global page those three slices live at three non-adjacent offsets (inside the full x block, the full B block, and the full C block respectively). So a contiguous copy of a prefix of P's page would splice the wrong columns together. The fix: read each sub-projection slice as its own descriptor — 3 conv reads (the "3-read") plus the ssm state — with no permutation on P.

Worked example: P_TP = 1 → D_TP = 2 (tp_ratio = 2)

Take a Mamba2 conv state with global columns x = 8, B = 2, C = 2 (counting columns, not bytes — real bytes are columns × conv_rows × dtype). The prefill rank holds the whole 12-column page; each decode rank holds half of every sub-projection: x = 4, B = 1, C = 1.

P's full conv page — who reads what (D_TP = 2)

D0 reads (green outline) D1 reads (red outline)
D0's local page is [x0 x1 x2 x3 | B0 | C0] — bytes at offsets 0–3, 8, 10 of P's page. D1's is [x4 x5 x6 x7 | B1 | C1] — offsets 4–7, 9, 11. Neither is a contiguous run.

remote_conv_offsets(local_offset, tp_ratio) computes exactly these three (offset, len) reads per rank (ssm_conv_transfer_utils.py:62-103). With remote_conv0 = 4×2 = 8 (P's whole x) and remote_conv1 = 1×2 = 2 (P's whole B):

D rankx readB readC read
D0 (off=0) (0, 4) (8+0, 1) (8+2+0, 1) = 10
D1 (off=1) (4, 4) (8+1, 1) = 9 (8+2+1, 1) = 11

A 4th read grabs the ssm temporal state, which is sharded on its heads dimension: D rank r reads conv_size_remote + r × ssm_len (nixl/worker.py:1088-1096). Contrast the naive whole-page copy: D0 copying P's first 6 bytes gets [x0 x1 x2 x3 x4 x5] — it picks up x4/x5 where B0/C0 belong and misses them entirely. Correct layout is only recoverable as separate slices.

The other direction (P_TP > D_TP, tp_ratio < 0).

Now D holds the full state and each P rank holds a fraction, so D gathers the same sub-projection from several P ranks and concatenates the pieces into its local x (then B, then C) region. remote_conv_offsets flips: it returns P-sized offsets (0, conv0/|ratio|), (conv0/|ratio|, conv1/|ratio|), … and D reads each P rank's whole (smaller) page, placing the slices side by side. Still per sub-projection, still no single contiguous copy — the interleave is just on the write side instead of the read side.

The conv state is always TP-sharded this way, even when the model's attention KV is replicated across ranks (nixl/worker.py:1065-1066). These 4-per-layer regions are what num_ssm_regions = len(block_len_per_layer) × 4 counts in 4.3, and they stride over logical_blocks (next), not num_blocks.

Why "always" sharded — padding vs. replication.

Attention takes the REPLICATE path only because of GQA/MQA: when num_kv_heads < tp_size a KV head can't be given to every rank, so it is duplicated — several ranks hold byte-identical KV (the M3 key-only indexer). Mamba has an analogous small dimension, n_groups (the B/C groups), but resolves it the opposite way: at layer-build time it pads n_groups up to a multiple of tp_size (extra_groups_for_head_shards, mamba_utils.py:201-210), so conv_dim always divides evenly and each rank stores a distinct conv_dim / tp slice (mamba_utils.py:173-180); the temporal state is likewise divide(num_heads, tp_size). Even in the n_groups = 1 case the duplicated group data lands as separate per-rank slices of a padded buffer — never two ranks pointing at the same bytes. So SSM cache is never byte-identical across ranks, and the connector has no if replicate branch for it: conv and ssm state are unconditionally SPLIT, divided by TP with a per-rank local_offset.

Example models.

The connector supports two SSM families (ssm_conv_transfer_utils.py): Mamba2 — e.g. NVIDIA Nemotron-H / Nemotron Nano 2, IBM Bamba, Codestral Mamba — with conv sub-projections (x, B, C); and Gated DeltaNet (GDN) — e.g. Qwen3-Next — with (Q, K, V). These are hybrids: most layers are SSM, a few are full attention (Nemotron Nano 2 keeps ~4 attention layers; Qwen3-Next runs a 3:1 GDN-to-attention ratio). The attention layers land in the FA section, the SSM layers in the SSM section — the exact split 4.3 exists to handle.

4.5 Logical vs physical blocks

Three closely-related counts appear throughout the connector, set once by _sync_block_size_with_kernel (nixl/worker.py:454-471). You configure a logical block_size (tokens per block, say 128). But an attention kernel may demand a smaller physical / kernel block size (say 64); select_common_block_size picks the common kernel size across the active backends. When they differ the connector switches to speaking physical blocks: 

kernel_block_size = select_common_block_size(block_size, backends)   # e.g. 64
_logical_num_blocks = num_blocks                                     # count BEFORE rescale
if block_size != kernel_block_size:
    physical_blocks_per_logical = block_size // kernel_block_size    # 128 // 64 = 2
    block_size  = kernel_block_size                                  # now physical
    num_blocks *= physical_blocks_per_logical                        # physical count

FA reads/writes at kernel granularity, so its descriptors run over num_blocks = 4; the per-layer SSM state is one blob per logical block, so its descriptors run over logical_blocks = num_blocks // 2 = 2. The same ratio drives get_mapped_blocks, which expands a logical block id into its physical ids when P and D run different block sizes (nixl/worker.py:2246-2265).

Llama, DeepSeek-V3, and Kimi K2 all have ratio = 1 — the kernel block size equals the configured one, so logical and physical blocks coincide and logical_blocks == num_blocks. The ratio only exceeds 1 when a kernel locks a smaller block size than you asked for (the 128→64 case above), which is also where get_mapped_blocks and the physical_blocks_per_logical scaling in the block-id pairing (5.4) earn their keep.

4.6 In-block layout: why HND?

One more layout choice sits inside each region's block: the order of the head (H) and token (N) axes. There are two options — and for non-MLA models the NixlConnector forces HND (nixl/connector.py:115-131):

  • HND = (num_kv_heads, block_size, head_dim) — a head's data (all tokens in the block) is contiguous.
  • NHD = (block_size, num_kv_heads, head_dim) — a token's heads are contiguous, so any single head is strided across the block.

The reason NIXL cares: a transfer descriptor is a contiguous (addr, len), and in hetero-TP a decode rank reads only its head-slice at a per-rank offset (the SPLIT path of 4.3). That slice is one clean descriptor only if the rank's heads are contiguous — i.e. only under HND.

One block, 4 heads × 4 tokens — rank A wants heads 0–1
HND
head-major h0h0h0h0 h1h1h1h1 h2h2h2h2 h3h3h3h3
NHD
token-major h0h1h2h3 h0h1h2h3 h0h1h2h3 h0h1h2h3
rank A's heads (0–1) other ranks' heads
Under HND, rank A's heads are the first 8 cells — one contiguous run, one descriptor at offset rank_offset × per_head_block_len. Under NHD the same heads are scattered 2-on-2-off across every token — no single contiguous slice exists.

So NHD "will not support Decode/Prefill TP_ratio > 1" (nixl/worker.py:605-622). At tp_ratio = 1 it's harmless — each D rank reads the whole block from its P counterpart, no head-slicing — which is why the constraint only bites in hetero-TP. Three corollaries:

  • MLA is exempt. get_required_kvcache_layout returns None for MLA (nixl/connector.py:122-127): a single shared latent has no head axis to split, so the whole-block (REPLICATE-style) read works under any layout.
  • Escape hatch. A device stuck on NHD can set enable_permute_local_kv to stage a permuted HND copy in a host buffer, restoring hetero-TP (nixl/worker.py:609-622).
  • Validated at handshake. kv_cache_layout rides in NixlAgentMetadata (the 3.3 example shows "HND"), so a P/D layout mismatch is caught and reconciled, not silently transferred wrong.

4.7 Remote (lazy, per engine)

Adding a remote agent only happens after the ZMQ handshake. Per remote engine, per remote rank, D calls (nixl/worker.py:1309-1312): 

remote_agent_name = self.nixl_wrapper.add_remote_agent(
    nixl_agent_meta.agent_metadata

self.dst_xfer_side_handles[engine_id][remote_tp_rank] = (

The "prepared xfer dlist" is NIXL's pre-baked descriptor table for the remote agent's memory. It's cached per (engine_id, remote_tp_rank). Every subsequent request to the same engine reuses it — that's why later requests are cheap.

5. Where the block IDs come from

The READ needs two lists: where the data lives on P (remote_block_ids) and where it should land on D (local_block_ids). Neither travels over NIXL, and neither rides the ZMQ handshake — the handshake only carries layout metadata (section 3.3). Both block lists are plain KV indices that piggy-back on the kv_transfer_params dict attached to the request. The "remote" naming is purely point of view: remote_block_ids are P's own local block ids, called remote only because D is the one reading them.

5.1 P mints the list when prefill finishes

When P's prefill request completes, the scheduler's request_finished (nixl/scheduler.py:580-674) is handed the request's local KV block ids. Rather than free them it decides to delay the free — so D still has something to read — unpads any SWA/HMA placeholder blocks with get_sw_clipped_blocks, and ships them inside the returned kv_transfer_params: 

return delay_free_blocks, dict(
    do_remote_prefill=is_p_node,
    remote_block_ids=block_ids,          # P's OWN local blocks
    remote_engine_id=self.engine_id,     # which agent D should READ
    remote_request_id=request.request_id,
    remote_host=self.side_channel_host,  # where D handshakes
    remote_port=self.side_channel_port,
    tp_size=...,
    remote_num_tokens=request.num_computed_tokens,
)

On the same step P arms the lease: the request id goes into _reqs_need_send with an expiry timestamp — the very entry that section 8's heartbeat later refreshes.

5.2 The params travel back to D

This dict rides the prefill response back to the orchestrator; the P/D proxy attaches it to the decode request it then issues. By the time the request lands on D's scheduler it is sitting in request.kv_transfer_params with do_remote_prefill=True and the full remote_block_ids list. NIXL itself never moves these — they arrive the same way the request text does.

Example: kv_transfer_params for Llama 3.1 8B

For a 120-token prompt on the same Llama-3.1-8B prefill engine as 3.3 (block_size = 16, TP = 1), the dict that lands on D looks like: 

request.kv_transfer_params = {
    "do_remote_prefill": True,                 # D should pull, not compute
    "do_remote_decode":  False,
    "remote_block_ids":  [12, 13, 14, 15, 16, 17, 18, 19],  # P's OWN local blocks
    "remote_engine_id":  "prefill-3f9c…",      # which P agent to READ from
    "remote_request_id": "cmpl-7a2f…",         # P's id for this request
    "remote_host":       "10.0.2.7",           # P's side-channel host
    "remote_port":       5557,                 # VLLM_NIXL_SIDE_CHANNEL_PORT
    "tp_size":           1,                    # P's TP → tp_ratio at D
    "remote_num_tokens": 120,                  # prefill length P computed
}
  • remote_block_ids has ceil(120 / 16) = 8 entries — P's own physical blocks holding the prompt's KV, "remote" only from D's seat (5.1).
  • remote_engine_id + remote_host/remote_port are what D feeds the 3.3 handshake to fetch P's NixlAgentMetadata and register it as a remote agent.
  • remote_num_tokens and tp_size drive D's pull count (5.3) and the tp_ratio of the READ (5.4, 4.6).

5.3 D allocates its own half and pairs them

D never reuses P's indices — its KV cache is a different tensor on a different GPU. It computes its own destination blocks in three steps:

  • get_num_new_matched_tokens validates the params and returns how many tokens must be pulled: min(remote_num_tokens, num_prompt_tokens) − num_computed_tokens — where num_computed_tokens already subtracts whatever D holds in its own local prefix cache (nixl/scheduler.py:393-429).
  • The scheduler allocates blocks for those external tokens, then update_state_after_alloc records D's freshly-allocated, not-already-cached blocks as local_block_ids via get_unhashed_block_ids_all_groups(), SWA-clipped (nixl/scheduler.py:461-501). A full local prefix hit yields an empty list — "pull nothing, but still notify P so it frees."
  • build_connector_meta → add_new_req_to_recv packs both halves into one ReqMeta: local_block_ids (D's blocks) and remote=RemoteMeta(block_ids=remote_block_ids, …) (P's blocks) (nixl/metadata.py:197-211).

5.4 Pairing is positional, with a tail-trim

At READ time the two lists are zipped index-by-index, per KV-cache group. They can differ in length: P sent its whole block list, but D may already hold a prefix locally. Because a cache hit is always a prefix, the blocks still to fetch are P's tail, so _apply_prefix_caching end-trims the remote list (nixl/worker.py:2292-2316): 

num_local_blocks = len(local_block_ids[i])
assert num_local_blocks <= len(remote_group)
if num_local_blocks < len(remote_group):
    remote_block_ids[i] = remote_group[-num_local_blocks:]   # keep the tail

6. The READ transfer

On start_load_kv, after the handshake has completed, _read_blocks_for_req (nixl/worker.py:2021-2095) computes the full plan and emits one READ per remote rank that holds a piece of this request's KV. 

read_specs = [
    ReadSpec(
        remote_rank=rank,
        local_block_ids=[...],   # where on D the data goes
        remote_block_ids=[...],  # where on P the data lives
    )
    for rank in plan.all_source_ranks
]

for spec in read_specs:
    self._read_blocks(read_spec=spec, ...)

It is all-at-once per request per remote rank. No per-layer batching. The save_kv_layer and wait_for_layer_load hooks on the base class are explicit no-ops for NIXL (nixl/connector.py:255-267). All of one request's blocks go in one NIXL transfer per source rank.

Hetero-TP means multiple READs per request

If TPD < TPP, a single D rank gathers from multiple P ranks. That becomes one READ per (D_rank → P_rank) pair. See PARALLELISM.html for the head-dimension math.

7. Completion: polling + NIXL notifs

NIXL transfers are non-blocking: nixl_wrapper.transfer(handle) returns immediately and the work happens in the NIXL backend. D learns of completion two ways depending on which side it's on.

7.1 D side: poll xfer state

Each scheduler step, get_finished calls _pop_done_transfers (nixl/worker.py:1868-1913): 

for handle in handles:
    xfer_state = self.nixl_wrapper.check_xfer_state(handle)
    if xfer_state == "DONE":
        self.nixl_wrapper.release_xfer_handle(handle)
    elif xfer_state == "PROC":
        in_progress.append(handle)
    else:
        self._handle_failed_transfer(req_id, handle)

Once all handles for a request are DONE, the request appears in done_recving and the local scheduler can consume the prompt as if it had been local all along.

7.2 P side: receive notifs from D

D piggy-backs a notification on the READ (notif_msg=notif_id in the call above). The notif body is "{remote_request_id}:{D_world_size}". P polls in get_finished via self.nixl_wrapper.get_new_notifs() (nixl/worker.py:1792-1845) and counts how many D ranks have notified: 

self.consumer_notification_counts_by_req[req_id] += 1
if self.consumer_notification_counts_by_req[req_id] == consumers_per_producer:
    notified_req_ids.add(req_id)

Only once all D consumers have notified does P add the request to done_sending. The consumers_per_producer number is derived from the TP layout: with TPD = 4 and TPP = 2, each P rank serves two D ranks, so each P rank must wait for two notifs per request.

8. The lease + heartbeat lifetime contract

What if D never notifies — crashed mid-load, network died? The naive scheme leaks P's KV blocks forever. NIXL prevents that with a lease + heartbeat:

  • P tracks each in-flight request in self._reqs_to_send: dict[req_id, expiry_time]. The initial expiry is kv_lease_duration (default 30s, configurable; nixl/worker.py:219-223).
  • D periodically sends a heartbeat notif of the form "HB:req1,req2,..." (nixl/worker.py:2013 and 1994-2020) listing every request whose load it has in flight.
  • P's notif loop recognizes the HB: prefix and resets each named request's expiry to now + _lease_extension, where _lease_extension = kv_lease_duration * 2 // 3 (nixl/worker.py:1847-1866).
  • In get_finished, P walks the time-sorted lease map and evicts any request whose lease has expired (nixl/worker.py:1771-1788). Evicted requests are reported as done_sending with a logged warning so the scheduler can free their blocks.

9. Failure modes

Transfer fails (D side)

check_xfer_state returns neither DONE nor PROC: _handle_failed_transfer (nixl/worker.py:1915-1931) marks the request's block ids as invalid (so the scheduler will recompute instead of trusting the half-written KV), puts the request into _failed_recv_reqs, releases the NIXL handle, and records the failure.

Handshake fails (D side)

Future's done_callback (nixl/worker.py:729-740) catches the exception. Subsequent requests targeting that engine never get added to _ready_requests and are marked failed. Useful when P has died — the failure surface is a well-defined ZMQ connect error, not a hung worker.

D died mid-load (P side)

No HBs arrive. Lease ticks down to zero. P logs a warning and adds the request to done_sending — the blocks are freed and the warning carries the count of how many consumers did manage to read. Block reclamation is eventually consistent.

Compatibility hash mismatch

On handshake, D computes a SHA-256 of (model, block layout, attention backend, ...) and compares to P's. Mismatch ⇒ RuntimeError (nixl/worker.py:537-553). Can be disabled with enforce_handshake_compat=false in the KV transfer config — useful in dev, dangerous in prod.
