KV Cache · Transfer · PARALLELISM

TP, PP, DP Across P and D

How NIXL, the direct Mooncake pipe, and the Mooncake store each handle heterogeneous tensor parallelism between prefill and decode. Head-dimension splitting, GQA dedup, MLA replication, Mamba's three-projection conv state. PP and DP mostly fall out for free.

TL;DR

TP is the hard one. Two TP sizes don't have to match: prefill might run TP=4 on H100s, decode might run TP=2 on smaller GPUs. All three transports support this, but they reach the goal three different ways. NIXL uses a precomputed TPMapping with explicit head-slot indices and rank offsets. The Mooncake pipe computes a _get_tp_ratio and offsets within registered regions. The Mooncake store doesn't precompute at all — it just asks "which TP shards exist in the store?" and only loads if all of them are there. PP comes for free in all three. DP is "many P groups, many D groups; route at the bootstrap layer."

1. The problem statement

Prefill and decode get to run on different hardware, with different parallelism strategies. Typical asymmetries:

  • TPP > TPD: P sharded across 4 GPUs, D across 2. Each D rank must gather from multiple P ranks.
  • TPP < TPD: P sharded across 2 GPUs, D across 4. Each P rank produces KV for multiple D ranks — or, equivalently, each D rank takes a fraction of one P rank's KV. This requires head-dimension splitting.
  • TPP = TPD: the easy case — 1:1 mapping. Often still needs validation that the per-rank layout matches.

The KV cache for a layer is laid out as [num_blocks, block_size, num_kv_heads / tp_size, head_dim] (FlashAttention HND) or a similar permutation. Splitting along num_kv_heads / tp_size is what TP does; the connectors must reverse the same split when P and D disagree.

Hetero TP example: TPP=2, TPD=4
P rank 0 holds:   heads [0, 1, 2, 3]      (half of all KV heads)
P rank 1 holds:   heads [4, 5, 6, 7]      (other half)

D rank 0 wants:   heads [0, 1]            (quarter)
D rank 1 wants:   heads [2, 3]            (quarter)
D rank 2 wants:   heads [4, 5]            (quarter)
D rank 3 wants:   heads [6, 7]            (quarter)

So:    D0  <-- read 1st half of P0's heads
       D1  <-- read 2nd half of P0's heads
       D2  <-- read 1st half of P1's heads
       D3  <-- read 2nd half of P1's heads
Sidebar — block-length variables in the worked examples

Two variables, local_kv_block_len and remote_kv_block_len, show up in every worked example below. The concept is simple (bytes per KV block per rank), but the names mislead in two ways. This is the single place that explains both.

1. The concept: bytes per KV block per rank

For a KV cache tensor of shape [num_blocks, block_size, num_kv_heads/tp_size, head_dim], one block on one rank is: 

local_kv_block_len = block_size × (num_kv_heads / tp_size) × head_dim × element_size
                                    ↓
                                    per-rank head count

Per layer; multiply by num_layers for the full payload per block per rank. Concrete numbers for Llama-3-70B (num_kv=8, head_dim=128, fp16, block_size=16):

tp_size num_kv_heads / tp_size local_kv_block_len (per layer)
2 4 heads / rank 16 × 4 × 128 × 2 = 16 KB
4 2 heads / rank 16 × 2 × 128 × 2 = 8 KB
8 1 head / rank 16 × 1 × 128 × 2 = 4 KB

With 80 layers (Llama-3-70B), the per-block payload that crosses the wire is 80 × local_kv_block_len. At TP=2 that's 1.28 MB per block per rank.

2. K and V layout: kv_block_len may be half of block_len

If K and V share a single tensor (blocks-first layout, e.g., FlashInfer), the registered TransferRegion.block_len covers both halves; kv_block_len = block_len // 2 is one half (mooncake_connector.py:121). With FlashAttention's K/V-first layout, K and V live in separate tensors and kv_block_len == block_len. The planning functions consume kv_block_len, so they don't care which layout the model uses.

3. The "local" naming trap

"Local" means the side that's computing the plan, which is not the same side across connectors:

Function "local" is "remote" is Where
NIXL _build_fa_remote D (reader of the READ) P (target of the READ) nixl/worker.py:1135
Mooncake _compute_sender_transfer_plan P (sender of the WRITE) D (target of the WRITE) mooncake_connector.py:140

So local_kv_block_len refers to a different concrete value depending on which connector you're reading. Concretely, across the four worked examples (Llama-3-70B sizing):

Example TP layout local_kv_block_len is… Value / layer
§2.1 NIXL TPP=4, TPD=2 D's block (4 heads), divided by num_attn_reads=2 per READ → 2 heads on the wire 16 KB → 8 KB per READ
§2.2 NIXL TPP=2, TPD=4 D's block (2 heads); only one READ per rank, no division 8 KB (= r in the example)
§3.1 Mooncake pipe TPP=2, TPD=4 P's block (4 heads); D's slice is remote_kv_block_len = 2 heads = r P: 16 KB   D: 8 KB
§3.2 Mooncake pipe TPP=4, TPD=2 P's block (2 heads) = r; D is remote_kv_block_len (4 heads) P: 8 KB   D: 16 KB

Notice that in §2.2 and §3.1 — same TP layout, TPP=2, TPD=4 — local_kv_block_len means 8 KB in the NIXL doc but 16 KB in the Mooncake doc. "Local" flipped sides; the bytes stayed put.

4. The extra NIXL quirk: remote_kv_block_len is also slice-sized

In NIXL specifically, remote_kv_block_len is not P's full block. It's set to local_block_len // block_size_ratio (nixl/worker.py:1154). When P and D agree on block_size (the usual case, block_size_ratio == 1), this collapses to "D's block size" — the slice size used per READ, not P's full block. That's why in §2.2, rank_offset = rank_offset_factor × remote_kv_block_len moves D1 to offset r (8 KB), not 2r (16 KB), inside P0's larger block. The variable is mis-named; treat it as "transfer slice size" rather than "P's block size."

One-line takeaway

Both variables mean "bytes per KV block per rank," local_* for whichever side is computing the plan and remote_* for the other side — with NIXL additionally normalizing remote_kv_block_len to the per-transfer slice size, not the peer's full block.

2. NIXL: precomputed TPMapping

NIXL's solution lives in vllm/distributed/kv_transfer/kv_connector/v1/nixl/tp_mapping.py. For each remote engine, on first handshake, D computes a complete plan: 

@dataclass(frozen=True)
class TPMapping:
    source_ranks_per_group: tuple[tuple[int, ...], ...]
    all_source_ranks: tuple[int, ...]
    rank_to_attention_slot: dict[int, int]   # which head-slot does each remote rank fill?
    rank_offset_factor: int                  # offset into remote KV when D_TP > P_TP

The decision tree in compute_tp_mapping (nixl/tp_mapping.py:65-142):

  1. TPD ≥ TPP (or MLA): each local rank reads from exactly one remote rank. Source rank = tp_rank * remote_tp_size // tp_size.
  2. TPD < TPP: each local rank reads from multiple remote ranks. Source ranks = [tp_rank * abs_tp .. tp_rank * abs_tp + abs_tp] where abs_tp = remote_tp_size // tp_size. Then GQA dedup collapses ranks that hold the same heads (see §6).

The rank_offset_factor is the head-dimension offset needed when multiple D ranks read from one P rank. Each needs a different slice. Use: 

# worker.py _build_fa_remote (line 1159-1168)
local_block_len = local_block_len // num_attn_reads
rank_offset = plan.rank_offset_factor * remote_kv_block_len

for block_id in range(num_blocks):
    block_offset = block_id * page_size
    addr = base_addr + block_offset + rank_offset
    result.append((addr, local_block_len, nixl_agent_meta.device_id))

What's elegant: NIXL doesn't move data twice. The local descriptor table records "read local_block_len // num_attn_reads bytes starting at base_addr + block_offset + rank_offset". The NIXL READ goes straight from the right slice of P's memory into the right slice of D's memory.

2.1 Worked example: TPP=4, TPD=2, 8 KV heads

What "sharded across N GPUs" means here

TP shards every tensor in the model — Q, K, V, O projections, FFN gate/up/down, layer norms partially. But none of those shardings cross the wire during a KV transfer; they're internal to each rank's forward pass. The only sharding the connector cares about is the KV cache tensor itself, whose shape per rank is [num_blocks, block_size, num_kv_heads / tp_size, head_dim]. So when we say "P is sharded across 4 GPUs, 2 KV heads per rank," we specifically mean each P rank's KV cache holds 2 of the model's 8 KV heads (each with the full head_dim=128) — a contiguous slice along the num_kv_heads axis.

Llama-3-70B-style: 8 KV heads, head_dim=128. P sharded across 4 GPUs (2 KV heads per rank). D sharded across 2 GPUs (4 KV heads per rank). Walk through what compute_tp_mapping produces for D rank 0: 

tp_rank = 0, tp_size = 2 (D), remote_tp_size = 4 (P)
total_num_kv_heads = 8

→ tp_size (2) < remote_tp_size (4):  "P TP > D TP" branch

abs_tp     = 4 // 2 = 2
start      = 0 * 2 = 0
heads      = [0, 1] * 8 // 4 = [0, 2]
attn_ranks = [0, 1]                # D0 reads from P ranks 0 and 1

head_to_slot[0] = 0                # P0 fills slot 0 of D0's KV (heads 0,1)
head_to_slot[2] = 1                # P1 fills slot 1 of D0's KV (heads 2,3)

rank_offset_factor = 0             # tp_size ≤ remote_tp_size branch

What goes on the wire for one request on D rank 0:

READ 1:  P0's KV blocks  ---->  D0 KV slot 0 (heads 0, 1)
READ 2:  P1's KV blocks  ---->  D0 KV slot 1 (heads 2, 3)

P side — TP=4, 2 KV heads / rank, block = r bytes each

P1 heads 2,3

P2 heads 4,5

Colors match head-pairs. The slicing is on D's side: each D rank's 2r-byte block is divided into two r-byte slots, and head_to_slot tells NIXL which slot each remote P rank fills. local_block_len // num_attn_reads = 2r / 2 = r is the per-READ size; rank_offset on the source side is 0 (each P sends its whole block).

Two NIXL READs for D0; D1 issues the symmetric pair reading P2 and P3. Total wire transfers per request across the D pool: 4, all happening in parallel through the NIXL backends. Notice: zero copies. The local descriptor's (addr, len) already points at the correct half-slot of D0's block, so NIXL writes data into its final resting place.

2.2 Worked example: TPP=2, TPD=4, 8 KV heads (the split direction)

The inverse layout: P sharded across 2 GPUs (4 KV heads per rank), D across 4 GPUs (2 KV heads per rank) — same TP layout as the Mooncake pipe example in §3.1, but with pull semantics. Each D rank reads a slice of one P rank's larger block, rather than gathering from two P ranks. Walk compute_tp_mapping for D rank 0: 

tp_rank = 0, tp_size = 4 (D), remote_tp_size = 2 (P)
total_num_kv_heads = 8

→ tp_size (4) ≥ remote_tp_size (2):  "D TP ≥ P TP" branch

attn_ranks = [0 * 2 // 4] = [0]    # D0 reads from P0 only (one rank)

head_to_slot[0 * 8 // 2] = 0       # P0 fills slot 0 of D0's KV
                                   # (D0 has only one slot — it reads from
                                   #  only one P rank)

rank_offset_factor:
  tp_size (4) > remote_tp_size (2), not MLA      → not the ≡ 0 branch
  tp_size (4) > total_num_kv_heads (8)? No       → else branch
  rank_offset_factor = tp_rank % (tp_size // remote_tp_size)
                     = 0 % (4 // 2)

  2        [1]               0              first half of P1  (heads 4,5)
  3        [1]               1              second half of P1 (heads 6,7)

Colors match head-pairs (compare with the §2.1 diagram — the same colors live on the same heads). The slicing is now on P's side: each P rank's 2r-byte block is divided into two r-byte halves, and rank_offset_factor picks which half each D rank reads. Two D ranks point at the same P, but at different offsets, so they receive disjoint head-pairs.

where r = local_kv_block_len is D's per-block byte size (2 heads worth). The local descriptor that NIXL uses for the READ has its source address computed as P_base + block_offset + rank_offset, where rank_offset = rank_offset_factor * remote_kv_block_len (see _build_fa_remote in nixl/worker.py:1159-1168). With rank_offset_factor ∈ {0, 1} that naturally picks the first or second half of P's 4-head block.

Notice the inversion against §2.1: there, each D rank issued multiple READs (D0 read from both P0 and P1, filling two half-slots of its own block). Here, each D rank issues exactly one READ but takes only a fraction of P's larger block. The rank_offset_factor is what makes the head-dim split work — without it, D0 and D1 reading from the same P0 would get identical data and clobber each other's different head ranges.

Comparison with Mooncake pipe §3.1 (same TP layout)

Both connectors move the same four head-pairs through the same four (P, D) pairs, but with opposite control-plane initiators. NIXL does four D-initiated READs with rank_offset_factor selecting the source slice of P's block. The Mooncake pipe does four P-initiated WRITEs with src_offset selecting the same source slice. Same byte movement, same final memory layout — just different sides of the wire holding the pen.

3. Mooncake pipe: _get_tp_ratio + region offsets

The direct pipe takes the same idea but expresses it in terms of a signed ratio (mooncake/mooncake_connector.py:88-106): 

def _get_tp_ratio(local_tp_size: int, remote_tp_size: int) -> int:
    if local_tp_size >= remote_tp_size:
        return local_tp_size // remote_tp_size      # positive
    return -(remote_tp_size // local_tp_size)       # negative

Positive: one local rank maps into a larger remote region. Negative: one local rank gathers from multiple remote regions. Both are handled by the same planning function (mooncake/mooncake_connector.py:140-174): 

def _compute_sender_transfer_plan(
    local_tp_rank, local_tp_size, remote_tp_rank, remote_tp_size,
    local_kv_block_len, remote_kv_block_len, producer_cache_replicated,
) -> tuple[bool, int, int, int]:
    tp_ratio = _get_tp_ratio(local_tp_size, remote_tp_size)
    if tp_ratio == 1:
        return True, 0, 0, local_kv_block_len
    if tp_ratio > 0:
        if producer_cache_replicated:
            return local_tp_rank % tp_ratio == 0, 0, 0, local_kv_block_len
        return True, 0, (local_tp_rank % tp_ratio) * local_kv_block_len, local_kv_block_len
    # tp_ratio < 0
    if producer_cache_replicated:
        return True, 0, 0, local_kv_block_len
    ratio_abs = -tp_ratio
    return True, (remote_tp_rank % ratio_abs) * remote_kv_block_len, 0, remote_kv_block_len

The returned tuple is (should_send, src_offset, dst_offset, transfer_len). Read it as: "Should this P rank send to that D rank at all? If so, where in the source region does the copy start, where in the dest, and how big is it?"

There's also a region-expansion step (_expand_transfer_regions, mooncake/mooncake_connector.py:109-137) that splits K and V into separate regions when the layout is blocks-first (FlashAttention) so that the planning logic above can treat them uniformly.

Validation upfront (mooncake/mooncake_connector.py:193-244) confirms that region lengths agree with the TP ratio. Mismatch ⇒ the whole batch errors with a clear message at handshake time, not silent corruption at transfer time.

3.1 Worked example: TPP=2, TPD=4, 8 KV heads

Flip the previous scenario: P sharded across 2 GPUs (4 KV heads per rank), D across 4 GPUs (2 KV heads per rank). Each P rank must push KV to two D ranks, splitting along the head dimension. local_tp_size = 2 (P), remote_tp_size = 4 (D)

P0 (local_tp_rank=0) sending to D0 (remote_tp_rank=0):

  → (should_send=True, src_offset=0, dst_offset=0, transfer_len=r)

  src_offset = (1 % 2) * r = r
  → (should_send=True, src_offset=r, dst_offset=0, transfer_len=r)

3.2 Worked example: TPP=4, TPD=2, 8 KV heads (the gather direction)

Now the inverse: P sharded across 4 GPUs (2 KV heads per rank), D across 2 GPUs (4 KV heads per rank) — same setup as the NIXL example in §2.1, but with push semantics. Each D rank gathers from two P ranks. Walk _compute_sender_transfer_plan for the four (P rank, D rank) pairs, where r = local_kv_block_len is now P's per-block byte size (2 heads worth): 

local_tp_size = 4 (P), remote_tp_size = 2 (D)
→ tp_ratio = 4 // 2 = 2     (positive: gather case)

Each P rank pairs with exactly one D rank (remote = local // tp_ratio):
  P0,P1 → D0      P2,P3 → D1

For positive tp_ratio (non-replicated cache), the plan is:
  src_offset = 0
  dst_offset = (local_tp_rank % tp_ratio) * r
  transfer_len = r

P0 (local_tp_rank=0) sending to D0:
  dst_offset = (0 % 2) * r = 0
  → (should_send=True, src_offset=0, dst_offset=0, transfer_len=r)

P1 (local_tp_rank=1) sending to D0:
  dst_offset = (1 % 2) * r = r
  → (should_send=True, src_offset=0, dst_offset=r, transfer_len=r)

What goes on the wire (four batch_transfer_sync_write calls):

P0  -->  D0:  src_offset=0,  dst_offset=0,  len=r   (heads 0, 1)
P1  -->  D0:  src_offset=0,  dst_offset=r,  len=r   (heads 2, 3)
P2  -->  D1:  src_offset=0,  dst_offset=0,  len=r   (heads 4, 5)
P3  -->  D1:  src_offset=0,  dst_offset=r,  len=r   (heads 6, 7)

Compare with the NIXL §2.1 diagram — identical TP layout, identical colors, identical byte movement. Only the arrow direction is different: here P drives with dst_offset picking the destination slot; in NIXL D drives with head_to_slot picking the same destination slot. Two pushes (or two pulls) per D block — whichever side has the larger block, that's the side with the variable offset.

Notice the inversion against §3.1: there, src_offset varied (P sliced its own 4-head block into two halves) and dst_offset was always 0. Here it's the opposite — src_offset is always 0 (each P rank pushes its whole block) and dst_offset varies (two different P ranks write into different halves of D's larger block). Comparison with NIXL §2.1 (same TP layout)

Both connectors move the same four head-pairs across the same four (P, D) pairs, but in opposite directions on the wire. NIXL does four READs issued by D (D pulls 2× per rank). The Mooncake pipe does four WRITEs issued by P (P pushes 1× per rank). Same byte volume, same final memory layout, different control-plane initiator.

4. Mooncake store: candidate-key lookup

Before walking the lookup logic, fix the shape of the store's keyspace in your head — everything else follows from this. The store is content-addressed; every block lives under a string key built by PoolKey.to_string() (mooncake/store/data.py:58-67): 

"{model_name}@tp_rank:{tp_rank}@pcp{pcp_rank}@dcp{dcp_rank}@pp_rank:{pp_rank}@group:{group_id}@{chunk_hash}"

A worked rendering for Llama-3-70B on TP=2, PP=1:

llama-3-70b@tp_rank:0@pcp0@dcp0@pp_rank:0@group:0@a3f2c1d4...
llama-3-70b@tp_rank:1@pcp0@dcp0@pp_rank:0@group:0@a3f2c1d4...
                ^                              ^      ^
                |                              |      |
                |                              |      `--- vLLM prefix-cache block hash
                |                              |
                |                              `--- KV cache group id (HMA: full attn,
                |                                   SWA, Mamba, ...)
                |
                `--- NOT the raw TP rank!  This is head_or_tp_rank
                     (worker.py:826-839):
                       head_or_tp_rank = tp_rank // put_step
                                         (when tp_size > num_kv_head)
                       head_or_tp_rank = tp_rank
                                         (otherwise)

The single most important field for hetero-TP is tp_rank, but it's not what the name suggests. The value stored there is the KV-head index, not the physical TP rank. When tp_size > num_kv_head (heavy TP under GQA), multiple physical ranks collapse to the same head_or_tp_rank — the put_step slice on the put path (store/worker.py:511) distributes the chunk writes among them so each chunk is stored exactly once per head.

Concretely, the per-chunk key count in the store is always min(tp_size, num_kv_head) × pp_size, regardless of how oversharded the writer was. That's the invariant the consumer-side lookup relies on.

Now the lookup logic. The store doesn't precompute. There's no "where does P's KV head k live?" because D never asks — D asks the store. The store has keys for every (model, head_or_tp_rank, pp_rank, hash) tuple that P put. D simply iterates over the head indices it cares about: 

# mooncake/store/worker.py:1210-1236
tp_count = min(self.tp_size, self.num_kv_head)
for g_idx, db in enumerate(self.token_dbs):
    for chunk_id, h in enumerate(group_hashes):
        for tp in range(tp_count):
            for pp in range(self.pp_size):
                md = dataclasses.replace(db.metadata, tp_rank=tp, pp_rank=pp)
                candidate_keys.append(PoolKey(md, h.hex()).to_string())
res = self.store.batch_is_exist(candidate_keys)

Then dedup: a chunk hash is "present" iff all expected (tp, pp) shards exist (mooncake/store/worker.py:1225-1231): 

expected_per_key = max(1, tp_count * self.pp_size)
present_count: dict[tuple[int, bytes], int] = {}
for gh, exists in zip(candidate_meta, res, strict=True):
    if exists == 1:
        present_count[gh] = present_count.get(gh, 0) + 1
exists_set = {gh for gh, c in present_count.items() if c >= expected_per_key}

This is a striking design choice. The store doesn't need to know what the producer's TP layout was: it just does the necessary fetches, and if any shard is missing (because the producer never wrote it, or because a TP layout swing leaves a gap), it falls back to recompute. The cost is iterating tp_count × pp_size candidate keys per chunk.

4.1 Worked example: TPP=8, TPD=4, 4 KV heads

The store's hetero-TP trick lives in how it builds the key on the producer, not in the consumer's lookup logic. Look at the metadata construction (mooncake/store/worker.py:826-839): 

if self.num_kv_head < self.tp_size:
    self.put_step = self.tp_size // self.num_kv_head
    self.head_or_tp_rank = self.tp_rank // self.put_step    # dedup: many ranks → one key
else:
    self.head_or_tp_rank = self.tp_rank
    self.put_step = 1

self.metadata = KeyMetadata(
    ...
    tp_rank=self.head_or_tp_rank,    # key's tp_rank is the *head index*, not the raw rank

That's what lets P and D talk to the store with different TP sizes: as long as they agree on num_kv_head, they agree on the key.

And as a side benefit, the same content-addressed scheme means cross-run cache hits work for free. 4.2 Worked example: TPP=8, TPD=2, 4 KV heads (lookup passes, fetch mismatches)

Now shrink D instead of growing it: keep P heavily oversharded (TP=8) but make D very compact (TP=2). Same num_kv_head=4. This case is the most instructive failure mode — the lookup actually passes, but the per-key byte sizes don't agree between P and D.

P side — identical to §4.1:

(2) Even for the two keys D does fetch, D's per-rank-slot expects 2-heads-worth of bytes while the store holds 1-head-worth — Mooncake errors on the size mismatch. The root cause: when tp_size < num_kv_head on either side, that side's per-rank slot packs multiple heads, and the store's "one key = one KV head's bytes" abstraction breaks. 4.3 Worked example: TPP=2, TPD=8, 4 KV heads (lookup fails outright)

Now flip it: P undersharded (TP=2, each rank holds 2 KV heads), D oversharded (TP=8, ranks pair up to replicate). Same num_kv_head=4. This case does not get a cache hit, and walking through why is the cleanest way to see the store's constraint.

P side — each rank's key packs multiple heads:

num_kv_head=4, tp_size=2  →  4 < 2 is False, so:
                                head_or_tp_rank = tp_rank
                                put_step = 1

  P rank 0 → head_or_tp_rank=0   (holds KV heads 0 AND 1, packed)
  P rank 1 → head_or_tp_rank=1   (holds KV heads 2 AND 3, packed)

After all puts, store holds 2 keys per chunk hash:
  "...@tp_rank:0@...@<h>"   (payload bytes = heads 0+1 concatenated)
  "...@tp_rank:1@...@<h>"   (payload bytes = heads 2+3 concatenated)

Notice the asymmetry with the previous example: there's no put_step dedup here, because each rank already holds multiple heads, not a fraction. The key just labels "which P rank's blob this is," and the blob contains whatever heads that rank owned — the store has no idea those bytes are actually two heads concatenated.

D side — expects one key per KV head:

num_kv_head=4, tp_size=8  →  4 < 8 is True, so:
                                put_step = 8 // 4 = 2
                                head_or_tp_rank = tp_rank // 2

  D rank 0,1 → head_or_tp_rank=0   (replicated full KV head 0)
  D rank 2,3 → head_or_tp_rank=1   (replicated full KV head 1)
  D rank 4,5 → head_or_tp_rank=2   (replicated full KV head 2)
  D rank 6,7 → head_or_tp_rank=3   (replicated full KV head 3)

Why the dedup check fails here

D expects the keyspace to be partitioned by KV head — one key per head. P only produced two keys, each holding two heads. The expected_per_key check fail-closes the whole chunk, forcing a recompute rather than risking a torn load.

The fix is on the producer side, not the consumer. If you want hetero-TP cache hits, the producer must shard at least as finely as num_kv_head: tp_sizeP ≥ num_kv_head. Then P's keys partition the keyspace by head, and any D with tp_sizeD a multiple/divisor of num_kv_head can reconstruct what it needs.

Rule of thumb: both sides must satisfy tp_size ≥ num_kv_head. Above that threshold each rank holds at most one KV head, and the store's "one key = one head's bytes" invariant holds for both producer and consumer. Below it, each rank packs multiple heads into one key and the keyspace stops being head-indexed — lookups either over- or under-count what's actually there. Note that the failure modes are asymmetric: when only D is undersharded (§4.2) the lookup misleads you into thinking the cache is hot before the fetch fails; when only P is undersharded (§4.3) you fail fast at lookup. The first is worse to debug.

5. MLA: replicated cache short-circuit

Multi-Latent Attention (Deepseek-style) is special: its compressed KV is small enough that vLLM replicates it across TP ranks instead of sharding. So MLA has the same KV on every TP rank.

Implications for the transfer plane:

  • NIXL: compute_tp_mapping hits the MLA branch (nixl/tp_mapping.py:79-84) and uses a single remote rank for any local rank — no head-dimension split, no offset. With TPP=4 and TPD=2, D's rank 0 reads from P's rank 0 only, D's rank 1 reads from P's rank 1 only.
  • Mooncake pipe: there's a producer_cache_replicated flag threaded through the planning function. When true, only one local rank actually transfers (the local_tp_rank % tp_ratio == 0 rank); others no-op — replicated data, no need to ship twice.
  • Mooncake store: each TP rank still puts its own key, even though the bytes are identical — this gives the store its bandwidth budget back through dedup (batch_is_exist catches the duplicate before put).

6. GQA: deduplicate redundant shards

Grouped-Query Attention has fewer KV heads than Q heads. With num_kv_heads < tp_size, the convention is to replicate KV heads across TP ranks: 8 ranks but only 4 KV heads means ranks (0,1) share KV head 0, ranks (2,3) share KV head 1, etc.

NIXL handles this in compute_tp_mapping (nixl/tp_mapping.py:88-95): 

heads = np.arange(start, start + abs_tp) * total_num_kv_heads // remote_tp_size
_, unique_idx = np.unique(heads, return_index=True)
attn_ranks = (start + np.sort(unique_idx)).tolist()

heads computes which logical KV head each remote rank holds; np.unique keeps only the first rank per unique head. The result is the minimal set of remote ranks D must read from to cover all needed heads. Without this, D would issue redundant READs for the same bytes.

Both Mooncake variants achieve the same thing organically: the pipe via the same arithmetic on TP ratios, the store via tp_count = min(self.tp_size, self.num_kv_head) which caps the iteration at the actual number of KV heads.

7. Mamba: three-projection conv state

Mamba's KV "cache" is a recurrent state (SSM state plus conv state). With TP, conv state is decomposed into three projections — x, B, C — each TP-sharded along its own axis. When P and D have different TP sizes, each projection must be re-sharded individually.

NIXL handles this in _build_mamba_remote (nixl/worker.py:1054-1097): 

local_offset = self.tp_rank % effective_ratio
conv_offsets = self._conv_decomp.remote_conv_offsets(local_offset, tp_ratio)
# Each D rank reads its TP-shard of each conv projection

The SSM (recurrent) state is treated separately from conv state — ssm_ranks and attn_ranks are computed independently in compute_tp_mapping. A full-attention + Mamba hybrid model produces transfers from different remote ranks for the two block groups.

Mamba pre-emption hazard

Mamba state is non-block-aligned and not rewindable like attention KV. If P pre-empts and re-prefills, the state recomputation must reach the exact same numeric value as before, or D will get a torn transfer. The transfer plane assumes this works — see MAMBA.html for the assumptions it relies on. Don't disaggregate Mamba models you don't understand yet.

8. PP across P and D

Pipeline parallelism is much easier than TP for the transfer plane. Each PP stage owns a contiguous range of layers; the KV for layer L lives only on the PP rank that owns L. So a P cluster with PP=2 has rank 0 holding KV for layers [0..N/2) and rank 1 holding KV for layers [N/2..N).

All three transports embed the PP rank in their addressing:

  • NIXL: a P PP rank registers only its own subset of KV memory. The handshake exchanges per-layer info via block_lens. D's TP mapping naturally targets "P's rank that owns this layer's KV".
  • Mooncake pipe: bootstrap registry indexes by (dp_rank, tp_rank, pp_rank) (mooncake_utils.py:38-41); each PP rank registers its own ZMQ address.
  • Mooncake store: the key includes @pp_rank:k. PP is just another dimension of the key space.

The trickiness with PP shows up only when PPP ≠ PPD — e.g., P has PP=2, D has PP=1. Then D's single rank must fetch layer ranges from two P ranks. In practice this is rare (P and D usually agree on layer partitioning) and handled by the same per-rank descriptor machinery as TP.

9. DP and engine routing

Data parallelism runs multiple engines. Each engine has its own TP/PP layout, model state, and KV cache. With P/D disaggregation, you typically have multiple P engines and multiple D engines:

  • A frontend router decides which D engine gets the request.
  • That request's prefill happened on some P engine. The kv_transfer_params identifies which one (engine ID, host, port).
  • D's worker contacts that specific P engine. All connectors handle this by treating each engine pair as an independent handshake / agent.

In Mooncake pipe, the bootstrap server actually keys by dp_rank first, so multiple DP groups can share a single endpoint: 

# mooncake_utils.py:91-96
if payload.dp_rank not in self.workers:
    self.workers[payload.dp_rank] = EngineEntry(
        engine_id=payload.engine_id,
        worker_addr={},
    )

In the store, DP isn't even visible at the connector layer — every engine independently puts and gets keyed by content hash, and the store dedups across DP groups for free.

The big picture

All three connectors solve the same problem, but they place the complexity in different places. NIXL and Mooncake pipe pay it at handshake time (compute a plan, validate layouts, error early). Mooncake store defers it to lookup time (ask the store, take what's there). Each choice has consequences: precomputed plans are faster per request but rigid; lookup-driven plans tolerate cluster churn but make a chattier control path. Match the choice to your deployment.
