KV Cache: Prefix Caching Today & SWA Limits

How vLLM's two-tier prefix cache (local + connector) is wired, and why a tempting "share by suffix" scheme for sliding-window attention is silently unsound.

TL;DR

Part 1. Prefix caching has two stacked layers: KVCacheManager (local GPU) and an optional KVConnector (external). The connector reports additional tokens beyond the local hit, and the scheduler allocates fresh blocks for the connector to fill on the worker.

Part 2. "Share blocks across requests with the same sliding-window suffix" looks attractive but is wrong for any transformer of depth ≥ 2. K/V values at a position depend on the residual stream from layers below, whose receptive field reaches past the per-layer attention window. Today's chained hash + R→L scan returns exactly the hits that are provably correct — no more.

Part 1 — How prefix caching works today

There are two prefix caches in vLLM: the local GPU prefix cache (owned by KVCacheManager) and an optional external prefix cache (owned by a KVConnector). They're stacked, not separate paths — the connector adds tokens beyond what the local cache hit.

Cast of classes

Local cache side vllm/v1/core/

  • KVCacheManager — request-level API the scheduler calls. Wraps coordinator + block pool.
  • KVCacheCoordinator (Unitary/Hybrid) — fans the lookup across one or more KV-cache groups.
  • SingleTypeKVCacheManager — per-attention-type find_longest_cache_hit. FullAttention scans L→R, SWA R→L, etc.
  • BlockPool + BlockHashToBlockMap + FreeKVCacheBlockQueue — the physical layer: a hash→block index over an LRU free list, ref-counted blocks.
  • KVCacheBlock + hash_block_tokens — chained hash: SHA256(parent_hash, token_ids, extra_keys).

Driver & external cache

  • Scheduler.schedule() in vllm/v1/core/sched/scheduler.py — orchestrates the whole flow per step.
  • KVConnectorBase_V1 in vllm/distributed/kv_transfer/kv_connector/v1/base.py — two roles: scheduler-side (lookup, metadata) and worker-side (transfer).

Concrete connectors: Mooncake Store, LMCache, NIXL, SimpleCPUOffloadConnector, OffloadingConnector.

Per-request flow when prefix caching is involved

When a WAITING request is being promoted in Scheduler.schedule() (Phase 2), starting at scheduler.py:610:

  1. Local cache lookup 
    new_computed_blocks, num_new_local_computed_tokens = (
    self.kv_cache_manager.get_computed_blocks(request)
)

    get_computed_blocks → Coordinator.find_longest_cache_hit → per-group SingleTypeKVCacheManager.find_longest_cache_hit → walks block hashes through BlockHashToBlockMap. Returns the longest matched prefix as a list of already-cached KVCacheBlocks and a token count.

  2. External cache lookup (only if a connector is configured) 
    if self.connector is not None:
    ext_tokens, load_kv_async = self.connector.get_num_new_matched_tokens(
        request, num_new_local_computed_tokens
    )

    The connector reports how many additional tokens beyond the local hit it can supply (from peer GPU, CPU offload, NIXL fabric, LMCache, Mooncake, etc.) and whether the load needs to be async.

  3. Async path (scheduler.py:656) — if load_kv_async, schedule num_new_tokens=0 and the request stalls in a WAITING_FOR_REMOTE_KVS state until the connector reports transfer done. Only then does it become eligible for real allocation.
  4. Allocation 
    self.kv_cache_manager.allocate_slots(
    request, num_new_tokens,
    num_new_computed_tokens=num_new_local_computed_tokens,
    num_external_computed_tokens=num_external_computed_tokens,
    new_computed_blocks=new_computed_blocks,
    ...,
)
    • BlockPool.touch() reactivates locally-hit blocks (removes them from the free queue, increments ref_cnt).
    • BlockPool.get_new_blocks() pops new blocks for the uncached remainder — including the range the connector promised to fill. Evicts cached blocks via _maybe_evict_cached_block if needed.
    • The new "external" blocks are allocated empty; the connector will fill them on the worker.
  5. Connector hook for staging the transfer 
    self.connector.update_state_after_alloc(
    request, blocks, num_external_computed_tokens
)

    Tells the connector exactly which physical blocks it must populate and what tokens they correspond to.

  6. Per-step metadata 
    meta = self.connector.build_connector_meta(scheduler_output)

    Serialized into SchedulerOutput and shipped to workers.

Worker-side: the transfer actually happens

On each forward pass:

  1. bind_connector_metadata(meta) then start_load_kv(forward_context) — kicks off async receives for any pending external prefix.
  2. Per attention layer: wait_for_layer_load(layer_name) blocks until that layer's KV has landed in the paged buffer (model_executor/layers/attention/kv_transfer_utils.py:50). Attention then runs against blocks the manager allocated in step 4.
  3. On the producer side, save_kv_layer writes to the connector after each layer; wait_for_save() flushes at end-of-pass.

Closing the loop: blocks become local cache entries

After the forward pass, the scheduler calls KVCacheManager.cache_blocks() for completed prefill chunks. BlockPool.cache_full_blocks() hashes each filled block (hash_block_tokens chained from the parent) and inserts it into BlockHashToBlockMap — so the next request gets a local hit on this prefix without needing the connector at all.

Key idea External cache becomes local cache as a side effect of normal forward execution. The connector is purely additive — it skips compute when someone else already has the KV.

Async cleanup

When a request finishes, request_finished(request, block_ids) returns (delay_free_blocks, kv_transfer_params). If the connector is still pushing blocks somewhere (e.g., back to LMCache, or to a decode peer), the scheduler keeps ref_cnt > 0 on those blocks until get_finished() reports the transfer done — preventing use-after-free.

Layering summary

Scheduler.schedule()
  ├── KVCacheManager.get_computed_blocks()       # local hit (N tokens)
  │     └─ Coordinator → SingleTypeManager → BlockHashToBlockMap
  ├── connector.get_num_new_matched_tokens()      # external hit (+M tokens)
  ├── KVCacheManager.allocate_slots()             # touch local + new blocks
  │     └─ BlockPool.touch / get_new_blocks
  ├── connector.update_state_after_alloc()        # which blocks to fill
  └── connector.build_connector_meta()            # ship plan to workers

Worker forward
  ├── start_load_kv → wait_for_layer_load(L)      # attention waits per layer
  └── save_kv_layer(L) → wait_for_save()          # producer side

Post-forward
  ├── KVCacheManager.cache_blocks()               # external→local promotion
  └── (on finish) connector.request_finished()    # delay_free_blocks if async send

Part 2 — What SWA prefix caching can and cannot do

An earlier draft proposed a suffix-digest hash for SWA blocks: hash the window-bounded tail and absolute block position, and let two requests with the same suffix at the same positions share K/V. That is unsound for any transformer of depth ≥ 2. This section explains why and what the legitimate engineering wins are.

The correctness wall: K/V receptive field

At every layer L, K_L[p] = W_K · h_{L-1}[p] and V_L[p] = W_V · h_{L-1}[p]. K/V values are a function of the layer's input hidden state, not of the input tokens directly. The hidden state's dependency on tokens grows with depth:

Each SWA layer adds W−1 tokens of reach to the cone. After L = ⌈(N+1)/(W−1)⌉ SWA layers, every prior token is in the receptive field. One Full-attention layer anywhere below short-circuits the whole stack to [0, p] — that's why hybrid models (Gemma-3, etc.) cannot share suffixes even at layer 2.
The conflation that breaks suffix-digest The attention mask at layer L bounds which K/V entries Q_L[p] mixes with at that layer. The K/V values themselves are computed from h_{L-1}, whose receptive field reaches further back than the mask. The earlier draft conflated these two.

Two requests with identical window-bounded suffix tokens at identical positions but divergent earlier prefixes do not produce identical K/V at any SWA layer beyond the first of a pure-SWA stack. In Gemma-3's hybrid (5 SWA : 1 full), the moment any SWA layer sits above any full-attention layer, its K/V depends on the entire prefix.

At layer 1, the SWA mask at pos 8 reads positions 1..8 — identical tokens for A and B? No. A reads (1,2,3,4,5,6,7,8); B reads (99,99,99,99,5,6,7,8). Even though the slot at pos 8 looks shareable by "same recent tail," the layer-1 K/V at pos 8 already depends on tokens 1..4 — which differ. Block 2 (pos 8..11) cannot be reused.

Block 2 spans positions 8..11. Claim from the draft: A's and B's block 2 K/V are bit-identical because tokens at positions 4..11 match. They are not:

  • At SWA layer 1 in a pure-SWA stack: h_1[8] for A depends on tokens at [1..8] = (2,3,4,5,6,7,8,9); for B it depends on (99,99,99,5,6,7,8,9). Different.
  • Therefore K_2[8] = W_K · h_1[8] differs between A and B. Likewise for positions 9, 10, 11.
  • In Gemma-3, divergence happens even faster: one full-attention layer below pulls the entire prefix into h_{L-1}[8..11], so every SWA layer above has K/V depending on tokens 0..3.
Silent failure mode Returning A's cached block 2 K/V on a B request inserts numerically wrong values. Subsequent attention reads them. There is no kernel-level signal — the model just gets quietly worse on long conversations with shared tails. This is exactly the failure mode prefix caching is supposed to never have.

Why the symmetric refutation applies

The earlier draft refuted bounded-chain hashing by graph reachability: block i's hash transitively reaches block 0 via the W-stride chain, regardless of how many hops away.

The same argument refutes suffix-digest hashing, applied to the model's residual stream rather than the hash chain. Suffix-digest's input edges don't reach token 0, but the model's computation edges do. A correct fingerprint must cover a function's full dependency set; otherwise hits between distinct dependency-set values are aliasing collisions.

What today's design already extracts correctly

The current chained-hash + R→L scan in SlidingWindowManager.find_longest_cache_hit (single_type_kv_cache_manager.py:548) returns the legitimate hits:

  • Same-prefix continuation — multi-turn chat, branching from a shared system prompt. The prefix matches up to divergence, K/V is provably identical, the chained hash hits. The R→L scan terminates as soon as it has ceil((W−1)/B) contiguous matched blocks.
  • Window eviction via remove_skipped_blocks() — blocks that fall out of the SWA window are freed. SWA memory win, independent of hashing.
  • Cross-instance prefix sharing through connectors — identical prefixes hit across machines because connectors use the same chained hash.
Not leaving hits on the table The hits today's design omits are hits that would be unsound.
Code: where the R→L scan terminates 
# single_type_kv_cache_manager.py
# Search from right to left and early stop when a match is found.
for i in range(max_num_blocks - 1, -1, -1):
    if cached_block := block_pool.get_cached_block(...):
        # ... insert null blocks for positions before i
        computed.extend([block_pool.null_block] * i)
        computed.append(cached)
        break
Earlier slots get filled with null_block because the SWA layer literally won't read them — and those positions' K/V would have been wrong anyway under suffix-digest.

Aside: L→R vs R→L scanning in find_longest_cache_hit

FullAttentionManager → L→R

A full-attention layer at position p attends to all positions [0, p]. Once any block diverges from cache, every subsequent block must also be wrong (its parent hash differs). Scan from block 0, stop at first miss. 

block_size=4, blocks B0..B2
cache: B0=HIT, B1=HIT, B2=MISS
L→R: B0 hit → B1 hit → B2 miss → stop.
Matched = 8 tokens.

SlidingWindowManager → R→L

SWA at position p only attends to [p − W + 1, p]. The block at the request end only needs ceil((W−1)/B) contiguous matched blocks ending at its position; earlier matches are irrelevant. Scan from the last block, accumulate contiguous hits, stop when you have W of them.

Position encoding: a smaller wall behind the receptive-field wall

RoPE applied to K bakes absolute position into the cached tensor for almost every modern SWA model (Llama, Mistral, Gemma-3, Qwen). With the receptive-field wall correctly stated, RoPE is just a tightening — even on a pure-ALiBi or learned-relative model where RoPE wouldn't exclude same-suffix-different-position cases, suffix-digest still aliases across divergent prefixes because the K/V values themselves differ. RoPE excludes a strictly smaller set than the receptive field does.

Where the legitimate engineering wins are

1. Sparser SWA block allocation (the biggest win)

HybridKVCacheCoordinator allocates blocks in lock-step across groups. An SWA layer fundamentally only needs ceil(W/B) blocks live at any moment per request. Today the allocator over-allocates and remove_skipped_blocks evicts after the fact — peak allocation is still proportional to context length per SWA layer.

A streaming allocator that never allocates blocks for out-of-window positions in SWA groups would shrink peak memory without changing correctness or hashing. For Gemma-3 with 5:1 SWA:full ratio, peak SWA-group memory drops from O(N) per layer to O(W) per layer.

2. R→L scan micro-optimization

The TODO at single_type_kv_cache_manager.py — skip-ahead by ceil(W/B) blocks rather than one at a time when there's a miss. Orthogonal to hashing.

3. Connector SWA coverage

Cross-instance K/V reuse for SWA group blocks is just as valid as for full-attention blocks (both rely on identical prefixes). Confirm Mooncake / LMCache / NIXL index SWA group blocks symmetrically (using the existing chained hash with group_id plumbed through). No hash change needed.

What this doesn't help: Mamba / RWKV / pure linear attention

Mamba's (h, conv_state) at position p is a function of all prior tokens with exponentially-decaying weights, not a literal window. There's no token-level "fall-off" point. MambaManager.find_longest_cache_hit already does what's possible: longest-prefix match with checkpoint replay, R→L scan. Windowed hashing doesn't apply.

The only way to extend prefix sharing for true linear attention is state checkpointing across requests — and that's already chained-hash-equivalent. The wins there are connector-side. See LINEAR_ATTENTION.html.

Test ideas before any future windowed-hash proposal

Failures here are silent Receptive-field aliasing has no kernel-level trip-wire. Any new SWA hash scheme must be guarded by:
  • Two-request K/V comparison at every SWA layer: A = [X*N, Y*M], B = [X'*N, Y*M] with X ≠ X'. Compute K/V at every SWA layer for the Y range. They must differ; any scheme that returns a hit for B after A primed the cache is unsound.
  • Numerical equivalence test: run B's prefill cold, then again with the cache primed by A's run. Outputs must be bit-identical to cold. If a windowed hash is wired in incorrectly, they won't be — and the difference may be small enough that eval scores barely move.

Build order

  1. Sparser SWA block allocation — measure peak memory delta on long-context Gemma-3 / Mistral workloads. No correctness risk.
  2. R→L scan micro-opt — close the TODO.
  3. Connector SWA audit — confirm Mooncake / LMCache / NIXL handle SWA group blocks correctly for identical-prefix sharing.
There is no step 4 that recovers a suffix-digest-style "win." It does not exist for transformers of depth ≥ 2.

Risks if anyone proposes a windowed hash again

  • Reviewers must demand the K/V-comparison + numerical-equivalence tests above before any new SWA hash scheme lands.
  • Anyone proposing per-layer or per-group hash variants must enumerate the dependency set of each cache entry's contents and prove the hash covers it. "The mask only reads W tokens" is not the dependency set; the dependency set is the receptive field of the residual stream feeding K/V.
  • extra_keys (LoRA, MM hash, cache salt) must remain part of every group's key regardless of scheme. See KV_CACHE_ASSUMPTIONS.html §13.
  • Hybrid models (Gemma-3) make the constraint stricter, not looser: even if a windowed hash were sound for pure-SWA layers, the moment a full-attention layer appears below an SWA layer, that SWA layer's K/V depends on the full prefix and must use the chained hash.
