A Beginner's Guide to vLLM's KV Cache

The on-ramp. Builds intuitions from scratch with small examples, so the other docs read as confirmations rather than revelations.

TL;DR

The cache exists because attention is repetitive: same prefix ⇒ same K/V. Blocks (16 tokens) are the unit of allocation, kernel reads, and hashing. A chained hash gives provably-correct prefix matches. Hybrid models split layers into groups by lifecycle, and a column-packed tensor layout pools physical memory across groups. The single-threaded scheduler is what lets ref-counting, the free list, and the hash table all stay lock-free.

1. The problem: attention is repetitive, on purpose

When a transformer generates text token by token, each new token's attention layer needs to look back at every previous token. At step 100 the model attends to tokens 0..99; at step 101 it attends to 0..100; and so on.

If you computed everything from scratch each step, you'd redo the K and V projections for tokens 0..99 a hundred times. Those projections don't change — token 5's K vector at layer 7 is the same on step 100 as it was on step 6. So we cache them.

What the KV cache is At each layer, for every token we've seen, we save the K and V tensors that attention computed. Next step, instead of recomputing them, we read them out of memory. Every design decision below is in service of "store K and V efficiently and find them again quickly."

A concrete picture

Cache size depends on how the model arranges its attention heads. Three common designs, sized to a 32-layer model with hidden size 4096 and head dim 128, fp16:

Full MHA

  • 32 KV heads
  • 1 token, 1 layer: 32 × 128 × 2B = 8 KiB
  • 1 token, all 32 layers: 32 × 2 × 8 KiB = 512 KiB
  • 4k context: ~2 GiB

GQA (today's default)

  • 32 Q heads, 8 KV heads
  • 1 token, 1 layer: 2 KiB
  • 1 token, all layers: 128 KiB
  • 4k context: ~512 MiB (4× smaller)

MLA (DeepSeek)

  • Latent dim 512 + RoPE dim 64
  • 1 token, 1 layer: ~1.1 KiB

2. Why we allocate in blocks, not per token

Naively you'd think one slot per token. The real design hands out fixed-size blocks — typically 16 tokens at a time. A 33-token request gets 3 blocks, not 33 slots, and the third block has 15 empty positions waiting to be filled.

1. Scheduler bookkeeping

Allocating and freeing blocks needs O(1). With per-token slots you'd need a bitmap or per-request arena and either fragmentation or scan cost. Fixed blocks ⇒ one free list of identical slots.

2. Attention kernel

GPU kernels read K/V in tiles. FlashAttention issues one memory load per 16-token chunk. Per-token allocation would scatter those 16 tokens across the GPU, forcing 16 indirect lookups.

3. Prefix-cache hash

vLLM caches prefixes by hashing block contents. Hashing a fixed-size unit is cheap and chainable. A token-level hash would multiply hash-table entries by 16× with no benefit.

Analogy: a hotel books rooms, not square feet. The room has 4 beds and you might use 1, but treating every square foot as bookable would dwarf the wasted space.

3. The prefix cache: same prefix, same K/V

Determinism is the whole story K and V at position p are deterministic functions of the effective inputs at positions 0..p. Same effective input in (model weights, LoRA, MM features, cache namespace, KV group), same result out.

So if request A has been processed and its blocks are still in GPU memory, request B with the same prefix can point at A's blocks and skip computing them.

Real-world payoff

The classic case is a chat application with a long system prompt:

System: You are a helpful assistant. [...3000 more tokens...]
User: Hi

System: You are a helpful assistant. [...same 3000 tokens...]
User: What's the weather?

These share 3000+ tokens. Prefix caching skips the prefill for them on every new conversation.

How do we know two prefixes are identical?

We can't compare token sequences directly — that's O(N) per lookup. Instead vLLM hashes each block, chaining each hash through the previous: 

block 0 hash = H(NONE_HASH, "The cat sat on the")
block 1 hash = H(block_0_hash, " mat. The mat was")
block 2 hash = H(block_1_hash, " green. It was a")

The real hash key also includes extra keys when tokens alone aren't enough: LoRA/adapters, multimodal feature identifiers, prompt embeddings, request cache_salt.

    global NONE_HASH
    hash_seed = os.getenv("PYTHONHASHSEED")
    if hash_seed is None:
        NONE_HASH = BlockHash(os.urandom(32))   # <-- random!
    else:
        NONE_HASH = BlockHash(hash_fn(hash_seed))
This prevents one tenant from poisoning another's cache by guessing hashes — the same trick CPython uses for hash().

4. Why we hash after the forward, not before

You might think: "If I know the tokens, I can hash up front and look them up before running the model." vLLM does compute the hash up front for lookup — but it only inserts a (hash → block_id) entry into the cache table after the worker has actually written the K/V contents.

Subtle but critical The hash is a key into K/V content, not into block IDs. If you inserted the hash before computation, two requests racing through prefill at the same time could both think they have valid K/V at a given block when really the block hasn't been written yet. You'd hand out garbage to whoever looked up second.

So vLLM splits it in two: cheap CPU-side hash up front for lookup, deferred insertion after the GPU finishes writing. Every K/V you read from the cache is K/V that was actually computed.

→ KV_CACHE_ASSUMPTIONS.html §5.

5. What sharing buys, and where it stops

Prefix caching only works for identical prefixes. If A is [1, 2, 3, 4] and B is [1, 2, 3, 5], they share through token 3; as soon as one block diverges, every subsequent block diverges too (its hash includes the parent's hash).

A natural question: "What about similar suffixes? In sliding-window attention (SWA), where each token only attends to the last W tokens, if both requests have the same last W tokens, why can't they share?"

No, never — for any transformer of depth ≥ 2 Even though the attention mask at layer L only reads the last W positions, the K/V values at those positions depend on the residual stream from earlier layers, which depends on tokens further back. A 1-layer SWA network could share suffixes; a 2-layer SWA network cannot. The receptive field of K/V contents expands with depth even though the per-layer mask doesn't.

This is the receptive-field wall. It rules out a whole class of "clever" cache-sharing schemes that look like they should work and silently corrupt outputs if implemented. The current chained hash + R→L scan returns exactly the hits that are provably identical — no more.

→ KV_CACHE.html "The correctness wall" has the full argument with worked counterexample.

6. Why different attention types live in different "groups"

Modern models often mix attention types in one stack. Gemma-3 alternates SWA layers with full-attention. Jamba mixes Mamba layers with attention. These types disagree about how their cache behaves:

Property Full attention Sliding window Mamba
When can K/V be freed?When request endsWhen position falls out of windowAfter each step (default mode)
How to find cache hits?Scan L→RR→L, take last W blocksR→L, take latest only
What's a "block"?16 tokens of K/V16 tokens of K/VOne snapshot of recurrent state

You can't put these into a single shared block table. Each cell would mean different things in different rows. So vLLM partitions layers into groups where every layer in a group agrees on those answers.

Why not one block table per layer?

That's the safe extreme. It also wastes huge GPU memory: an 80-layer model at 128k context burns ~10 GiB on block-table metadata alone (the block table is on GPU because the kernel reads it directly). Layers within the same attention type have identical lifecycles — their block tables would be byte-identical — so making them share is free.

Example: request A with 12 tokens and block_size=4 owns three logical blocks. All full-attention layers (L0, L7, L31) want the same logical mapping. K/V contents differ per layer, but "which physical block stores logical block 2 for A?" is identical.

SWA layers have a different lifecycle, but the sharing logic holds within the SWA group:

full-attention group table: [47, 89, 153]
SWA group table:            [NULL, 89, 153]

Why split SWA layers across multiple groups in Gemma-3?

If Gemma-3 has 5 SWA layers per 1 full-attention layer, putting all 50 SWA layers in one group with 10 full-attention layers in another means "a block" in the SWA group would represent 5× more memory than in the full-attention group. The scheduler could no longer ask "do I have N free blocks?".

vLLM's solution: split the 50 SWA layers into 5 groups of 10. Now every group has 10 layers, every block represents the same physical memory, and one global free list works for everyone.

→ KV_CACHE_ALLOCATION.html "Why Groups Exist At All" for the full reasoning chain.

The three invariants you can read off this example
  1. Block tables are the same size across groups — both are 4 entries long for a 16-token request, because both groups agree on block_size = 4.
  2. NULL entries fill SWA's expired positions — the SWA layer literally won't read those positions, so no physical block is allocated there.
  3. Block IDs come from one global pool — G1 holds {1, 2, 3, 4}, G2 holds {NULL, NULL, 5, 6}, all drawn from the same BlockPool with no overlap. That's the scheduler invariant that makes column packing collision-free.

Same physical address, three lifecycles. The slot isn't tied to a layer or group — whoever currently holds block_id = 42 writes there. Between owners (T2), the bytes stay but are unreferenced. This is why ref counting matters: it tracks who is reading, not who wrote.

Why this matters (it's not the allocation count)

Per-group layout would still have 6 block tables. Column packing's win is cross-group physical-memory pooling: one slot 42 reused across groups, not 6 slot-42s sitting in 6 partitioned tensors. 8. Why blocks are stored block-major in memory

Inside one block (e.g., block 42 in Tensor 0), the layout is:

[2, block_size, num_kv_heads, head_size]
 ^   ^           ^             ^
 |   |           |             head dim
 |   |           num KV heads
 |   token-within-block (0..15)
 K vs V

The 16 tokens are laid out contiguously. This is "block-major", the opposite of "token-major" (where token 0 of block 0 would sit next to token 0 of block 1).

Why block-major? The kernel reads a block at a time. FlashAttention issues one contiguous load for all 16 tokens' K and V. Block-major makes that load a single linear stride. Token-major would force scattered reads.

Block-major also makes copies cheap: one memcpy per block per layer of page_size bytes.

→ KV_CACHE_ASSUMPTIONS.html §3.

9. The free list is doubly-linked, and that's not arbitrary

When a block becomes unused, it goes onto a free queue: candidates for eviction. The free queue is a hand-rolled doubly-linked list — not Python's collections.deque.

Why? You sometimes need to pull a block out of the middle:

Scenario

  • Request A finishes; blocks (still holding valid K/V) go to the free queue.
  • Request B arrives with the same prefix — hits the cache, wants blocks currently in A's freed list but not yet evicted.
  • B's block lookup pulls those blocks out of the middle in O(1) and increments their ref count.

A deque only supports O(1) operations at the ends. Middle removal would be O(N).

The eviction-order trick

When request A frees its blocks, they're added in reverse order (tail block first, head last). Tail blocks are the least likely to be hit by future requests; head blocks (system prompt, shared prefix) might serve many. Reverse insertion puts tails at the front of the eviction queue.

10. Reference counting without locks

When two requests share a cached block, vLLM tracks a ref_cnt per block — literally block.ref_cnt += 1 on a Python attribute. No atomics, no locks.

Why this is safe The scheduler is single-threaded. Every code path that touches ref_cnt runs inside Scheduler.schedule() or Scheduler.update_from_output(), and there's exactly one scheduler thread per engine. No race possible.

This is load-bearing for performance: at thousands of blocks per step, even a threading.Lock per touch would show up in profiles. The single-threaded scheduler also makes the doubly-linked free list safe and the hash table lock-free.

If anyone proposes to parallelize the scheduler … they'll touch all three of these at once.

→ KV_CACHE_ASSUMPTIONS.html §8 and the "Patterns" section at the bottom.

11. When memory runs out: preempt the youngest

Eventually a request can't allocate the blocks it needs. The scheduler preempts another running request — kicks it out of running state, frees its blocks, sends it back to WAITING.

Counterintuitively, FCFS preempts the youngest The oldest running request has the most computed K/V (longest context, most blocks held). Preempting it would throw away tens of thousands of tokens of work. Preempting the youngest throws away the least.

Preempted requests retry, often hitting the cache for their own already-computed prefix on the second try. Preemption is conservative and reversible; eviction (dropping cache entries entirely) is more aggressive.

→ KV_CACHE_ASSUMPTIONS.html §12.

12. Two sentinels: null blocks and PAD_SLOT_ID

null block (is_null=True)

Appears in block tables for SWA or Mamba groups when a request has logical positions that the layer won't read. Null blocks have ref_cnt=0 permanently and never enter the free queue.

PAD_SLOT_ID = -1

Appears in the per-token slot_mapping for tokens that exist in the batched tensor but aren't real (CUDA-graph padding, decode positions past a request's end). Backends treat -1 as invalid and skip the write.

Why both, and why -1? Null blocks live in the block table (group-level metadata) and mean "this position has no storage." PAD_SLOT_ID lives in the slot mapping (per-token) and means "this token isn't real." Different problems, different sentinels. -1 specifically because 0 is a valid block ID and a valid slot ID.

→ KV_CACHE_ASSUMPTIONS.html §4.

13. Linear attention (Mamba, RWKV) is a different problem

Everything so far assumed softmax attention: every past token's K/V is addressable by index. Linear attention doesn't work that way. It compresses the entire history into a fixed-size state tensor: 

S[p]   = f(S[p-1], x[p])           # state evolves
out[p] = g(S[p], x[p])             # output read from state
No positional structure You can't ask the state "what was K at position 47?" — the information has been irreversibly mixed into the running state. You can only resume from a checkpoint.
AspectSoftmax attentionLinear attention
Memory per requestO(N)O(1) running, O(N/block_size) with prefix cache
Position lookupYesNo
Partial sharingPer-block (every 16 tokens)Per-checkpoint only
Hit granularityContinuous run of cached blocksLatest single checkpoint

vLLM still uses the same BlockPool, same hashing, same group abstraction for Mamba layers — but the contents of a "block" mean something completely different. A Mamba block holds "one snapshot of the recurrent state," and "block size" becomes the interval between snapshots, not the number of cached tokens.

MambaManager.find_longest_cache_hit reflects this: scans right-to-left and returns just one matching checkpoint, filling all earlier slots with null blocks.

→ LINEAR_ATTENTION.html and MAMBA.html for the full story.

Steps 1-2 happen on CPU before the GPU is touched. Step 6 is the only GPU-heavy step — it processes only the new block. Step 7 is the deferred insert that makes prefix caching correct (see §4).
  1. Lookup the prefix cache. Compute block hashes: 
    block_0 hash = H(NONE_HASH, [A B C D])
block_1 hash = H(block_0, [E F G H])
block_2 hash = H(block_1, [I J K L])
    Walk through BlockHashToBlockMap. Suppose blocks 0 and 1 hit (physical block IDs 47, 89). Block 2 misses.
  2. External cache. If a connector is configured, ask whether peers/CPU/storage have block 2. Suppose no — proceed.
  3. Allocate. Touch blocks 47, 89 (pull out of free queue, increment ref counts). Pop a fresh ID — say, 153. Block table: [47, 89, 153].
  4. Schedule. Write block table to GPU. Compute slot mapping for tokens needing processing (only block 2).
  5. Forward pass. Kernel reads K/V for cached prefix from 47 and 89, computes K/V for block 2, writes to block 153's slots.
  6. Cache the new block. After forward fills block 153, vLLM inserts (block_2_hash → 153) into BlockHashToBlockMap.
  7. Continue decoding. Each new token writes K/V to whichever block has space. When a block fills, hash it and add to the table.
  8. Finish. Decrement ref counts. Blocks whose ref count hits zero go to the free queue (in reverse order). If a future request hits their hash, they're pulled out and reused.

15. The big takeaways

  1. The cache exists because attention is repetitive — same prefix means same K/V, so don't compute it twice.
  2. Blocks are the unit of everything. They make allocation O(1), kernels efficient, and hashing chainable.
  3. The chained hash is the key correctness mechanism. Anything weaker silently corrupts outputs.
  4. Groups partition layers by lifecycle. Different attention types need different block tables, scan directions, eviction rules.
  5. The uniform-block-size invariant is what lets one global block ID space serve all groups. Column-packed allocation builds on it.
  6. The single-threaded scheduler is load-bearing for ref counting, the free list, and the lock-free hash table.
  7. Linear attention reuses the allocator but has totally different semantics. Mamba blocks store state checkpoints, not K/V.
  8. Failures here are silent. No kernel-level alarm if a wrong block gets returned. That's why the code is dense with assertions.
