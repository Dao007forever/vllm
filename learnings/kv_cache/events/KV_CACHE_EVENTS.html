KV Cache Events — vLLM, MooncakeStore & Nvidia Dynamo

What vLLM emits when a KV block is cached or evicted, what the MooncakeStore connector adds on top, and the concrete wiring it takes to feed those events into Nvidia Dynamo's KV-cache-aware router. Code references target the tree at vllm/distributed/kv_events.py and .../mooncake/store/.

TL;DR

vLLM emits exactly three KV-cache event types — BlockStored, BlockRemoved, AllBlocksCleared — from the BlockPool, batched per engine step and published over a ZMQ PUB socket (default tcp://*:5557) as msgpack.

The MooncakeStore connector reuses the same BlockStored struct but tags it medium="cpu", aggregates per-worker events into common events, and merges them into the same publish stream. It emits stores only — no removes.

Dynamo integration is mostly already done for you: vLLM's event schema was contributed by the Dynamo team, so its three event types map 1:1 onto Dynamo's. The cleanest path is Dynamo's KvEventPublisher in ZMQ-relay mode, pointed at vLLM's :5557. The one knob that matters: block_hashes must be cumulative sequence hashes as signed i64 — which vLLM already does by default via VLLM_KV_EVENTS_USE_INT_BLOCK_HASHES=1.

Part 1 — vLLM's native KV cache events

KV-cache events are an observability / external-indexing side channel. They do nothing for vLLM's own correctness — the prefix cache works whether or not events are enabled. Their job is to tell an external consumer (a router, a global prefix index, a metrics sink) which content-addressed blocks each engine currently holds, so that consumer can route a new request to the engine that already has the longest matching prefix.

One source of truth All three event types are defined in vllm/distributed/kv_events.py. They are msgspec.Struct subclasses with array_like=True and tag=True — so on the wire each event is a compact tagged array, not a JSON object. The class name is the tag.

1.1 The three event types

BlockStored

Emitted when one or more full blocks become cached (content-addressable). Carries everything an external indexer needs to reconstruct the block key.

medium = "GPU"

BlockRemoved

Emitted when a cached block is evicted from the pool (LRU reuse). Only the block hashes + group are needed — the consumer already knows the content.

medium = "GPU"

AllBlocksCleared

Emitted on a full prefix-cache reset. An empty struct — "drop everything you think I have." Consumer wipes its index for this worker.

BlockStored schema

FieldTypeMeaning
block_hasheslist[bytes|int]Cumulative sequence hashes, one per new block. Each hash folds in all tokens from the start of the sequence up to and including that block — this is what makes cross-request prefix matching work.
parent_block_hashbytes|int | NoneHash of the preceding block in the chain. None for the first block in the sequence.
token_idslist[int]The actual token IDs covered by the new blocks (range num_cached_blocks*bs : num_full_blocks*bs).
block_sizeintTokens per block.
lora_id / lora_nameint|str|NoneLoRA adapter identity folded into the hash (so adapters never alias the base model). lora_id is deprecated in favor of lora_name.
mediumstr|None"GPU" for native events; "cpu" for MooncakeStore. Lets a consumer classify by tier.
extra_keyslist[tuple|None]|NonePer-block extra hash inputs (MM identifiers, cache_salt, prompt-embedding hashes…) so external consumers can reconstruct the block hash themselves.
group_idxint|NoneKV-cache group (hybrid models: full-attn vs sliding-window vs Mamba groups hash into different namespaces).
kv_cache_spec_kind / _sliding_windowstr|int|NoneCache-spec metadata so a consumer can filter/classify groups as it learns them. Store events carry it; remove events don't need it.

BlockRemoved is just { block_hashes, medium, group_idx }; AllBlocksCleared has no fields. All three define __hash__ so they can be counted by the aggregator (see §1.6).

1.2 Where they are emitted

Every event originates in the BlockPool (vllm/v1/core/block_pool.py), gated behind enable_kv_cache_events. The pool appends to an internal kv_event_queue list; nothing is published from here.

EventTrigger siteWhat just happened
BlockStoredcache_full_blocks() — block_pool.py:315One or more freshly-filled blocks got inserted into cached_block_hash_to_block. The event records the new hashes, parent, token range, and per-block extra keys.
BlockRemoved_maybe_evict_cached_block() — block_pool.py:392A block was popped off the free queue for reuse and its old hash had to be unregistered. The eviction is the removal.
AllBlocksClearedreset_prefix_cache() — block_pool.py:485Every block hash was reset (admin reset / cache flush).
Eviction ≈ removal, lazily A BlockRemoved is only produced when a block is recycled (get_new_blocks → _maybe_evict_cached_block), not the instant a request frees it. A freed-but-still-cached block stays in the index (and is still reusable) until pressure forces its reuse. Consumers must not assume "request finished" implies "blocks removed."

1.3 Collection → batch → publish

The Scheduler drains the queue once per engine step and publishes a single batch. The hop chain:

1 BlockPool appends to self.kv_event_queue  # during alloc/evict/reset
2 KVCacheManager.take_events() → block_pool.take_events()  # atomically drains + clears — kv_cache_manager.py:510
3 Scheduler.update_from_output()  # scheduler.py:1554

The publisher is created once in the scheduler from KVEventsConfig (scheduler.py:137), via EventPublisherFactory.create(). If events are disabled the factory returns a NullEventPublisher (a no-op), so the take_events() calls all short-circuit.

1.4 Wire format & config

ZmqEventPublisher (kv_events.py:274) runs a daemon thread that pulls batches off an in-memory queue, msgpack-encodes them, and sends a 3-frame multipart message:

frame 0   topic       # default "" (subscribe-all)
frame 1   seq         # monotonic uint64, 8 bytes big-endian — for gap detection
frame 2   payload    # msgpack of EventBatch = [ts, [events…], data_parallel_rank]
  • Reliability: at-least-once with monotonic seq. A ROUTER replay socket (optional replay_endpoint, default :5558) keeps the last buffer_steps batches in a deque; a subscriber that detects a gap requests missed batches by starting sequence number.
  • Back-pressure: a high-water mark (hwm=100k) and bounded queue (max_queue_size=100k) — if a consumer can't keep up, events drop rather than blocking the engine.
  • Data parallelism: each DP rank runs its own publisher and stamps data_parallel_rank on every batch; the endpoint port is offset by the rank (offset_endpoint_port), so rank k publishes on 5557+k.

Config lives in vllm/config/kv_events.py (KVEventsConfig):

# Enable + publish over ZMQ
--kv-events-config '{"enable_kv_cache_events": true, "publisher": "zmq", "endpoint": "tcp://*:5557"}'

When enable_kv_cache_events is true and publisher is unset, __post_init__ defaults it to "zmq"; otherwise "null".

1.5 The block-hash subtlety (this is what makes routing work)

The single most important field for any external router is block_hashes. Two properties make it usable:

  1. Cumulative / chained. A block's hash folds in its parent's hash, so identical prefixes across two different requests produce identical block hashes — that's the whole basis for prefix-overlap routing.
  2. Reproducible across processes. So that worker A and worker B compute the same hash for the same prefix. This requires a shared seed.

ExternalBlockHash is bytes | int (kv_cache_utils.py:52). maybe_convert_block_hash() (:77) decides which: 

def maybe_convert_block_hash(hash_bytes):
    if not envs.VLLM_KV_EVENTS_USE_INT_BLOCK_HASHES:
        return hash_bytes              # raw 32-byte sha256
    return int.from_bytes(hash_bytes, "big") & ((1 << 64) - 1)  # unsigned 64-bit
Default is already Dynamo-friendly VLLM_KV_EVENTS_USE_INT_BLOCK_HASHES defaults to 1 / True (vllm/envs.py:240). So out of the box vLLM emits a truncated unsigned 64-bit int, whose bit pattern is exactly the signed i64 a radix-tree indexer wants. You do not normally have to set anything here.
PYTHONHASHSEED across workers The chain is seeded by NONE_HASH, initialized in init_none_hash() (kv_cache_utils.py:97). If PYTHONHASHSEED is unset, the seed is os.urandom(32) — different per process, so two workers will hash the same prefix differently and the router's cross-worker matching breaks. For multi-worker routing every engine must run with the same fixed PYTHONHASHSEED (and identical model / config).

1.6 Multi-worker aggregation

Under TP/PP a "block" is sharded across N workers, so N workers each emit the "same" store event. KVEventAggregator (kv_events.py:116) counts events by their __hash__ and exposes get_common_events() — events seen from all workers — so a downstream index records each logical block once. This is the machinery the MooncakeStore connector reuses.

Part 2 — What MooncakeStore emits

The MooncakeStoreConnector (.../mooncake/store/connector.py) uses MooncakeDistributedStore as a shared, off-GPU KV pool. When a request's blocks are written to the store, the connector emits its own BlockStored events so an external index knows the prefix is also available in the shared tier — not just on a single GPU.

What it emits, and what it doesn't

✓ Emits

  • BlockStored with medium = "cpu"
  • Built in the worker during batch_put — store/worker.py:562
  • parent_block_hash chained within a group (prev_key_per_group), matching native semantics
  • block_size = original_block_size, token IDs for the range

✗ Does not emit

  • No BlockRemoved — eviction from the store is internal to Mooncake (CPU/disk tiering) and not surfaced to vLLM.
  • No AllBlocksCleared.
  • No extra_keys / group_idx / spec metadata (the store-side struct is the minimal form).

The aggregation path is different from native

Native GPU events come straight off the BlockPool. Connector events take a longer road because they originate on the worker side and must be funneled back to the scheduler and deduplicated across workers:

1 Worker writes blocks → builds BlockStored(medium="cpu"), buffers via update_kv_event()  # worker.py:619
2 get_kv_connector_kv_cache_events() wraps them in a MooncakeStoreKVEvents (num_workers=1)  # connector.py:266
3 Events ride back in KVConnectorOutput → scheduler-side update_connector_output()  # connector.py:196
    # merges per-worker batches, increments the worker count
4 connector.take_events() calls .aggregate() → keeps only events seen by all workers  # connector.py:211
5 Scheduler appends them to the same KVEventBatch as GPU events → same ZMQ socket

MooncakeStoreKVEvents (connector.py:47) is a thin KVConnectorKVEvents wrapper around the same KVEventAggregator. So the end result is a single, unified event stream on :5557 in which a consumer distinguishes tiers by the medium field: GPU = on this engine's HBM, cpu = in the shared Mooncake pool.

Why "common events"? A block is only truly "stored in the shared pool" once every shard has written its slice. Counting and keeping only events emitted by all workers prevents the index from advertising a prefix that's only partially persisted.

Part 3 — Integrating with Nvidia Dynamo KV-cache-aware routing

The punchline first vLLM's KV-event infrastructure was contributed by the Nvidia Dynamo team. The three event types, the ZMQ multipart framing, and the [ts, [events], dp_rank] msgpack payload were designed to be exactly what Dynamo's router consumes. So "integration" is mostly configuration and a relay, not new code in vLLM.

3.1 What Dynamo's KV router needs

Dynamo's KV router picks the worker with the best prefix overlap for an incoming request, trading that against decode load. Its cost function is roughly cost = overlap_score_weight · prefill_blocks + decode_blocks. To compute overlap it maintains a KvIndexer backed by a radix / prefix tree of cached blocks, keyed by block hash, one branch per worker. To keep that tree current it consumes a live store/remove event stream from every worker:

  • Store events → insert (parent_hash → block_hash) edges into the tree under that worker.
  • Remove events → prune those nodes.
  • Cleared → drop the worker's whole branch.

Block hashes must be cumulative sequence hashes (so a shared prefix lands on a shared path) and deterministic across workers. vLLM satisfies both, as covered in §1.5.

3.2 Event mapping is 1:1

vLLMDynamoNotes
BlockStored→BlockStored / KvCacheStoreDataDynamo reads block_hashes, parent_block_hash, token_ids, block_size, lora_name.
BlockRemoved→BlockRemovedJust block_hashes.
AllBlocksCleared→AllBlocksClearedEmpty.

Dynamo even documents the ZMQ wire format as "compatible with SGLang / vLLM": the same 3 frames (topic, 8-byte big-endian seq, msgpack [ts, [events], dp_rank]), each event tagged by type.

One real serialization nuance vLLM's structs are array_like=True, tag=True, so each event is actually a tagged array ([tag, field0, field1, …]), not a {"type": …} dict — Dynamo's docs describe the logical dict form. The Dynamo relay decoder is built against vLLM's exact field order, so this works in practice; just don't expect to hand-parse the payload as JSON objects.

3.3 Two integration modes

A. ZMQ relay — recommended for vLLM

Dynamo's KvEventPublisher runs in relay mode: it subscribes to vLLM's existing PUB socket and forwards events onto the Dynamo event plane (NATS or ZMQ) → KvIndexer. No engine code changes.

from dynamo.llm import KvEventPublisher
KvEventPublisher(
    component=component,
    kv_block_size=16,          # must match vLLM block size
    zmq_endpoint="tcp://127.0.0.1:5557",
    zmq_topic="",              # all topics
)
vs
B. Direct publishing

For a custom engine with no ZMQ event system: call publish_stored() / publish_removed() yourself. Not needed for vLLM — vLLM already has the ZMQ publisher, so relay is strictly less work and avoids duplicating the hashing logic.

There is also a higher-level path used by Dynamo's own vLLM backend: DynamoStatLoggerPublisher (in components/src/dynamo/vllm/publisher.py), which hooks vLLM's StatLoggerBase to pull KV updates. The ZMQ relay is the decoupled, lower-overhead choice when you just want routing.

3.4 What it actually takes — checklist

  1. Enable events in vLLM. --kv-events-config '{"enable_kv_cache_events": true, "publisher": "zmq", "endpoint": "tcp://*:5557"}'.
  2. Keep int block hashes (default). Leave VLLM_KV_EVENTS_USE_INT_BLOCK_HASHES=1 so hashes are i64, not 32-byte blobs.
  3. Pin a shared PYTHONHASHSEED on every engine, with identical model and config, so cross-worker prefix hashes line up (§1.5).
  4. Match block size. Pass Dynamo kv_block_size / --kv-cache-block-size equal to vLLM's block size; mismatched block sizes make the radix tree meaningless.
  5. Point Dynamo's relay KvEventPublisher(zmq_endpoint=…:5557) at each engine, one per dp_rank (remember the 5557+rank offset).
  6. Run the router in KV mode. --router-mode kv (or DYN_ROUTER_MODE=kv); tune --router-kv-overlap-score-weight.
  7. (Optional) handle gaps. vLLM exposes a replay ROUTER socket; Dynamo's indexer does its own gap detection (last_recovered_event_id + tree dump), so durable replay is available but not required.

3.5 Gaps & gotchas

IssueWhy it matters
medium / group_idx not in Dynamo's core schemaDynamo's radix tree is fundamentally a per-worker, GPU-prefix model. vLLM's medium="cpu" (Mooncake) and hybrid group_idx are extra dimensions Dynamo's basic indexer doesn't natively branch on — a shared-pool prefix may be treated like any other cached prefix for that worker. Verify behavior before relying on tiered routing.
No store-side BlockRemovedMooncakeStore never emits removes (§2). The router's view of the shared tier only grows from vLLM's perspective; stale entries rely on Dynamo's own TTL/eviction, not vLLM signals.
Lossy transportPUB/SUB is fire-and-forget with an HWM that drops under back-pressure. A slow router silently loses events → index drift. Use the replay socket or Dynamo's recovery, and watch the publisher's queue.
Eviction lagBlockRemoved fires on reuse, not on free (§1.2). The router can briefly believe a worker holds a prefix it has logically released — usually harmless (overlap is a hint, not a guarantee) but worth knowing when debugging "why did it route there."
Hash-seed driftThe most common silent failure: forgetting PYTHONHASHSEED on one worker. No error — just zero cross-worker overlap and round-robin-like routing.
