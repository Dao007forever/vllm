EAGLE + SWA: caching the lookahead block

Why a memory optimization for sliding-window prefix caching (#42258) silently broke all prefix-cache hits under EAGLE/MTP speculative decoding, and how #44082 caches exactly the one extra block EAGLE needs — no more.

TL;DR

In a hybrid model, a sliding-window (SWA) group only needs the last tail blocks of each lcm_block_size-aligned segment to serve a prefix-cache hit. #42258 stopped writing the other blocks of each segment into the prefix-cache hash map to save memory. But EAGLE/MTP needs tail + 1 contiguous cached blocks — it matches one block past the aligned boundary, then drops it. That lookahead block lives at the first position of the next segment — exactly the position the mask skipped. Result: EAGLE + SWA found zero cache hits. #44082 fixes this by caching that single extra block (and only that one), keeping the memory win for the non-EAGLE path.

1. Background: SWA prefix caching in hybrid models

A hybrid model mixes attention types — e.g. some layers use full attention, some use sliding-window attention (SWA). Each type is a separate KV cache group with its own block_size. Prefix caching must produce a hit length that is valid for every group at once, so the cache-hit length is required to be a multiple of the LCM of all groups' block sizes.

Source: vllm/v1/core/kv_cache_coordinator.py (HybridKVCacheCoordinator) 
self.lcm_block_size = lcm(*(g.spec.block_size for g in self.attention_groups))

This lcm_block_size defines an aligned segment. Our running example (taken from the PR's regression tests):

QuantityValueMeaning
full attn block_size324 × SWA block
SWA block_size8the small group
lcm_block_size (alignment)32 tokens= 4 SWA blocks
per_segment4 blocksalignment / block_size
SWA sliding_window8 tokens1 block of context
tail1 blockcdiv(window−1, block_size)

The key insight for SWA: within a per_segment = 4-block window, a sliding window of 8 tokens only ever reads back tail = 1 block. The other 3 blocks of each segment fall outside the window for any aligned boundary — they can never participate in an aligned cache hit.

2. The #42258 optimization: the segment mask

Since those earlier blocks can never serve a hit, #42258 decided not to write them into the prefix-cache hash map at all (saving hash-map entries and the bookkeeping that comes with them). It did this with a per-block cache mask in SlidingWindowManager:

Before (#42258): single_type_kv_cache_manager.py _cache_block_mask 
per_segment = alignment_tokens // self.block_size      # 4
tail = cdiv(self.sliding_window - 1, self.block_size)  # 1
if tail >= per_segment:
    return None                # cache everything (no savings possible)
skip = per_segment - tail      # 3
return [i % per_segment >= skip for i in range(num_cached_blocks, num_full_blocks)]
#       only the LAST `tail` block(s) of each 4-block segment are cached

For our numbers, skip = 3, so block i is cached iff i % 4 ≥ 3 — only blocks 3, 7, 11, … (the last block of each segment):

Each segment is 4 SWA blocks = 32 tokens (one aligned unit). The sliding window is 1 block, so only block 3, 7, 11… (the segment tail) is ever read back. The other three are pure overhead — #42258 stops caching them. This is correct… until EAGLE.

3. What EAGLE needs: tail + 1 contiguous blocks

EAGLE (and MTP) speculative decoding runs a draft head that needs the KV of one extra token past the matched prefix as its starting context. In the prefix-cache lookup, vLLM handles this by matching one additional block and then dropping the last matched block — the "lookahead" block. For SWA, the lookup widens its required contiguous run by one:

After #44082: shared helper used by both lookup and mask 
@classmethod
def _contiguous_blocks_for_hit(cls, window_size, block_size, use_eagle) -> int:
    blocks = cdiv(window_size - 1, block_size)   # tail
    if use_eagle:
        blocks += 1     # match one extra block past the boundary, then drop it
    return blocks

So with EAGLE, the SWA group needs need = tail + 1 = 2 contiguous cached blocks. After matching those 2 and dropping the last, it is left with 1 block ending exactly on the aligned boundary. The extra (dropped) block sits one position past the boundary — the first block of the next segment.

Why "one block past the boundary"? The cache hit length (after the drop) must be a multiple of alignment_tokens = 32. So the matched run is aligned + block_size long: it ends one block beyond an aligned boundary. That trailing block — index 4, 8, 12… (i.e. i % per_segment == 0) — is the lookahead block EAGLE drops.

4. The bug: the lookahead block was never cached

Look at where the lookahead block lives versus what the mask cached. The lookahead block is at i % 4 == 0 (block 4, 8, …). But the #42258 mask only cached i % 4 ≥ 3 (block 3, 7, …). The lookahead block was never written to the hash map. EAGLE could therefore never find 2 contiguous cached blocks — so EAGLE + SWA found no prefix-cache hit at all.

To reuse the 32-token prefix ending at the segment-0 boundary, EAGLE must match b3 + b4 (2 contiguous blocks) then drop b4. b3 is cached; b4 is not. The contiguous run is length 1, short of the required 2, so the match fails and falls through to zero hit.
Symptom A model that combines sliding-window attention with EAGLE/MTP speculative decoding (where the cache-hit alignment exceeds the SWA window) gets a 0% prefix-cache hit rate — every request recomputes its full prompt, even on an exact repeat. No crash, no error: just silently lost performance.

5. The fix: cache exactly one extra block

#44082 keeps the memory win but caches the single lookahead block EAGLE needs. Two coordinated changes do this.

5a. reachable_block_mask — a shifted, widened run

_cache_block_mask is renamed to reachable_block_mask (a classmethod now) and reworked around the shared _contiguous_blocks_for_hit helper, so the mask and the lookup can never drift apart. A block is reachable iff it falls in the need-wide run ending at a boundary's right edge, where EAGLE shifts that edge one block to the right:

After #44082: single_type_kv_cache_manager.py SlidingWindowManager.reachable_block_mask 
per_segment = alignment_tokens // kv_cache_spec.block_size          # 4
need = cls._contiguous_blocks_for_hit(
    window_size=kv_cache_spec.sliding_window,
    block_size=kv_cache_spec.block_size,
    use_eagle=use_eagle,                                            # tail (+1 if eagle)
)
if need >= per_segment:
    return None
shift = 1 if use_eagle else 0
return [
    i >= shift and (i - shift) % per_segment >= per_segment - need
    for i in range(start_block, num_blocks)
]

The EAGLE mask caches 2 of every 4 blocks instead of 1 — the tail and the next segment's first block — while still skipping the middle two. Non-EAGLE behavior is byte-for-byte unchanged.

5b. reachable_block_mask parameter-by-parameter, with worked examples

This is the heart of the fix, so it is worth slowing down. The method is a @classmethod — it holds no instance state; everything it needs arrives as an argument. It is called from SingleTypeKVCacheManager.cache_blocks like this:

Call site: single_type_kv_cache_manager.py cache_blocks 
block_mask = self.reachable_block_mask(
    num_cached_blocks,   # -> start_block
    num_full_blocks,     # -> num_blocks
    alignment_tokens,    # -> alignment_tokens  (= coordinator's lcm_block_size)
    self.kv_cache_spec,  # -> kv_cache_spec
    self.use_eagle,      # -> use_eagle  (propagated by the coordinator, see §7)
)
ParameterTypeWhat it is
start_blockint First block index to produce a flag for. Equals num_cached_blocks — how many blocks of this request were already cached on a previous step. Caching is incremental, so the mask only covers the newly-full blocks.
num_blocksint One past the last block index (num_full_blocks). The returned list has length num_blocks − start_block: element j describes absolute block start_block + j.
alignment_tokensint The aligned-segment size in tokens (the coordinator's lcm_block_size). Asserted > block_size — the caller only builds a mask when a segment spans more than one block.
kv_cache_specKVCacheSpec Source of block_size and sliding_window. Asserted to be a SlidingWindowSpec.
use_eaglebool Whether this group caches the EAGLE lookahead block. This is the instance flag self.use_eagle, propagated to every manager in an EAGLE spec group (§7).

Returns: a list[bool] aligned to blocks [start_block, num_blocks) — True = write this block into the prefix-cache hash map. Or None, the sentinel meaning "cache every non-null block" (the base-class default for full attention; SWA returns it only when masking can save nothing).

The three derived quantities

per_segment = alignment_tokens // block_size      # blocks per aligned segment
need        = tail (+1 if use_eagle)              # contiguous blocks a hit consumes
shift       = 1 if use_eagle else 0               # push the run's right edge past the boundary
threshold   = per_segment - need                  # the comparison's right-hand side

Picture the blocks chopped into per_segment-sized segments. An aligned cache hit ends on a segment boundary. The blocks a hit actually consumes are a need-wide contiguous run whose right edge sits at the boundary — on the last block before the boundary for plain SWA (shift=0), or one block past it for EAGLE (shift=1, the lookahead block). A block is reachable iff it lands inside such a run for some boundary, which is exactly: 

i >= shift  AND  (i - shift) % per_segment >= per_segment - need

The i >= shift guard only matters for EAGLE at i = 0: block 0 has no segment before it, so it can never be a lookahead block. Without the guard, (0 - 1) % per_segment wraps to per_segment - 1 in Python and would wrongly mark block 0.

Returned list: [True, False, False, True, True], describing blocks 4–8. Note i=4 (a lookahead block) is correctly marked even though it is the first element of this partial mask — proof that the formula keys off the absolute block index, not the list position.

The None short-circuit

If need ≥ per_segment, the run is at least as wide as a whole segment, so every block is reachable and masking saves nothing — the method returns None ("cache everything"). Example: sliding_window=32, block_size=8 ⇒ tail=cdiv(31,8)=4 = per_segment, so even the non-EAGLE need=4 ≥ 4 returns None. This is also why a small window like our example (tail=1) is where the optimization — and therefore the original bug — actually bites.

5c. cache_blocks — extend the cached range by one block

The mask only decides which blocks within the cached range get written. But the coordinator floors the cached range to the aligned boundary, so the lookahead block (which lives past the boundary) was never even offered to the mask. The coordinator now extends the range by one block for EAGLE groups:

After #44082: kv_cache_coordinator.py HybridKVCacheCoordinator.cache_blocks 
aligned_num_computed_tokens = (
    num_computed_tokens // self.lcm_block_size * self.lcm_block_size
)
for manager in self.single_type_managers:
    num_tokens_to_cache = aligned_num_computed_tokens
    # EAGLE groups match one block past each aligned boundary and drop
    # it, so make that lookahead block eligible to be cached.
    if manager.use_eagle and aligned_num_computed_tokens > 0:
        num_tokens_to_cache = min(
            num_computed_tokens,
            aligned_num_computed_tokens + manager.block_size,
        )
    manager.cache_blocks(request, num_tokens_to_cache,
                         alignment_tokens=self.lcm_block_size)
Why both changes are needed cache_blocks widens the range of blocks passed to the manager (now reaching the lookahead block); reachable_block_mask decides which blocks in that range to actually write (the tail + the lookahead, not the skipped middle). One without the other either never sees the block or writes back the blocks we wanted to skip.

6. The use_eagle → drop_eagle rename

The find_longest_cache_hit parameter use_eagle is renamed to drop_eagle across all manager subclasses. This is not cosmetic — it disentangles two genuinely different things that used to share one name:

NameKindMeaning
self.use_eagle instance attribute This group is an EAGLE/MTP group. Fixed for the manager's lifetime. Read by the cache-write path (cache_blocks / reachable_block_mask).
drop_eagle per-call argument On this lookup pass, drop the last matched block. Can be False even for an EAGLE manager — e.g. after the drop has already been applied earlier in the hybrid fixed-point convergence loop.
The fixed-point loop in kv_cache_coordinator.py — where they diverge 
drop_eagle = use_eagle and idx not in eagle_verified
# ... match with drop_eagle, then:
if drop_eagle:
    eagle_verified.add(idx)   # next pass for this group: drop_eagle=False

The hybrid coordinator re-scans groups until the hit length stops changing. An EAGLE group (use_eagle=True) drops its lookahead block on the first pass, then is marked verified, so subsequent passes use drop_eagle=False even though it is still an EAGLE group. The old single name use_eagle conflated "is an eagle group" with "drop on this pass," which is exactly the distinction the cache-write path now depends on.

7. SpecGroup: carrying the eagle bit explicitly

Previously the coordinator tracked which attention groups were EAGLE in a separate eagle_attn_group_indices set, recomputed by scanning membership. #44082 replaces this with a SpecGroup NamedTuple that carries the use_eagle bit alongside the spec and member group ids, then propagates that bit onto each SingleTypeKVCacheManager instance:

kv_cache_coordinator.py 
class SpecGroup(NamedTuple):
    spec: KVCacheSpec
    group_ids: list[int]
    manager_cls: type[SingleTypeKVCacheManager]
    use_eagle: bool      # True iff ANY member group is EAGLE/MTP

# ... after grouping by spec:
for group in self.attention_groups:
    if group.use_eagle:
        for gid in group.group_ids:
            self.single_type_managers[gid].use_eagle = True

Notice the propagation loop sets use_eagle = True on every manager in the spec group — even members that are not themselves EAGLE groups. That looks wrong at first glance. The next two sections explain when groups land in the same SpecGroup and why the propagation is necessary.

7a. When are two groups batched into one SpecGroup?

Batching happens once at startup in HybridKVCacheCoordinator.verify_and_split_kv_cache_groups. It walks the KV cache groups in order and, for each, looks for an already-seen SpecGroup whose spec is equal to this group's spec. Equal → append to it; otherwise → start a new SpecGroup:

kv_cache_coordinator.py verify_and_split_kv_cache_groups 
for i, g in enumerate(self.kv_cache_config.kv_cache_groups):
    spec = g.kv_cache_spec
    use_eagle = i in self.eagle_group_ids
    for idx, group in enumerate(self.attention_groups):
        if group.spec == spec:                 # <-- the batching condition
            assert manager_cls is group.manager_cls
            group.group_ids.append(i)          # join existing SpecGroup
            if use_eagle and not group.use_eagle:
                self.attention_groups[idx] = group._replace(use_eagle=True)
            break
    else:
        self.attention_groups.append(          # new SpecGroup
            SpecGroup(spec, [i], manager_cls, use_eagle)
        )

The whole rule is the single line group.spec == spec. Specs are frozen dataclasses (vllm/v1/kv_cache_interface.py), so == is field-by-field equality: two groups batch together iff they are the same spec subclass and every field matches — block_size, num_kv_heads, head_size, dtype, kv_quant_mode, and (for SWA) sliding_window.

Two groups…Batched?Why

Why batch at all? Groups with the same spec have byte-identical cache geometry, so vLLM runs one cache-hit scan for all of them at once (passing the full group_ids list to find_longest_cache_hit) instead of repeating the same scan per group. That shared scan is precisely what forces the EAGLE bit to be all-or-nothing — the subject of the next section.

7b. Why a non-EAGLE sibling must also cache the lookahead block

With the batching rule in hand: two SWA groups with identical specs collapse into one SpecGroup and are looked up jointly in a single scan. Here is the concrete config from test_eagle_grouped_swa_siblings_use_same_cache_mask:

Group idNameSpecis_eagle_group
0fullFullAttention, block_size 32—
1swa_mainSlidingWindow, block_size 8, window 8no
2swa_mtpSlidingWindow, block_size 8, window 8 (same object)yes

Groups 1 and 2 have the same SlidingWindowSpec, so they collapse into one SpecGroup with group_ids = [1, 2]. This pairing is realistic: group 1 is the main model's sliding-window layer; group 2 is an MTP/EAGLE draft head's sliding-window layer with the identical window — one is EAGLE, one is not, but their cache geometry is the same.

The mechanism: a joint lookup requires the block in all siblings The SWA scan in find_longest_cache_hit probes the block pool with the whole id list at once: 
block_pool.get_cached_block(block_hashes[i], kv_cache_group_ids=[1, 2])
This returns a hit only if block i is cached for both group 1 and group 2. The two siblings therefore share one hit length — a block is "present" only when every member cached it.

Now the problem is visible. The EAGLE group 2 needs the lookahead block (b4, b8, …) cached. But if only group 2 cached it and the non-EAGLE group 1 used the narrow tail-only mask, then get_cached_block(b4, [1, 2]) would miss — group 1 doesn't have b4. The joint contiguous run breaks at the boundary, and EAGLE finds no hit. The fix is exactly the propagation loop above: because the spec group's use_eagle is True, both managers get use_eagle = True, so group 1 also caches b4/b8.

Without the fix, group 1 (non-EAGLE) would skip b4, so the joint probe get_cached_block(b4, [1, 2]) fails and EAGLE's 2-block run collapses. With the fix, the EAGLE bit propagates to group 1, both cache b4, and the joint lookahead match succeeds. The test asserts get_cached_block(b4, [1, 2]) and get_cached_block(b8, [1, 2]) both hit, and the replay yields a 8 × block_size = 64-token cache hit.
One-line takeaway Siblings sharing a spec are cached and matched as a unit, so "drop the EAGLE lookahead block" can't be decided per-group — it's all-or-nothing across the spec group. If any member is EAGLE, every member must cache the lookahead block, or the joint lookup can never see it.

8. Tests & the alternative PR

Three new regression tests in tests/v1/core/test_prefix_caching.py:

  • test_eagle_swa_alignment_caches_extra_block — primes a 16-block prompt, frees it, replays it; asserts the second request gets a non-zero hit (the headline bug) and the hit lands on a 32-token boundary.
  • test_eagle_swa_boundary_caches_post_boundary_block — with a 5-block prompt, asserts both block 3 (tail) and block 4 (lookahead) are in the cache, and the replayed hit is 4 × block_size = 32 tokens.
  • test_eagle_grouped_swa_siblings_use_same_cache_mask — two SWA groups sharing a spec (one EAGLE) must both cache the lookahead block.

The drop_eagle rename also touches test_single_type_kv_cache_manager.py and the Mooncake store coordinator (which calls find_longest_cache_hit directly).

vs. the alternative fix (#42784) A competing PR fixed the same bug by disabling the SWA cache mask entirely when EAGLE is active — caching every SWA block and forfeiting the #42258 memory savings. #44082 instead caches only the one additional block EAGLE actually needs, and additionally handles grouped SWA siblings and the block_size ≠ alignment_tokens (Gemma-style) path where the post-pop re-alignment matters.
Mental model in one line #42258 said "a block is cacheable iff it's in the segment tail." #44082 corrects this to "…in the need-wide run ending at the boundary's right edge" — and for EAGLE that edge is shifted one block right, sweeping in the lookahead block that the drop will discard.
