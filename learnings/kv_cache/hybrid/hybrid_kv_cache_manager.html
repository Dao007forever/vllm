vLLM v1 — Hybrid KV Cache Manager

How HybridKVCacheCoordinator manages prefix caching when a single model mixes multiple attention types (full / sliding-window / chunked-local / Mamba). Code pointers are to vllm/v1/core/ in this repo.

TL;DR

A hybrid model has more than one attention type per request. The manager splits layers into KV cache groups (one per attention type), and the HybridKVCacheCoordinator fans every allocation, free, cache, and lookup call across all groups. Cache-hit length is the longest prefix all groups agree on — found by a small fixed-point loop that converges because length only ever shrinks. The length is rounded to the LCM of group block sizes so it is a clean block boundary in every group simultaneously. Hybrid caching is real prefix sharing — the suffix-digest shortcut for SWA is unsound past depth 2 and the manager intentionally does not attempt it.

1. Why "hybrid"?

Most modern long-context models do not use one attention type throughout. A typical pattern interleaves a cheap local attention (sliding-window or chunked-local) with a smaller number of full-attention layers, getting most of the modeling power of full attention at a fraction of the K/V memory. Some models also splice Mamba state-space layers into a transformer stack. All of these are "hybrid" from the KV cache's perspective.

Model Pattern Group makeup
Gemma-3 5 SWA : 1 full (every 6th layer is full) 1 full-attention group + 5 sliding-window groups
Llama-4 3 chunked-local : 1 full 1 full + 3 chunked-local groups
Jamba / Zamba / Mamba2-Hybrid Mamba + sparse full-attention 1 full + N Mamba groups
Mistral All sliding-window (homogeneous) Single group → UnitaryKVCacheCoordinator, not hybrid

Each attention type has fundamentally different needs from the cache: full attention reads every prior position so its blocks cannot be evicted while the request is alive; sliding-window only needs the last W tokens so blocks past that horizon are reclaimable; Mamba keeps state checkpoints at block boundaries. One uniform block table cannot serve all three. The hybrid manager exists to give each type its own logical block table while sharing the same physical block pool.

One physical pool, many logical tables Each KV cache group has its own SingleTypeKVCacheManager and its own req_to_blocks mapping. They all draw from the same BlockPool, but the per-group block lists for the same request are independent — group A's block 3 has nothing to do with group B's block 3.

2. Three coordinator flavors

The factory get_kv_cache_coordinator() at kv_cache_coordinator.py:610-658 picks one of three concrete classes. They all inherit a common KVCacheCoordinator base that fans allocate / free / cache / get-blocks over the per-group managers — the subclasses differ only in how cache hits are computed.

NoPrefixCache

Used when prefix caching is disabled. Supports any number of groups (including 0). Its find_longest_cache_hit returns 0 hits.

No block-size or context-parallelism restrictions.

Unitary

Exactly one KV cache group. Most homogeneous models (Llama-3 full, Mistral sliding-window-only). Cache hit is a single SingleTypeKVCacheManager.find_longest_cache_hit.

Asserts hash_block_size == block_size.

Hybrid

Two or more groups with different specs. Cache hit is an iterative fixed-point across types, aligned to the LCM block size.

DCP / PCP must be 1 — context parallelism is not yet supported with hybrid attention.

def get_kv_cache_coordinator(...):
    if not enable_caching:
        return KVCacheCoordinatorNoPrefixCache(...)
    if len(kv_cache_config.kv_cache_groups) == 1:
        return UnitaryKVCacheCoordinator(...)
    return HybridKVCacheCoordinator(...)

3. KV cache groups

A KV cache group is a set of model layers that share the same KVCacheSpec (attention type, block size, head dim, dtype, etc.). The grouping is done up front, before any request hits the manager, by _get_kv_cache_groups_uniform_page_size() in kv_cache_utils.py:1057.

The grouping rule is: find the smallest repeated pattern of layer types, use the count of the minority type as the group size, and split each attention type's layers into that many groups. For Gemma-3's 5:1 SWA:full pattern across 30 layers (5 full, 25 SWA): the minority is full with 5 layers, so group size = 5; that yields 1 full-attention group (5 layers) plus 5 sliding-window groups (5 layers each).

All 5 full-attention layers (L5, L11, L17, L23, L29) share one block table. The 25 SWA layers fan out into 5 groups of 5 layers each — within a group, layers share a block table; across groups, they do not. Each group's page size in bytes must match (a hard invariant of the current design — see kv_cache_utils.py:1086-1115), so allocation can treat all groups as one homogeneous pool of physical blocks.

Why N SWA groups instead of 1? Each SWA layer needs its own block table because the same request is at different positions across layers (different K/V values). One layer's blocks can't be reused by another layer. Splitting full's 5 layers and SWA's 25 layers into groups of 5 keeps every group the same shape, so allocation is one number of blocks across N parallel block tables.

4. Block-size alignment (LCM & GCD)

A cache hit length must be a multiple of every group's block_size at once — there is no partial-block representation in the manager. The smallest such number is the LCM of all group block sizes. The coordinator stores this as self.lcm_block_size and uses it in two places:

  • Cache hits: find_longest_cache_hit always returns a length that is a multiple of lcm_block_size.
  • Caching new blocks: cache_blocks(num_computed_tokens) rounds down to lcm_block_size before delegating to each per-group manager: 
    num_computed_tokens = num_computed_tokens // self.lcm_block_size * self.lcm_block_size
for manager in self.single_type_managers:
    manager.cache_blocks(request, num_computed_tokens, alignment_tokens=self.lcm_block_size)

The companion hash_block_size is at the other end of the chain: by construction it divides every group's block_size, so a request's block_hashes can be re-grouped to any group's frame by concatenating consecutive hashes — that is what BlockHashListWithBlockSize does, lazily, on access. 

hash_block_size  |  group.block_size  |  scheduler/lcm_block_size

   ↑ smallest                              ↑ largest
   one hash per N tokens                   the only length a cache hit
                                           can take in the hybrid coordinator

The scheduler block size from resolve_kv_cache_block_sizes() equals the coordinator's lcm_block_size when there are multiple groups. They are the same number, looked at by the same constraint, in different layers of the stack. Section 12 of the Mooncake architecture page works through the divisibility chain in detail.

5. find_longest_cache_hit overview

Lookup is a single call from the scheduler:

Scheduler get_computed_blocks(req)
KVCacheManager forwards to coordinator
Coordinator find_longest_cache_hit(block_hashes, max_len)
Per-type manager.find_longest_cache_hit(...) per group

The return value is a pair: a per-group list of cached KVCacheBlock objects (one list per group, in kv_cache_groups order), and the longest common hit length in tokens. The hit length is the same number for every group — but each group's block list may differ in which slots are nulled out (SWA fills only the trailing positions; chunked-local nulls everything before the last chunk; etc.).

6. The fixed-point algorithm

Different attention types accept different prefix lengths. Full attention is downward-closed: if length L is a hit, every length below L is also a hit. Sliding-window only cares about the last ceil((W-1)/B) contiguous blocks, so the hit it returns for length L is independent of the prefix before its window. The trick is finding the largest L that satisfies all types simultaneously. 

# kv_cache_coordinator.py:503-607 (paraphrased)
hit_length = max_cache_hit_length
while True:
    curr_hit_length = hit_length
    for spec, group_ids, manager_cls in self.attention_groups:
        if isinstance(spec, FullAttentionSpec) and full_already_done:
            # full is downward-closed; trim to current hit_length, don't re-scan
            curr_hit_length = curr_hit_length // spec.block_size * spec.block_size
            continue
        hit_blocks = manager_cls.find_longest_cache_hit(
            block_hashes=_get_block_hashes(spec),
            max_length=curr_hit_length,
            ...
            alignment_tokens=self.lcm_block_size,
        )
        curr_hit_length = len(hit_blocks[0]) * spec.block_size
        ... store hit_blocks ...
    if curr_hit_length >= hit_length:
        break                          # fixed point reached
    hit_length = curr_hit_length       # length shrunk; reconsider every type
    if is_simple_hybrid:
        break                          # one iteration suffices

Why this terminates

Each pass either leaves hit_length unchanged (then we break) or shrinks it strictly. Lengths are non-negative integers bounded below by 0 and aligned to lcm_block_size from above, so monotone descent must terminate. In the typical 2-type hybrid (1 full + 1 other) one pass is provably sufficient — the is_simple_hybrid short-circuit skips the convergence check entirely.

Why full attention runs first

verify_and_split_kv_cache_groups() sorts the attention groups so that FullAttentionSpec comes first. Full's L→R scan stops at the first miss, returning a tight upper bound. Subsequent types only need to confirm that bound or reduce it, never enlarge it — so the first pass already does almost all the work.

The EAGLE quirk

EAGLE/MTP speculative decoding requires dropping the last matched block (the drafter must run one block beyond the verified prefix). The coordinator tracks which attention-group indices are EAGLE groups and applies the drop exactly once per candidate hit_length. If the length shrinks on a later iteration, the EAGLE-verified set is cleared so the drop applies afresh — preventing the bug from issue #32802 where the drop was applied twice.

7. Per-type scan directions

Each SingleTypeKVCacheManager.find_longest_cache_hit walks the request's hash list in the direction that matches its attention pattern. The differences look small but they encode deep correctness properties.

FullAttention L→R

Position p attends to [0, p]. Once any block diverges, every subsequent block's parent hash also differs, so all later blocks must be misses. Scan from block 0, stop at first miss. 

cache: B0=HIT  B1=HIT  B2=MISS  B3=HIT
L→R:   hit    hit    miss → stop
matched = 2 blocks  (B3's hit is irrelevant — parent diverged)

SlidingWindow R→L

Position p attends only to [p − W + 1, p]. The last block only needs ceil((W−1)/B) contiguous matched blocks ending at its position. Scan from the end, accumulate contig hits, stop once W of them. 

W_blocks = 2 (sliding_window=8, block_size=4)
cache: B0=HIT  B1=MISS  B2=HIT  B3=HIT  B4=HIT
R→L:   B4 hit (1) → B3 hit (2 — enough) stop
result blocks = [NULL, NULL, NULL, B3, B4]
matched = 5 blocks worth (20 tokens) — but earlier slots are NULL

ChunkedLocal chunk-boundary L→R

Llama-4-style local attention only attends within a chunk of fixed size. Blocks fully outside the current chunk are marked computed with null blocks; only blocks inside the active chunk are looked up. 

chunk_size=8, block_size=4, max_len=15
out-of-window: blocks 0..1 → NULL NULL (auto-computed)
in-window:     block 2 → check cache
result = [NULL, NULL, B2?] or [NULL, NULL] on miss

Mamba R→L, last-match-only

Mamba caches state at block boundaries. You want the latest checkpoint matching the prefix, so scan from the end and stop at first hit. Earlier matches are unhelpful — you'd have to replay forward. 

cache: B0=HIT  B1=HIT  B2=MISS  B3=HIT  B4=MISS
R→L:   B4 miss → B3 hit → stop
result = [NULL, NULL, NULL, B3]  (replay from B3 forward)
Why the null-fill is correct for SWA SWA attention at the last position literally won't read positions outside its window — those blocks are never dereferenced during the forward pass. The cached "tail" only needs to cover the window, so leaving earlier slots as null_block is semantically correct, not a hack.

8. Walkthrough — Gemma-3 hybrid hit

Setup: Gemma-3 with block_size=16, sliding_window=64 (so W_blocks = ceil(63/16) = 4). One full-attention group plus five SWA groups, all with the same block_size=16, so lcm_block_size = 16 and hash_block_size = 16. A request with 8 full blocks (tokens 0..127) arrives; the local cache state is shown below.

Step 1 — full-attention group runs first (L→R)

FullAttentionManager.find_longest_cache_hit walks the chain and stops at the first miss.

hashes: H0 H1 H2 H3 H4 H5 H6 ✗ H7

Full hit = 6 blocks = 96 tokens. curr_hit_length = 96.

Step 2 — each SWA group runs (R→L, capped at 96 tokens)

For a SWA group, find_longest_cache_hit looks at the last 6 hash slots (indices 0..5 since max_num_blocks = 96/16 = 6) and scans right-to-left for 4 contiguous hits. Suppose this group has H2, H3, H4, H5 cached, with H0 and H1 also cached but H1 is not contiguous with the W-block tail.

SWA group A: H0 H1 H2 H3 H4 H5

R→L from index 5: H5 hit (contig=1), H4 hit (2), H3 hit (3), H2 hit (4 ≥ W → stop). Result blocks (length 6): [NULL, NULL, H2, H3, H4, H5]. Hit length stays 96.

Step 3 — another SWA group reduces the bound

Suppose SWA group B has a miss at H4:

SWA group B: H0 H1 H2 H3 H4 ✗ H5

R→L: H5 hit (1), H4 miss → contig resets to 0, H3 hit (1), H2 hit (2), H1 hit (3), H0 hit (4 ≥ W → stop). Match ends at index 0; result has 4 trailing hits → 4 blocks = 64 tokens. curr_hit_length drops from 96 → 64.

Step 4 — reconsider full attention at the new length

The outer loop detects that curr_hit_length (64) < hit_length (96). Set hit_length = 64 and restart. Full attention is downward-closed so it just trims its block list: num_blocks = 64 / 16 = 4, drop H4 and H5 from the cached list.

SWA group A re-scans with max_length=64: index range 0..3, R→L, H3 hit, H2 hit, H1 hit, H0 hit → 4 contig blocks at index 0 → 64 tokens. No change. SWA group B re-scans with max_length=64: same as above, 64 tokens. No change. curr_hit_length (64) ≥ hit_length (64): fixed point. Return 64 tokens and the per-group block lists.

What just happened The fixed-point loop forced the full group to honor SWA group B's lower bound. Without re-trimming, the full group would have claimed 96 tokens matched while SWA B only had 64 — and the request would have been resumed against an inconsistent prefix.

Step 5 — caching after the forward pass

When the forward pass finishes for this request, the scheduler calls cache_blocks(num_computed_tokens=…). The hybrid coordinator rounds that number down to a multiple of lcm_block_size = 16 (in this case, no-op — it already is) and then asks each per-group manager to hash any newly-filled blocks into the prefix-cache hash map. Future requests with the same prefix will get a real hit on these blocks across all groups.

10. Limitations & correctness walls

Context parallelism (DCP / PCP)

HybridKVCacheCoordinator asserts both DCP and PCP world sizes are 1. Per-type managers (sliding-window, chunked-local, Mamba) also reject CP. Hybrid + CP is not yet implemented.

Partial-block cache hits

The coordinator always rounds hit length down to lcm_block_size. There is no representation for "matched 12 out of 16 tokens of this block." This is why the LCM exists in the first place.

One full + one other (lookup constraint)

The fixed-point loop is general, but _get_kv_cache_groups_uniform_page_size currently only supports patterns of one attention type, or full plus exactly one other type. Three-way hybrids are blocked at config time, not at coordinator time.

The SWA receptive-field wall

For depth ≥ 2 transformers, SWA layer K/V depends on the entire prefix through the residual stream — not just the window. So "same suffix at same positions but different earlier prefix" is not the same K/V. The manager intentionally does not implement suffix-digest hashing; the chained hash is the only sound key.

Why suffix-digest hashing is unsound At every layer L, K_L[p] = W_K · h_{L-1}[p]. The receptive field of h_{L-1}[p] grows with depth: one full attention layer anywhere below position L pulls in the entire prefix. Returning request A's cached SWA block on request B (different early prefix, same suffix) inserts numerically wrong K/V values. No kernel-level signal — the model just gets quietly worse on shared-tail long-context workloads. See KV_CACHE.md Part 2 for the full proof.

11. Where the engineering wins are

Three improvements are sound and have measurable upside for hybrid models; a fourth (suffix-digest hashing) does not exist.

  1. Sparser SWA block allocation. Today, SWA groups allocate a block per block_size tokens of prefix and rely on remove_skipped_blocks to free out-of-window blocks after the fact. Peak memory is still O(N) per SWA layer. A streaming allocator that never allocates out-of-window blocks for SWA groups would shrink peak memory to O(W) per layer — for Gemma-3's 5:1 ratio on long contexts, a substantial reduction. Pure allocation-layer change, no impact on lookup.
  2. R→L scan micro-optimization in SlidingWindowManager. The TODO at single_type_kv_cache_manager.py:578-582: on a miss, advance by ceil(W/B) blocks rather than one. Lowers worst-case from O(max_num_blocks) to roughly O(max_num_blocks / W_blocks + W_blocks) — meaningful for low hit-rate scenarios.
  3. Connector coverage for SWA groups. Mooncake / LMCache / NIXL primarily index full-attention blocks. For Gemma-3-style traffic with shared system prompts, ensuring connectors symmetrically index SWA blocks (using the existing chained hash) recovers cross-instance compute. No hash change needed — just plumbing per-group keys through.
