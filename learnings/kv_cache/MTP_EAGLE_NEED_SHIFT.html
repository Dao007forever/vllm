MTP/EAGLE: need and shift

A visual walk-through of why sparse SWA prefix-cache retention keeps a wider tail when MTP/EAGLE is enabled. Parameters are adapted from the PR test (block_size=8, alignment_tokens=32, retention_interval=0, a 127-token prompt), with sliding_window=16 so the window spans two blocks and the tail-versus-anchor distinction is visible (the test itself uses sliding_window=8, the degenerate one-block case).

TL;DR

MTP/EAGLE drafting heads take the target model's hidden states as input, but the prefix cache only stores K/V — never hidden states. So on a full cache hit the last block must be recomputed to regenerate that hidden state, which means the cache hit has to give one block back. Under sparse SWA retention you cannot simply lop the last block off, because the returned hit must stay on an aligned boundary. The fix is to match one extra block past the boundary and then drop it: need += 1 and shift = 1. The returned prefix length is unchanged; a block was sacrificed to force the recompute.

Reading the two knobs: need = width, shift = position

need is how many contiguous cached blocks must sit at the match's right edge for the hit to count. SWA can only resume at a position if the KV of the previous sliding_window − 1 tokens is still present, and that spans cdiv(sliding_window − 1, block_size) blocks — the width of the retained run. Here a 16-token window spans two 8-token blocks, so it is 2 (cdiv(15, 8) = 2); a window equal to the block size would make it 1, and a 33-token window would make it 4. EAGLE/MTP adds + 1 for the sacrificial block.

shift is where that run's right edge sits relative to the aligned boundary: 0 puts it on the last block ending at the boundary (b11 → 96); 1 pushes it one block past the boundary (b12 → 104), which is then dropped back to 96.

So does SWA cache more blocks at the end? Yes — but the hit doesn't grow

Three things retain or store more under EAGLE/MTP: need += 1 keeps one extra block per boundary ({b10, b11, b12} instead of {b10, b11}); shift = 1 places that extra block after the boundary; and the coordinator writes it into the hash map too, caching aligned_num_computed_tokens + block_size for EAGLE groups so the lookahead block actually exists on replay.

What does not change is the returned prefix length. b12 is matched and then dropped, so the served hit is still 96 tokens in both modes. The net is more blocks retained and stored, identical prefix returned — the extra block exists only so the drop-for-recompute can land back on an aligned boundary.

1. Why a block gets dropped at all

This is the piece that the need/shift arithmetic hides. MTP and EAGLE are speculative-decoding drafters: a small head runs on top of the target model and proposes the next few tokens so they can be verified in one step. That head is autoregressive on the target's last-layer hidden state — to draft the token after position t, it needs the hidden state the target produced at position t. (DeepSeek-style MTP heads behave the same way for cache purposes, which is why they share this code path.)

The prefix cache does not store hidden states

A KV-cache block holds only the per-layer keys and values for its tokens. It never holds the final hidden states that the drafting head consumes. So when an entire prompt is served from cached K/V, the forward pass that would have produced the seed hidden state never runs — and the drafter has no input for its first proposal.

The fix (introduced in #17137) is to shorten the cache hit by one block, forcing the model to recompute that block's tokens on the next forward pass and regenerate their hidden states. The original comment says it plainly: 

if self.use_eagle and len(computed_blocks) > 0:
    # Drop the last matched block if (1) eagle is enabled and
    # (2) there is a cache hit.
    # This is to recompute the last block to get the required
    # hidden states for eagle drafting head.
    computed_blocks.pop()
This is a different drop from the num_tokens - 1 cap

get_computed_blocks already caps every hit at num_prompt_tokens - 1 so the last token is recomputed for its logits. The EAGLE drop is an additional block sacrificed for hidden states. The two coincide in lucky cases, but EAGLE must guarantee the recompute regardless of where the prompt length lands.

2. Why SWA needs shift, not just a pop()

For dense full attention the drop is trivial: every block is cached, so computed_blocks.pop() just hands the last block back and the block before it is still there. Sparse SWA retention breaks that, because of two hard constraints that fight each other:

  1. The returned hit must land on an aligned boundary. allocate_slots requires num_computed_tokens to be block-aligned, and the coordinator only accepts hits that are a multiple of scheduler_block_size (the alignment_tokens, here 32). A hit ending at token 96 is legal; one ending at 88 is not.
  2. A block must still be dropped to force the hidden-state recompute (section 1).

If SWA retained only the normal tail ({b10, b11}, ending at 96) and then popped the last block, the hit would collapse to token 88 — no longer aligned, so the coordinator throws the whole hit away. You would get a worse result than not caching at all. The resolution is to do the drop on a block that sits past the boundary:

  1. shift = 1 — match one block past the aligned boundary, so the matched run's right edge is at token 104 (block b12), not 96.
  2. need += 1 — retain one extra block so the real SWA tail ({b10, b11}) survives once the extra block is dropped.
  3. drop the extra block — pop b12. The returned hit falls back exactly onto the aligned boundary, 96, and a block was still sacrificed to trigger the recompute.
The "0% prefix cache hit with MTP" bug

Matching past the boundary only works if that lookahead block was written into the prefix-cache hash map in the first place. The original SWA mask kept just the normal tail, so the boundary anchor b12 was never cached — the EAGLE match could never complete its run to the anchor, and every replay silently dropped to a 0-token hit. That is the literal symptom users saw with MTP enabled, fixed in #44082: the coordinator bumps num_tokens_to_cache to aligned_num_computed_tokens + block_size for EAGLE groups, and the retention mask explicitly keeps the lookahead block. When sliding_window ≤ alignment_tokens that extra block lives at the next segment's first position, which the plain SWA mask had left uncached.

“If we drop it anyway, why cache it?” — it is the anchor, not dead weight

The drop happens to the match result, not to the cache. The match itself can only place its right edge on a block that is physically cached — find_longest_cache_hit scans the hash map for a contiguous run of cached blocks. b12 is the block that starts at the aligned boundary, so it is the only legal right-edge anchor that leaves an aligned hit (96) after the pop. No cached b12, no anchor.

And skipping it does not cost you one block — it costs you everything. With b12 absent the scan tries to anchor on b11, but the alignment guard (post_pop_blocks * block_size % alignment_tokens != 0: here 88 % 32 ≠ 0) refuses, because dropping b11 would leave a misaligned 88. No other run reaches need contiguous blocks, so the hit collapses to 0 — the bug above, not a graceful one-block haircut.

The irony: the replaying request never even loads b12’s KV. It is matched, then immediately popped, then recomputed from 96 to regenerate hidden states. b10 and b11 are the real tail whose KV the window actually reads; b12 serves only as an existence proof in the hash map — “the prefix truly was computed through 104, so returning an aligned 96 and recomputing from there is safe.” The producer computed it anyway; retention just declines to throw it away. One extra block held against eviction buys a full aligned hit instead of zero.

Then why not make b12 a marker-only anchor — a hash entry with no stored KV — since it is dropped anyway? In the sparse regime its KV genuinely is never read (the chart below proves it), so a marker would in fact be correct. vLLM keeps a real block for two mundane reasons, not because the data gets reused: first, the cache has no “hash without a block” concept — the map is hash → physical block, and “found by hash” means “data resident and pinned,” so a phantom anchor would need a parallel registry, its own eviction/refcount handling, would break that invariant, and would not fit the free-queue ordering #43447 relies on; second, it is one block, trivial against block_size = 256 and million-token contexts.

Real reuse of the post-boundary block happens only in the dense case (alignment = block_size, the common non-hybrid setup), where there is no sparse mask at all: every block is cached and the “drop” just pops the last contiguous one, so the dropped block is an ordinary cached block that a longer hit will load. The dedicated, never-loaded anchor is unique to the sparse hybrid regime this page is about — there, storing it is about cache mechanics, not data reuse. (An earlier draft of this page claimed the anchor doubles as a tail for a later boundary; working the geometry out below shows that it cannot.)

Why the anchor is never a window tail (sparse regime)

Same parameters as the rest of the page (block_size = 8, alignment_tokens = 32, sliding_window = 16 ⇒ need_base = 2, EAGLE need = 3, shift = 1, per_segment = 4), extended to a longer replay so two boundaries — 96 and 128 — are visible.

Each retained group is need_base = 2 loaded tail blocks plus one dropped anchor: {b10, b11} + b12 for boundary 96 and {b14, b15} + b16 for boundary 128. A window tail always spans the need_base blocks ending at an aligned boundary, while an anchor sits one block past a boundary — in the gap between groups (b12, the anchor for 96, is followed by the un-retained b13, and is nowhere near the {b14, b15} tail for 128). The anchor of boundary B is block B / block_size; for it to fall inside the tail of a later boundary B′ you would need B′ − B ≤ need_base · block_size. But boundaries are alignment = per_segment · block_size apart, and sparse retention only happens when need < per_segment (i.e. need_base + 1 < per_segment). Those two cannot both hold, so no tail ever reaches an anchor: in the sparse regime the anchor's KV is provably never loaded — it is retained purely to be findable.

3. Why the latest replay boundary is 96

Exact prompt replay is capped at num_prompt_tokens - 1, then rounded down to the cache-hit alignment: 

latest = (num_prompt_tokens - 1) // alignment_tokens * alignment_tokens
       = (127 - 1) // 32 * 32
       = 126 // 32 * 32
       = 96

4. Normal SWA: need=2, shift=0

A 32-token aligned segment contains four 8-token SWA blocks. With a 16-token sliding window (two blocks), normal SWA needs the last two blocks before each aligned boundary. For the 96-token boundary, those are blocks 10 and 11.

5. MTP/EAGLE: need=3, shift=1

MTP/EAGLE lookup asks SWA to prove one extra block after the returned boundary. The returned hit is still 96 tokens, but the SWA match must extend to 104 tokens. With retention_interval=0 in the PR test, the replay exception retains exactly blocks 10, 11, and 12. That is why the replay tail is range(prompt_end_block - need, prompt_end_block).

What changed from the regular segment rule?

With positive segment retention, the same need=3, shift=1 rule would retain triples such as b2+b3+b4, b6+b7+b8, and b10+b11+b12. The PR test sets retention_interval=0, so the chart below focuses only on the latest replay triple: b10+b11+b12.

The useful invariant

shift moves the right edge of the SWA match one block past the aligned boundary. need widens the retained run so the normal SWA tail is still present. Together they keep the extra proof block without changing the final returned prefix length — so SWA can hand back a block (the recompute that regenerates the drafter's seed hidden state, section 1) while the returned hit still lands on a legal aligned boundary. The proof block is the price SWA pays to satisfy the same drop that full attention gets for free with a single pop().
