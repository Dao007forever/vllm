vLLM v1 + Mooncake Store Architecture

Focus: pure vLLM v1 scheduler path, non-V2 GPUModelRunner with KVConnectorModelRunnerMixin, and MooncakeStoreConnector. Code pointers are from /home/zhewen/repos/vllm at commit 1c78f76c29.

TL;DR

Think of the KV connector as a two-sided plugin. The scheduler side decides what prefix is already external, which GPU blocks were allocated, and what metadata the worker should see. The worker side receives that metadata in KVConnectorModelRunnerMixin, then returns a KVConnectorOutput containing completed sends/receives. For Mooncake Store specifically, start_load_kv() and wait_for_save() are intentionally no-ops: its real load/store work is issued from worker-side get_finished() to overlap with compute.

1. Mental model

Scheduler owns request truth

The scheduler owns Request states, local KV allocation, preemption, and when a request is safe to run again. A connector may delay frees or mark a request as waiting for remote KV, but the scheduler still owns self.requests and queue transitions.

scheduler.py KVCacheManager

Connector scheduler plans

MooncakeStoreScheduler does not move bytes. It performs external lookup, tracks per-request save/load intent, and emits MooncakeStoreConnectorMetadata for the worker.

get_num_new_matched_tokens ReqMeta

Model runner brackets forward

In non-V2 GPU execution, GPUModelRunner inherits KVConnectorModelRunnerMixin. The mixin binds connector metadata before forward and polls get_finished() after forward, even when no model tokens are run.

KVConnectorModelRunnerMixin

Mooncake worker moves bytes

MooncakeStoreWorker owns Mooncake store setup, lookup server, one ChunkedTokenDatabase per KV group, and background send/recv threads that call Mooncake batch put/get APIs.

MooncakeStoreWorker MooncakeDistributedStore
One-sentence architecture Scheduler-side connector code produces a per-step I/O plan; V1 model-runner mixin transports that plan into the worker; Mooncake Store worker translates the plan into store keys and GPU addresses; scheduler consumes completion signals on the next update_from_output().

3. Engine loop

The engine loop is the backbone. It asks the scheduler for a SchedulerOutput, sends that object to workers, gets a ModelRunnerOutput, then lets the scheduler update request state.

1. Scheduler scheduler.schedule()
Creates SchedulerOutput with connector metadata.
2. Executor execute_model(scheduler_output)
Sends the output to all worker processes.
3. Worker GPUModelRunner.execute_model()
Runs model forward and connector worker hooks.
4. Scheduler update_from_output()
Consumes generated tokens and KVConnectorOutput. 
EngineCore.step()
  scheduler_output = scheduler.schedule()
  future = model_executor.execute_model(scheduler_output, non_block=True)
  model_output = future.result() or model_executor.sample_tokens(...)
  scheduler.update_from_output(scheduler_output, model_output)

5. V1 model runner mixin

If use_v2_model_runner is false, gpu_worker.py instantiates vllm.v1.worker.gpu_model_runner.GPUModelRunner. That class inherits KVConnectorModelRunnerMixin. The mixin is the bridge between SchedulerOutput and worker-side connector logic.

Normal forward path

  1. GPUModelRunner enters maybe_get_kv_connector_output().
  2. Mixin binds scheduler_output.kv_connector_metadata.
  3. Mixin calls start_load_kv(get_forward_context()).
  4. Model forward runs inside the context.
  5. On exit, mixin calls wait_for_save() and get_finished(...).
  6. Mixin attaches completed send/recv sets to KVConnectorOutput.

No-forward path

Even when total_num_scheduled_tokens == 0, the V1 runner calls kv_connector_no_forward() if a KV connector exists. That still binds metadata and calls get_finished(), which is necessary for background Mooncake loads/saves to make progress.

Pointer: gpu_model_runner.py:4004-4021 and kv_connector_model_runner_mixin.py:38-55. 

KVConnectorModelRunnerMixin._get_kv_connector_output(...)
  output = KVConnectorOutput()
  kv_connector = get_kv_transfer_group()
  kv_connector.bind_connector_metadata(scheduler_output.kv_connector_metadata)
  kv_connector.start_load_kv(get_forward_context())
  try:
      yield output                 # model forward happens here
  finally:
      kv_connector.wait_for_save()
      output.finished_sending, output.finished_recving =
          kv_connector.get_finished(scheduler_output.finished_req_ids)
      output.kv_cache_events = kv_connector.get_kv_connector_kv_cache_events()
      kv_connector.clear_connector_metadata()
Mooncake Store twist The mixin still calls start_load_kv() and wait_for_save(), but Mooncake Store implements both as no-ops. Its loads and stores are issued from get_finished() instead. This is why understanding Mooncake Store requires reading the worker get_finished() method, not just the generic connector lifecycle.

8. Scheduler-to-worker metadata

MooncakeStoreConnectorMetadata is one object per engine step. It is not just a list of current scheduled requests: it also carries unfinished request IDs and preemptions, and each ReqMeta tells the worker whether to load, save, both, or only poll completion. 

MooncakeStoreConnectorMetadata
  unfinished_request_ids: set[str]
  preempted_req_ids: set[str]
  requests: list[ReqMeta]              # one per request scheduled this step
                                       # (or with a pending load)
Where ReqMeta sits Scheduler → ReqMeta.from_request_tracker(...) in mooncake/store/scheduler.py → packed into MooncakeStoreConnectorMetadata.requests → worker iterates and routes: if req.can_save: send_thread.add_request(req), if req.load_spec.can_load: recv_thread.add_request(req).

Fresh or resumed prefill

The scheduler creates a RequestTracker from the request's prefill token range and allocated block IDs. It builds ReqMeta with optional LoadSpec, block hashes, and a save watermark.

Decode / chunked continuation

The scheduler extends the existing tracker with new block IDs, advances the token length, and emits more save metadata only while the request is still inside the tracked prefill range.

9. ReqMeta deep dive

ReqMeta is the per-request "work order" the scheduler emits on every engine step. It is the single most important data structure for understanding Mooncake Store: every save and every load is parameterized by one of these. See MOONCAKE.md for the long-form note. 

@dataclass
class ReqMeta:
    req_id: str
    token_len_chunk: int
    block_ids: tuple[list[int], ...]
    block_hashes: list[BlockHash]

    can_save: bool | None = None
    load_spec: LoadSpec | None = None
    is_last_chunk: bool | None = None
    current_event: torch.cuda.Event | None = None

    token_ids: list[int] | None = None
    original_block_size: int | None = None

Companion: RequestTracker (persistent across steps)

ReqMeta is rebuilt every step. RequestTracker persists — the scheduler keeps one per in-flight request in self._request_trackers. Its job is to remember "how far have we saved this request so far?" so the next step knows where to resume.

RequestTracker field Meaning
req_id vLLM request ID.
token_len Token length covered by the current prefill chunk; grows as chunked prefill progresses.
allocated_block_ids Per-group block IDs for this request in the GPU paged cache. Updated as new blocks are allocated.
num_saved_tokens High-watermark of tokens already saved to Mooncake. Drives the "anything new to save?" check.
token_ids Cached token-id list, used to populate KV cache events.
prefill_end_tokens Snapshot of the prefill range. For a fresh request = len(prompt). For a preempt-resume request it includes previously-generated tokens that have to be re-prefilled. Used to compute is_last_chunk.

Fields by purpose

Identity & addressing

req_id — dedup key for in-flight saves, cancel key under store pressure, completion key in get_finished().

block_ids: tuple[list[int], ...] — per-KV-cache-group lists of GPU paged-cache block IDs. A tuple of lists, not a flat list, because of HMA: each KVCacheGroup (full-attn, sliding-window, Mamba, ...) has its own block pool.

block_hashes — vLLM's prefix-cache block hashes computed at the engine's hash_block_size, reused as global Mooncake store keys. A single hash → a single PoolKey rendered as model@tp:i@pcp..@dcp..@pp:j@group:g@<hex>. Content-addressed: a store hit from one engine run can be served to another.

Save plan

token_len_chunk — how many tokens to consider for save this step. Block-aligned when discard_partial_chunks=True (default): num_tokens_to_save = token_len // block_size * block_size. The send thread later re-floors this to the HMA LCM block size before issuing I/O.

can_save — gate for "anything new whole-block to put?": chunk_boundary = cdiv(num_saved_tokens + 1, block_size) * block_size; skip_save if the new chunk does not reach the next boundary. Forced off when kv_role == "kv_consumer" (pure consumer of P/D disagg never writes back).

is_last_chunk — True when this chunk reaches the end of the prefill range (token_len >= floor(prefill_end / block_size) * block_size). The send thread uses it to know when to surface the request as finished_sending so the scheduler can release blocks that request_finished() previously deferred.

Load plan

load_spec: LoadSpec | None — non-None means this request wants to load KV from Mooncake into freshly allocated GPU blocks.

@dataclass
class LoadSpec:
    vllm_cached_tokens: int    # already in local prefix cache
    kvpool_cached_tokens: int  # found in Mooncake
    can_load: bool             # blocks allocated & ready
    token_len: int = 0         # actual range, set by worker
  • vllm_cached_tokens / kvpool_cached_tokens — filled in get_num_new_matched_tokens after client.lookup(). Subtracts one if 100% hit so vLLM still generates from at least one real token.
  • can_load — flipped True in update_state_after_alloc once destination blocks exist.
  • token_len — stamped by the worker in get_finished() right before handing the request to the recv thread. Recv uses this, not token_len_chunk.
  • Recv also uses vllm_cached_tokens as a mask offset to avoid re-fetching what the local prefix cache already covers.

Side channel: events & KV-event reporting

current_event: torch.cuda.Event | None — not set by the scheduler. The worker's get_finished() records one event on the compute stream after forward and stamps it on every save ReqMeta before queuing into the send thread: 

current_event = torch.cuda.Event()
current_event.record()           # compute stream
for req in meta.requests:
    if req.can_save:
        req.current_event = current_event
        send_thread.add_request(req)

The send thread synchronize()s on it before reading GPU memory; without this barrier the Mooncake put would race the forward writing KV into the same blocks (torn KV).

token_ids — chunk's tokens, for emitting BlockStored.token_ids on the KV event bus.

original_block_size — the engine's user-visible cache_config.block_size, not the LCM-stretched effective _block_size the connector uses internally. External KV-event consumers index by the user-visible size.

Construction logic

RequestTracker
  ├── token_len             # grows each step (chunked prefill)
  └── num_saved_tokens      # high-watermark; updated when can_save=True

ReqMeta.from_request_tracker:

  num_tokens_to_save = floor(token_len / block_size) * block_size      # if discard_partial
  chunk_boundary     = ceil((num_saved_tokens + 1) / block_size) * block_size
  skip_save          = (caller forced skip) OR num_tokens_to_save < chunk_boundary

  if skip_save and no load_spec: return None
  if not skip_save: tracker.num_saved_tokens = num_tokens_to_save

  ReqMeta(
    req_id              = tracker.req_id
    token_len_chunk     = num_tokens_to_save             # save range this step
    block_ids           = tracker.allocated_block_ids    # per-group blocks
    block_hashes        = caller-supplied                # prefix-cache hashes
    can_save            = not skip_save
    load_spec           = caller-supplied (or None)      # only when external hit
    is_last_chunk       = caller-supplied                # uses prefill_end_tokens
    token_ids           = tracker.token_ids              # for KV events
    original_block_size = engine's cache_config.block_size
    current_event       = None                           # worker fills in later
  )

Easy-to-confuse pair token_len_chunk (the save range) vs load_spec.token_len (the load range) — independent ranges living on the same ReqMeta. The save path reads the former; the load path reads the latter. If you mix them up while debugging, neither path will line up with the block IDs.
Reading order for ReqMeta

Save: from_request_tracker → KVCacheStoreSendingThread._handle_request. The scheduler decides what is saveable; the send thread translates ranges to addresses, dedups against batch_is_exist, calls batch_put_from_multi_buffers.

Load: get_num_new_matched_tokens → update_state_after_alloc → worker.get_finished (stamps load_spec.token_len) → KVCacheStoreRecvingThread._handle_request.

Load path

  1. get_finished() finds each request with load_spec.can_load.
  2. It sets load_spec.token_len to the external cached length.
  3. It enqueues the request on KVCacheStoreRecvingThread.
  4. The recv thread uses coord.load_mask() to skip chunks the local KV spec would not populate.
  5. It turns each chunk into Mooncake keys and GPU address/size lists.
  6. It calls batch_get_into_multi_buffers() and marks the request finished.

Store path

  1. get_finished() records a CUDA event when any request can save.
  2. Each save request receives that event and is enqueued on KVCacheStoreSendingThread.
  3. The send thread aligns token length to coord.lcm_block_size.
  4. It uses coord.store_mask() to avoid storing chunks that no future hit would need.
  5. It deduplicates with batch_is_exist().
  6. It synchronizes the CUDA event before reading GPU memory.
  7. It calls batch_put_from_multi_buffers() and decrements outstanding store jobs.

12. HMA and block-size frames

Mooncake Store with HMA is easiest to reason about if you keep three token frames separate. Mixing these frames is the usual source of wrong conclusions.

Frame Meaning Where used
native group block size Each KV cache group's own physical/spec block size. ChunkedTokenDatabase per group, coordinator masks, address mapping.
hash_block_size Granularity of Request.block_hashes. Request block hasher, BlockHashListWithBlockSize, key generation.
scheduler_block_size Safe prefix length alignment visible to the scheduler. Scheduler num_computed_tokens, Mooncake hit length, HMA convergence.
Resolver rule Single group returns cache_config.block_size * dcp * pcp for both scheduler and hash block sizes. Multiple groups use LCM of group block sizes for scheduler alignment, and either explicit hash_block_size or GCD of group block sizes for hash granularity.

The divisibility chain

The three frames are not independent — they are constrained to nest cleanly: 

hash_block_size  |  group block_size  |  scheduler_block_size

   ↑ smallest                                ↑ largest
   one hash covers this many tokens          scheduler rounds prefix lengths
                                             to this many tokens

Every group block_size is a multiple of hash_block_size (so consecutive hashes concatenate into a group-level block hash via BlockHashListWithBlockSize). The scheduler_block_size = lcm(group block sizes), so every prefix length the scheduler reports is a clean block boundary in every group simultaneously. The largest unit (scheduler) is the one Mooncake's send thread also floors against — that is the coord.lcm_block_size you saw in section 10's store path.

Worked examples

Scenario hash_block_size group block_size scheduler_block_size Why
Single-group full-attn, cache.block_size=16, no DCP/PCP 16 {16} 16 Single group: scheduler = hash = block_size · dcp · pcp.
Single-group, block_size=16, dcp=2 32 {16} (physical) 32 Single-group branch multiplies by DCP/PCP. Scheduler frame and hash frame both become 32.
Gemma-3 (5 SWA : 1 full), both groups block_size=16 16 {16, 16} 16 Multiple groups but uniform: lcm(16,16)=16, gcd(16,16)=16. All frames collapse.
Hybrid: full-attn block_size=32, SWA block_size=16 16 {32, 16} 32 gcd(32,16)=16 hashes; lcm(32,16)=32 alignment. The full-attn group concatenates 2 hashes per block; SWA uses 1 hash per block.
Same hybrid, but user sets cache.hash_block_size=8 8 {32, 16} 32 Explicit override wins. Full concatenates 4 hashes per block; SWA concatenates 2. Resolver asserts both are divisible by 8.
Mamba + full-attn hybrid, mamba_cache_mode="align", both block_size=16 16 {16, 16} 16 "Align" forces Mamba's block size to equal cache_config.block_size, so divisibility holds. Mamba participates in the normal gcd/lcm dance and Mamba prefix caching is active (one state checkpoint per aligned block boundary).
Mamba + full-attn hybrid, mamba_cache_mode ≠ "align" ("all" or "none") = scheduler {cache.bs, Mamba's own size} LCM of the above Mamba's block diverges from cache_config.block_size (e.g. it caches one state per request, not per block_size tokens). Divisibility breaks; resolver backs off so hash_block_size = scheduler_block_size — no finer hashing.
Multi-group, prefix caching off, no KV connector = scheduler varies LCM Block hashes are only consumed by prefix caching and connectors. With both inactive, the resolver collapses hash to scheduler to skip the work.

Walking through the hybrid {32, 16} case end-to-end

Concrete tokens. Block sizes: full-attn group = 32, SWA group = 16, hash_block_size = gcd(32, 16) = 16, scheduler_block_size = lcm(32, 16) = 32. 

Tokens:        [0..15] [16..31] [32..47] [48..63] [64..79] ...

block_hashes:    H0      H1       H2       H3       H4         # one per hash_block_size (16)

SWA group blocks (block_size=16):
  block 0: H0       block 1: H1       block 2: H2 ...

Full-attn group blocks (block_size=32, via BlockHashListWithBlockSize):
  block 0: H0||H1   block 1: H2||H3   block 2: H4||H5 ...

Scheduler num_computed_tokens (must be multiple of 32):
  valid:    0, 32, 64, 96, ...
  invalid:  16, 48 — full-attn group would land mid-block

When Mooncake's lookup returns "I have 48 cached tokens" but scheduler_block_size = 32, the scheduler rounds down to 32 — because 48 would leave the full-attn group's block 1 half-populated and the manager has no partial-block representation.

How each frame is used at runtime

hash_block_size (smallest)

  • Driven by get_request_block_hasher(block_size=hash_block_size) in core.py:202-211.
  • One BlockHash in Request.block_hashes per hash_block_size tokens.
  • The Mooncake PoolKey's chunk_hash is the hex of either one of these hashes (when group block size equals hash block size) or a concatenation of consecutive ones (when group block size is larger).

group block_size (middle)

  • Each KVCacheGroup.kv_cache_spec.block_size — the physical paging unit for that group's GPU tensor.
  • One ChunkedTokenDatabase per group, parameterized by this size, drives token-range → GPU-address translation.
  • SingleTypeKVCacheManager subclasses (FullAttentionManager, SlidingWindowManager, ...) all iterate in this frame.

scheduler_block_size (largest)

  • LCM of all group block sizes.
  • Scheduler's num_computed_tokens is always a multiple of this — guarantees a clean block boundary in every group.
  • Mooncake's send thread re-floors token_len_chunk to this before issuing batch_put: token_len = token_len_chunk // lcm_block_size * lcm_block_size.
  • External lookup return value is also clamped to this in MooncakeStoreScheduler.get_num_new_matched_tokens.
The trap cache_config.block_size is the user-facing knob, but it is not any of these three frames directly. It is the seed for them: single-group multiplies it by DCP·PCP to get the scheduler size; multi-group uses it as the basis for each group's spec block size, then derives scheduler/hash from those. When a connector emits BlockStored events it ships original_block_size = cache_config.block_size so external consumers can re-index in the user-visible frame — never in the LCM-stretched effective frame.

The coordinator is the HMA bridge. It mirrors vLLM cache-manager hit logic over an ExternalCachedBlockPool, not over local allocated blocks. Lookup feeds it an existence set from Mooncake keys. Load and store paths ask it for masks so each group only transfers chunks that the corresponding KV spec would actually use.

    -> which chunks to load locally

Practical debugging tip When a request appears stuck, ask which boundary failed: scheduler never emitted load metadata, worker never enqueued the recv, recv completed but finished_recving did not reach scheduler, or scheduler received it but did not promote WAITING_FOR_REMOTE_KVS.
