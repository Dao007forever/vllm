KV Cache Allocation & Group Block IDs

How vLLM groups KV-cache layers, why groups are shaped to have uniform physical block sizes, and how logical block IDs map to physical KV memory.

TL;DR

A group is the equivalence class of layers that share a KV-cache lifecycle. Each group has its own block table, but all groups draw block IDs from one shared BlockPool. The column-packed tensor layout turns that "global ID" into a physical slot reused across groups — not just a logical reservation label. Five constraints stack: lifecycle → group separation → uniform layer count → uniform page size → shared BlockPool → column packing.

Mental model

vLLM separates two ideas:

  • Logical block ID: an integer managed by the scheduler/block pool, such as 10.
  • Physical KV memory: the actual K/V tensor storage used by a specific layer or KV-cache group.

KV-cache groups have separate block tables, but their block IDs come from one shared scheduler-side BlockPool. In the current column-packed layout, that global ID is also a physical slot in the preallocated KV-cache pool: block ID 10 means slot 10 in each column tensor, not a group-local slot.

Layout-dependent In a hypothetical per-group physical layout, the scheduler could still hand out globally unique logical IDs, but ID 10 would map to group_0_tensor[..., 10, ...] or group_1_tensor[..., 10, ...] — different byte ranges. Column packing turns global uniqueness into cross-group physical-memory pooling.

For a hybrid model, one request may hold different block IDs in different groups:

group 0 table: [10, 11, 12]
group 1 table: [13, 14, 15]
group 2 table: [16, 17, 18]

Inside a group, all layers share the same block table, but each layer has its own KV-cache tensor view:

group 1, layer 7,  block 13 -> K/V memory for layer 7
group 1, layer 13, block 13 -> K/V memory for layer 13

The actual key is effectively (layer_name, block_id). The group selects the block table and attention metadata.

Why groups exist at all

Before "why must group blocks be uniform-sized" comes the prior question: why have groups at all?

A group is the equivalence class of layers that share a KV-cache lifecycle. Different attention types disagree about three things; any one is enough to forbid sharing a block table:

Eviction rule

Full attention: K/V at position p available forever. SWA: free K/V at p once past p + W. Mamba: state checkpoints at block boundaries. One block table can encode one rule.

Scan direction

FullAttentionManager.find_longest_cache_hit walks L→R; SlidingWindowManager walks R→L; MambaManager walks R→L through state checkpoints. These fall out of attention-pattern correctness.

Block size

Attention typically block_size=16. Mamba uses a different block size driven by SSM kernel state cadence. They cannot index into the same block table.

The code shape mirrors this: each attention type gets its own SingleTypeKVCacheManager subclass (FullAttentionManager, SlidingWindowManager, ChunkedLocalAttentionManager, MambaManager, CrossAttentionManager) with its own find_longest_cache_hit and remove_skipped_blocks.

What about per-layer block tables instead?

It works but it's wasteful, because layers of the same attention type have identical lifecycles. Sharing costs nothing semantically and saves real GPU memory.

Why block tables live on GPU and why int32 The block table is a (max_num_reqs, max_num_blocks_per_req) int32 tensor on GPU because Triton kernels read it directly via block_table_ptr. Int32 because entries are logical block IDs (indices in [0, num_blocks)), not memory addresses. Even 60M blocks fits comfortably under 2³¹ ≈ 2.1B.
Asymmetry: int32 block table, int64 slot mapping 
# block_table.py:357 (Triton kernel)
tl.load(block_table_ptr + ...).to(tl.int64)
# slot_id = block_id * block_size + offset, upcast to int64
Slot mapping is a per-token tensor (kilobytes), so int64's cost is negligible and PyTorch's index-tensor API expects int64. The block table is large (megabytes), so int32 saves real memory there.

Concrete numbers

Typical production config: max_num_reqs=512, max_model_len=131072, block_size=16:

max_num_blocks_per_req = 131072 / 16 = 8192
one table = 512 * 8192 * 4 bytes = 16 MiB on GPU

An 80-layer model with per-layer tables would burn 80 × 16 MiB = 1.28 GiB on metadata alone. With groups, that drops to one table per group (e.g., 6 tables for Gemma-3 hybrid → 96 MiB), a 13× reduction.

At 1M context, per-layer table = 128 MiB, so 80 layers = 10 GiB before any K/V is stored. The savings ratio is n_layers / n_groups — most load-bearing on deep models with few group types.

How does layer-specific data work if metadata is shared?

Each group has one block table, one cache-hit walk result, one slot-mapping tensor. But each layer still has its own K/V bytes. How?

Metadata is logical, the K/V tensor is physical Dispatch happens at the forward() call boundary, not inside the kernel.

When the model runner steps through layers:

  • Layer L's attn.forward() is called with layer L's own kv_cache tensor (looked up via kv_caches[layer_name]).
  • The attn_metadata argument carries the group-shared block_table and slot_mapping.

What about one giant group per attention type?

Within a single attention type, why subdivide (Gemma-3's 50 SWA layers split into 5 groups of 10) instead of one giant SWA group? The uniform-page-size constraint (next section) forces splitting an attention type's layers into multiple groups when there are more layers than the smallest other type.

The hierarchy of reasons:

  1. Different attention types → must be different groups (lifecycle).
  2. Same attention type, similar count → one group per type (sharing is free).
  3. One group block-table entry is shared by every layer in that group, but each layer still needs its own K/V payload for that token block.
  4. Same attention type, more layers than smallest type → split into equal-sized groups so block-pool slots stay uniform-sized.
  5. Uniform group block size → one shared BlockPool can hand block IDs to any group.
  6. One shared BlockPool plus per-group block tables → column-packed storage can share physical block slots across groups while keeping separate block tables for different cache lifecycles.

Why groups need uniform physical block size

vLLM's allocator wants a simple invariant: one block allocated from the shared BlockPool corresponds to the same amount of physical KV memory, regardless of which group uses it.

A group block is the allocation unit, so its physical size is proportional to the number of layers in that group: 

group block bytes = layers_per_group * page_size_per_layer

Suppose a model has 10 full-attention + 50 SWA layers. If only two groups:

Group 0: 10 full layers    Group 1: 50 SWA layers
Group 0 block = KV memory for 10 layers
Group 1 block = KV memory for 50 layers   (5x bigger!)
Why this breaks the scheduler The scheduler could no longer ask "request needs 3 more blocks; do we have at least 3 free blocks?" — it would need different-sized chunks, fragmentation, per-group free capacity. Centralized scheduling falls apart.

vLLM's solution: split the 50 SWA layers into 5 groups of 10:

Group 0: 10 full layers
Group 1..5: 10 SWA layers each
1 block ~= KV memory for 10 layers
Code: the assumption is documented 
# vllm/v1/core/kv_cache_utils.py
# Physical memory per block: Must be the same across all KV cache groups.
# Breaking this assumption is non-trivial due to memory fragmentation
# concerns when allocating blocks of different sizes.

This assumption ties to centralized scheduling. vLLM profiles memory once at startup, preallocates KV-cache tensors, and gives the scheduler a fixed logical block capacity.

The structural mechanism: shared column tensors

The "fragmentation" framing is the consequence. The mechanism sits in the allocator at _get_kv_cache_config_uniform_page_size:

# General case:
# We will have group_size memory pools, each is shared by one layer from
# each group. As layers of different groups have different block table,
# they will use different parts of the shared Tensor.
group_size = max(len(group.layer_names) for group in kv_cache_groups)
page_size = get_uniform_page_size(...)
num_blocks = get_num_blocks(vllm_config, group_size, available_memory, page_size)
for i in range(group_size):
    shared_by = []
    for j in range(len(kv_cache_groups)):
        if i < len(kv_cache_groups[j].layer_names):
            shared_by.append(kv_cache_groups[j].layer_names[i])
    kv_cache_tensors.append(
        KVCacheTensor(size=page_size * num_blocks, shared_by=shared_by)
    )

group_size here is the number of layers per group (not the number of groups). The outer loop iterates layer-slots within a group; the inner loop walks across groups. Each KVCacheTensor is shared by one layer from each group.

Each column is one physical tensor of size page_size × num_blocks. Each row is one group's row of layers — every group contributes exactly one layer to every tensor. 30 layers, 5 tensors — layer count drops out, block count stays the same.
Block ID 42, seen from three vantage points

Scheduler view

Block 42 is a KVCacheBlock with ref_cnt ≥ 1. It's currently held by one group's block table.

Physical view

In every column tensor, byte offset 42 × page_size is reserved for block 42. Tensor 0's bytes are for one layer; Tensor 1's for another. Same index, different bytes.

Layer view

L5 sees its bytes as K, V tensors of shape [block_size, n_kv_heads, head_dim]. L0 sees its own bytes the same way. No layer can see another layer's bytes.

"Block 42" is a row index shared across all column tensors — not a single physical address. Each tensor independently has a slot 42. The scheduler invariant (one holder at a time) ensures the row is consistent — whoever holds block 42 owns slot 42 in all column tensors at once.

Packing is column-major across the layer grid, but each physical tensor is block-major: column_tensor[block_id]. The layer_slot has already selected which column tensor this layer uses. Every tensor has size page_size * num_blocks and exactly num_blocks slots. A block ID B means slot B in every tensor — no per-group offset needed.

Inside one block: byte layout and per-layer reinterpretation

A "block" in the column tensor is page_size bytes at offset block_id * page_size. The layout within those bytes is the layer's KV cache shape:

one block = page_size bytes
          = [2, block_size, num_kv_heads, head_size] * dtype_bytes
            ^   ^           ^             ^
            |   |           |             head dim
            |   |           number of KV heads (after TP/MQA/GQA)
            |   token position within the block (0..block_size-1)
            K vs V (unbind(0) splits these in flash_attn.py)
Sharing is across groups, not within A column tensor is shared by exactly one layer per group. Layers within the same group are in different column slots → different column tensors → different physical memory. 
column tensor 0 (raw allocation, page_size * num_blocks bytes):
  offset 0           page_size       42*page_size      43*page_size
  |<- block 0     ->|<- block 1   ->|...|<- block 42 ->|<- block 43 ->|...

Block 42 (page_size bytes) — interpreted by each layer's view as:

  L5's view  (group 0, FULL)  : [2, block_size, n_kv_heads, head_size]
  L0's view  (group 1, SWA)   : [2, block_size, n_kv_heads, head_size]
  L1's view  (group 2, SWA)   : [2, block_size, n_kv_heads, head_size]
  ... (one view per layer in shared_by)

Each kv_caches[layer_name] is a .view(dtype).view(kv_cache_shape).permute(...) of the same raw bytes. PyTorch handles address arithmetic via strides; no copy.

No collision because of the scheduler invariant At any single moment, the global block ID 42 appears in at most one group's block table. Block IDs are popped from one global queue and ref-counted as held. Even though L5, L0, L1, L2, L3, L4 could all reference block 42, only one does at a time.

Within a group, every layer is in a different column slot. For group 0 with 5 layers (L5, L11, L17, L23, L29), they occupy slots 0..4 → Tensors 0..4 → five distinct allocations. Block 42 in Tensor 0 (L5's storage) and block 42 in Tensor 1 (L11's storage) are separate page_size-byte regions.

Worked example: hybrid Full + SWA, with NULL blocks

The diagrams above show six full-attention column tensors in abstract. Concrete numbers help see where NULL blocks come from and why every group's block table is the same width. Here's a small hybrid case — two groups, two layers each:

Setup
  • 4 layers, 2 groups, group_size = 2 (2 column tensors)
    • G1 = {L1, L3} — Full attention FULL
    • G2 = {L2, L4} — SWA, sliding_window = 8 SWA
  • block_size = 4, request length = 16 tokens

Both groups' block tables are the same width

T0 hosts L1 (G1) and L2 (G2); T1 hosts L3 and L4. L1 and L2 share the tensor but never collide — L1 uses slot 8 (block 2) for pos 4 while L2 has nothing there (PAD). At pos 8, L1 uses slot 12 (block 3) and L2 uses slot 20 (block 5) — same tensor, different rows. The global block-pool invariant is exactly what makes this collision-free.

G1 holds block IDs {1, 2, 3, 4}; G2 holds {5, 6} plus two null_block sentinels. All real IDs are drawn from the same global BlockPool — the scheduler guarantees they don't collide.

G2's first two table entries are NULL because the SWA layers' window ends at position 15 and only reaches back to position 8. Positions 0..7 are provably unreadable by L2/L4, so the scheduler refuses to back them with physical blocks. The NULL sentinels stay in the table to keep the array the same width as G1's — they are the SWA group's "this position has no storage" marker.

Slot-ID translation at the same logical positions

The formula is identical across groups (it's just block_id * block_size + offset); only the table lookup differs:

Token position Group block_index Table entry → block_id slot_id Outcome
pos 8 (inside SWA window) G1 8 // 4 = 2 table[2] = 3 3 * 4 + 0 = 12 L1, L3 write K/V at slot 12 of their column tensors
G2 2 table[2] = 5 5 * 4 + 0 = 20 L2, L4 write K/V at slot 20 of their column tensors
pos 4 (outside SWA window) G1 1 table[1] = 2 2 * 4 + 0 = 8 L1, L3 write at slot 8
G2 1 table[1] = NULL PAD_SLOT_ID = -1 L2, L4 skip the write (kernel honors -1 as "don't touch")
What this example demonstrates
  1. Block tables are the same size across groups — both are 4 entries long. This is what makes one global slot-mapping kernel work for all groups (just swap the table pointer per group, same formula).
  2. NULL entries are the SWA group's way of saying "no physical storage here." They translate to PAD_SLOT_ID in the slot mapping and the attention kernel skips them. See KV_CACHE_ASSUMPTIONS §4.
  3. Block IDs come from one global pool. G1's {1, 2, 3, 4} and G2's {5, 6} were popped from a single FreeKVCacheBlockQueue. No two groups simultaneously hold the same block ID — that's the scheduler invariant that lets column packing share physical memory across groups.
  4. L1's slot 8 and L3's slot 8 are in different physical tensors. L1 lives at column slot 0 of G1, L3 at column slot 1 of G1 → different column tensors. L1 and L2 (different groups) might share a column tensor, but L1 holds block ID 2 there and L2 holds block ID 5 there → different slot IDs, no collision.

In code, the NULL sentinel translates to a flag on the KVCacheBlock object (is_null=True) and the scheduler invariant is enforced by the global FreeKVCacheBlockQueue — an ID is either on the free queue or in exactly one group's table at any moment, never both.

How Mamba/SSM layers fit the uniform-page invariant

SWA was the easy hybrid case: it still stores per-token K/V, just with a shorter lifetime. Mamba (and GatedDeltaNet / GDN, RWKV, other linear attentions) is the hard case — its “cache” isn't K/V at all. A Mamba layer keeps two small recurrent state tensors per request (see MAMBA.html):

  • conv_state — the rolling depthwise-conv window, shape ~(d_inner, d_conv−1).
  • ssm_state — the recurrent summary, shape ~(d_inner, d_state).

Their summed byte size is the Mamba layer's natural page_size_bytes — and it has nothing to do with block_size, because a Mamba block isn't “16 tokens of K/V,” it's one snapshot of state. Left alone, that page would be a different size than an attention page, which would break the one-block-is-one-uniform-size invariant the shared BlockPool depends on (§Why groups need uniform physical block size).

The fix: pad the Mamba page up to the attention page vLLM first raises the attention block_size until one attention page is at least as large as one Mamba state page, then pads the Mamba page with dead bytes so the two are exactly equal. After that, a Mamba block and an attention block are byte-for-byte the same allocation unit — the BlockPool can't tell them apart.
Code: alignment + padding 
# vllm/platforms/interface.py (verify_and_update_config path)
# 1. Force attention block_size up so attn_page ≥ mamba_page:
attn_block_size = kernel_block_alignment_size * cdiv(
    mamba_page_size, kernel_block_alignment_size * attn_page_size_1_token)
...
# 2. Pad the mamba page to *exactly* the attention page:
attn_page_size = cache_config.block_size * attn_page_size_1_token
assert attn_page_size >= mamba_page_size
if attn_page_size != mamba_page_size:
    cache_config.mamba_page_size_padded = attn_page_size   # <- the pad
MambaSpec.page_size_bytes returns page_size_padded when set, so every consumer downstream (allocator, model runner, connector) sees the padded, attention-equal size. The padding percentage is logged at startup.

One block per request, not one per 16 tokens

Uniform page size makes a Mamba block cost the same as an attention block, but a request needs far fewer of them. How many depends on mamba_cache_mode (MambaSpec.max_memory_usage_bytes):

ModeBlocks per requestMeaning
"none" (default)1 + num_specOne live state snapshot; no prefix reuse.
"align"2 + num_specSnapshot + one aligned checkpoint.
"all"cdiv(max_len, block) + num_specFull prefix caching of checkpoints.

conv_state and ssm_state are two views of one buffer

Per layer, the allocator hands out one contiguous raw tensor of num_blocks × page_size_bytes. conv_state and ssm_state are not separate allocations — they are two torch.as_strided views into that single buffer at different storage offsets:

Code: striding two states out of one tensor 
# vllm/v1/worker/gpu_model_runner.py (initialize_kv_cache_tensors)
elif isinstance(kv_cache_spec, MambaSpec):
    raw_tensor = kv_cache_raw_tensors[layer_name]   # ONE buffer
    storage_offset_bytes = 0
    for shape, dtype in zip(kv_cache_spec.shapes, kv_cache_spec.dtypes):
        tensor = torch.as_strided(raw_tensor.view(dtype), ...,
                                  storage_offset=storage_offset_bytes // dtype_size)
        state_tensors.append(tensor)
        storage_offset_bytes += stride[0] * dtype_size
    kv_caches[layer_name] = state_tensors            # [conv_view, ssm_view]
The trade vLLM is making A few percent of padding, on a handful of Mamba layers each holding one block per request, is cheap. The alternative — giving Mamba its own physical pool with its own page size — would re-introduce the differently-sized-blocks fragmentation the uniform invariant exists to avoid, and would statically strand memory between the attention and Mamba pools. Padding pushes the waste onto the side where it costs least.

How NIXL transfers SSM state (independent of the allocator)

Uniform page size is not what makes SSM transferable It's tempting to think the mamba_page_size_padded trick is what lets the NIXL connector move SSM state. It isn't. The padding is a BlockPool / column-packing concern, applied in platform config for any hybrid model regardless of whether a connector is attached. NIXL operates on whatever physical tensors the allocator produced and would work just as well under a per-group allocator.

NIXL registers each layer's physical buffer as its own region and tracks a per-layer block length — it is explicitly built for SSM and attention layers having different page sizes and different block counts. Mamba layers carry _logical_num_blocks while attention layers carry num_blocks, and a _physical_blocks_per_logical_kv_block ratio reconciles them.

Code: per-layer block lengths, not a global page size 
# vllm/distributed/kv_transfer/kv_connector/v1/nixl/worker.py (register_kv_caches)
# "physical page_size may differ when kernel requires a specific block size.
#  This leads to SSM and FA layers having different num_blocks.
#  `_physical_blocks_per_logical_kv_block` ratio is used to adjust for this."
self.block_len_per_layer = list[int]()      # <- one entry per layer
...
num_blocks = (self._logical_num_blocks       # mamba: fewer, larger blocks
              if isinstance(layer_spec, MambaSpec)
              else self.num_blocks)          # attention: more, smaller blocks

What is true and worth knowing about SSM transfer:

One physical region per Mamba layer, logically doubled
  • Each Mamba layer's contiguous buffer is registered as its own NIXL region (its own base address), distinct from every attention region. Descriptors are partitioned: attention (“FA”) descriptors first, SSM descriptors after.
  • conv+ssm are registered as a single region, then logically split in two so conv and ssm can be transferred/indexed independently — mirroring the “one buffer, two views” allocation above.
Code: SSM region registration 
# vllm/distributed/kv_transfer/kv_connector/v1/nixl/worker.py (register_kv_caches)
if self.transfer_topo.virtually_split_kv_in_blocks:
    # ... for Mamba layers, we register SSM+Conv as a single region and
    # then duplicate it logically to be able to index SSM/Conv separately.
    self.num_regions *= 2

Two SSM-specific constraints the connector enforces:

  • No cross-layer block layout. NixlConnector.prefer_cross_layer_blocks returns False as soon as any group is a MambaSpec (nixl/connector.py) — hybrid SSM models don't yet support that packing.
  • Heterogeneous TP (P-TP ≠ D-TP) splits the conv state into sub-projections for a fine-grained “3-read” RDMA transfer, which requires the DS conv-state layout (VLLM_SSM_CONV_STATE_LAYOUT=DS); see ssm_conv_transfer_utils.py.

What NIXL actually depends on is unrelated to the uniform-page invariant: each region's byte layout (block length, block count) must be known, and producer and consumer must agree on it — enforced by a compatibility hash, plus an “all regions same total size” assertion on the non-MLA path. Those hold under either allocation strategy. See transfer/NIXL.html for the connector internals.

Why column packing instead of per-group or per-layer

The same total memory could be cut three ways:

Layout# tensorsPer-tensor size
Per-layer30page_size * num_blocks_per_layer
Per-group6page_size * 5 * num_blocks_per_group
Column (current)5page_size * num_blocks

The columns layout is chosen because:

  1. The global BlockPool is physically pooled, not just globally named.

    Each column tensor has num_blocks slots, and a block ID B means slot B in every column. The scheduler can pop block IDs from one queue and hand any of them to any group. In per-group, a globally unique ID would still be just a logical label — B in group 0 vs group 1 would be different byte ranges. Every private tensor would need to reserve the full ID-range as slots even if its working set is much smaller.

  2. Layer view shape stays kernel-friendly.

    With column packing, each layer's view is [num_blocks, 2, block_size, num_kv_heads, head_size], indexed directly by block ID — what FlashAttention expects. Per-group layout would produce [num_blocks, layers_per_group, ...] and force every backend to take an extra layer-index argument.

  3. Layers sharing a block table want separate buffers.

    All 5 layers in group 0 share group 0's block table, so when group 0 holds block ID B, all 5 write to slot B. They cannot share a tensor without colliding. Layers from different groups always hold different block IDs (scheduler invariant), so they can share a tensor. The column axis is exactly the axis that can collapse into shared storage.

  4. Padding for uneven groups is cheap.

    The if i < len(kv_cache_groups[j].layer_names) guard lets the last column be shared by fewer layers when groups have unequal sizes (e.g., 13 FULL + 12 SWA padded to group_size=13).

  5. Fewer allocations. 5 large vs 30 small. Secondary, but consistent.

Gemma-3-style hybrid grouping

Gemma-3-style models commonly use a repeated pattern like:

SWA, SWA, SWA, SWA, SWA, FULL

For N repetitions: 5N SWA + N full-attention layers.

vLLM does NOT create groups by contiguous layer ranges It creates groups that preserve a uniform layer count per group: 
SWA group 0: layer 0,  layer 6,  layer 12, ...
SWA group 1: layer 1,  layer 7,  layer 13, ...
SWA group 2: layer 2,  layer 8,  layer 14, ...
SWA group 3: layer 3,  layer 9,  layer 15, ...
SWA group 4: layer 4,  layer 10, layer 16, ...
FULL group:  layer 5,  layer 11, layer 17, ...

Each group has N layers. The grouping is for memory / block-table layout, not execution order. Forward execution still runs layers in normal order: 0, 1, 2, 3, ...

(See KV_CACHE_ASSUMPTIONS.html §14 for why the split is strided, not contiguous — pipeline parallelism.)

Startup memory sizing

The worker keeps one block table per group, with int32 IDs:

        (self.max_num_reqs, max_num_blocks),
        dtype=torch.int32, device=device,

    self.block_tables.append(block_table)

Attention metadata aliasing

During input preparation, vLLM starts with metadata for group 0 and swaps in the group-specific block table and slot mapping for later groups. Crucially, the resulting attention metadata is assigned to every layer in the attention group as a reference to the same object, not a copy: 

for layer_name in attn_group.layer_names:
    attn_metadata_dict[layer_name] = attn_metadata_i   # <-- aliased

Per-layer attn_metadata_dict lookup returns identical metadata for every layer in the group, so there's only one block_table tensor and one slot_mapping tensor in memory per group.

Final lookup shape:

layer_name -> layer KV tensor                              (per-layer)
layer_name -> attention metadata (shared within group)     (per-group via aliasing)
metadata   -> group block table, group slot_mapping        (per-group)
block id   -> physical slot inside that layer/group KV tensor

Summary

  • vLLM splits hybrid models into KV-cache groups so each group block has uniform physical size.
  • KV-cache groups have separate block tables, but their block IDs are allocated from one shared BlockPool.
  • Physical lookup is layer-specific after the group block table selects the block ID: (layer_name, block_id) → physical K/V memory.
  • Mamba/SSM layers join the same invariant by padding their state page up to the attention page (mamba_page_size_padded); conv_state and ssm_state are two strided views of one per-layer buffer, and a request needs only one block (default mode).
  • NIXL transfer is independent of that padding: it registers one region per Mamba layer (logically split into conv/ssm) and tracks per-layer block lengths, explicitly handling SSM and attention having different page sizes and block counts — so it works under a per-group allocator too.
