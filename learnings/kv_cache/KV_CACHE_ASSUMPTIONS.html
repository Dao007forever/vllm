KV Cache: Hidden Assumptions & Constraints

Fifteen invariants that look arbitrary in the code until you ask "what would break if we did the obvious other thing?"

TL;DR

The KV cache subsystem rests on a small number of load-bearing decisions: blocks (not tokens), block-major memory, post-forward hashing, distinct null/PAD sentinels, single-threaded scheduler, one global block ID space, and a chained hash with extra-keys. Most "obvious" alternatives produce silent corruption — wrong numbers in valid tensor slots with no kernel-level trip-wire. That's why the surrounding code reads like a contract.

1. Why allocate in fixed-size blocks at all (not per-token)

The unit of allocation is a block of block_size tokens. The design makes three trades at once:

O(1) allocation

FreeKVCacheBlockQueue is a doubly-linked list of KVCacheBlock objects because the scheduler needs O(1) push/pop and O(1) middle-removal.

Kernel tiles

FlashAttention / FlashInfer issue one TMA / vector load per block_size-tokens chunk. Per-token allocation would force scatter-gather.

Hashable unit

Prefix caching keys content by block hash; you can only hash a fixed unit and chain it. Token-level hashes would multiply hash-table entries by 16×.

The price: internal fragmentation in the last block, acceptable because the next decode step fills it.

Example: 33-token request, block_size=16 
block 0: tokens 0..15
block 1: tokens 16..31
block 2: token 32 plus 15 empty future slots
The scheduler accounts for 3 fixed slots, not 33 independently managed token slots.

2. Why block_size cannot be chosen freely

Three layers each impose a constraint, and they don't agree:

When manager ≠ kernel, BlockTable silently splits each manager block into block_size / kernel_block_size kernel blocks via map_to_kernel_blocks. The scheduler still thinks in manager blocks; the kernel sees twice as many.

What breaks if you ignore this Pick a manager block size the kernel doesn't accept → worker raises ValueError("kernel_block_size N must divide kv_manager_block_size M") on init. Subtler: choose a hash block size that doesn't divide one group's block size and prefix-cache lookup would silently return the wrong span — guarded by an assertion in kv_cache_coordinator.py, so it crashes loudly.
Example 
manager block B, offsets 0..31
  -> kernel block 2B,   offsets 0..15
  -> kernel block 2B+1, offsets 0..15
If the manager block were 24, that same backend could not split it into 16-token kernel blocks without a partial.

See KERNEL_BLOCK_ALIGNMENT.html for the CUTLASS MLA-specific 128-token tile assertion.

3. Why slot layout is block-major, not token-major

The slot mapping kernel computes:

slot_ids = block_numbers * block_size + local_block_offsets

Tokens within a block are contiguous in physical memory. Token-major would be natural if you only thought about "logical position → physical address." But:

  • A FlashAttention tile at position p reads K[block_table[p // block_size]][0 : block_size, :] — one contiguous load. Token-major would force scattered gathers.
  • The per-layer K/V tensor view is [num_blocks, 2, block_size, num_kv_heads, head_size] — block-major is the only layout where block_id is a direct first-axis index.
  • Copying a block (preemption, swap-out, P/D transfer) is one contiguous memcpy of page_size bytes per layer.
Example: block_size=16, group table [42, 99], position 19 
block_numbers = 99
local_block_offsets = 3
slot_id = 99 * 16 + 3 = 1587
The 16 slots for block 99 are contiguous: 1584..1599.

4. Why the null block / PAD_SLOT_ID = -1 exists

Two distinct sentinels you might confuse:

is_null=True on a KVCacheBlock

Appears in block tables for SWA / Mamba groups when a request has gaps (out-of-window positions). Has ref_cnt permanently zero, never enters free queue, never gets cached.

PAD_SLOT_ID = -1

Written into slot_mapping for tokens that exist in the batched tensor but aren't real (CUDA-graph padding, positions past the request end). Writes to slot -1 go into a no-op address.

Why distinct? Block 0 is a real allocatable block; "block id == 0" can't be a sentinel. Slot 0 belongs to block 0 token 0; same problem. Hence -1, out of range for both block IDs (uint) and slot IDs.

What breaks if PAD_SLOT_ID were 0 Every CUDA-graph-padded decode token would write its phantom K/V over the real K/V at block 0 token 0 of the first request. Silent corruption.

5. Why blocks are hashed after the forward pass, not before

BlockPool.cache_full_blocks() runs in KVCacheManager.cache_blocks only after the scheduler step has executed. The naive design would hash up-front — you know the tokens before you run the model.

The hash isn't a key into the block, it's a key into KV content Two requests with identical token prefixes have identical KV only if computation actually happened on those tokens with the same weights/dtype/config. Hashing before the forward would let a request share KV that another request hasn't actually computed yet — a write-write race in the GPU buffer.

The two-phase split: lookup hash computed eagerly on CPU (via update_block_hashes), insertion deferred until after the GPU writes. Lookup is cheap, insertion is safe.

Example Requests A and B have the same first 16 tokens and are scheduled in the same prefill wave. B may compute the same hash as A on CPU, but B cannot hit A's block yet because A's worker has not written the K/V contents. The (hash -> block_id) entry becomes valid only after A's forward finishes and cache_blocks inserts it.

6. Why only full blocks get cached

Partial blocks are never inserted into BlockHashToBlockMap. The hash input is (parent_hash, block_tokens, extra_keys) — if you hashed a partial block, the next decode token would land in that block, the hash would change, and the prior insertion would be a stale orphan.

The fix would be deletion-on-extend, but that needs a back-reference plus a write on every decode step. Easier to wait for the block to fill.

Side effect: prefix cache hit granularity is block_size tokens — a 3-token shared suffix in a 16-token block is invisible to the cache.

Example: block_size=16 
request length 15 -> 0 cacheable blocks
request length 16 -> 1 cacheable block
request length 31 -> 1 cacheable block
request length 32 -> 2 cacheable blocks

7. Why the free queue is doubly-linked, not a deque

FreeKVCacheBlockQueue is a hand-rolled doubly-linked list with sentinel head/tail. The docstring says it's for O(1) middle-removal — why?

When a request hits the prefix cache, the cached blocks already exist in the free queue (freed but not yet evicted). BlockPool.touch has to pull them out of the middle. A collections.deque only supports O(1) ends — this would be O(N) over tens of thousands of blocks.

The eviction tiebreak

Within blocks freed at the same time, blocks with more hash tokens (the tail of a chain) are at the front. BlockPool achieves this by reversing the block order when freeing a request's blocks. Tail blocks evict first because they only help one specific request continuation. Head-of-chain blocks (system prompts, shared prefixes) survive longer because more requests will hash-hit them.

Code: the fake sentinels reduce branching 
# kv_cache_utils.py: FreeKVCacheBlockQueue
self.fake_free_list_head = KVCacheBlock(block_id=-1)
self.fake_free_list_tail = KVCacheBlock(block_id=-1)
# Implementation guarantees fake head and tail are NEVER popped,
# so each real block in the queue has prev and next blocks.
The class is hand-rolled in pure Python because every cache hit touches it — closing the perf gap to deque (C) by not allocating Python objects during list ops.
Example: freed blocks [10, 11, 12] 
front of eviction queue -> 12, 11, 10 -> back
If a new request later hits block 11 as the second block of a shared system prompt, touch(11) removes it from the middle in O(1), while block 12 remains easier to evict.

8. Why ref counting has no atomics

block.ref_cnt += 1 and the matching decrement in free_blocks are plain Python attribute writes. No lock, no atomic, no compare-and-swap. This works because the scheduler is single-threaded and is the only writer.

Load-bearing for performance The scheduler runs once per step on the critical path. Acquiring even a threading.Lock per block touch would be a measurable hit at thousands of blocks per step.
What breaks if you parallelize the scheduler ref_cnt double-decrement → still-referenced block to free queue → next allocation hands out a slot another request is reading → silent corruption. Any future multi-threaded scheduler design needs to either shard by request id or atomicize the block pool.
Example: shared block 20 Touch for B: 1 -> 2. A finishes: 2 -> 1. Block 20 must not enter the free queue until B finishes. A lost or double decrement puts a live block back into circulation.

9. Why NONE_HASH is randomized per process by default

init_none_hash seeds the hash chain with os.urandom(32) unless PYTHONHASHSEED is set. Why not just use a fixed sentinel?

Same reason CPython randomizes hash() Prevent adversarial cache poisoning. With a fixed seed, a request prefix that produces a known collision with a sensitive cached prefix would let one tenant read another's cached KV. Per-process randomization makes the chain values unpredictable across runs.

The cost: cache cannot be reused across process restarts (e.g., a connector that persisted hashes to disk would see all-misses after restart). Set PYTHONHASHSEED if you want cross-process reuse — explicit opt-in for the security tradeoff.

Code: init_none_hash 
# kv_cache_utils.py
def init_none_hash(hash_fn):
    global NONE_HASH
    hash_seed = os.getenv("PYTHONHASHSEED")
    if hash_seed is None:
        NONE_HASH = BlockHash(os.urandom(32))
    else:
        NONE_HASH = BlockHash(hash_fn(hash_seed))

10. Why all groups use the same num_blocks, even when SWA doesn't need them

num_blocks from get_num_blocks is global. An SWA group provably never holds more than ceil(W / block_size) live blocks per request, so allocating num_blocks slots for it looks wasteful.

It isn't waste, it's the column-packing invariant Block ID B means "slot B in every column tensor." If groups had different block counts, the scheduler couldn't hand any block ID to any group without per-group offset arithmetic. The "wasted" SWA slots are shared by full-attention layers in the same column tensor. Total memory is still available_memory.

The genuine waste is peak per-request blocks held: HybridKVCacheCoordinator allocates ceil(num_tokens / block_size) blocks in lock-step across all groups, even though SWA groups only need W's worth. See KV_CACHE.html "Sparser SWA block allocation".

11. Why workers get clamped to the minimum num_blocks across ranks

# unify_kv_cache_configs
min_num_blocks = min(c.num_blocks for c in kv_cache_configs)
for c in kv_cache_configs:
    c.num_blocks = min_num_blocks

Workers profile their own GPU memory; ranks may differ. The scheduler, however, hands out one global block ID space every worker must honor. If worker A has 10000 blocks and worker B has 9800, block 9900 would address valid memory on A and out-of-bounds on B.

Pessimistic clamping The only safe choice without per-worker block ID namespaces. Cost: throwing away 200 blocks on A — small vs the alternative (per-worker tables, ID translation on cross-worker transfers).

12. Why preempt the latest request, not the oldest

When allocate_slots returns None (out of blocks), the scheduler preempts:

if self.policy == SchedulingPolicy.PRIORITY:
    preempted_req = max(self.running, key=lambda r: (r.priority, r.arrival_time))
else:
    preempted_req = self.running.pop()  # last appended

FCFS preempts the most recently arrived running request. Counterintuitive — the oldest has the most computed KV (longest context, most blocks). Preempting it means re-running prefill on tens of thousands of tokens. Preempting the youngest is cheaper to recover from.

Preempted requests go back to WAITING, not killed: their cache entries may be hit by the very requests that bumped them. Preemption is conservative; eviction is aggressive; they're separate mechanisms acting on the same blocks.

Example 
old request:   12000 computed tokens, hundreds of blocks
young request:   128 computed tokens, a few blocks
Preempting the old frees more blocks immediately, but resuming may cost a long prefill replay. Preempting the young usually costs much less work.

13. Why extra_keys exists and what gets in it

hash_block_tokens takes extra_keys alongside the parent hash and tokens. Without extra_keys, two requests with identical token sequences but different conditioning would alias:

Source of aliasingWhere it's added
LoRA — different adapters → different K/V from same tokensLoRA name appended to first block's extra_keys
Multi-modal — image embeddings vary by inputMM hash + per-block-offset
cache_salt — opaque request-supplied stringFirst block only

cache_salt exists not for correctness but for experimental isolation: when benchmarking two prompt variants on the same prefix, you don't want one to warm the cache for the other.

What breaks if extra_keys were dropped A request switching LoRA adapters mid-prefix would hit cached blocks computed with the previous adapter's weights. Quietly wrong outputs.
Example 
tokens: [128000, 791, 3931, ...]
request A: LoRA adapter = sql
request B: LoRA adapter = python
Without LoRA in the key, B could reuse K/V produced with A's adapter weights.

14. Why the strided group split (layers[i::num_groups]) instead of contiguous slicing

_get_kv_cache_groups_uniform_page_size splits a same-type layer list as layers[i::num_groups] (every Nth layer) rather than layers[i*g:(i+1)*g]. Both yield equal-sized groups; only one survives pipeline parallelism.

Pipeline parallelism partitions layers into contiguous stage ranges. If a group held a contiguous slice, it could land entirely on one PP stage — other stages would have an empty group, padded to keep the per-stage layer count equal.

Example: 12 SWA layers, 3 groups, PP stages [0..5] and [6..11]

15. Why delay_free_blocks exists for connector finishes

When a request completes, its blocks normally go straight back to the free queue. But if a connector is still pushing those blocks somewhere (LMCache write, P/D send to a decode peer), the underlying KV memory must not be reused yet.

request_finished returns (delay_free_blocks, kv_transfer_params); if true, the scheduler holds the blocks until the connector reports get_finished is done.

Use-after-free guard, not optimization Without it, the connector races against BlockPool.free_blocks → get_new_blocks → worker writes new K/V for the same physical slot. The receiver would see a torn block — half old request, half new.

The flag is per-block and decided by the connector. Synchronous connectors finish before request_finished returns, so they answer false.

Example A producer request finishes with blocks [200, 201], and a connector is still sending those blocks to a decode worker. If vLLM freed them immediately, the next request could allocate block 200 and overwrite it while the connector DMA is still reading. delay_free_blocks=True keeps block 200 owned until the connector reports completion.

Patterns that show up across these

A few of these aren't independent — they're applications of the same underlying constraint.

Single-threaded scheduler

Load-bearing for ref-count integrity (§8), the doubly-linked free queue (§7), and the lack of locking around BlockHashToBlockMap. Any plan to parallelize the scheduler hits all three at once.

Global block ID space

Load-bearing for column-packed tensors (KV_CACHE_ALLOCATION), the min_num_blocks clamp (§11), and the choice not to split block IDs by group. Makes preemption (§12) and connector transfers (§15) a single global problem.

The chained hash

Load-bearing for prefix cache correctness across divergent prefixes (KV_CACHE), extra_keys semantics (§13), cache_salt, and NONE_HASH randomization (§9). All flavors of "scope the cache key correctly."

Block-as-unit-of-everything

Ties hash granularity (§6), allocation granularity (§1), kernel tile size (§2), slot layout (§3), and eviction order (§7). Block size is not a tunable; it's the joint where four kernels meet.

The recurring failure mode if any of these were violated Silent KV corruption — wrong numbers in the right tensor slots. There is no kernel trip-wire; the model produces slightly worse outputs, possibly only on specific prompts, possibly indistinguishable from normal sampling noise. That is why the surrounding code is dense with assertions and the comments read like contracts.
