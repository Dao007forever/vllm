Kernel Block Alignment Constraint

Why the CUTLASS MLA kernel asserts block_num % (128 / PAGE_SIZE) == 0, and how vLLM keeps that true.

TL;DR

CUTLASS MLA on SM100 (Blackwell) processes KV in fixed 128-token tiles. Each tile spans 128 / PAGE_SIZE pages. The kernel's inner loop has no boundary check (warp divergence would hurt throughput), so it requires the page-table width block_num to be pre-aligned to tile units. vLLM solves this by inflating block_size to a multiple of 128 — which costs prefix-cache resolution. An alternative path 2 (page-table padding with NULL_BLOCK_ID) preserves fine granularity but isn't the current default.

The assertion

Source: vllm/v1/attention/backends/mla/cutlass_mla.py:203 
_, PAGE_SIZE, D_ckv = kv_c_and_k_pe_cache.shape
assert block_num % (128 / PAGE_SIZE) == 0

This fires when, for example, PAGE_SIZE=32 and block_num=295:

128 / PAGE_SIZE = 128 / 32 = 4   (pages per kernel tile)
295 % 4 = 3  ≠ 0  →  AssertionError

Why the kernel needs this

sm100_cutlass_mla_decode processes the KV cache in tiles of 128 tokens. Each tile consists of 128 / PAGE_SIZE consecutive pages from the page table. The inner loop iterates over these page groups without a boundary check: 

for tile_idx in range(block_num // pages_per_tile):
    pages = page_table[:, tile_idx * pages_per_tile : (tile_idx + 1) * pages_per_tile]
    # load 128 tokens, compute attention partial sum
Why no boundary check? Adding a bounds check would cause warp divergence and hurt throughput. The kernel requires the page-table width to be pre-aligned.

If block_num is not divisible by pages_per_tile, the last tile is partial — it would read past the end of the page table into garbage memory:

block_num = 295  →  295 / 4 = 73.75 tiles  →  last tile is 3/4 full  →  reads 32 garbage slots
block_num = 296  →  296 / 4 = 74.0  tiles  →  every tile is full     →  safe

Path 1: block_size inflation (current approach)

Source: vllm/platforms/interface.py:620 
kernel_block_alignment_size = max(
    min(s.base if isinstance(s, MultipleOf) else s
        for s in backend_cls.get_supported_kernel_block_sizes()),
    cache_config.block_size,
)

For CUTLASS MLA, get_supported_kernel_block_sizes() returns [128], so kernel_block_alignment_size = 128. The attention block_size is then forced to a multiple of 128:

attn_block_size = kernel_block_alignment_size * cdiv(
    mamba_page_size,
    kernel_block_alignment_size * attn_page_size_1_token,
)

If block_size becomes 128 and max_model_len = 9440:

block_num = 9440 / 128 = 73.75 → ceiled to 74 (padded by vLLM)
74 % (128 / 128) = 74 % 1 = 0  ✓
Why this works Inflation guarantees alignment because block_size itself is a multiple of the kernel tile size, so block_num = max_model_len / block_size is already in kernel-tile units — the modulus check trivially passes.

Path 2: page-table padding (the alternative)

Instead of inflating block_size, pad the page table to the next aligned width at construction time:

pages_per_tile = 128 // block_size           # e.g. 128 // 32 = 4
block_num_padded = cdiv(block_num, pages_per_tile) * pages_per_tile
# 295 → cdiv(295, 4) * 4 = 74 * 4 = 296
page_table = F.pad(page_table, (0, block_num_padded - block_num), value=NULL_BLOCK_ID)

Padding with NULL_BLOCK_ID causes the kernel to read the null block (whose K/V is zeroed), contributing nothing to the attention sum. This is safe because attention weights at null slots are zeroed out by sequence-length masking.

Trade-off
  • Path 1 (current): simpler, inflates block_size, degrades prefix-cache resolution.
  • Path 2: preserves fine block_size = 32 for better prefix-cache hits but adds null-block padding logic.

Path 1 inflates the unit. Path 2 inflates the count. Either way block_num × PAGE_SIZE reaches a multiple of 128. vLLM chose Path 1 because it composes more cleanly with the Mamba-page-size-matching rule in resolve_kv_cache_block_sizes.

Connection to the block-size inflation problem

This constraint is one of the two reasons vLLM's resolve_kv_cache_block_sizes inflates attention block_size in hybrid models (alongside Mamba page size matching). Even in a pure-attention model with CUTLASS MLA, block_size must be a multiple of the kernel tile size (128 tokens for CUTLASS MLA on SM100).

Backend get_supported_kernel_block_sizes() Effective alignment
FlashInfer MLA[32, 64]32 or 64 tokens
CUTLASS MLA (SM100)[128]128 tokens
FlashMLA[64]64 tokens
Triton[MultipleOf(1)]any
Practical impact The minimum block_size vLLM can use for a model depends on which attention backend is selected, not just the model architecture. Switching from CUTLASS MLA to FlashInfer MLA allows block_size = 32 instead of 128, recovering 4× finer prefix-cache granularity.

Worked example end-to-end

Model: DeepSeek-V3 (MLA), max_model_len = 9440, backend = CUTLASS MLA.

With CUTLASS MLA

attn_page_size_1_token = 512 (bytes)
kernel_block_alignment_size = 128

block_size = 128  (smallest multiple of 128 that fits)
block_num  = ceil(9440 / 128) = 74

74 % (128 / 128) = 74 % 1 = 0  ✓  assertion passes

With FlashInfer MLA instead

kernel_block_alignment_size = 32

block_size = 32
block_num  = ceil(9440 / 32) = 295

295 % (128 / 32) = 295 % 4 = 3  ✗
   — would fire IF CUTLASS MLA were used,
      but FlashInfer MLA has no such assertion —
      it directly supports PAGE_SIZE = 32
When you'll see this error The "block_num=295 % (128/32=4) != 0" error only appears when CUTLASS MLA is the active backend but the configured block_size is smaller than 128 (e.g. 32 from a previous config or an override). Fix:
  • Use block_size = 128 (default when CUTLASS MLA is selected), or
  • Switch to a backend that natively supports smaller page sizes (FlashInfer MLA, FlashMLA).
