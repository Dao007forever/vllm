CUDA Graph and hash_block_size

What CUDA Graph capture does, why it pads batches, and how that interacts (or doesn’t) with the prefix-cache hashing block size.

TL;DR

CUDA Graph records a fixed-shape sequence of kernel launches and replays it with near-zero CPU overhead. To keep shapes fixed, vLLM captures graphs at a discrete set of token counts (e.g. 1, 2, 4, …, 256, 272…) and pads every real batch up to the next captured size at runtime. Padded tokens are routed to PAD_SLOT_ID = -1, an invalid slot that kernels skip, so padding never touches the real KV cache.

CUDA Graph is a GPU shape concern; hash_block_size is a CPU scheduler / prefix-cache concern. They are orthogonal: CUDA Graph capture sizes do not need to align with hash_block_size, and a partial block below hash_block_size is simply not hashed — its KV is written to real slots but the block is never registered in the prefix cache and therefore can never be reused.

What CUDA Graph is, and why vLLM uses it

A CUDA Graph is a recorded DAG of GPU operations (kernel launches, memcopies, stream events) that can be replayed as a single submission. The big win is CPU overhead: launching hundreds of small kernels per decode step from Python costs more than the kernels themselves on modern GPUs. Replaying a captured graph collapses that to one launch.

The constraint A captured graph is shape-frozen. Tensor sizes, strides, and device pointers used during capture must match exactly on replay. Per-step batch_size and num_tokens are part of those shapes, so vLLM cannot capture “one graph for all batches”.

vLLM’s answer is to capture a menu of graphs at selected token counts and pad each runtime batch up to the nearest captured size. The padded tokens are dummies; the trick is making sure they do not corrupt state — especially the KV cache.

CUDA graph modes: FULL, PIECEWISE, NONE, and the combos

Not every kernel in the model is graph-safe. Attention in particular consumes per-step CPU-computed metadata (cumulative sequence lengths, FA3 scheduler_metadata, paged KV pointers…) whose contents change every step. CUDA Graph records pointer values, not pointer dereferences — if attention’s metadata is captured once and replayed unchanged on every later step, the model produces garbage. vLLM gives you three primitive runtime modes plus two combos to navigate that trade-off:

ModeWhat gets capturedBest forLimitation
NONE Nothing — pure eager. Debugging, backends without graph support. Full per-step Python launch overhead.
PIECEWISE One small graph per inter-attention segment. Attention runs eagerly between graph replays. Any batch shape — prefill, chunked prefill, mixed. Attention launches still cost CPU overhead.
FULL A single graph wrapping the whole model including attention. Uniform decode (every request has the same query length). Attention metadata must be stable for the captured (num_tokens, num_reqs); not all backends support it; fails on cascade attention etc.
FULL_DECODE_ONLY FULL graphs for uniform-decode steps; mixed steps run as NONE. Decode-heavy serving (e.g. the decode side of a P/D split). Mixed batches lose graph benefit; prefill is slower.
FULL_AND_PIECEWISE (v1 default) FULL for uniform decode, PIECEWISE for everything else. General serving — best blended throughput. Highest capture-time + graph-memory cost (two graph menus).

The mode is one config knob; the dispatcher picks which captured graph to replay each step based on the runtime batch shape.

Breakable graphs: how PIECEWISE splits the model

PIECEWISE works by cutting the model graph at ops that are unsafe to capture. Cuts produce partitions; each partition is a separately captured CUDA graph; the unsafe ops run as plain kernel launches between graph replays. This is the “breakable graph” mechanism.

The two partition paths

Dynamo split (use_inductor_graph_partition = False, default)

  • Splits the FX graph at splitting_ops before Inductor sees it.
  • Each subgraph is independently compiled by Inductor and wrapped in vLLM’s CUDAGraph wrapper.
  • Inductor fusions only see one subgraph at a time, so cross-attention fusions are lost.

Inductor partition (use_inductor_graph_partition = True)

  • Inductor compiles the full graph, runs all passes/fusions, then partitions at codegen time.
  • Emits a single call function that wraps cudagraph-safe ops into N+1 partition functions, leaving the N unsafe ops outside.
  • Preserves whole-model fusions; supports FULL and PIECEWISE without compiling twice.

Either way, a model with N attention layers ends up with N+1 captured graph segments per capture size (the pre-first-attention chunk, each inter-attention chunk, and the post-last-attention chunk). With 40 attention layers and 50 capture sizes that’s ~2050 small graphs — cheap individually but they add up in startup time.

Why FULL needs num_reqs in the key

FA3’s scheduler-metadata buffer is sized by num_reqs, not just total num_tokens. Capture a FULL graph at num_tokens = 64, num_reqs = 8 and the buffer layout assumes 8 sequences; replay it with num_reqs = 16 and the metadata addresses point at half-sized records. PIECEWISE sidesteps this entirely because attention is outside the graph and rebuilds its metadata fresh each step:

Source: vllm/v1/cudagraph_dispatcher.py:204–207 
# Only relax for PIECEWISE mode. FULL mode needs exact num_reqs
# because FA3's scheduler_metadata computation depends on it.
if cudagraph_mode.mixed_mode() == CUDAGraphMode.PIECEWISE:
    batch_desc = replace(batch_desc, num_reqs=None, uniform=False)
self.add_cudagraph_key(cudagraph_mode.mixed_mode(), batch_desc)

This is why FULL_AND_PIECEWISE reserves FULL for uniform decode only: when every request contributes the same query length, num_reqs can be derived from num_tokens (num_reqs = num_tokens / uniform_decode_query_len), so a single num_tokens key is enough.

Inside a transformer layer: what attention does, what the rest does

The reason PIECEWISE can leave attention out of the graph without breaking anything is structural: in a transformer, attention is the only op that mixes information across tokens or knows about request boundaries. Every other op is point-wise per-token. Pinning that distinction down makes the request-dim shape variance fall out naturally.

Each transformer layer has exactly one attention call. With N layers and PIECEWISE mode, this pattern repeats N times, producing N + 1 captured partitions per capture size (the leading embedding partition, the N − 1 “O_proj → MLP → next-layer QKV_proj” partitions, and the trailing “O_proj → MLP → lm_head” partition).

Non-attention partition: pure per-token compute

OpInputOutputTouches request boundaries?
RMSNorm x [num_tokens, H] x [num_tokens, H] No — normalizes each row independently.

None of these ops know — or need to know — that token 0 belongs to request A and token 50 to request B. They’re GEMMs and element-wise kernels that walk the leading num_tokens axis. Their kernel-launch parameters (grid sizes, tile shapes) depend on num_tokens_padded, which is why padding makes them captureable but request-dim shapes are irrelevant to them. This is the entire content of every captured PIECEWISE partition.

Attention partition: the one op that joins tokens to requests

Attention is doing four things at once:

  1. Write the new (K, V) for each of this step’s tokens into the global KV cache, using slot_mapping[batch_pos] as the destination index.
  2. For each query token, figure out which request it belongs to using query_start_loc — req i owns batch positions [query_start_loc[i], query_start_loc[i+1]).
  3. Walk back through history: read seq_lens[i] historical positions for the owning request, looking up each position’s physical block via block_tables[i, pos // block_size].
  4. Reduce: softmax(QKT)·V over those historical positions, producing one output vector per query token.
InputShapeRole
Q, K, V [num_tokens, num_heads, head_dim] This step’s new query/key/value, output of QKV_proj.
K_cache, V_cache [num_blocks, block_size, num_kv_heads, head_dim] Global pool. Read for historical context, written for the new step.
slot_mapping [num_tokens_padded] (token-dim) Where to write new (K, V) for each token. -1 for padded tokens.
block_tables [num_reqs, max_blocks] in PIECEWISE; [num_reqs_padded, max_blocks] in FULL Per-request list of physical block IDs for historical KV.
seq_lens [num_reqs] in PIECEWISE; [num_reqs_padded] in FULL Per-request history length to read.
query_start_loc [num_reqs + 1] in PIECEWISE; [num_reqs_padded + 1] in FULL Maps each batch position to its owning request.

The output is again token-dim — attn_out [num_tokens, num_heads, head_dim] — which flows into the next captured non-attention partition (starting at O_proj). Attention is the only place where request-dim metadata enters the layer. That’s precisely why the request-dim padding matters only when attention is captured (FULL) and is dropped when attention runs eagerly (PIECEWISE).

The boundary, restated Captured non-attention partitions: per-token compute over token-dim tensors. Eager (or captured-in-FULL) attention call: per-request lookups into block-paged storage, then a per-token reduction over each request’s history. The partition boundary at attention is also the boundary between “sees only num_tokens” and “sees only num_reqs-shaped metadata.”

Now walk a single transformer layer top-to-bottom:

  1. Embedding lookup — captured, per-token. input_ids [2] → x [2, 4]. Say x = [[0.5, −0.2, 0.1, 0.3], [−0.4, 0.6, 0.0, 0.2]]. Two independent table lookups; the kernel walks num_tokens = 2 rows and has no idea row 0 is req 0 and row 1 is req 1. Same kernel would work for one row, ten rows, or 256.
  2. RMSNorm — captured, per-token. Each row normalized by its own RMS: x_norm[i] = x[i] / sqrt(mean(x[i]²)) * γ. Still [2, 4]. Row 0 and row 1 do not interact.
  3. QKV_proj — captured, per-token. One GEMM: x_norm [2, 4] × W_qkv [4, 12] = qkv [2, 12]. Split: Q [2, 4], K [2, 4], V [2, 4]. Pretend the result is Q = [[q0_a, q0_b, q0_c, q0_d], [q1_a, q1_b, q1_c, q1_d]] and similarly for K and V. The graph partition ends here.
  4. Attention — eager (PIECEWISE), per-request. This is the only op that touches block_tables, seq_lens, and query_start_loc. Four sub-steps:

    (a) Write new (K, V) to cache.

    K_cache.flat[189] = K[0]   # token at position 5 of req 0 → block 47, offset 1
V_cache.flat[189] = V[0]
K_cache.flat[355] = K[1]   # token at position 3 of req 1 → block 88, offset 3
V_cache.flat[355] = V[1]

    (b) Resolve each query’s owning request via query_start_loc. Position 0 falls in [0, 1) → req 0. Position 1 falls in [1, 2) → req 1.

    (c) For query 0 (req 0, seq_len = 6), read 6 historical keys/values.

    # Block lookups:
# pos 0..3 → block_tables[0, 0] = 12, offsets 0..3
# pos 4    → block_tables[0, 1] = 47, offset 0
# pos 5    → block_tables[0, 1] = 47, offset 1   (just-written)

K_ctx_0 = stack(K_cache[12, 0..3], K_cache[47, 0..1])   # shape [6, 4]
V_ctx_0 = stack(V_cache[12, 0..3], V_cache[47, 0..1])   # shape [6, 4]

scores_0 = softmax(Q[0] @ K_ctx_0.T / sqrt(4))          # shape [6]
attn_out[0] = scores_0 @ V_ctx_0                         # shape [4]

    (d) For query 1 (req 1, seq_len = 4), read 4 historical keys/values.

    # All four positions are in block_tables[1, 0] = 88, offsets 0..3
K_ctx_1 = K_cache[88, 0..3]                              # shape [4, 4]
V_ctx_1 = V_cache[88, 0..3]                              # shape [4, 4]

scores_1 = softmax(Q[1] @ K_ctx_1.T / sqrt(4))           # shape [4]
attn_out[1] = scores_1 @ V_ctx_1                         # shape [4]

    Output: attn_out [2, 4]. Notice that req 0 attends over 6 positions, req 1 attends over 4. The same Python call handled both by reading seq_lens and walking each request’s block_tables row independently. Nothing in the eager attention launch is fixed-shape with respect to num_reqs — if a third request showed up next step, the call accepts seq_lens [3] / block_tables [3, …] without re-capture.

  5. O_proj — captured, per-token. attn_out [2, 4] × W_O [4, 4] = out [2, 4]. Yet another per-row GEMM. The next graph partition begins here.
  6. Residual + RMSNorm — captured, per-token. h = x + out, then RMSNorm(h). Still [2, 4].
  7. MLP — captured, per-token. gate = h @ W_gate and up = h @ W_up, act = SiLU(gate) * up, down = act @ W_down. All operate on the num_tokens = 2 rows independently.
  8. Final residual — captured. h_out = h + down. Layer output: [2, 4]. Feeds into the next layer’s RMSNorm.

If next step there are 3 requests at different positions, the captured partitions don’t need to change — same num_tokens (assuming it’s still 2 or pads to 2), same captured kernel grid. But the eager attention call now receives seq_lens [3], block_tables [3, …], query_start_loc [4], and rebuilds its scheduler metadata fresh for 3 requests. That’s the whole reason PIECEWISE works.

Concrete walk-through: one prefill + one decode in the same batch

Same tiny model (H = 4, 1 head, head_dim = 4, block_size = 4, PIECEWISE). Two requests, but now with very different shapes — the kind of mix that makes PIECEWISE earn its keep:

Total num_tokens = 5 + 1 = 6. With capture sizes [1, 2, 4, 8, …], the dispatcher rounds 6 up to num_tokens_padded = 8 — two trailing padding slots. The packed buffers:

Token-dim tensors (positions, slot_mapping, hidden states) are padded to 8: padding slots use position = 0 and slot = −1. Request-dim tensors (query_start_loc, seq_lens, block_tables) keep the real num_reqs = 2 — PIECEWISE doesn’t pad those (see why). The final entry of query_start_loc is 6, not 8, so attention won’t walk into the padded slots.

Walk the layer:

  1. Embedding + RMSNorm + QKV_proj — captured, per-token. input_ids [8] → x [8, 4] → Q, K, V [8, 4]. The kernels iterate all num_tokens_padded = 8 rows; they do not know that the first 5 are one request’s prefill, the 6th is a different request’s decode, and the last 2 are dummy padding. They compute hidden states for the padded rows too — that compute is wasted, but it doesn’t corrupt anything because the padded outputs never reach the KV cache. Graph partition ends here.
  2. Attention — eager. Four sub-steps:

    (a) Write new (K, V) into the cache — padded slots are skipped.

    for batch_slot in 0..7:
    slot = slot_mapping[batch_slot]
    if slot == -1:
        continue                       # ← padded slots 6 and 7
    K_cache.flat[slot] = K[batch_slot]
    V_cache.flat[slot] = V[batch_slot]
# Actual writes: 6 of them.
#   slots 84..87, 136  (req 0's positions 0..4 land in blocks 21, 34)
#   slot 355           (req 1's position 3 lands at offset 3 of block 88)
# K[6..7], V[6..7] are computed but discarded.

    (b) Resolve query → request via query_start_loc = [0, 5, 6]. Batch slots [0, 5) → req 0; batch slot 5 → req 1. Batch slots 6 and 7 are past query_start_loc[-1] = 6, so the kernel never visits them — this is how attention “skips” padded queries without needing a separate mask.

    (c) Req 0 (prefill): causal self-attention over 5 positions. Each of the 5 query tokens attends to positions [0, query_pos] of req 0 only — standard causal mask:

    # Build req 0's full KV context from block_tables[0] = [21, 34], seq_len = 5:
K_ctx_0 = stack(K_cache[21, 0..3], K_cache[34, 0..0])   # shape [5, 4]
V_ctx_0 = stack(V_cache[21, 0..3], V_cache[34, 0..0])   # shape [5, 4]

# All 5 of req 0's queries attend over K_ctx_0 / V_ctx_0
# with a [5, 5] causal mask:
Q_0 = Q[0..4]                                            # shape [5, 4]
scores = (Q_0 @ K_ctx_0.T / sqrt(4)).masked_fill(mask, -inf)
scores = softmax(scores)                                 # shape [5, 5]
attn_out[0..4] = scores @ V_ctx_0                        # shape [5, 4]

    (d) Req 1 (decode): single-query attention over 4 positions.

    # Req 1's KV context from block_tables[1] = [88, _], seq_len = 4:
K_ctx_1 = K_cache[88, 0..3]                              # shape [4, 4]
V_ctx_1 = V_cache[88, 0..3]                              # shape [4, 4]

Q_1 = Q[5]                                               # shape [4]
scores_1 = softmax(Q_1 @ K_ctx_1.T / sqrt(4))            # shape [4]
attn_out[5] = scores_1 @ V_ctx_1                         # shape [4]

# attn_out[6..7] are never written — left as whatever garbage
# was in the persistent buffer. The next graph partition will run
# GEMMs on those rows but the final logits gather drops them.

    Output is attn_out [8, 4] — same token-dim shape as captured, even though only the first 6 rows hold meaningful values. The same eager attention call handled 5 prefill queries with a causal mask AND 1 decode query with no mask in one launch: the kernel reads query_start_loc to know where each request’s queries start, reads seq_lens to know how far back each one reaches, and applies the causal mask only where query_len > 1 (i.e., when more than one row of Q belongs to the same request).

  3. O_proj → residual → RMSNorm → MLP → residual — captured, per-token. [8, 4] → [8, 4] throughout, including the 2 padded rows whose values are garbage. The MLP processes the prefill tokens, the decode token, and the padded rows row-by-row with the same GEMMs — no special case. Only the final logits gather (driven by cu_num_logits) drops the padded rows, so the wasted compute never affects output.

The captured partitions used here are the same captured graph objects as in the previous walk-through, just replayed at the size-8 graph instead of size-2. Different mix of requests, different attention work — same compiled partitions wrapping it:

The captured non-attention partitions only care about num_tokens_padded = 8. Whether those 8 tokens are 7 prefill + 1 decode, 4 reqs × 2 tokens (spec decode), or any other shape that sums to 8 — the partitions execute the same GEMMs. Eager attention picks up the rest, reading block_tables, seq_lens, and query_start_loc fresh each call.

Why this matters in production Real serving traffic is overwhelmingly mixed: continuous batching schedules new prefills alongside running decodes every step. Without PIECEWISE, every such step would dispatch to NONE (eager everywhere) because no FULL graph matches the exact (num_tokens, num_reqs, query-length-distribution) combination. PIECEWISE recovers the captured-launch speedup for the ~90% of compute that is per-token (norms, gemms, MLP) while letting attention’s per-step metadata stay flexible.

Runtime dispatch and fallback

Each step the dispatcher looks up the best graph for the current batch, preferring FULL over PIECEWISE over NONE:

Source: vllm/v1/cudagraph_dispatcher.py:311–328 
if CUDAGraphMode.FULL in allowed_modes:
    batch_desc_to_check = batch_desc
    if batch_desc_to_check in self.cudagraph_keys[CUDAGraphMode.FULL]:
        return CUDAGraphMode.FULL, batch_desc_to_check

if CUDAGraphMode.PIECEWISE in allowed_modes:
    # also check if the relaxed key exists for more "general"
    # piecewise cudagraph
    batch_desc_to_check = replace(batch_desc, num_reqs=None, uniform=False)
    if batch_desc_to_check in self.cudagraph_keys[CUDAGraphMode.PIECEWISE]:
        return CUDAGraphMode.PIECEWISE, batch_desc_to_check

assert CUDAGraphMode.NONE in allowed_modes
return CUDAGraphMode.NONE, BatchDescriptor(num_tokens)

Common dispatch paths in FULL_AND_PIECEWISE:

  • Uniform decode step (e.g. 8 requests × 1 token = 8 tokens, no spec decode): rounds to next capture size, matches a FULL key with both num_tokens and num_reqs derivable → FULL.
  • Mixed prefill + decode (e.g. 50-token chunked prefill + 7 decodes = 57 tokens): not uniform → FULL lookup misses → falls back to PIECEWISE with the relaxed key (num_reqs = None).
  • Cascade attention (passed in invalid_modes={FULL} by the runner): FULL is forbidden, so PIECEWISE is used even for uniform decode.
  • Batch over max_cudagraph_capture_size (e.g. a huge prefill): no captured graph → NONE, eager execution.
Capture-time cost trade Each captured graph holds its own pinned input buffers and workspace. FULL_AND_PIECEWISE captures both menus, plus LoRA specializations and decode-vs-mixed variants. For a 40-layer model with the default ~50 capture sizes, that’s easily a couple of GB of graph memory and tens of seconds of warmup. NONE skips all of it but pays per-step launch overhead forever after.

            self._bs_to_padded_graph_size[bs] = start
        else:
            self._bs_to_padded_graph_size[bs] = end

The pattern is: any batch size in the half-open interval (start, end] rounds up to end, while exact captured sizes map to themselves.

Result Attention kernels treat slot_id = −1 as a skip: they do not read or write the KV cache for that token. So padding inflates compute work modestly, but leaves the cache untouched. Padded tokens never write to a real cache block, never shift the last partial block, never trigger a hash.

What is hash_block_size?

hash_block_size is the granularity at which a request’s token sequence is chunked into hash blocks for the prefix cache. Each hash block’s tokens are fed into a hash function whose output is the cache key. Two requests with the same prefix produce identical block hashes, so the second request can reuse the already-computed KV.

Source: vllm/config/cache.py:54–65

by it. This enables prefix-caching keys to be computed at the finest common

What is num_kv_cache_groups?

The slot-mapping shape above is [num_kv_cache_groups, num_tokens_padded]. A KV cache group is a set of model layers that share the same block table because they share the same attention spec (block size, head dim, dtype, sliding-window length, etc.). The KV cache manager allocates blocks once per group; the runner reuses that block table for every layer in the group.

Tie-in to hash_block_size When num_kv_cache_groups > 1, groups may have different physical block_size values. That is exactly why the hybrid path picks hash_block_size = gcd(group_block_sizes) (kv_cache_utils.py:624) — a single hashing granularity that divides every group’s block, so the same block hashes are valid across all groups.

Why CUDA Graph does not affect hash_block_size

The two concepts use the word “block” but live in different layers:

CUDA Graph capture size

  • Unit: tokens in this step’s batch
  • Layer: GPU shape / launch overhead
  • Set by: cudagraph_capture_sizes in compilation config
  • Visible to: model forward pass on GPU
  • Mechanism: pad up, route extras to PAD_SLOT_ID

hash_block_size

  • Unit: tokens in a single prefix-cache key
  • Layer: CPU scheduler / prefix-cache lookup
  • Set by: cache_config.hash_block_size or derived
  • Visible to: scheduler when computing request.block_hashes
  • Mechanism: num_tokens // hash_block_size hashes

There is no code path where the CUDA Graph dispatcher reads hash_block_size, and no code path where the block hasher reads the CUDA Graph capture menu. The two systems share no shape constraint. Capture sizes do not need to be multiples of hash_block_size; conversely hash_block_size can be 8 or 16 while capture sizes go …, 240, 256, 272, ….

Don’t confuse this with the kernel block alignment constraint A different constraint — the CUTLASS MLA kernel’s requirement that block_num % (128 / PAGE_SIZE) == 0 — does couple the physical block size to a kernel-internal tile, and that one can force inflation of the KV block size. See KERNEL_BLOCK_ALIGNMENT for that story. CUDA Graph capture is unrelated to it.

Partial blocks below hash_block_size

The question “what happens when we compute a partial block below hash_block_size” has two valid readings; both end at the same place.

Reading 1: the request only has fewer tokens than a full hash block

The hasher early-exits before producing any hash: 

start_token_idx = len(request.block_hashes) * block_size
num_tokens = request.num_tokens
if start_token_idx + block_size > num_tokens:
    return []          # ← no new full block to hash

The single-type KV-cache manager registers the same thing on the caching side — only the floor of complete blocks is committed:

Source: vllm/v1/core/single_type_kv_cache_manager.py:298–302 
num_cached_blocks = self.num_cached_block.get(request.request_id, 0)
num_full_blocks = num_tokens // self.block_size
if num_cached_blocks >= num_full_blocks:
    return

Reading 2: chunked-prefill / decode is writing into a still-partial block

Same logic applies: each scheduling step recomputes num_full_blocks = num_tokens // hash_block_size. The current trailing partial block physically has its KV written into real cache slots (the slot mapping for those positions is a real, non-sentinel slot id, because the block was allocated for this request), but it is not yet hashed and therefore not yet visible to the prefix cache.

Trailing partial block — hashed on completion, not before
tokens: 0123 4567 891011 1213
block#: block 0 (hashed) block 1 (hashed) block 2 (hashed) block 3 (partial)
hash_block_size = 4, num_tokens = 14 → num_full_blocks = 14 // 4 = 3. Blocks 0–2 (positions 0–11) are full and hashed. Block 3 holds only positions 12–13; its KV is written into real cache slots but the block is not hashed — not until two more tokens arrive to complete it. If the request ends here, block 3’s KV is discarded with the block when it is freed.

Concretely, the trailing partial block:

  • Has KV written to real, valid cache slots (the block was already allocated for this request).
  • Has no hash entry in the prefix cache — no other request can match it.
  • Will be hashed on the first scheduler step after enough new tokens land to complete it.
  • Will be lost if the request finishes before the block fills: the block is freed back to the pool, its slots reused, and the partial KV is gone.
Why this is fine Hashing a partial block would either need padding to a fixed length (changing the hash and breaking prefix matching) or a variable-length hash key (more work per lookup, larger key table). The current design trades a small reuse miss at the tail of every request for a simple fixed-size hash key.

Worked example end-to-end

Configuration: block_size = 16 (single KV-cache group, so hash_block_size = 16), cudagraph_capture_sizes = […, 88, 96, 104, …]. Suppose 100 actual tokens are scheduled this step across a few requests.

  1. Dispatcher pads num_tokens = 100 up to the next captured size, 104. The forward pass will run the size-104 graph.
  2. Slot mapping is built with shape [num_kv_groups, 104]: the first 100 entries are real slots; the trailing 4 are PAD_SLOT_ID = −1.
  3. Attention writes KV for positions 0–99 into their real slots and skips positions 100–103.
  4. Scheduler hashes new full blocks on the CPU side using request.all_token_ids and hash_block_size = 16. For a request whose total tokens jumped from 30 to 46 this step: 30 // 16 = 1 already hashed, 46 // 16 = 2 total full blocks → one new hash is registered for the block spanning tokens 16–31. Tokens 32–45 sit in a partial block; the request has real KV in those slots but the block has no hash yet.
  5. Next step the request adds 5 more tokens (total = 51). Now 51 // 16 = 3 full blocks: the block spanning 32–47 becomes the new third hash. Tokens 48–50 become the new partial tail.

Three things the attention kernel does with it:

  1. Locate each query’s owning request: for batch position b, find i such that query_start_loc[i] ≤ b < query_start_loc[i+1].

  3. Branch on per-request query length: pure decode (qᵢ = 1) takes a different code path than prefill (qᵢ > 1).

Where is the KV cache read from? — block_tables

The slot_mapping tensor above tells attention where to write the new (K, V) for each of this step’s queries — one slot per query. The read side — the historical KV the kernel attends over — is a separate tensor: block_tables, shape [num_kv_cache_groups, num_reqs_padded, max_num_blocks]. Each row is the per-request list of physical block IDs that hold this request’s context.

Continuing example A, suppose physical block_size = 16. Each request needs ceil(seq_len / 16) blocks of context:

block_tables[0] for the size-4 decode graph (one KV cache group)
batch slot: 0 (req A) 1 (req B) 2 (req C) 3 (pad)
block 0: 12 88 301 0
block 1: 47 52 442 0
block 2: 99 7 501 0
block 3: 14 — 678 0
… … … … …
Each column is a request’s ordered list of physical block IDs. Block IDs are pool indices into the KV cache tensor — they have no relationship to batch position. Padded batch slot 3 has its row zeroed by gather_block_tables.

Inside the attention kernel, each captured query in batch slot i reads its context like this (pseudo-code per layer, per KV cache group): 

q = query[i]                              # newly computed Q for this token
seq_len = seq_lens[i]                     # e.g. 48 for req B
for pos in 0 .. seq_len - 1:              # historical positions
    block_idx   = pos // block_size       # which logical block
    block_offset = pos %  block_size      # offset within that block
    phys_block  = block_tables[group, i, block_idx]   # ← READ
    k = K_cache[group, phys_block, block_offset]
    v = V_cache[group, phys_block, block_offset]
    accumulate(q, k, v)

# write this step's new KV
slot = slot_mapping[group, query_start_loc[i]]        # ← WRITE
K_cache.flat[slot] = new_k
V_cache.flat[slot] = new_v

Two tensors, two jobs
  • block_tables[group, req_idx, :] — READ addresses. Resolves logical position → physical block. Built/refreshed by the scheduler when blocks are allocated; gathered onto a per-batch slice each step.
  • slot_mapping[group, batch_pos] — WRITE addresses. One slot per new query token in this step. Computed each step from positions and block_tables.
The CUDA graph captures pointer values for both tensors; on each replay the runner writes new contents into the same persistent buffers, so the captured graph reads up-to-date routing without being re-captured.

Same story for the padded slot 3: seq_lens[3] = 0 makes the read loop iterate zero times, slot_mapping[0, 3] = -1 makes the write a no-op, and the resulting fake logit is dropped by the gather (it has no entry in cu_num_logits).

What the CUDA graph actually captures about block_tables (and every other input)

A common mental model gotcha: “If block_tables contents change every step, surely the graph captures them and we re-capture every step? Or is block_tables excluded from the graph?” Neither — block_tables is captured in exactly the same way as input_ids, positions, slot_mapping, seq_lens, and query_start_loc. What CUDA graphs record about an input tensor argument is a device pointer + tensor shape, not the underlying bytes.

AspectCaptured into the graph?How it stays correct on replay
Tensor memory address Yes — baked in at capture time, for every tensor passed to a captured kernel. The runner always uses the same persistent buffer object; new contents are written into that buffer.
Tensor shape & stride Yes — baked in, but only for tensors that actually flow through captured kernels. Padding rounds the runtime batch up to a captured size so the shape matches. (See below for why some tensors aren’t padded in PIECEWISE.)
Tensor contents (bytes) No — not part of the recording. The runner overwrites the buffer each step before calling replay; the captured kernels then read whatever is there now.
Kernel launch params (grid size, block size, shared-mem size, scalar args) Yes — baked in. These derive from tensor shapes & the captured config, so padding makes them invariant.

So “the graph includes block_tables” just means a pointer to a fixed-shape buffer is encoded in the captured kernel calls. The block IDs themselves are read from that buffer at replay time:

→
each step

→
replay

The mechanism is identical for every input. The reason gather_block_tables writes into self.input_block_tables rather than allocating a new tensor each step is precisely to preserve the captured address:

Source: vllm/v1/worker/gpu/block_table.py:140–145 
def get_dummy_block_tables(self, num_reqs: int) -> tuple[torch.Tensor, ...]:
    # NOTE(woosuk): The output may be used for CUDA graph capture.
    # Therefore, this method must return the persistent tensor
    # with the same memory address as that used during the model's forward pass,
    # rather than allocating a new tensor.
    return tuple(block_table[:num_reqs] for block_table in self.input_block_tables)
So what is excluded from FULL graphs? Not block_tables — that’s fine because it fits the persistent-buffer pattern. What can’t be captured in FULL safely is anything whose shape, kernel-launch parameters, or internal pointer arrays are derived from per-step CPU computation that varies between (num_tokens, num_reqs) configurations. FA3’s scheduler_metadata is the canonical example: its layout depends on num_reqs, so different num_reqs values need different captured graphs. PIECEWISE solves this by leaving attention out of the graph entirely — the attention kernel rebuilds its metadata fresh in eager mode each step, while the rest of the model enjoys captured launches.

The same logic explains the seemingly redundant query_start_loc padding (entries past num_reqs repeat the last value) and the seq_lens = 0 for padded slots: the captured kernel reads a fixed-shape [num_reqs_padded + 1] array, and the runner must populate it with values that make the padded queries no-ops. The kernel cannot “skip” padded slots based on a dynamic count — it reads what the buffer says, every replay.

Why this is consistent with how graphs work

PIECEWISE captures only the non-attention partitions of the model. Those partitions touch the token-dimension tensors (input_ids, hidden states of shape [num_tokens_padded, hidden_dim], etc.) but never the request-dimension tensors. The captured kernels in those partitions encode num_tokens_padded in their launch params, so that dimension must stay fixed across replays.

The request-dimension tensors are only consumed by attention. In PIECEWISE, attention runs eagerly between graph segments — a regular kernel launch whose launch params are recomputed on the host every step. Eager launches happily accept different shapes every call, so the runner is free to skip padding on those tensors. Skipping the padding has two practical benefits:

  • Smaller copies and lookups every step (a 64-row block table instead of a 256-row one when only 64 reqs are live).
  • No need to populate dummy padded rows with seq_lens = 0 and zeroed block IDs.

FULL captures attention inside the graph, so the request dimension also gets baked in — that dimension must match the captured shape on replay, which is exactly why num_reqs is part of the FULL graph key but not the PIECEWISE key.

The sharpened rule What CUDA Graph requires fixed across replays is the shape of every tensor that appears as an argument to a captured kernel. In PIECEWISE the captured kernels are non-attention only, so they never see block_tables / seq_lens / query_start_loc — those tensors are free to vary. In FULL the captured kernels include attention, so every input tensor’s shape is locked in by the capture key.

Per-request prefix-cache hashing is still independent

Even though all three requests share one graph, the hasher runs per request on the CPU after the step. Continuing example A:

  • req A (101 tokens, hash_block_size = 16): 101 // 16 = 6 full blocks. If 6 were hashed already, no new hash; otherwise the new ones are registered.
  • req B (48 tokens, hash_block_size = 16): 48 // 16 = 3 full blocks — a new hash on this step because position 47 just completed the third block.
  • req C (201 tokens, hash_block_size = 16): 201 // 16 = 12 full blocks, plus a 9-token partial tail that is not hashed.

Three separate calls to request_block_hasher; the dummy fourth slot has no Request object and contributes nothing. The fact that CUDA Graph rounded the batch up to four queries is invisible to the hasher.

The capture-cost trade Each captured graph holds onto its own pinned input buffers and workspace. With ~40 default capture sizes plus separate FULL and PIECEWISE menus and optional LoRA specializations, capture can use several GB and add startup time. That is the reason the dispatcher uses a discrete menu with rounding rather than capturing every integer batch size: the runtime cost of one padded forward pass is much smaller than the static cost of carrying 256 graphs in memory.

Scenario: 20 tokens with hash_block_size = 32

A common worry: if a request only has 20 tokens but hash_block_size = 32, does CUDA Graph still work? Yes — the two systems do not interact.

ConcernWhat it seesValue
CUDA Graph dispatcher num_tokens_after_padding — rounds 20 up to the next captured size (e.g. 24) 24
Slot mapping 20 real slot ids + 4 × PAD_SLOT_ID kernel writes 20, skips 4
Block hasher request.num_tokens // hash_block_size = 20 // 32 0 full hash blocks
Prefix cache reuse no hashes registered yet none until 32+ tokens

Concretely:

  • The forward pass runs the captured graph closest above 20 — for the default capture menu, size 24. Positions 0–19 write real (K, V) into allocated cache slots; positions 20–23 route to PAD_SLOT_ID = -1 and are skipped by attention.
  • The scheduler computes 20 // 32 = 0 full hash blocks, so request.block_hashes stays empty and nothing is registered in the prefix cache.
  • The 20 tokens of KV are physically present in the GPU cache — they participate in this request’s own future attention — but no other request can match them, because no hash key exists.
  • If decode continues and crosses position 32, the first full hash block (positions 0–31) gets hashed on that step. The block then becomes reusable for future requests sharing the same 32-token prefix.
  • If the request finishes at 20 tokens, the block is freed back to the pool and the KV is discarded. The CUDA Graph layer is entirely unaffected by this.
The general rule The CUDA Graph dispatcher rounds up by token count, never by hash block. The hasher rounds down by hash block, never by captured graph size. num_tokens < hash_block_size is a perfectly valid runtime state: nothing breaks, you just don’t get a prefix-cache entry until the request grows past the threshold.

Takeaways

  • CUDA Graph is a GPU-shape optimization, not a cache-management one. It pads runtime batches to one of a discrete set of captured token counts.
  • PIECEWISE breaks the graph at attention so that attention metadata stays dynamic; FULL captures everything in one graph but requires stable metadata, which is why it’s reserved for uniform-decode batches. The default FULL_AND_PIECEWISE uses FULL for uniform decode and PIECEWISE for mixed batches.
  • Captured non-attention partitions see only num_tokens; attention sees both num_tokens and the request-dim metadata (block_tables / seq_lens / query_start_loc). Attention is the one op that joins tokens to requests — that’s why it’s graph-unsafe and why PIECEWISE can drop request-dim padding.
  • Padding is safe for the KV cache because padded positions are routed to PAD_SLOT_ID = −1 and attention kernels skip that sentinel.
  • hash_block_size is a scheduler-side concept for prefix-cache hashing. It is not derived from, nor constrained by, CUDA Graph capture sizes.
  • num_kv_cache_groups reflects the model’s attention-type pattern: 1 for dense full-attention models, >1 for hybrids (full + SWA, full + Mamba, full + chunked local). It is the leading axis of slot_mappings and the list length of block_tables; it is also what motivates the GCD rule for hash_block_size in hybrid models.
  • Partial blocks below hash_block_size are “materialized but invisible”: their KV occupies real slots in the GPU tensor but they have no hash entry, cannot be reused by other requests, and are discarded if the request ends before the block fills.
  • Don’t conflate this with kernel block alignment. The CUTLASS MLA constraint (KERNEL_BLOCK_ALIGNMENT) does couple physical block size to a kernel tile, but it is a separate mechanism with separate consequences.
