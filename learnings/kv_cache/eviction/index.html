KV-Cache LRU Eviction

How the Hybrid Memory Allocator orders, evicts, and rescues prefix-cache blocks — the reverse-on-free trick, the hash/no-hash split, and how SWA, Mamba, and RETENTION_INTERVAL change the rules.

TL;DR

Free blocks live in a doubly-linked LRU queue. The head is evicted first, the tail last. Allocation pops from the head; freeing appends to the tail. A block whose request finished but still carries a block_hash is an eviction candidate: it sits in the queue but is still in the prefix-cache hash map, so it can still score a hit — until it reaches the head and gets popped (real eviction) or a new request touches it (rescue).

To keep prefix-cache hit rate high, vLLM frees a request's blocks in reverse (tail-first) and splits freed blocks by hash: blocks that can never match (no hash) are prepended to the very front (evicted first); cacheable blocks are appended to the back. The net effect: a request's reusable prefix blocks survive longest, its request-specific suffix blocks die first.

SWA and Mamba additionally free blocks mid-request (they only need a tail window / a single recurrent state), and cache sparsely. VLLM_PREFIX_CACHE_RETENTION_INTERVAL thins SWA's cached checkpoints even further to free pool capacity for full-attention prefixes.

The cast: four data structures

All eviction logic lives in the BlockPool and the structures it owns. Everything below is per-engine (per worker), in plain Python — no GPU work happens here, only bookkeeping of which physical block holds whose KV.

The key insight that makes prefix caching work Freeing a block (ref_cnt → 0) and evicting it from the prefix cache are two separate events. Between them, the block is an eviction candidate: physically reclaimable, but still hashed and still matchable. The whole LRU machinery exists to delay that second event for the blocks most likely to be reused.

There is no separate priority list or timestamp. A block's eviction priority is its position in FreeKVCacheBlockQueue. Closer to the head ⇒ evicted sooner.

State 4 is the heart of prefix caching. A finished request's blocks are free (reclaimable) yet still cached (matchable). The longer a valuable block survives in State 4, the higher the hit rate. Every trick on this page is about controlling how fast each block drifts toward the head (Exit A).

Eviction vs. rescue (touch)

Eviction happens lazily, at allocation time

There is no background evictor. Eviction is a side effect of get_new_blocks: pop n blocks from the head, and for each, call _maybe_evict_cached_block — if it still carries a hash, drop that hash from BlockHashToBlockMap and reset_hash(). Only now is the cached prefix gone.

block_pool.py:333–400 
def get_new_blocks(self, num_blocks):
    ret = self.free_block_queue.popleft_n(num_blocks)   # take LRU victims
    for block in ret:
        self._maybe_evict_cached_block(block)         # drop its hash if any
        block.ref_cnt += 1
    return ret

Rescue: a hit pulls a candidate back out

When a new request's find_longest_cache_hit matches a candidate block, the block must leave the free queue before it can be reused. touch does exactly that: if ref_cnt == 0 it removes the block from the queue (the O(1) middle splice), then bumps ref_cnt. The block is now shared, hash intact, no recomputation needed.

block_pool.py:402–417 
def touch(self, blocks):
    for block in blocks:
        if block.ref_cnt == 0 and not block.is_null:
            self.free_block_queue.remove(block)   # rescue from eviction
        block.ref_cnt += 1

Trick #1: reverse-on-free keeps prefixes alive

Blocks are allocated front-to-back: block 0 holds the first prompt tokens (often a shared system prompt), the last block holds the request-specific tail. For future hit rate, the front blocks are gold (many requests share them) and the tail blocks are nearly worthless (unique to this request).

So when a request finishes, vLLM frees its blocks in reverse order — tail block first. Because free_blocks appends to the queue tail (the MRU / evict-last end), feeding it tail-first lands the request-tail nearest the head (dies first) and the shared prefix nearest the tail (dies last).

The reversal — single_type_kv_cache_manager.py:401–409 
def free(self, request_id):
    # Free blocks in reverse order so that the tail blocks are freed first.
    self.block_pool.free_blocks(reversed(self.pop_blocks_for_free(request_id)))

"(1) The least recently used block is at the front. Trick #2: hash / no-hash split inside free_blocks

Reversal orders blocks within a request. free_blocks adds a second axis: a partial (non-full) block never got a hash, so it can never match in the prefix cache. Keeping it around only wastes a survive-long slot. So free_blocks sorts the freed blocks into two bins and places them at opposite ends:

block_pool.py:419–440 
def free_blocks(self, ordered_blocks):
    blocks_with_hash, blocks_without_hash = [], []
    for block in ordered_blocks:
        block.ref_cnt -= 1
        if block.ref_cnt == 0 and not block.is_null:
            (blocks_without_hash if block.block_hash is None
             else blocks_with_hash).append(block)
    # no-hash blocks can never match → evict them first (front)
    self.free_block_queue.prepend_n(blocks_without_hash)
    # cacheable blocks → keep them longest (back)
    self.free_block_queue.append_n(blocks_with_hash)

HMA & align mode

The Hybrid Memory Allocator (HMA, the HybridKVCacheCoordinator) drives models that mix attention types — e.g. full attention + sliding window, or attention + Mamba. Each type is its own KV cache group with its own block_size and its own SingleTypeKVCacheManager, but they all share one BlockPool and therefore one free queue. Eviction is global; the per-group managers only decide what to free and what to cache.

Alignment: making heterogeneous groups agree on a prefix

Groups have different block sizes, so a "prefix of length N tokens" lands on different block boundaries in each group. HMA forces every cross-request hit to a length that is a multiple of scheduler_block_size = the LCM of all group block sizes (kv_cache_utils.py:593–627). That shared alignment_tokens value is threaded into every group's find_longest_cache_hit and cache_blocks, so all groups commit to and resume from the same token boundaries.

scheduler_block_size = lcm(block_size of every group)
e.g. full attn = 32, SWA = 8  →  lcm = 32
every cross-request cache hit length ≡ 0 (mod 32)

Mamba "align" cache mode

For attention+Mamba hybrids, --mamba-cache-mode picks how Mamba participates in prefix caching (config/cache.py:133–141):

modemeaningblocks reserved / request
noneprefix caching off for Mambapage × (1 + spec)
allcache the Mamba state at every i·block_size (default when PC on)page × (cdiv(max_len,bs) + spec)
align cache the state only at the last token of each scheduler step and at i·block_size; Mamba's block_size is forced equal to the attention block_size so hashes line up page × (2 + spec) — only ~2 live blocks ever

In align mode Mamba's block_size == cache_config.block_size, which keeps the hash granularity divisible across groups; otherwise hashing backs off to scheduler_block_size (kv_cache_utils.py:636–644). The page × 2 reservation is the eviction story below: Mamba never holds the whole sequence, just the current state block plus one source block.

SWA eviction

Sliding-window attention only attends to the last sliding_window tokens, so blocks that fall out the back of the window are dead weight. SWA therefore evicts on two timelines.

(a) Mid-request: remove_skipped_blocks

As the request decodes, blocks that slid out of the window are freed back to the pool immediately and replaced with the null block in the request's block list. The number skipped is max(0, num_computed_tokens − sliding_window + 1).

t0–t3 are outside the window → their blocks are freed mid-flight (get_num_skipped_tokens(7) = 4). SWA's peak live blocks ≈ one window, which is why it carries a recycling-aware admission cap (single_type_kv_cache_manager.py:770–796).

remove_skipped_blocks walks the skipped range back-to-front, frees each real block, and stops at the first already-null block (everything before it was nulled on a prior call) (single_type_kv_cache_manager.py:479–520).

(b) On hit: right-to-left search

A SWA prefix hit doesn't need the whole prefix — only the last window's worth of contiguous cached blocks. find_longest_cache_hit scans right to left, needs sliding_window_contiguous_blocks in a row, and returns [NULL, NULL, …, blk, blk] with everything before the window nulled out (single_type_kv_cache_manager.py:619–704).

(c) Sparse caching: reachable_block_mask

Because only the tail window is ever consulted at any aligned boundary, most interior blocks of a long prompt can never serve a hit. Caching them would just burn hash-map slots and add eviction pressure. SWA overrides reachable_block_mask to mark only the need contiguous blocks just before each boundary as cacheable; cache_full_blocks skips the rest (treated like null blocks) (single_type_kv_cache_manager.py:706–768, block_pool.py:211–283). RETENTION_INTERVAL tunes how many of those tails survive — see below.

Mamba eviction

Mamba is recurrent: it carries a single fixed-size state, not a per-token KV history. So "the past" is one block, not N. get_num_skipped_tokens(n) = n − 1 — everything except the most recent state is skippable (single_type_kv_cache_manager.py:1206–1212).

The 2-block dance in align mode

In align mode the new step's state is computed from the previous step's block, so for one step two blocks must coexist: the source (previous) and the destination (current). last_state_block_idx tracks the block from two steps ago; once the current state has moved past it, that block is freed and nulled (single_type_kv_cache_manager.py:1021–1049). That is exactly the page × (2 + spec) reservation.

On hit: last match only

Since only the latest state matters, find_longest_cache_hit scans right-to-left and early-stops on the first match, returning [NULL]×i + [cached] (single_type_kv_cache_manager.py:973–1019).

Cross-step hazard: can't reuse a state cached this very step A Mamba block's state cached by another request in the current scheduler step isn't materialized yet, so it can't be reused until next step. get_num_blocks_to_allocate detects this (cached_blocks_this_step) and returns num_gpu_blocks + 1 — an impossible demand — to force the scheduler to defer the request to the next step (single_type_kv_cache_manager.py:1067–1075).

How RETENTION_INTERVAL helps

VLLM_PREFIX_CACHE_RETENTION_INTERVAL tunes how sparsely SWA checkpoints are cached. It only affects sliding-window groups — setting it on a model with no SWA group is rejected, and the value must be a non-negative multiple of scheduler_block_size (kv_cache_coordinator.py:30–57; env at envs.py:282,1056).

First: what is a "checkpoint"?

A checkpoint is a resumable prefix boundary for the SWA group: an aligned prefix length L (a multiple of scheduler_block_size) together with the run of cached SWA blocks that lets a future request rebuild the window at L and continue — i.e. score a prefix-cache hit there.

Why SWA needs the concept at all: full attention keeps the entire prefix, so every block boundary is automatically resumable. SWA keeps only a sliding window, so a boundary is resumable only if the window's worth of blocks ending at it are still cached. That run is exactly need blocks:

need = cdiv(sliding_window − 1, block_size)  (+1 if EAGLE)   # single_type_kv_cache_manager.py:606

So a checkpoint physically is those need contiguous blocks ("the tail") sitting just before boundary L. Everything between checkpoints is interior context that can never be the tail of an aligned boundary, so it could never serve a hit anyway — no point caching it.

How checkpoints are created (and the default, interval = None)

Checkpoints are minted at cache time, not eviction time. When a request commits its full blocks, cache_blocks → reachable_block_mask returns a per-block True/False mask, and cache_full_blocks only assigns a block_hash to mask-True blocks and inserts them into the prefix-cache map. Mask-False blocks never get a hash → never matchable → first to be evicted (no-hash → prepend_n to the queue front). Creating a checkpoint = marking its need tail blocks True.

Default path, retention_interval is None — single_type_kv_cache_manager.py:737–753 
segment_tokens = alignment_tokens          # == scheduler_block_size
per_segment    = scheduler_block_size // block_size
# mark the last `need` blocks of EVERY segment:
if (i - shift) % per_segment >= per_segment - need:
    mask[i] = True

So the default puts one checkpoint at every scheduler_block_size boundary — the densest setting. Two consequences worth noting:

  • The replay-boundary special case (part 2 below) is skipped when interval is None (if retention_interval is not None): every boundary, including the prompt's end, is already a checkpoint, so there is nothing extra to pin.
  • If the window spans a whole segment (need ≥ per_segment — e.g. a pure SWA model where scheduler_block_size == block_size so per_segment == 1), the function short-circuits to return None = cache every block. Sparsification only bites when the SWA block is finer than the scheduler block (hybrid) or you set an interval larger than one segment.
start_block / end_block are not a window

The mask is computed over [start_block, end_block) = [num_cached_blocks, num_full_blocks) (single_type_kv_cache_manager.py:342) — the blocks that became full since the last step. cache_blocks runs once per scheduler step and only re-hashes that fresh slice, then advances num_cached_block. So all blocks from token 0 to the current token still pass through the mask — just incrementally, across calls, never re-examined twice.

The checkpoint decision uses the absolute index i ((i - shift) % per_segment); only the output is offset (mask[i - start_block]). So checkpoints land at the same absolute positions regardless of where a slice begins. What actually keeps SWA from caching every block is the mask's sparsity (above) and, separately, remove_skipped_blocks freeing out-of-window blocks during the live request — not these two bounds.

What the interval changes

RETENTION_INTERVAL simply swaps what segment_tokens is. A larger segment means fewer, sparser checkpoints; 0 drops periodic checkpoints entirely and keeps only the one explicitly-pinned replay boundary. For very long prompts, "a tail at every 32-token boundary" adds up to many cached SWA blocks competing in the shared pool against full-attention prefixes (which must be kept in full). Thinning SWA checkpoints frees pool capacity for those more-reusable prefixes and cuts eviction churn, at the cost of coarser SWA resume granularity.

The cost: where a partial-prefix follow-up can resume

Suppose a follow-up request shares the first 700 tokens (same document, new question). The SWA hit lands on the largest checkpoint at or below the 64-aligned match length (640). Because HMA takes the minimum hit across all groups, that SWA resume point caps the whole request — even though full attention cached all 700 tokens.

Two contributions feed the mask (single_type_kv_cache_manager.py:737–768): (1) segment tails — the need blocks before each segment boundary (segment = alignment_tokens when None, else interval; suppressed when 0); and (2) the replay-boundary tail — get_computed_blocks caps hits at num_prompt−1 (to recompute the last token's logits), so a full re-submission can only resume at the latest fine-aligned boundary. Sparse retention would skip it, so it is always pinned explicitly.
