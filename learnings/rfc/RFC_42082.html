RFC #42082 — Standardize KV-Cache Layouts

A beginner-friendly walkthrough of the proposal to give every attention backend one canonical KV-cache shape, so the KV-transfer (“connector”) code stops needing per-backend special cases.

TL;DR

Today each attention backend (GQA, MLA, Mamba, …) stores its KV cache in its own shape, and the code that copies KV cache between machines has to special-case every one of them (if is_mla, if is_mamba, …). This RFC proposes a single logical 5-D shape [layers, blocks, heads, states, content] that all backends map onto, a stride_order knob that decides the physical memory order separately from that logical shape, and a rule that each backend slices its own tensor for transfer. The payoff: the connector becomes fully backend-agnostic.

Background: the words you need first

This RFC sits at the intersection of three subsystems. If you are new to vLLM, anchor on these definitions before reading the proposal — every later section reuses them.

KV cache

When a model generates token t, attention needs the Key and Value vectors of every earlier token. Recomputing them each step is wasteful, so vLLM stores them. That store is the KV cache. It is split into fixed-size blocks (a.k.a. pages), each holding block_size token positions.

Attention backend

The kernel + memory format that implements attention. FlashAttention/GQA, MLA (DeepSeek-style latent attention), Mamba (state-space / recurrent), etc. Crucially, each one lays out its KV bytes differently.

KV connector

In P/D disaggregation, one set of GPUs does prefill (reads the prompt, fills the KV cache) and another set does decode (generates tokens). The KV cache must be transferred from prefill to decode — often over the network with RDMA (e.g. the NIXL connector). That transfer code is the connector.

TP (tensor parallelism)

A model too big for one GPU is split across several. With TP=2, the attention heads are divided in half — GPU 0 holds heads 0–n/2, GPU 1 holds the rest. The prefill and decode sides can even run at different TP degrees (“hetero-TP”).

The one-sentence why The connector copies KV bytes between machines. To copy bytes correctly it must know where the bytes are — and right now that knowledge is scattered across every attention backend.

The problem this RFC fixes

The connector currently asks each attention backend two questions: get_kv_cache_shape() (“what shape is your tensor?”) and get_kv_cache_stride_order() (“in what memory order?”). Because the answers differ per backend, the connector fills up with branches and the two subsystems become tightly coupled.

Symptom 1 — flag soup

Connector code is littered with is_mla / is_mamba checks to decide how to interpret the bytes.

Symptom 2 — tight coupling

The connector reaches into attention-backend internals like flash_attn.py to learn the layout. Change a backend → risk breaking transfers.

Root cause There is no shared vocabulary for “where is head 3 of block 12 of layer 5?” Each backend answers in its own dialect, so the connector must speak all of them.

Idea 1: one canonical logical shape

The core proposal: define a single logical (semantic) shape that every backend must describe its KV cache in. It has five dimensions:

[ num_layer_slots , num_blocks , num_heads , num_states , <state_content> ]
       L             B            H            N           C

L — layers

How many transformer layers share this tensor.

B — blocks

The paged-attention blocks (the pool of pages).

H — heads

Attention heads — or 1 for “headless” backends like MLA.

N — states

Token positions per block — or 1 for recurrent state (Mamba).

C — state_content

The backend-specific payload (e.g. [2, head_dim] for K&V). Always contiguous.

Why 5 dims?

It is the smallest shape expressive enough to hold attention and recurrent state, while exposing heads & blocks for slicing.

“Logical” is the key word This shape says what the dimensions mean, not how they sit in memory. The memory order is decided separately (Idea 2). That separation is what makes the whole thing flexible.

How each backend maps onto it

The trick is that the unusual backends fold their quirks into the num_heads=1 or num_states=1 slots. Here is the mapping table from the RFC:

Worked example — GQA, one block

Picture a single block of a standard attention layer with block_size=4, num_kv_heads=2. The block holds 4 token positions (N), each with 2 heads (H), each head storing a K and a V vector (C = [2, head_dim]).

GQA: one block = N states × H heads × (K,V)
state N0 h0: K | V h1: K | V
state N1 h0: K | V h1: K | V
state N2 h0: K | V h1: K | V
state N3 h0: K | V h1: K | V
num_states=4, num_heads=2, state_content=[2, head_dim]. Each cell is one “[H,N,C]” element the connector might copy.

Worked example — MLA & Mamba (the “1” tricks)

MLA — headless

MLA compresses all heads into one shared latent. So num_heads = 1. It still has block_size token positions, so N = block_size, and the content is the latent vector.

vs
Mamba — recurrent

Mamba keeps a single rolling state per sequence, not one-per-token. So there is no “token position” axis → num_states = 1. The recurrent state itself lives in the content dim.

Worked example — DeepSeek V4 (why num_states = block_size/4)

DeepSeek V4’s “c4a” variant is MLA-like — it is headless (num_heads = 1: one shared latent instead of per-head K&V) — but with one extra twist: it compresses every 4 token positions into a single stored state. That compression ratio is exactly the tokens_per_state = 4 field from Idea 3 (the “4” in “c4a”).

Crucially, num_states is never stored — it is derived at allocation time as block_size / tokens_per_state (see Idea 3), because block_size is a deployment choice. With tokens_per_state = 4 that becomes block_size / 4. So a block that addresses block_size token positions only needs block_size/4 physical state slots — each slot is one compressed latent standing in for 4 consecutive tokens.

DeepSeek V4 (c4a): block_size=8, tokens_per_state=4 → num_states = 8/4 = 2
state S0 tokens 0–3 → 1 latent
state S1 tokens 4–7 → 1 latent
num_heads=1 (headless), num_states = 2, state_content = [latent_size]. The block still addresses 8 token positions, but stores only 2 latents because every 4 tokens collapse into one. Halve the compression to tokens_per_state=2 and the same block would hold 4 states; set it to 1 and you are back to plain MLA with 8.
Contrast with plain MLA MLA is also headless, but it keeps tokens_per_state = 1 → num_states = block_size (one latent per token). DeepSeek V4’s c4a is essentially “MLA + 4× temporal compression,” which is the only reason its num_states shrinks to block_size/4. Same headless content dim, fewer state slots.

Idea 2: logical shape vs physical layout (stride_order)

The logical shape is always [L, B, H, N, C]. But you may want the bytes stored in a different order for performance. A stride_order is just a permutation that says “walk the logical dimensions in this physical order.”

Analogy A spreadsheet has the same logical cells whether you store it row-major or column-major on disk. stride_order is choosing row-major vs column-major — the meaning is identical, the access cost is not.

Why does the connector care about physical order? Because a network transfer is fastest when the bytes it wants are contiguous — one big RDMA read beats thousands of tiny scattered ones. Different transfer patterns want different things contiguous, so the RFC defines four named layouts.

The four layouts & when each wins

NamePhysical orderstride_orderWhat ends up contiguousBest for
LBHNC default [L,B,H,N,C] (0,1,2,3,4) identity a head’s data head-level TP transfers, hetero-TP disagg
LBNHC [L,B,N,H,C] (0,1,3,2,4) a state (token) slice block-level transfers, hetero-block-size disagg
BLHNC [B,L,H,N,C] (1,0,2,3,4) one block across all layers big P/D transfers when TP & block size match
BHLNC [B,H,L,N,C] (1,2,0,3,4) one (block, head) across all layers big P/D transfers for hetero-TP

Visual: where does L (the layer axis) move?

The whole game is where the layer dimension sits. In the two “L-first” layouts each layer is a separate region; in the “B-first” layouts all layers for a block are glued together so you can transfer a block in one shot.

LBHNC (default) — layer-major: each layer is its own region
layer 0 blk0 h0 blk0 h1 blk1 h0 blk1 h1
layer 1 blk0 h0 blk0 h1 blk1 h0 blk1 h1
A single head’s bytes are contiguous → easy to grab “just head 1” for tensor-parallel transfers.
BLHNC — block-major: one block carries all its layers together
block 0 L0 h0 L0 h1 L1 h0 L1 h1
block 1 L0 h0 L0 h1 L1 h0 L1 h1
All layers of block 0 are adjacent → the connector transfers an entire block across every layer in one RDMA read. Wins when both sides agree on TP and block size.
Same data, two layouts Both diagrams describe the identical logical tensor [L,B,H,N,C]. Only the byte order differs. The connector reads stride_order to know which one it is looking at — it does not need to know which backend produced it.

Why layer-major is the default but block-major is an opt-in

The two “winning” layouts are pulled by two different consumers. The attention kernel runs the model one layer at a time (finish layer 0, then layer 1…), so its natural nesting is layers → blocks — layer-major. The connector moves a whole sequence’s KV addressed by the block table, so its natural nesting is blocks → layers — block-major. Same tensor, opposite preferred axis order.

ConsumerNatural orderLayoutWhy
Compute (attention kernel) layer-major LBHNC default forward pass runs layer-by-layer; each layer wants its own contiguous region. Also gives the finest-grain (per-head) contiguity, which head-level TP slicing needs.
Bulk transfer (connector) block-major BLHNC P/D copies sequences addressed by block; gluing a block’s layers together turns L small reads into one RDMA read per block.
So which do you store? LBHNC is the logical default: it matches compute, gives the finest contiguity for slicing, and satisfies the HMA contiguity rule, so it always works. BLHNC is a physical opt-in you flip to via stride_order only when the workload is dominated by big P/D transfers and both sides agree on TP and block size — that agreement is what lets the block-of-all-layers blob line up on each end.

Constraint for shared tensors (HMA)

Some models mix attention types (HMA = hybrid memory allocation). vLLM may pack several layer groups into one underlying KVCacheTensor when they share the same page_size_bytes. Those layers can have different H, N, or C sizes. That puts a rule on which layouts are legal:

  1. The per-layer-per-block content [H, N, C] must be contiguous in memory (layout[0,0,...].is_contiguous()). This rules out LBNHC and BHLNC; only LBHNC and BLHNC qualify.
  2. If H, N, and C happen to match across all layers in the tensor, then any layout is valid again.
Why step 2 is the escape hatch The contiguity rule in step 1 exists only to make differently-shaped layers coexist in one buffer. When shapes differ, the tensor is ragged — the only universal handle is “layer i, block j’s page,” one contiguous page_size_bytes blob you reinterpret with that layer’s own shape; that forces [H,N,C] to stay together (LBNHC splits it as N,H,C; BHLNC wedges L in between). But if every layer has the same H, N, C, the buffer is a genuine uniform rectangular [L,B,H,N,C] array — nothing ragged to protect — so the rule has nothing to guard and all four permutations become legal again.
Concrete case — DeepSeek V4 Its layers use mixed compression ratios, so they have different N and C sizes yet share one tensor. The default LBHNC layout satisfies the contiguity rule, so it “just works.”

Idea 3: the spec describes itself

Each backend already has a KVCacheSpec object. The RFC adds three fields so the shape can be reconstructed from the spec alone — no need to call into the attention backend. 

@dataclass
class KVCacheSpec:
    num_heads: int          # heads (1 if headless)
    tokens_per_state: int   # -1 infinite, 1 standard, N compressed
    state_content_size: int # bytes per state per head

Notice num_states is not stored — it is derived as block_size / tokens_per_state at allocation time, because block_size is a deployment choice not known when the spec is built.

Reading tokens_per_state 1 = one token per stored state (normal). 4 = four tokens compressed into one state (DeepSeek V4). -1 = recurrent / “infinite” — the whole sequence collapses to one rolling state (Mamba).

Idea 4: specs slice themselves for TP

Here is the cleverest part. When transferring between two machines with different TP, you must figure out which heads overlap. Today the connector does raw byte math. The RFC moves that knowledge into the spec: each spec implements slice_for_tp_transfer(). 

class KVCacheSpec:
    def slice_for_tp_transfer(self, my_tensor, my_tp, my_rank,
                              other_spec, other_tp, other_rank) -> list[Tensor]:
        """Return the slices of my_tensor that need to be transferred.
        Takes BOTH sides' TP configs, since tp_size is a deployment
        property, not part of the spec."""
        ...

GQA

Slices by heads — computes which head range on the source overlaps the destination rank, returns that slice.

Mamba Conv

Returns 3 slices (the x / B / C components of the conv state) — same interface, different internal structure.

MLA

May be a single unsliced latent — nothing to split because it is headless.

Why pass both sides’ TP?

The source has src_tp × num_heads total heads; the destination has dst_tp × num_heads. The spec computes the overlap of head ranges and converts it to local indices. The GQA implementation even handles replication when one side has fewer ranks (dst_rank %= src_tp).

TP slicing: src TP=1 (4 heads) → dst TP=2 (2 heads each)
source rank 0 h0 h1 h2 h3
dst rank 0 h0 h1   dst rank 1 → h2 h3
To fill dst rank 1 (heads h2,h3), the source spec returns src_tensor[:, :, slice(2,4)]. The spec — not the connector — computed that overlap.

The payoff: a generic connector

With the three ideas in place, the connector no longer contains a single backend-specific branch. It asks both specs to slice themselves, then copies contiguous chunks. 

# 1. TP slicing — computed ONCE at setup (which heads to transfer)
src_slices = src_spec.slice_for_tp_transfer(
    src_cache, src_tp, src_rank, dst_spec, dst_tp, dst_rank)
dst_meta   = make_meta_kv_cache(dst_spec, num_blocks, dst_block_size)
dst_slices = dst_spec.slice_for_tp_transfer(
    dst_meta, dst_tp, dst_rank, src_spec, src_tp, src_rank)

# 2. Per-block transfer — handles hetero block sizes naturally
for src_blk, src_states, dst_blk, dst_states in block_mapping:
    for src_s, dst_s in zip(src_slices, dst_slices):
        src_block = src_s[:, src_blk, :, src_states, ...]
        dst_block = dst_s[:, dst_blk, :, dst_states, ...]
        for src_chunk, dst_chunk in jointly_contiguous_chunks(src_block, dst_block):
            self._rdma_transfer(src_chunk, dst_chunk)
  1. TP slicing once. Both specs return slices in the same semantic order, so src_slices and dst_slices pair up by zip — even if the two sides use different stride_orders.
  2. Meta tensor for the destination. make_meta_kv_cache builds a device='meta' tensor — it carries shape/stride info but allocates no GPU memory. It exists purely to compute destination offsets.
  3. jointly_contiguous_chunks decomposes each slice pair into regions contiguous on both sides, automatically coalescing interleaved layers into bigger transfers when the layout allows.
  4. Hetero block sizes fall out for free: the block_mapping carries per-side block index + state slice, so prefill block size ≠ decode block size still works.

PR #42374 — Part 1 in depth

The RFC is the blueprint. PR #42374 (“[Core][WIP][1/N] Standardize kv layout”, body literally just says “part 1 of #42082”) lays the foundation: it builds the shared vocabulary in one place and then deletes the per-backend dialects that the connector used to depend on.

The diff shape tells the story +1,677 / −3,382 across 98 files — a net removal of ~1,700 lines. When a refactor deletes twice as much as it adds, it is collapsing duplicated code into a single definition. That is exactly what “standardize” means here.

Which of the four ideas does Part 1 deliver?

Part 1 is deliberately scoped. It ships the shape vocabulary (Ideas 1–3) and rips out the old per-backend shape methods. The self-slicing connector (Idea 4) is left for later stages.

Before this PR, every attention backend implemented two methods the connector and allocator called: get_kv_cache_shape() and get_kv_cache_stride_order(). Each backend answered in its own way — that is the “dialect” problem from the motivation.

The new vocabulary, as actually committed

1. The KVCacheLayout enum — Idea 2 made concrete

The four named layouts from the layouts section are now an enum whose value is the stride permutation — with helper properties the connector reads instead of hard-coding layout strings. 

class KVCacheLayout(Enum):
    LBHNC = (0, 1, 2, 3, 4)  # identity (default)
    LBNHC = (0, 1, 3, 2, 4)
    BLHNC = (1, 0, 2, 3, 4)
    BHLNC = (1, 2, 0, 3, 4)

    @property
    def is_layer_compact(self):   # L is outermost?  (LBHNC, LBNHC)
        return self.value[0] == 0

    @property
    def is_block_contiguous(self): # [H,N,C] contiguous in a block?
        return self.value[-3:] == (2, 3, 4)

is_block_contiguous is precisely the shared-tensor constraint turned into a one-line check: only LBHNC and BLHNC pass it.

2. Three spec properties — Idea 3 made concrete

KVCacheSpec gains the RFC’s three fields as properties (the RFC’s state_content_size landed as state_content_size_bytes). Each subclass fills them in: 

class KVCacheSpec:
    @property
    def num_heads(self) -> int: ...               # 1 if headless
    @property
    def tokens_per_state(self) -> int: ...        # -1 recurrent, 1 std, N compressed
    @property
    def state_content_size_bytes(self) -> int: ...

# AttentionSpec (GQA):  C = K and V packed together
    state_content_size_bytes = 2 * head_size * dtype_size
# MLAAttentionSpec:     headless latent, no factor of 2
    num_heads = 1
Naming cleanup that proves the point MLA’s DeepSeek-V4 field compress_ratio was renamed to tokens_per_state throughout kv_cache_interface.py. The RFC’s vocabulary literally replaced the old per-model term — storage_block_size is now block_size // tokens_per_state.

3. reshape_kv_cache() — Idea 1 made concrete

This single function replaces every backend’s shape logic. Give it a flat int8 buffer + a spec + a layout, and it returns one [B, H, N, C] view per layer slot — works for GQA, MLA, Mamba alike: 

def reshape_kv_cache(raw, spec, num_blocks, num_layer_slots, layout, block_size=None):
    # 1. logical byte-shape from the SPEC (not the backend)
    logical = (num_layer_slots, *compute_layer_kv_cache_shape_bytes(spec, num_blocks, block_size))
    # 2. reorder to physical layout via the enum's stride_order
    physical = tuple(logical[i] for i in layout.stride_order)
    cache = raw.view(physical).permute(*inv_order)   # back to logical [L,B,H,N,C]
    return [cache[i] for i in range(num_layer_slots)]   # one view per layer

def num_states_for(block_size, tokens_per_state):
    return 1 if tokens_per_state == -1 else block_size // tokens_per_state

Note num_states is computed here at allocation time (-1 recurrent → 1), exactly as the RFC said it would be — derived, never stored.

What is inv_order? It is the inverse permutation of layout.stride_order. stride_order scrambles the logical [L,B,H,N,C] axes into the physical order the bytes are actually stored in; inv_order un-scrambles physical back to logical, so every caller gets a consistent [L,B,H,N,C] view no matter which layout produced the bytes. Compute it as argsort(stride_order) — the permutation q with q[stride_order[i]] = i. For the default LBHNC layout stride_order is the identity, so inv_order is identity too and the permute is a no-op; it only does real work for the non-default layouts.

Example run — GQA buffer in BLHNC layout

Take a tiny GQA tensor: num_layer_slots=2, num_blocks=3, num_heads=2, block_size=4 with tokens_per_state=1 (so num_states=4), and a state_content of 16 bytes per state. Store it block-major (BLHNC) so whole blocks transfer in one shot. 

reshape_kv_cache(
    raw,                          # flat int8 buffer of 2*3*2*4*16 = 3072 bytes
    spec,                         # GQA: num_heads=2, tokens_per_state=1, content=16B
    num_blocks=3,
    num_layer_slots=2,
    layout=KVCacheLayout.BLHNC,   # stride_order = (1, 0, 2, 3, 4)
    block_size=4,
)

# num_states = num_states_for(block_size=4, tokens_per_state=1) = 4 // 1 = 4
# logical    = (L, B, H, N, C) = (2, 3, 2, 4, 16)     # axis indices (0,1,2,3,4)
# stride_order (BLHNC)            = (1, 0, 2, 3, 4)
# physical = (logical[1],logical[0],logical[2],logical[3],logical[4])
#          = (3, 2, 2, 4, 16)  ->  [B, L, H, N, C]   # how the bytes actually sit

# inv_order = argsort(stride_order) = inverse of (1,0,2,3,4) = (1, 0, 2, 3, 4)
cache = raw.view(3,2,2,4,16)        # interpret flat bytes as [B,L,H,N,C]
            .permute(1,0,2,3,4)    # un-scramble -> [L,B,H,N,C] = (2,3,2,4,16)

return [cache[0], cache[1]]           # 2 per-layer views, each [B,H,N,C] = (3,2,4,16)

The key step is the permute(*inv_order): the raw bytes are block-major (B outermost), but after permuting by inv_order the returned views are indexed [L,B,H,N,C] like every other backend — the caller never has to know it was stored BLHNC.

When inv_order ≠ stride_order BLHNC just swaps the first two axes, so it happens to be its own inverse. That is a coincidence of that layout, not a rule. For BHLNC (stride_order = (1,2,0,3,4)) the inverse is argsort((1,2,0,3,4)) = (2,0,1,3,4) — genuinely different. inv_order is always the true inverse, never just a copy of stride_order.

The recurrent & connector edges line up too

Mamba binds N=1, H=1

Mamba’s new bind_kv_cache() unpacks a raw [B, 1, 1, C] int8 view into its conv/SSM state tensors — literally the num_heads=1, num_states=1 row from the mapping table.

NIXL speaks the new names

The NIXL connector dropped its old "HND"/"NHD" strings for "LBHNC"/"LBNHC", and replaced a hard-coded layout == "HND" check with KVCacheLayout[name].is_block_contiguous. First taste of the connector becoming layout-driven.

Still WIP PR #42374 is marked [WIP] and is open, touching 98 files including many test updates. It establishes the vocabulary and removes the old methods, but the fully generic, self-slicing connector (Idea 4) arrives in later stages. Read it as “phase 1 of N,” not the finished product.

Takeaways

1. One logical shape

[L, B, H, N, C] — every backend folds its quirks into the H=1 / N=1 slots.

2. Layout ≠ meaning

stride_order picks physical byte order independently, to make the right things contiguous for a given transfer.

3. Specs are self-describing

Three fields reconstruct the shape; num_states is derived from block_size.

4. Specs slice themselves

slice_for_tp_transfer() moves TP/head math out of the connector and into each backend’s spec.

The one thing to remember The connector should answer “where is head 3 of block 12 of layer 5?” in one language. This RFC defines that language (logical shape + stride_order) and makes each backend translate itself into it — so the transfer code becomes backend-agnostic.
