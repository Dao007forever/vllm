PR #44560 - Async Remote KV Load Reservations

How the scheduler avoids the case where two requests both async-load remote KV blocks, each parks a prefix in GPU memory, and then neither has enough remaining blocks to finish prefill.

TL;DR

PR #44560 fixes the wedge by making async remote KV-load admission reserve capacity for already in-flight prefills. When request A is already loading or chunk-prefilling, the scheduler computes how many more KV blocks A still needs to hold its full prompt. A new async KV load for request B is allowed to allocate its loaded-prefix blocks only from free_blocks - reserved_for_A. If B would consume A's reserved space, B stays WAITING and holds no blocks. That is the key: B cannot block A, so at least A can finish prefill and make forward progress. Compared with PR #40968, this is a prevention strategy rather than a repair strategy: #40968 lets the wedge form and then laterally preempts a non-running block holder; #44560 denies the risky async-load admission up front.

The blocking pattern

The dangerous state appears with async KV connector loads, usually in P/D-style flows. The request has a remote prefix-cache hit, so vLLM allocates KV blocks for the externally computed prefix and marks the request WAITING_FOR_REMOTE_KVS. While the transfer is in flight, the request is not running a forward pass, but its allocated blocks are already unavailable to everyone else.

Request A

A async-loads remote KV for a prefix. It holds those prefix blocks while waiting for the receive to complete. Later, it still needs extra blocks for the rest of its prefill.

Request B

B also async-loads remote KV and parks its own prefix blocks. If the pool cannot fit A's and B's full sequences together, B can consume the exact capacity A needed to continue.

Why this wedges Both requests hold blocks while doing no local forward progress. When their remote receives complete, each needs more blocks to finish prefill, but the other request is occupying the headroom. Because these requests are not in running, the usual running-request preemption path may have no victim to free.

Before #44560

The existing full_sequence_must_fit admission check answered only one question: can this request's full sequence fit against the blocks that are free right now? For async loads, though, the actual allocation was only for the remote prefix blocks. The future blocks needed to finish the sequence were not held aside.

Before: latent capacity was not protected

→
2A holds prefix blocks
→
3A's remaining blocks are only implicit
4B loads prefix

→
6A and B can now block each other

PR #40968 proposed resolving the same family of stalls after they form, by laterally preempting a non-running block holder. PR #44560 instead prevents the wedge from forming: do not admit the second async load if its initial prefix allocation would eat space already needed by in-flight prefills.

What #44560 changes

The PR adds a small reservation model around async KV-load admission.

  1. Track in-flight prefills. The scheduler adds self._inflight_prefills, containing requests that are still in prefill and have already entered the scheduler: in-progress async KV loads and chunked prefills that still need more prompt work.
  2. Compute remaining blocks for each tracked request. _request_remaining_blocks(request) asks the KV-cache coordinator how many more blocks that request needs to allocate to hold its full sequence, capped at max_model_len and using the same block accounting machinery as normal allocation.
  3. Sum the reservations. _inflight_prefill_reserved_blocks() sums those remaining block counts. The candidate async load is not in the set yet, so the sum represents older in-flight work that must not be starved.
  4. Gate only async KV-load allocations. When load_kv_async is true, the scheduler passes reserved_blocks into KVCacheManager.allocate_slots(). The cache manager then checks the candidate's actual allocation against free_blocks - reserved_blocks.
  5. Drop finished work from the set. Requests are removed from _inflight_prefills when they stop being prefill chunks, are preempted, or are freed.
Files touched The scheduler logic is in vllm/v1/core/sched/scheduler.py. The allocation gate is in vllm/v1/core/kv_cache_manager.py. 
reserved = sum(remaining_blocks(req) for req in inflight_prefills)

if candidate_is_async_kv_load:
    available = free_blocks - reserved
    admit_candidate_only_if(candidate_prefix_blocks <= available)

Two-request walkthrough

The PR's scheduler test uses a compact version of the failure: A needs 8 blocks for its full prompt, B needs 5 blocks, both have a 4-block remote prefix, and the pool has only 9 usable blocks. Both prefixes fit together, but both full prompts do not.

Moment Without reservation With #44560
A is considered A async-loads 4 prefix blocks. A async-loads 4 prefix blocks and enters _inflight_prefills.
Scheduler considers B B can also allocate 4 prefix blocks because those blocks are currently free. A's remaining 4 blocks are reserved. B sees only free - 4 usable blocks.
B admission result B enters WAITING_FOR_REMOTE_KVS and holds prefix blocks. B stays WAITING and holds no blocks.
A's receive completes A may be unable to allocate the rest of prefill because B holds the missing capacity. A can allocate the reserved blocks and move to RUNNING.
The actual fix The fix does not need to cancel B's transfer or preempt B after the fact. It prevents B from holding any KV blocks in the first place unless doing so leaves enough capacity for older in-flight prefills to finish.

Comparison with #40968

PR #40968 and PR #44560 target the same scheduler pathology: non-running requests can hold KV blocks after an async remote KV load, and a head-of-line request can then fail admission because the blocks it needs are parked behind another request. They differ mainly in when they intervene.

Question PR #40968: lateral preemption PR #44560: reservation admission
Intervention point After the wedge has formed. If the target request cannot allocate blocks, the scheduler searches skipped waiting queues for a lateral victim and frees it. Before the wedge forms. A candidate async load is admitted only if it leaves enough free blocks for older in-flight prefills to complete.
Victim model Picks a non-running block holder: recv-done WAITING_FOR_REMOTE_KVS, or WAITING / PREEMPTED with loaded prefix progress. Running requests are deliberately excluded. No victim is needed in the normal fix path. The later async-load candidate simply remains WAITING and holds no blocks.
How it resolves A/B A and B may both load prefixes. When A later cannot allocate its remaining blocks, B can be laterally preempted, freeing B's prefix blocks so A can run. A loads first and reserves its remaining blocks. B is not allowed to load if B's prefix would consume A's reservation, so A can run once its receive completes.
Cost paid Usually preserves more async-load overlap. The cost is paid only on wedge-like allocation failure: preemption bookkeeping, block frees, clearing stale recv state, and retry. May reduce overlap in tight pools because later async loads can wait even while blocks are physically free. The benefit is simpler state: no cancel/preempt/retry path is required for the common wedge.
Complexity Adds candidate classification, victim search, lateral-preempt cleanup, and Request.has_executed so never-executed victims reenter the worker as new requests rather than resumed requests. Adds _inflight_prefills, remaining-block accounting, and a reserved_blocks argument to allocate_slots().
Policy flavor Optimistic. Admit more async work, overlap remote receives, and repair only if the bad capacity shape actually appears. Pessimistic. Protect completion capacity for older prefills first, even if that means delaying a later remote KV load.
Why #44560 is simpler for this specific bug The A/B wedge precondition is "both requests hold prefix blocks, but the pool cannot fit both full sequences." #44560 removes that precondition directly: the second request never gets to hold prefix blocks unless the older in-flight requests' remaining full-prefill capacity is still protected.

The tradeoff is exactly the one hinted in #40968's alternative section: reservation-based admission can serialize async loads behind earlier in-flight work under pressure, exposing remote-receive latency that might otherwise have been overlapped. #40968 preferred throughput in the common wedge-free case and paid only when a wedge occurred. #44560 chooses predictability and a smaller scheduler state machine: it may be more conservative, but it avoids preempting work that never needed to be admitted in the first place.

Is the policy pessimistic?

Yes, in the useful sense. It is a conservative admission policy for async remote KV loads: once a request is in-flight, its remaining prefill capacity is treated as unavailable to later async loads. That can reduce overlap in a tight KV pool, because a later remote load may wait even though there are physically free blocks at that instant.

It is also pessimistic because get_num_free_blocks() is a snapshot of the current block pool. The scheduler does not forecast that running prefills or decodes might finish soon and return blocks. If the candidate async load does not fit in current_free_blocks - reserved_blocks, it is held back now and retried in a later scheduling step after real frees have happened.

But it is not a blanket "one async load at a time" rule. It is capacity based:

Conservative

It prefers preserving progress for already admitted prefills over overlapping another remote load. This avoids deterministic wedges and predictable preemptions.

Not over-serializing

If the pool has enough slack beyond the in-flight reservations, the next async load is still admitted. The PR adds a sanity test for this case.

A subtle point: the reservation protects older in-flight work from the new candidate's initial async-load allocation. The candidate is added to the in-flight set only after it is admitted. So the policy is pessimistic about not stealing from older requests, but it does not globally solve a bin-packing problem for all future completions. That is intentional for this PR's scope: prevent mutual blocking by ensuring a newly admitted async load cannot be the thing that robs older prefills of their completion blocks.

Takeaways

  • Async remote KV loads are special because they can hold GPU KV blocks while doing no forward-pass work.
  • The old full-ISL check proved a candidate could fit at admission time, but it did not protect the unallocated remainder of already admitted prefills.
  • PR #44560 turns those unallocated remainders into an admission reservation for later async loads.
  • The policy is deliberately conservative in tight pools: wait before allocating B's remote prefix rather than allocate it and later discover A cannot complete.
  • Compared with lateral preemption, this is a prevention strategy. The scheduler keeps the wedge precondition from appearing instead of trying to repair it after both requests already hold blocks.
  • The gate uses current free blocks, not predicted future frees, so it can delay an async load that would have fit after a near-future decode or prefill completion.
