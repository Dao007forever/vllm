The GPU Model Runner — V1 vs V2

Both runners turn a SchedulerOutput into sampled tokens for one engine step. V1 is the battle-tested 7.6k-line monolith. V2 (“MRV2”) is the experimental rewrite that breaks the same job into small, testable modules. A guided tour of every step in a single step — and what changed.

TL;DR Both share the same two-phase shape: execute_model() runs the forward pass and stashes ephemeral state, then sample_tokens() samples + proposes drafts. The difference is structure, not the algorithm. V1 keeps request state, input prep, attention metadata, sampling and spec-decode all inside one class. V2 splits them into RequestState, InputBatch, BlockTables, ModelState, Sampler, a CudaGraphManager and helper modules — and adds staged writes + UVA buffers to cut host→device sync overhead.

1. Orientation — where each lives

A model runner lives inside the GPU worker process. Each engine step, the scheduler hands it a SchedulerOutput (which requests run, how many tokens each, block tables, grammar bitmasks) and it returns a ModelRunnerOutput (sampled token ids, logprobs, KV-connector output). It owns the persistent input batch, the KV-cache block tables, CUDA-graph capture, sampling, and speculative decoding.

V1 The monolith

  • vllm/v1/worker/gpu_model_runner.py
  • ~7,600 lines, one GPUModelRunner class
  • Helpers: gpu_input_batch.py (InputBatch, CachedRequestState), block_table.py, and a stack of mixins (LoRA / KV-connector / EC-connector)
  • The default, production path

V2 / MRV2 The rewrite

  • vllm/v1/worker/gpu/model_runner.py
  • ~1,600 lines for the class; ~5,200 lines across the gpu/ package
  • ~18 helper modules + sub-packages (sample/, spec_decode/, mm/, model_states/, pool/)
  • Marked experimental, under active development in its README.md
Naming note “V1” here means the classic runner that ships under vllm/v1/ — not the old V0 engine. “V2” / “MRV2” is the newer runner inside the vllm/v1/worker/gpu/ sub-package. Both target the same V1 engine; they are two implementations of the same role. Commits tag it [MRV2] / [Model Runner V2].

2. The shared shape: a two-phase step

Before diving into either, internalize the shape they both use. A single engine step is split into two method calls. execute_model() does everything up to and including the model forward and logits, then returns None after stashing an ExecuteModelState tuple on the runner. The worker then calls sample_tokens(), which unpacks that state and does sampling, draft proposal and bookkeeping.

Why split it? The grammar/structured-output bitmask for a request often isn't ready when the forward starts — it's computed on the CPU in parallel. Splitting lets the GPU run the (expensive) forward while the CPU finishes the bitmask, then sample_tokens() applies it just-in-time. The split also creates a clean seam for async scheduling: the sampled-token D2H copy is launched on a side stream and the result is collected later, so the next step's CPU prep overlaps it.

3. Scheduler ↔ Model Runner: the per-step handshake

The runner never calls the scheduler, and the scheduler never calls the runner. They live in different places — the Scheduler runs in the EngineCore process; each GPUModelRunner runs inside a worker. EngineCore is the broker: it drives one engine step by calling the scheduler, fanning the result out to every worker's runner through the model_executor, then feeding the runner's output back to the scheduler. Two plain data structures cross that boundary each step.

The contract's load-bearing trick: optimistic advance, then reconcile. schedule() pre-bumps request.num_computed_tokens by the scheduled count in _update_after_schedule() before the forward even runs (scheduler.py:1141) — that's what lets it schedule the next chunk immediately and stay one step ahead. When the runner's output comes back, update_from_output() corrects it: for spec-decode it rolls num_computed_tokens (and num_output_placeholders) back by the number of rejected draft tokens (scheduler.py:1562–1567). The scheduler plans optimistically; the runner reports reality; the scheduler trues up.

Where the contract bends for V1 vs V2

schedule() itself branches on use_v2_model_runner when it builds the output — so the two runners receive subtly different payloads:

V1 payload shape

  • Resumed (preempted-then-restarted) requests travel in a separate scheduled_resumed_reqs list. :1023
  • NewRequestData carries block ids but not the full token history.
  • Under PP-without-async, CachedRequestData.new_token_ids is populated so the last-stage worker sees the sampled tokens. :1242

V2 payload shape

  • Resumed requests are merged into scheduled_new_reqs; the resumed list is emptied. :1012–1014
  • NewRequestData additionally ships req._all_token_ids so RequestState can seed its UVA token buffer. :1015–1022
  • Sampled tokens flow worker-to-worker via the GPU broadcast path (input_batch.prev_sampled_token_ids), so new_token_ids is omitted except in PP-without-async. :1239–1242
  • V2+PP+async adds a decode-throttle cadence: a request can't decode again until current_step ≥ next_decode_eligible_step, matching the worker's sampled-token broadcast ring. :450, :277
The async / pipelined variant: step_with_batch_queue() core.py:519 With async scheduling or PP, EngineCore keeps several batches in flight. It schedules and launches the next forward before the previous step's output is processed. Sampling for a step can be deferred when its grammar bitmask depends on tokens still being computed (pending_structured_output_tokens); in that case the drafter's tokens are validated via update_draft_token_ids_in_output() (invalid drafts padded with -1) before the deferred bitmask is computed. core.py:612–629 This is why the runner copies sampled tokens out on a side stream (V2's AsyncOutput) — the scheduler may not read them until a later tick.

5. Model Runner V2 — the steps

Same two phases — but the monolithic _update_states / _prepare_inputs are decomposed into named methods that delegate to dedicated helper objects (self.req_states, self.block_tables, self.model_state, self.input_buffers, self.cudagraph_manager, self.pp_handler).

Phase 1 — execute_model() gpu/model_runner.py:1102

Where V1 has one _update_states, V2 calls them in sequence: update_pp_decode_requests() (collect sampled outputs broadcast from the last PP rank pp_size steps ago) · finish_requests() (drop finished/preempted) · free_states() (release encoder MM cache) · add_requests() (register new requests with req_states / model_state / block_tables / sampler / lora) · update_requests() (advance computed tokens, append blocks). :1112–1116

8. The big design shifts

① From one class to a graph of collaborators

V1's power and pain are the same thing: everything is reachable from self. That makes cross-cutting features (async spec-decode, M-RoPE, hybrid Mamba state) easy to bolt on but hard to test in isolation, and the file grew to ~7.6k lines. V2 inverts this — the runner becomes a thin orchestrator that delegates to RequestState, BlockTables, ModelState, Sampler, CudaGraphManager, and friends. Each is independently unit-testable.

② Staged writes + UVA buffers to hide H2D latency

Every step, request metadata (token ids, computed counts, block ids) must reach the GPU. V1 copies eagerly. V2's buffer_utils.py introduces staged writes: CPU-side edits accumulate in lists and are flushed as a single Triton kernel (apply_staged_writes()), and large rarely-read tensors are UVA-backed (unified memory, source-of-truth on CPU) to save VRAM. A round-robin UVA buffer pool lets multiple in-flight steps copy without stalling each other.

V1 eager writes

edit → copy_to_gpu → edit → copy_to_gpu …

Many small H2D transfers per step.

vs
V2 staged writes

stage_write × N → apply_staged_writes (1 kernel)

One fused transfer; fewer syncs.

③ Model quirks become polymorphism, not branches

Whisper needs cross-attention; Mamba/hybrid models carry SSM state; some models need M-RoPE. In V1 these are conditionals threaded through _prepare_inputs, _preprocess and the state updates. V2 lifts them into a ModelState hierarchy (DefaultModelState, WhisperModelState, MambaHybridModelState) chosen at load_model(). The runner calls model_state.prepare_attn() / get_mm_embeddings() / prepare_inputs() and lets the subclass do the right thing.

④ Cross-cutting concerns get their own objects

Pipeline parallel (PPHandler), data parallel (dispatch_cg_and_sync_dp), async output (AsyncOutput), CUDA graphs (CudaGraphManager + BatchExecutionDescriptor), and speculators (spec_decode/) each become a named module with a small interface — versus V1, where they're methods and attributes interleaved through the one class.

9. Takeaways

Same algorithm

Both: reconcile batch → prep inputs → attn metadata → forward → logits → (handoff) → grammar → sample → draft → output. If you understand one phase split, you understand both.

Different structure

V1 keeps state + behavior in one class; V2 splits them into ~18 modules and helper objects with narrow interfaces. The step list is shorter per-method because work moved into collaborators.

V2's perf bets

Staged block-table writes (one fused kernel), UVA-backed buffers + pools (less VRAM, fewer stalls), and an explicit AsyncOutput that overlaps the sampled-token copy with draft proposal.

Status

V1 is the default production path. V2 is experimental, under active development — read its README.md and git log for [MRV2] tags before relying on a detail; the package moves fast.
