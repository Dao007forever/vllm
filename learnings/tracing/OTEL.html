OpenTelemetry Tracing in vLLM v1

Where request spans come from, how timing crosses the scheduler/worker boundary, and a recipe for emitting your own span from inside EngineCore.

TL;DR vLLM emits one OTel span per request — "llm_request" — and it is created in the frontend process by OutputProcessor.do_tracing, after the request finishes, with its start_time backdated to arrival. The worker processes never emit request spans; the EngineCore process only measures per-request timing (monotonic event timestamps) and forwards it back over ZMQ. The EngineCore process already has a tracer initialized (core.py:1170), so emitting a span there is mostly a matter of calling @instrument or instrument_manual — with one sharp edge: EngineCore timestamps are time.monotonic(), but OTel wants epoch-nanoseconds.

1. The big picture: who emits the span

vLLM v1 runs as (at least) two cooperating processes connected by ZMQ. The default frontend is Python (FastAPI + AsyncLLM + OutputProcessor). There is also an experimental, opt-in Rust frontend (VLLM_USE_RUST_FRONTEND=1) which has no OpenTelemetry — it forwards trace_headers to the engine and exports Prometheus metrics only. Everything below describes the default Python path.

2. The tracing toolbox

Everything lives under vllm/tracing/. The public surface is small (vllm/tracing/__init__.py):

SymbolWhat it doesWhere used
init_tracer(module, endpoint, attrs) Builds a TracerProvider + OTLP exporter; also writes the endpoint into os.environ["OTEL_EXPORTER_OTLP_TRACES_ENDPOINT"] so child processes inherit it. Frontend — async_llm.py:116, llm_engine.py:68
maybe_init_worker_tracer(module, kind, name) Reads that env var; if set, calls init_tracer in this process. Returns None when no endpoint. EngineCore — core.py:1170; workers
@instrument(span_name=…) Decorator. Wraps sync/async fn in a current span; parent via _get_smart_context(); wraps body in propagate_trace_to_env(). Wall-clock start/end automatically. core.py:239 ("Prepare model"), core.py:900 ("EngineCoreProc init")
instrument_manual(name, start_time, end_time, attributes, context, kind) Creates a span with explicit epoch-ns timestamps and ends it immediately. Used to backdate spans. output_processor.py:766 (the llm_request span)
extract_trace_context(headers) Parses W3C traceparent/tracestate into an OTel Context — the parent link. output_processor.py:723
SpanAttributes String constants for gen_ai.* attribute keys (latency, tokens, request id, …). vllm/tracing/utils.py:15

Two layers, don't conflate them. When OTel is unavailable, @instrument returns the function unchanged and instrument_manual returns None — both no-op cleanly. When OTel is installed but no endpoint is set, the worker tracer is None, so trace.get_tracer() hands back the global no-op tracer: spans are created but never exported. Cheap, but you still pay for any attribute dict you build — so gate expensive work on the endpoint being configured (see §9).

3. Timing across the process boundary

This is the crux of "scheduler and worker processes". The per-request timeline is measured in EngineCore and assembled in the frontend:

  1. Frontend stamps wall-clock arrival_time (time.time()) when the request is created.
  2. EngineCore (scheduler) records EngineCoreEvents — QUEUED, SCHEDULED — using time.monotonic() (scheduler.py:942, 1217), and stamps an engine_core_timestamp on each output batch.
  3. Those events ride back to the frontend inside EngineCoreOutput over ZMQ.
  4. Frontend rebuilds RequestStateStats from the events (stats.py:404-422) and computes durations: queued = scheduled_ts − queued_ts, prefill = first_token_ts − scheduled_ts, decode = last_token_ts − first_token_ts.
  5. On finish, OutputProcessor.do_tracing (output_processor.py:712) builds the attribute dict and calls instrument_manual("llm_request", start_time=arrival_ns, context=extract_trace_context(trace_headers), kind=SERVER).
Why durations survive the process hop. queued_ts, scheduled_ts, first_token_ts, last_token_ts are all on the EngineCore monotonic clock, so their differences are valid even though they're consumed in another process. Only arrival_time (the absolute span start) is wall-clock. EngineCoreEvent's own docstring warns: monotonic timestamps "should not be compared with timestamps from other processes" (vllm/v1/engine/__init__.py:156-161).

4. EngineCore already has a tracer

You don't need to bootstrap anything. When the frontend has tracing enabled, init_tracer puts the OTLP endpoint into the environment, the EngineCore subprocess inherits it, and at startup it calls:

# vllm/v1/engine/core.py:1170 (inside run_engine_core)
maybe_init_worker_tracer("vllm.engine_core", "engine_core", process_title)

So a TracerProvider with a BatchSpanProcessor is live in the EngineCore process. The only question is how you open a span and what parent it gets.

5. Recipe A — @instrument (coarse, easy)

The lowest-effort option. Decorate any EngineCore method and you get one span per call, with wall-clock start/end filled in automatically (no clock math). This is exactly how init/warmup are traced today.

from vllm.tracing import instrument

class EngineCore:
    @instrument(span_name="EngineCore step")
    def step(self) -> tuple[dict[int, EngineCoreOutputs], bool]:
        ...
Watch the volume. step() runs once per forward pass — at decode that is one span per token-batch per iteration. Decorating the busy loop will flood your collector. Prefer decorating rare/coarse operations (init, profiling, a KV-transfer wait), or use Recipe B with explicit start/end so one span covers a whole request rather than one per step.

Parent. @instrument derives its parent from _get_smart_context() (otel.py:216): if a span is already active in this process it nests under it; otherwise it reads traceparent from os.environ. EngineCore's env only holds the process-level traceparent injected at spawn — not the per-request client trace. So an @instrument span is not tied to the originating HTTP request. For that, use Recipe B.

6. Recipe B — per-request span, parented to the client trace

To attach an EngineCore span to the same distributed trace as the client request, you must explicitly pull the context out of the request's trace_headers (the scheduler's Request object carries them — request.py:165) and hand it to a manual span. Mirror what OutputProcessor.do_tracing does, but inside EngineCore.

import time
from vllm.tracing import (
    SpanAttributes, SpanKind, extract_trace_context, instrument_manual,
)

# Capture a monotonic→epoch offset ONCE, e.g. in EngineCore.__init__ — see §7.
#   self._epoch_offset = time.time() - time.monotonic()

def _emit_engine_span(self, request, sched_ts, done_ts):
    # sched_ts / done_ts are EngineCore time.monotonic() readings.
    endpoint = self.vllm_config.observability_config.otlp_traces_endpoint
    if not endpoint:
        return                                  # nothing is exported; skip the work
    ctx = extract_trace_context(request.trace_headers)  # parent = client trace, or None
    start_ns = int((sched_ts + self._epoch_offset) * 1e9)
    end_ns   = int((done_ts  + self._epoch_offset) * 1e9)
    instrument_manual(
        span_name="engine_core.schedule_execute",
        start_time=start_ns,
        end_time=end_ns,
        attributes={
            SpanAttributes.GEN_AI_REQUEST_ID: request.request_id,
            SpanAttributes.GEN_AI_LATENCY_TIME_IN_SCHEDULER: done_ts - sched_ts,
        },
        context=ctx,
        kind=SpanKind.INTERNAL,
    )

Call it from step() (core.py:479) or from scheduler.update_from_output — wherever you can see both the Request (for trace_headers) and the timestamps you want to bracket. Since durations come from monotonic deltas, they are accurate regardless of the clock conversion below.

7. The clock gotcha — monotonic vs epoch

Do not pass a time.monotonic() value as start_time. OTel interprets start_time/end_time as nanoseconds since the Unix epoch. time.monotonic() is an arbitrary zero (often seconds since boot), so a span built from it would be dated to ~1970 + uptime — garbage on the timeline. This is precisely why the frontend's llm_request span uses wall-clock arrival_time for its start, and uses the monotonic event timestamps only for subtraction.

Two correct options:

Use wall-clock directly

If you can capture the bracket with time.time() at the point of interest, just pass those epoch-seconds×1e9. Simplest when you control the call site.

Convert monotonic → epoch

Capture an offset once: offset = time.time() − time.monotonic(), then epoch_ns = int((mono_ts + offset) * 1e9). Reuse existing monotonic event timestamps without re-instrumenting the scheduler.

The offset drifts slightly over long uptimes (the two clocks aren't perfectly coupled), but for span placement on a waterfall it is well within tolerance. Capture it once, not per-span.

8. Parenting — where your EngineCore span lands

OTel parent/child linkage comes from the trace context (trace-id + span-id), never from timestamps. So:

  • Feed context=extract_trace_context(request.trace_headers) → your span becomes a child of the client's span (the incoming HTTP traceparent).
  • The frontend's llm_request span also parents off those same client headers. So your EngineCore span and llm_request end up as siblings under the client root — they are not nested inside llm_request.
  • To nest under llm_request instead, you'd have to propagate that span's context to EngineCore (it isn't today — only the original client headers make the round trip). Sibling-under-client is the path of least resistance and is usually what you want.
  • Pass context=None (or omit) and _get_smart_context() takes over — you get the process-level trace or a fresh root, decoupled from the request.

9. Pitfalls & checklist

Do

  • Gate on observability_config.otlp_traces_endpoint before building attributes.
  • Use SpanAttributes.* constants, not raw strings.
  • Capture the monotonic→epoch offset once.
  • Prefer one span per request (explicit start/end) over one per step().
  • Pass trace_headers-derived context to join the client trace.

Avoid

  • Feeding time.monotonic() into start_time (§7).
  • Decorating the hot busy loop with @instrument (span flood).
  • Assuming is_tracing_available() means "exporter configured" — it doesn't.
  • Comparing EngineCore monotonic timestamps against frontend wall-clock for absolute placement.
  • Expecting the Rust frontend to emit OTel spans — it does not.
Minimal change, summarized. The tracer exists in EngineCore already. To emit a request-scoped span: (1) grab request.trace_headers → extract_trace_context; (2) convert your monotonic timestamps to epoch-ns via a once-captured offset; (3) call instrument_manual(..., context=ctx, kind=SpanKind.INTERNAL); (4) gate the whole thing on otlp_traces_endpoint being set.

10. Appendix — SGLang contrast: every process traces

SGLang makes the opposite design choice, and it flows from a different process architecture. vLLM centralizes scheduling in one EngineCore process and treats the GPU workers as dumb executors — so it only ever emits a span from the frontend. SGLang runs the scheduler SPMD: every GPU process is its own autonomous scheduler+worker, so tracing has to emit and stitch spans from all of them.

The extra vLLM process is the centralized scheduler (EngineCore, no GPU). SGLang has no such process — the 4 GPU processes self-schedule in lockstep.

Who coordinates the 4 GPUs in SGLang?

No single process. Coordination is collective, through the torch.distributed groups, not a boss:

  1. Ingest + broadcast. In recv_requests() (scheduler.py:1504), only attn_tp_rank == 0 reads requests off the ZMQ socket, then broadcast_pyobj(..., tp_cpu_group) sends them to ranks 1–3 over a gloo CPU group.
  2. Decide identically. All 4 ranks now hold the same request set and run the same scheduling logic → identical batches, no divergence.
  3. Execute together. The forward runs as a tensor-parallel collective over NCCL.

vLLM's EngineCore is instead the single decision-maker: it runs the scheduler and ships scheduler_output to the 4 workers via MultiprocExecutor (workers execute, never schedule). That dedicated decision-maker is the "+1" process — and the reason vLLM can emit its span from one place.

The trace_level dial

SGLang gates each span ("slice") and event on a per-request level snapshot of a process-global (global_trace_level, default 3 but --enable-trace defaults off). level 0 disables tracing for that request; __check_fast_return(level) skips any span whose level > trace_level.

The trade-off. SGLang gets a true per-process distributed trace with a live verbosity knob — at the cost of real machinery: pickling the trace context onto every cross-process request, a custom ID generator to dodge trace-id collisions across the TP processes, and per-thread monotonic→realtime calibration. vLLM sidesteps all of it by only ever emitting from the single frontend, accepting one coarse backdated span as the price.
