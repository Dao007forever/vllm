CUDA Graphs in vLLM

A from-scratch guide to what a CUDA graph is, why LLM inference needs them, what “breakable” / piecewise means, and how vLLM actually captures and replays them at runtime. Written for someone who has never touched the term before.

The one-paragraph version

Launching a GPU kernel from Python/C++ costs CPU time. A decode step in an LLM launches hundreds of tiny kernels, and each one is so fast that the GPU finishes before the CPU can queue the next — the GPU sits idle waiting for the CPU. A CUDA graph records that whole sequence of launches once and lets you fire it again with a single call, erasing the per-launch CPU overhead. The catch: a recorded graph is frozen — fixed shapes, fixed memory addresses, no data-dependent control flow. vLLM's job is to make the LLM forward pass fit inside that frozen box. Attention doesn't fit, so vLLM either splits the graph around attention (piecewise) or captures the whole thing (full) — that split is what “breakable” refers to.

Part 1 — What is a CUDA graph?

The problem: launch overhead

When you call a GPU operation — a matmul, an add, a normalization — the CPU doesn't do the math. It launches a kernel: it packages up the arguments and hands the work to the GPU's queue, then moves on. That hand-off is cheap but not free; it costs a few microseconds of CPU time per launch.

A single transformer decode step (generating one token) runs something like: for every layer — a QKV projection, a RoPE, an attention, an output projection, two norms, an MLP gate/up, an activation, an MLP down. Multiply by 32, 80, or more layers and you get hundreds to thousands of kernel launches, each tiny. With batch size 1 the GPU finishes each kernel in less time than the CPU needs to launch the next one. The result is a GPU that is mostly idle, throttled by how fast a single Python/C++ thread can issue launches. This is the classic CPU-bound decode problem.

The solution: capture once, replay many times

CUDA (the NVIDIA GPU API) lets you put the driver into a capture mode. While capturing, every kernel you launch is recorded into a graph object instead of being executed immediately — the nodes (kernels) and their dependencies are saved. When you're done you have a cudaGraph you can replay: a single API call that re-issues the entire recorded sequence to the GPU with almost no CPU involvement.

In PyTorch this is wrapped as torch.cuda.CUDAGraph() and the with torch.cuda.graph(g): ... context manager. You can see vLLM use exactly these primitives in vllm/compilation/cuda_graph.py:

cudagraph = torch.cuda.CUDAGraph()
with torch.cuda.graph(cudagraph, pool=..., stream=...):
    output = self.runnable(*args, **kwargs)  # recorded, not run
# later, on a hot path:
cudagraph.replay()  # re-fires every recorded kernel, one call
The price of capture: everything is frozen A replayed graph re-runs the exact kernels on the exact memory addresses it saw during capture. That imposes three hard rules that drive vLLM's entire design:
  • Static shapes. A graph captured for batch size 8 only works for batch size 8. Different sizes need different graphs.
  • Static memory addresses. Inputs must live in the same buffers every replay. You copy new data into the captured input tensors, then replay — you can't hand it a fresh tensor at a new address.
  • No data-dependent control flow. No Python if on a tensor value, no CPU↔GPU syncs, no dynamic loops. The graph is a fixed list of kernels.

vLLM enforces the static-address rule directly: in debug mode the wrapper records each input tensor's data_ptr() at capture time and asserts they're identical on replay (cuda_graph.py, __call__).

Part 2 — Why LLM inference makes this hard

The frozen rules above clash with two facts about serving LLMs:

  1. Batch sizes vary every step. vLLM batches whatever requests are ready. One step might have 3 sequences, the next 47. You can't capture a graph for every possible size, so vLLM captures a fixed set of sizes (e.g. 1, 2, 4, 8, 16, …) and pads a real batch up to the nearest captured size. These are the cudagraph_capture_sizes.
  2. Attention is inherently dynamic. Attention reads the KV cache, whose length differs per sequence and grows every step. The kernel does data-dependent work (variable sequence lengths, block tables, paged lookups). Many attention kernels can't be captured into a graph at all, and the ones that can need careful handling. This is the crux of the whole “breakable” story.

Separately, vLLM uses torch.compile to fuse and optimize the model into fast kernels. CUDA graphs and torch.compile are different things that work together: torch.compile decides which kernels exist and how fast each is; CUDA graphs decide how cheaply the sequence of kernels is launched. vLLM's compilation pipeline (vllm/config/compilation.py, CompilationMode) has levels from NONE (pure eager) up to VLLM_COMPILE — the custom Inductor backend that does piecewise compilation, shape specialization, and the graph splitting we're about to discuss.

Part 3 — What “breakable” means: splitting the graph

Since attention won't go inside a CUDA graph cleanly, vLLM breaks the model's computation graph at the attention boundaries. Think of the per-layer forward as a chain where attention ops are the “break points”:

Piecewise: break the graph at attention
CAPTURED norm + QKV
proj + RoPE
→
EAGER (break) attention
(reads KV cache)
→
CAPTURED out proj + norm
+ MLP
→
EAGER (break) next layer's
attention
The dense, static parts (norms, projections, MLP) are captured as CUDA graph pieces. Attention runs eagerly in between. The pieces between breaks are what get the launch-overhead savings.

The list of ops to break at is the config field splitting_ops. Its docstring (vllm/config/compilation.py) states it plainly:

splitting_ops — verbatim from source “A list of ops to exclude from cudagraphs, used in piecewise compilation. … If None, defaults to attention ops for piecewise cudagraphs. If empty list [], no ops are excluded (suitable for full cudagraphs).”

The default split ops are the attention / sequence-mixing custom ops: vllm::unified_attention_with_output vllm::unified_mla_attention_with_output vllm::mamba_mixer2 vllm::short_conv vllm::linear_attention …

Two ways to perform the break

vLLM has three mechanisms for breaking the graph, and this is where naming gets confusing. Here's the map:

MechanismWhen the break happensControlled by
Dynamo FX splitting (default) At torch.compile trace time. The FX graph is literally cut into separate subgraphs before Inductor compiles each one. See split_graph() in compilation/backends.py and should_split() in compilation/partition_rules.py. splitting_ops, use_inductor_graph_partition=False
Inductor graph partition At Inductor codegen time, after all fusion passes. Lets custom passes see the whole graph, then partitions at the end. use_inductor_graph_partition=True
Breakable CUDA graph (newer, experimental) At runtime, during capture. No pre-splitting at all — a single capture drives the whole forward and breaks the stream capture when it hits an attention op. See compilation/breakable_cudagraph.py. VLLM_USE_BREAKABLE_CUDAGRAPH env var

The newer “breakable cudagraph” in detail

The file vllm/compilation/breakable_cudagraph.py is worth understanding because its name is “breakable.” Its module docstring explains the idea (verbatim, inspired by an sglang design):

breakable_cudagraph.py — module docstring “Instead of pre-splitting the model into many pieces at attention boundaries, a single capture context drives the whole forward and intercepts attention / kv-cache custom ops at the dispatcher to end the current stream capture, run the op eagerly, and resume capture. The captured artifact is a list of zero-arg callables — the bound CUDAGraph.replay for graph segments, or the user fn for eager segments — replayed in order at inference time.”

The clever bit is the @eager_break_during_capture decorator. You wrap an attention kernel with it. Outside capture, it just runs the kernel. Inside capture, it ends the current graph segment, runs the kernel eagerly, records it for replay, and starts a fresh segment — an automatic break point, no FX surgery required.

A real constraint this exposes The decorated op must write into a caller-provided output buffer. If it returned a fresh tensor, that tensor would live at a new address every replay — and the next captured segment, which reads that address, would break. This is the static-address rule from Part 1 biting again. The docstring also warns that decorator order matters: PD-disaggregation's KV-transfer side effects must run in the eager segment, not get baked into the graph (which would hang on replay).

Part 4 — Piecewise vs Full: the CUDAGraphMode enum

Now the central abstraction. vLLM expresses the whole strategy as an enum, CUDAGraphMode, defined in vllm/config/compilation.py. Two of its members (PIECEWISE, FULL) are the “break the graph” vs “capture everything” choice; the compound members combine them for the prefill/decode split.

NONE

No CUDA graphs. Every kernel launched eagerly. Maximum flexibility, maximum CPU overhead. Used during the profile/warmup run, or when graphs are disabled.

PIECEWISE

Break at attention (Part 3). The dense pieces are captured; attention runs eagerly between them. Handles any batch (prefill, decode, mixed) because attention — the dynamic part — is never inside a graph.

FULL

Capture the entire forward pass, attention included, as one graph. Lowest overhead, but requires an attention backend whose kernels are CUDA-graph-safe, and works cleanly mainly for uniform (pure-decode) batches.

Compound modes

FULL_DECODE_ONLY = (FULL, NONE): full graph for decode, eager for everything else.
FULL_AND_PIECEWISE = (FULL, PIECEWISE): full graph for uniform decode, piecewise for prefill/mixed. This is the strong default — best of both.

The compound modes are literally tuples in the enum. A (decode_mode, mixed_mode) pair tells vLLM to use one strategy for uniform-decode batches and another for mixed/prefill batches. Verbatim definition:

class CUDAGraphMode(enum.Enum):
    NONE = 0
    PIECEWISE = 1
    FULL = 2
    FULL_DECODE_ONLY = (FULL, NONE)
    FULL_AND_PIECEWISE = (FULL, PIECEWISE)

Helper methods make the intent readable: decode_mode() and mixed_mode() pull the two halves apart; separate_routine() is true exactly when the value is a tuple; has_full_cudagraphs() and requires_piecewise_compilation() ask which machinery needs to be set up. Only NONE, PIECEWISE, and FULL are valid runtime modes — the compound modes are resolved down to one of those three per batch.

Why both, instead of just FULL everywhere? A full graph that includes attention is fastest, but only works when every kernel in the path is capture-safe and the batch shape is one you captured. Prefill batches have wildly varying token counts and attention patterns that don't fit. So vLLM uses FULL for the well-behaved uniform-decode case and falls back to PIECEWISE (or NONE) for the messy cases — that's exactly what FULL_AND_PIECEWISE encodes.

Part 5 — How it all runs: capture and dispatch

Capture happens once, at startup (warmup)

After the model loads and memory is profiled, vLLM does a warmup phase that captures graphs for every configured size. The entry point is capture_model() on the GPU model runner (vllm/v1/worker/gpu/model_runner.py). For each capture size it runs a dummy forward with fake inputs of that shape; the CUDAGraphWrapper sees a batch shape it hasn't recorded yet, captures it, and caches the resulting graph keyed by a BatchDescriptor. This is why vLLM takes “Graph capturing finished in N secs, took X GiB” on startup — graphs cost GPU memory.

STARTUP
Load model + torch.compile
→
WARMUP
capture_model(): dummy run per capture size
→
CACHED
One graph per (mode, BatchDescriptor)
→
SERVING
Each step: dispatch → replay

Dispatch happens every step, at runtime

On each forward pass, the CudagraphDispatcher (vllm/v1/cudagraph_dispatcher.py) looks at the current batch — how many tokens, is it uniform decode, is LoRA active — and decides a (runtime_mode, batch_descriptor) pair. That pair is stashed in the forward context, a thread-local object every layer can read.

The CUDAGraphWrapper around each captured piece then does something elegant: it reads the forward context and “blindly trusts” the mode. Its logic (verbatim behavior from cuda_graph.py):

  1. If there's no forward context (e.g. a vision encoder), just run eagerly.
  2. If runtime_mode == NONE, or the mode doesn't match this wrapper's own mode, run the underlying function directly. (This is how nested PIECEWISE and FULL wrappers avoid stepping on each other — only the matching one acts.)
  3. Otherwise, look up the BatchDescriptor in the cache. If there's no graph yet, capture one now. If there is, replay it and return the cached output.

The dispatcher rounds the real batch up to a captured size and selects a mode; the wrapper just replays the matching cached graph.

Part 6 — The config knobs you'll actually touch

All of these live on CompilationConfig in vllm/config/compilation.py.

FieldWhat it does
cudagraph_modeThe strategy: NONE / PIECEWISE / FULL / FULL_DECODE_ONLY / FULL_AND_PIECEWISE. The main lever.
cudagraph_capture_sizesWhich batch sizes to capture graphs for. None = inferred from the vLLM config (a default pattern like [1,2,4] + steps of 8, then 16).
max_cudagraph_capture_sizeLargest size to capture. Batches bigger than this fall back to eager / piecewise.
splitting_opsOps to break at for piecewise. None = attention ops; [] = no breaks (i.e. full).
use_inductor_graph_partitionBreak at Inductor codegen time instead of Dynamo FX time (see Part 3).
cudagraph_num_of_warmupsHow many warmup runs before the recorded capture.
cudagraph_copy_inputsIf True, copy inputs into the static buffers each call instead of assuming the same buffers. (PIECEWISE only.)
Mental model to keep
  • CUDA graph = record a kernel sequence once, replay with one cheap call. Kills launch overhead.
  • Frozen rules = fixed shape, fixed addresses, no dynamic control flow. Everything else follows from these.
  • Breakable / piecewise = cut the graph at attention so the dynamic part runs eagerly and the static parts still get captured.
  • FULL = no breaks; capture attention too. Faster, but needs capture-safe kernels and uniform shapes.
  • Mode + descriptor + dispatcher = the runtime machinery that picks and replays the right cached graph per step.
