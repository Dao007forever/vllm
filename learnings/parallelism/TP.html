Tensor Parallelism (TP)

The original Megatron-LM strategy: shard a single layer's weight matrix across GPUs and recombine activations with an AllReduce. The most fine-grained of vLLM's parallelism options — and the one with the highest per-step communication cost.

One-line model

TP shards weights along a chosen axis. Activations stay sharded for as long as possible; an AllReduce closes each Q·K·V → Output and Gate·Up → Down boundary. Per-step communication is ~4× AllReduce per transformer layer, every layer, every forward pass.

Why shard a layer at all

A 70B-parameter model in FP16 needs ~140 GB just for weights, and activations / KV cache add tens of GB more. No single GPU has that memory (an H100 has 80 GB). The first question is: can we keep the model on one device by sharding the work, not the data? The answer:

Data parallel only

Each GPU has a full copy of the model. Memory grows per replica. Doesn't help if the model itself doesn't fit on one GPU. Communication is once-per-step at most (for sync), but the memory wall blocks you first.

vs
Tensor parallel

Each GPU holds 1/N of every weight matrix. The model fits for any N. Communication is paid every layer, but the bandwidth budget on NVLink/NVSwitch is roughly 600–900 GB/s, which is usually enough for TP=4 to TP=8 in one node.

Column-parallel and Row-parallel

Megatron-style TP has exactly two primitive layers:

ColumnParallelLinear

Compute Y = X · A. Split A along its output dimension. Each rank holds A_i of shape [in, out/N] and produces Y_i = X · A_i of shape [B, out/N].

Input X is replicated across all ranks (no comm needed), output Y_i is sharded.

vllm/model_executor/layers/linear.py:410

RowParallelLinear

Compute Y = X · A. Split A along its input dimension. Each rank holds A_i of shape [in/N, out], gets X_i of shape [B, in/N], and produces a partial output Y_i = X_i · A_i.

Each rank's output is a partial sum. An AllReduce sums them to the full Y.

vllm/model_executor/layers/linear.py:1392

The trick: pair them, defer the AllReduce.

A ColumnParallel followed by a RowParallel does not need any communication between them. The ColumnParallel emits a sharded activation; the RowParallel consumes that sharded activation directly (its input is already partitioned along the row dimension). The AllReduce only happens at the end:

[B, H]
→
Y = X · A
ColumnParallel
Y sharded [B, H/N]
→
Z = Y · B
RowParallel
Z partial sum
AllReduce(Z)
→ full Z, replicated

This is the entire pattern. Every block in a transformer (MLP, attention) is one Column → Row pair plus an AllReduce.

Which weights are column- vs row-parallel

Not every weight matrix is sharded the same way. The transformer has exactly two Column→Row pairs per layer: QKV → O for attention, and gate/up → down for the MLP. In each pair, the column-parallel layer shards its output dim along a logical axis (heads, or intermediate features); the row-parallel layer shards its input dim along the same axis. That's how the sharded activation flows from one to the next without communication.

Weight Logical shape Sharded along Type vLLM class
W_qkv (fused Q/K/V) [H, (n_q + 2·n_kv)·head_dim] output (heads) Column-parallel QKVParallelLinear
W_O (o_proj) [n_q·head_dim, H] input (heads) Row-parallel RowParallelLinear
W_gate_up (fused gate/up) [H, 2·I] output (intermediate) Column-parallel MergedColumnParallelLinear
W_down (down_proj) [I, H] input (intermediate) Row-parallel RowParallelLinear

So W_qkv is sharded differently from W_O: same physical dimension (heads), opposite sides of the matmul. Same story for W_gate_up vs W_down along the intermediate dim. The choice of which side to shard is what determines whether the layer needs an AllReduce on the way out:

  • Column-parallel: input is replicated, output is sharded → no AllReduce needed (the next layer expects the sharded shape anyway).
  • Row-parallel: input is sharded, output is a partial sum → AllReduce required to recombine before the next replicated-input layer (LayerNorm, residual, the next block).

Worked example: the attention block

Same Column → Row pattern, but the sharded dimension is num heads:

StepOpWhat's shardedComm
1 QKVParallelLinear — ColumnParallel num_heads → num_heads/N per rank none
2 rotary position embedding applied to local heads only none
3 flash-attention each rank computes its heads' attention output none
4 o_proj — RowParallel concatenates heads → partial hidden none
5 AllReduce(o_proj_out) full hidden replicated AllReduce
KV cache implication. Each rank only holds the KV cache for its assigned heads. Memory for KV scales as num_heads/N × head_dim, not num_heads × head_dim. For GQA models (where num_kv_heads < num_heads), TP must divide num_kv_heads; otherwise some ranks would have no KV cache and need replication. This is where DCP enters the picture.

Why heads, not hidden, get sharded

Why not just shard hidden_dim directly, as in the MLP?

Because softmax is non-linear. The attention output for head h needs the full Q, K, V for that head — you can't compute softmax over a sliced K then sum partial outputs and get the right answer. Sharding by head keeps each head's computation whole and local. Sharding within a head would require splitting the softmax, which is exactly what context-parallel does, with extra communication.

How the KV cache inherits QKV's head sharding

A natural question after the above: TP shards every weight matrix in the model — QKV, O, gate, up, down. So why is the KV cache sharded specifically along num_kv_heads, and where does the cross-rank synchronization (AllReduce? ReduceScatter?) for populating it happen?

Answer: there is no separate KV-cache sharding mechanism. The cache layout falls out of where the column-parallel QKV projection drops its outputs. Trace one token through one layer:

  1. Input x is replicated across ranks (shape [B, S, H]).
  2. QKVParallelLinear is column-parallel. Each rank holds W_qkv_local of shape [H, (n_q + 2·n_kv) · head_dim / N].
  3. Each rank locally computes q_local, k_local, v_local = x · W_qkv_local. k_local shape is [B, S, n_kv/N, head_dim] — already sharded by KV head. No communication.
  4. k_local (and v_local) is written directly into this rank's slot in the paged KV cache. Again, no communication, no reshuffling. The cache layout is fixed by the output shape of the column-parallel projection that produced it.

Worked example: Llama-3-70B with TP=4 (GQA, 8 KV heads)

Concrete numbers help with one easy-to-miss point: q, k, v are sharded, just as much as W_qkv_local is. The single fused QKV matmul doesn't keep them whole; it just slices its output into three tensors after the GEMM, and each slice is already this rank's head shard.

Llama-3-70B: n_q=64 query heads, n_kv=8 KV heads (each KV head is shared by 8 query heads under GQA), head_dim=128, hidden H=8192. The logical full W_qkv concatenates the Q, K, and V projections along the output axis: 

full W_qkv shape:  [H,  (n_q + 2·n_kv) · head_dim]
                =  [8192, (64 + 8 + 8) · 128]
                =  [8192, 10240]
                            ↑
            structured as: [Q: 64·128 | K: 8·128 | V: 8·128]

Column-parallel sharding by N=4 doesn't naively chop the 10240 output axis into four equal chunks — that would put all KV heads on rank 3 and leave ranks 0–2 with no K or V. Instead, QKVParallelLinear shards Q, K, and V independently by head, then concatenates each rank's slice:

Rank Q heads owned K heads owned V heads owned W_qkv_local shape
0[0..15][0, 1][0, 1][8192, 2560]
1[16..31][2, 3][2, 3][8192, 2560]
2[32..47][4, 5][4, 5][8192, 2560]
3[48..63][6, 7][6, 7][8192, 2560]

Each rank's local output dim: 16·128 + 2·128 + 2·128 = 2048 + 256 + 256 = 2560. The matmul produces a single per-rank tensor: 

out  =  x · W_qkv_local  =  [B, S, 2560]   on each rank

That single output is then sliced along the last dim to recover Q, K, V separately. No collective, no reshape across ranks — just a local tensor view: 

q_local = out[..., :2048]      → reshape [B, S, 16, 128]   # 16 Q heads
k_local = out[..., 2048:2304]  → reshape [B, S,  2, 128]   # 2 KV heads
v_local = out[..., 2304:2560]  → reshape [B, S,  2, 128]   # 2 KV heads

So q_local, k_local, v_local each carry exactly this rank's 1/4 share of heads, with the full head_dim=128 per head. On rank 0, k_local holds the entirety of KV heads 0 and 1 and nothing else; rank 1 holds KV heads 2 and 3; and so on. Across all four ranks, every KV head is owned by exactly one rank — no replication, no missing heads.

That k_local of shape [B, S, 2, 128] is what gets appended to rank 0's KV cache slot. Rank 0's cache tensor per layer has shape [num_blocks, block_size, 2, 128] — exactly the n_kv/N × head_dim tail of the head-sharded layout, with no additional sharding logic required.

The same head shard is what FlashAttention then reads back from the cache on subsequent steps — the producer and consumer of the cache are both local to the rank, so the head-sharded layout never leaves the rank during normal forward execution.

Where do AllReduce / ReduceScatter actually fire?

Per transformer layer, only at the row-parallel boundaries. Not at the KV cache.

Boundary Standard TP + Sequence parallel
QKV projection (column-parallel write) none AllGather of x first (input is seq-sharded)
KV cache populate (store k_local, v_local) none none
KV cache read (FlashAttention) none none
Attention o_proj (row-parallel) AllReduce ReduceScatter
MLP down_proj (row-parallel) AllReduce ReduceScatter

Two collectives per layer in standard TP; both are AllReduce on partial sums emerging from row-parallel weights. Sequence parallelism (see below) swaps each AllReduce for a ReduceScatter + matching AllGather later — same total bandwidth, different intermediate shape. Neither AllReduce nor ReduceScatter touches the KV cache.

Why this matters for the KV transfer plane

Connectors that move KV between prefill and decode workers (see kv_cache/transfer/PARALLELISM) describe the transfer entirely in terms of "which heads live on which rank." That phrasing is precise — the cache layout is already final on each rank before any collective fires, so the connector never has to coordinate with a collective in flight or worry about partial-sum state crossing the wire. It just copies head shards.

Communication cost

Per transformer layer, the per-step bandwidth on each rank is:

  • 1 AllReduce after attention o_proj: payload B×S×H
  • 1 AllReduce after MLP down_proj: payload B×S×H

With L layers that's 2L AllReduces per forward pass. For Llama 3 70B (L=80, H=8192), one decode step (S=1) at B=64 sends: 

2 × 80 × 64 × 1 × 8192 × 2B(fp16)
  ≈ 167 MB on the wire per rank, per forward pass

On 600 GB/s NVLink that's about 0.28 ms. Multiply by the decode step rate to see why TP>8 across nodes (over slower interconnects) falls off a cliff.

Sequence-parallel optimisation

LayerNorm, residual adds, and dropout are elementwise — they don't mix the hidden dim. So instead of having all ranks redundantly compute them on the full [B, S, H], you can:

  1. Replace each AllReduce with a ReduceScatter (along the sequence dim).
  2. Run LayerNorm / residual on the per-rank sequence shard.
  3. AllGather before the next attention layer that needs the full sequence.

Total comm volume stays the same (ReduceScatter + AllGather = AllReduce), but the activation memory drops by 1/N, and you save the duplicated norm/residual work. See the AllReduce identity for the algebra.

When to choose TP vs PP vs DP

Choose TP when…

  • All ranks are on the same node (NVLink available).
  • You need the lowest end-to-end latency.
  • The model doesn't fit on one GPU but fits on one node.

Choose PP when…

  • Crossing nodes; interconnect can't sustain the AllReduce volume.
  • You can hide the bubble with enough in-flight microbatches.
  • Memory is your constraint, not latency.

Choose DP when…

  • The model already fits. You want more throughput.
  • For MoE: EP + DP together for wide deployments.
  • Per-replica latency is fine; you just want to serve more requests.
The composition rule: world_size = DP × PP × PCP × TP. They are orthogonal — vLLM builds one rank-grid at startup (parallel_state.py:1575) and projects the same physical GPUs into each parallelism's "group" by reshaping.
