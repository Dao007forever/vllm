Pipeline Parallelism (PP) & Chunked Pipeline Parallel (CPP)

Pipeline parallel splits the model by layer. Stage k holds layers [k·L/N, (k+1)·L/N) and ships activations to stage k+1. The unit of communication is the entire hidden state; the trade-off is bubbles. Chunked Pipeline Parallel (CPP) is the optimisation that overlaps chunked-prefill chunks across pipeline stages so multiple chunks are in flight at once, using async (non-blocking) Send to remove the serial hand-off bottleneck.

One-line model

PP is the right tool when you need to cross a node boundary. Per-step communication is just two Send/Recv per stage transition, instead of AllReduce per layer. The cost is the bubble: the time when pipeline stages are idle while data flows through. CPP is the cure for that bubble when chunked prefill is on: make the inter-stage Send asynchronous so stage k can start chunk N+1 while stage k+1 is still chewing on chunk N.

Pipeline stages

pipeline_parallel_size = N means the model's layers are sliced into N contiguous blocks. Each PP stage holds exactly one block on its TP group. For Llama-3 70B with 80 layers and PP=4:

PP stage 0 — embedding + layers 0–19
Reads: token IDs · Writes: hidden state
PP stage 1 — layers 20–39
Reads: hidden state from stage 0 · Writes: hidden state
PP stage 2 — layers 40–59
Reads: hidden state from stage 1 · Writes: hidden state
PP stage 3 — layers 60–79 + lm_head
Reads: hidden state from stage 2 · Writes: token logits

Each stage can still be a TP group internally. The full grid: 

world_size = DP × PP × PCP × TP

# PP=4, TP=2 on 8 GPUs:
PP stage 0: GPUs {0, 1}  (TP=2)
PP stage 1: GPUs {2, 3}
PP stage 2: GPUs {4, 5}
PP stage 3: GPUs {6, 7}

The PP group spans across-node ranks (one rank per stage), while each TP group stays within a single node. This is intentional: TP wants NVLink, PP can survive on a slower network because it only sends per-stage hidden states, not per-layer AllReduces.

The stage hand-off

At the boundary between stage k and stage k+1, every TP rank in stage k sends to its matching TP rank in stage k+1. The protocol is:

  1. Stage k finishes its layer block. Each TP rank holds its shard of the hidden state.
  2. The PP group's get_pp_group().send_tensor_dict({"hidden_states": h_local, "residual": r_local}, dst=k+1) is called.
  3. Stage k+1 calls recv_tensor_dict(src=k) to receive the hidden state. Each TP rank receives the matching shard.
  4. Stage k+1 begins its layer block immediately.

The bytes-per-send are B × S × H/TP per rank (the hidden state, sharded). For a 70B model at H=8192, TP=2, that's 8 KB per token per layer-block boundary — trivial compared to AllReduce volume.

Chunked Pipeline Parallel (CPP)

Disambiguation: "CPP" is used in the vLLM community for two related-but-not-identical things. (1) The narrow reading (rebel-jinhwan/vllm-rbln #5): chunked-prefill chunks driven through PP via async Send. (2) The broad reading (vLLM RFC #28912, also called Context Pipeline Parallelism or Sequence Pipeline Parallelism, after arXiv 2409.17264): a redesign that adds FLOPs-balanced chunking, an async scheduler, and async isend/ irecv so the four distinct PP bubbles can all be flattened. The narrow reading is one piece of the broad one.

CPP is the combination of two ideas that vLLM already has separately:

  1. Chunked prefill: split a long prefill into chunks of ~2k tokens so the per-step latency stays bounded and decode requests can run between chunks. (Scheduler knob: enable_chunked_prefill; not a parallelism strategy on its own.)
  2. Pipeline parallel: split the model by layer-blocks; each block is a stage.

Stick the two together naively and you get the sequential-chunks behaviour. Each chunk walks all PP stages before the next chunk starts:

The wasted gap: while chunk N is processed by stage 1, stage 0 has finished its job for chunk N and is sitting idle — even though chunk N+1 is queued and ready. The serial walk reintroduces the bubble that microbatching was supposed to eliminate.

The four PP bubbles (per RFC #28912)

The vLLM RFC is precise about why naively combining PP with chunked prefill isn't enough — it identifies four distinct sources of idle time, each needing a different fix:

BubbleCauseWhat removes it
Bubble 0 Blocking inter-stage Send: stage k waits for stage k+1 to receive before continuing. Async P2P (isend/irecv), the narrow-CPP fix.
Bubble 1 Synchronous scheduler overhead: each PP rank has its own FCFS scheduler that runs serially with the worker. Async scheduler — scheduling overlaps with the previous step's compute.
Bubble 2 Adjacent InputBatches differ in compute cost because schedulers split on token count, not FLOPs (attention is O(S2) on context, so a 2k chunk on a 100k context is far heavier than a 2k chunk on a 2k context). When stage k's chunk is heavier than stage k+1's previous chunk, stage k+1 finishes early, has nothing to receive yet, and idles. Downstream-stage idle. FLOPs-balanced chunking: budget batches by computational load (a function of chunk size and context length), not by token count.
Bubble 3 Mirror of Bubble 2: when stage k's chunk is lighter than stage k+1's, stage k finishes early but can't push its result downstream (K+1 is still busy). Even with async send, the bounded send-queue fills, the scheduler can't usefully start another chunk yet, and stage k stalls. Upstream-stage idle. Same fix (FLOPs balancing) plus SLO-based prioritisation to keep the steady-state chunk-queue full.

The narrow-CPP fix (async Send) only addresses Bubble 0. The broad CPP proposal aims for all four.

The CPP fix: async Send

Move the blocking send_tensor_dict() off the main thread. A background daemon thread holds the in-flight Send while the worker keeps going. The next chunk's forward pass starts immediately. Right before the worker would overwrite the intermediate-tensors buffer (which the background send still has a handle to), call wait_pending_send() to ensure the previous send has drained.

The pipeline as it should be: with PP=4 and 4 chunks in flight, all four stages stay busy in steady state. Stage 0 processes chunk 4 while stage 3 is finishing chunk 1. Reported scaling efficiency in the SGLang reference is ~83% at PP=4 — the gap to 100% being the start-up and drain bubbles at the chunk-queue ends.

The bubble-ratio formula (SGLang)

SGLang quantifies the leftover bubble exactly: 

Bubble Ratio = (P − 1) / (P − 1 + M)

where P = pipeline depth (number of PP stages) and M = number of chunks. Both terms come straight from the gantt above — here's the derivation:

Where the P − 1 comes from

The first useful output from the last stage appears only after P − 1 fill steps (stage 0 → 1 → ... → P−1). Symmetrically, when the input ends, it takes another P − 1 drain steps for the last chunk to walk through.

Per-device, fill and drain split that total: stage 0 is busy from the start and idles for the last P − 1 steps; stage P−1 idles for the first P − 1 steps and is busy at the end. Either way, each device sits idle for exactly P − 1 chunk-times.

Where the M comes from

Useful work per device = M chunks. The wall-clock for the whole pipeline is fill + work + drain — but the fill and drain overlap on different devices, so the elapsed time is: 

total time = M (useful) + (P − 1) (start fill or end drain)

Divide idle by total: (P − 1) / (P − 1 + M). When M ≫ P the ratio → 0; when M = 1 (one chunk, naive PP) the ratio is (P−1)/P — which for PP=4 is 75% bubble, matching the "GPU utilisation 25%" diagram earlier.

The formula tells you the two knobs: more chunks (smaller chunk size) drives M up; shallower pipeline drives P down. The first is preferred — per-chunk overhead has to be paid in CUDA-graph launch cost, but chunks themselves are nearly free to multiply.

Why fixed chunks underperform the formula in practice. Attention is O(S2) on context, so a fixed-size 6k chunk takes longer when it lands at position 100k than at position 6k. Later chunks run slower than earlier ones; downstream stages keep waiting on the slow tail. SGLang reports that with fixed chunks "the pipeline bubble ratio will be greater than the theoretical expectation."

Dynamic Chunking (DCK)

DCK shrinks each later chunk so that the wall-clock for processing it matches the wall-clock of the first chunk. Concretely, given prefix length L: 

Runtime(L + ΔL) − Runtime(L) = Runtime(initial chunk size)

Cumulative attention runtime is modelled as quadratic in L; SGLang solves the equation for ΔL and floors it to a multiple of max(--page-size, 64). A smoothing factor (SGLANG_DYNAMIC_CHUNKING_SMOOTH_FACTOR, default 0.75, sweet spot 0.6–0.85) damps the shrinkage so chunks don't become trivially small at the tail. This is the concrete fix for Bubble 2/3 from the four-bubble taxonomy above — "FLOPs-balanced chunking" in action.

Async P2P + multi-stream

The async-Send trick from the rbln implementation generalises in SGLang to two pieces:

  • async_send returns a P2PWork handle; the wait is deferred to a later _pp_commit_comm_work call so the CPU schedules the next batch while bytes are still in flight.
  • Dedicated CUDA streams (forward_stream and copy_stream) overlap GPU compute and D2H copies with the main sync stream — same idea as cudagraphs splitting the attention-and-comm phases.

Why this needs to be opt-in

Three things have to be true for CPP to be safe and useful:

  • Multiple in-flight batches per worker. vLLM V1 already supports this through step_with_batch_queue() with max_concurrent_batches = pp_size.
  • The Send buffer must not be overwritten in flight. The worker reuses fixed staging tensors (prefill_intermediate_tensors, decode_intermediate_tensors) to avoid allocation churn. The async path calls wait_pending_send() at the top of the next execute_model() step so the previous send is guaranteed drained before the buffer is reused. Skipping this would corrupt the data in flight.
  • Only one outstanding Send per worker. Bounded queue depth, no head-of-line stalls beyond a single chunk's send time.

The broad design (RFC #28912): FLOPs budgeting + async scheduler

Async Send alone removes Bubble 0. To remove the others, RFC #28912 proposes two phases of change beyond the narrow rbln fix:

Phase 1 — request-grained PP with FLOPs-balanced chunking

Stop using "token count" as the scheduler budget. Use a computational-load estimate that's a function of chunk size and context length (attention is O(S2) so a 2k chunk on a 100k context is much heavier than a 2k chunk on a 2k context).

  • Adds FLOPs estimation to the scheduler.
  • Switches the budget metric from max_num_batched_tokens to a load metric.
  • SLO-based scheduling to keep TTFT bounded.
  • Tracks PR vllm#20808.

Result: adjacent chunks have similar wall-clock cost, so Bubbles 2 and 3 (the per-stage hand-off mismatches) shrink.

Phase 2 — pack multiple chunks of one request per InputBatch

Give the priority scheduler more flexibility by allowing several chunks of the same request to coexist in one batch. Useful when one long prefill dominates a stage.

The author flags this "may impact certain dictionary variables keyed by req_id" — many vLLM internals assume one entry per request per batch, so this phase is invasive.

The async ingredient pair

The RFC is emphatic that async scheduler and async P2P are both required. Doing one without the other re-introduces the bubble:

  • Async P2P without async scheduler: stages don't block on Send any more, but each rank's scheduler still runs serially with its worker → Bubble 1 dominates.
  • Async scheduler without async P2P: the scheduler is ready early, but the Send is still blocking → Bubble 0 dominates.
  • vLLM's existing recv_tensor_dict() path in gpu_worker.py is synchronous; switching to isend/irecv hits PyTorch issue #122597, which is part of why the RFC predicts a long landing time.

Reference numbers

Three data points across implementations:

  • rbln implementation (narrow CPP): Llama-3.2-1B, PP=2, max_num_seqs=2 — async-Send-on vs off is essentially identical (789 vs 797 tok/s, 16 reqs at 512in/64out). The author flags the bubble is "too small" at this scale to demonstrate the optimisation.
  • RFC #28912 preliminaries: request-level PP had higher TTFT than vanilla PP or chunk-level PP — explicitly cited as evidence that "an asynchronous scheduler and asynchronous communication are critical prerequisites". Without them, finer-grained PP is bitten by per-iteration bubbles.
  • SGLang (broad CPP, fully overlapped) at scale — on H20 nodes (8×96 GB) with DeepSeek-V3.1 and Qwen3-235B-A22B-FP8:

The layer-partition tip

Put the heavier partition on higher PP ranks (SGLANG_PP_LAYER_PARTITION=15,15,15,16 typically beats 16,15,15,15 for DeepSeek-V3.1). The trailing rank's extra work overlaps with upstream P2P traffic, eating into the drain bubble. Asymmetry in the right direction is free latency.

Takeaway: CPP is a real win, but only after all four bubbles are addressed. The narrow rbln implementation is one piece; the RFC is the plan for the rest. Both were in flight at the time of writing; the RFC was closed but cited as design context for ongoing work (PR #20359, #23545, #20808).

How CPP composes with TP and decode

The SGLang blog frames PP and TP as complements rather than substitutes:

  • TP: AllReduce per layer, comm volume scales as 4·B·S·H·L — grows with every layer, bandwidth-bound across nodes.
  • PP: Send/Recv only at stage boundaries, comm volume B·S·H·(P−1) — independent of layer count.
  • Recommended pattern: TP (or CP) within a node over NVLink, PP across nodes to dodge cross-node bandwidth ceilings.
  • Note also: very wide TP runs into a quantization wall — FP8 block granularity doesn't divide MoE FFN weights cleanly past some TP size, so MoE deployments often hit a hard TP ceiling well before bandwidth becomes the binding constraint.

CPP targets the prefill phase where M ≫ P makes the bubble disappear. For decode, where each step has just one new token per request, there's nothing to chunk — so SGLang uses PD-disaggregated deployments: prefill nodes run heavy PP (e.g. PP8 TP8) and decode nodes run a totally different layout (PP1 TP8, PP1 DP16 EP16, etc.), letting each phase hit its own TTFT/TPOT targets. KV cache transfers between them happen chunk-by-chunk via backends like Mooncake instead of waiting for full prefill to finish.

Don't confuse CPP with chunked prefill. Chunked prefill alone is a scheduler trick to bound per-step latency; it does not require PP. CPP is what happens when chunked prefill plus PP cooperate via async Send, async scheduler, and FLOPs-balanced chunking so that the per-chunk pipeline behaves like a microbatch pipeline. Without those, you have PP + chunked prefill but not CPP — chunks still walk the pipeline serially.

PP vs TP — when to pick which

TPPP
Comm per layer 2 AllReduce (each 2B×S×H volume) 0 between stage interior; 1 Send/Recv at stage boundary
Comm hardware Needs NVLink or NVSwitch (intra-node) OK on Ethernet / IB (inter-node)
Memory split Weights split 1/N within each layer Different layers on different GPUs
Latency for batch=1 Best (every GPU is computing every layer) Worst (one GPU at a time)
Throughput at scale Limited by AllReduce bandwidth Limited by bubble
KV cache Sharded per head Each stage holds KV only for its layers

The pragmatic rule for multi-node serving: TP within a node, PP across nodes. So with 16 GPUs split across 2 nodes of 8 each, the natural config is TP=8, PP=2.

Limits and gotchas

Layer count must divide evenly

80 layers and PP=3 won't balance cleanly. vLLM rounds, but some stages end up larger. For Llama-3 70B prefer PP ∈ {2, 4, 5, 8, 10}.

The embedding/lm_head asymmetry

Stage 0 has the embedding; the last stage has lm_head + sampler. Their compute cost differs from a mid-stage of pure transformer blocks, so even with equal layer counts the wall-clock per stage isn't perfectly equal. Usually a sub-bubble effect, but visible at PP>8.

CPP correctness depends on send-buffer discipline

The async-Send optimisation only works because the worker waits for the previous send to drain before reusing the intermediate-tensors buffer. Skip that barrier and you get silent data corruption — a partially overwritten activation lands at the next stage with no error raised. If you're porting CPP to a new backend, this is the single most important invariant to enforce.

PP and prefix caching

Each PP stage runs its own scheduler-side bookkeeping but shares the same prefix-cache hashes (computed centrally). Stages downstream just skip layers for cache-hit tokens; no extra coordination needed.
