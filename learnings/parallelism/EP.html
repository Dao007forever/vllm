Expert Parallelism (EP)

The right way to scale MoE models. Each rank owns a subset of the experts, not a slice of every expert. Tokens travel to their chosen expert's rank via All2All. Without EP, a 256-expert MoE wastes most of every rank's memory on experts that won't be activated on this rank's tokens.

One-line model

TP shards a layer; EP shards an expert population. An MoE block with E experts and EP=N places E/N experts on each rank. Tokens are routed by an All2All to their owners, compute happens locally, and a reverse All2All combines results.

Why EP instead of TP for MoE

A typical MoE layer has many experts (e.g. DeepSeek-V3 has 256), but each token activates only top-k (often k=8). So most of the MLP weights are not used for any given token. There are two ways to spread the work:

TP over MoE (the naive way)

Shard every expert's MLP weights across N ranks (column/row split). Every rank stores 1/N of every expert. Each token's chosen expert still runs on all N ranks (one shard each).

Memory: E × W / N per rank.
Comm: one AllReduce per token, per layer.
Wasted FLOPs: every rank does work for every token, even ones whose expert is "mostly idle" on this rank.

vs
EP (the right way)

Give each rank E/N whole experts. Each token's expert is on exactly one rank; that rank does the whole MLP for the token.

Memory: E × W / N per rank (same).
Comm: 2 All2All per layer instead of AllReduce.
FLOPs: each token only computed once, on the right rank.

The wins are FLOP efficiency (you don't activate experts that no one routed to) and bandwidth shape: All2All bytes scale with the number of active tokens, not the number of experts. AllReduce over a TP'd MoE always pays for every expert.

The set-MoE-on-fire test. With 256 experts and top-8, only ~3% of expert weights matter for any given token. TP can't tell the difference; EP can. That's the whole reason EP exists.

Layout: where experts live

The placement strategy is configurable (expert_placement_strategy in parallel.py:164):

linear (default)

Contiguous experts to contiguous ranks. With E=8 and EP=4:

rank 0
e0, e1
rank 1
e2, e3
rank 2
e4, e5
rank 3
e6, e7

Good locality. Bad for models with expert-grouping patterns — if adjacent experts get correlated activation, one rank becomes a hot spot.

round_robin

Stride-N placement. With E=8, EP=4:

rank 0
e0, e4
rank 1
e1, e5
rank 2
e2, e6
rank 3
e3, e7

Spreads grouped experts across ranks. Used for grouped-expert models like Mixtral when EPLB is not enabled.

The MoE forward, end to end

Walk through a single MoE block on EP=4. Assume rank 0 starts with batch X_0 shape [B/N, H] (one DP shard's worth of tokens), top-k = 2.

1. Router — local on each rank
Linear projection X_0 · W_r → gates [B/N, E]. Top-k selection picks 2 expert IDs per token. Routing decisions are local; no comm.
2. All2All Dispatch
Sort tokens by (chosen_expert → owning_rank). Build per-destination buckets. all_to_all_single sends each token to its expert's rank.
Volume per rank: (k · B / N) · H — independent of E.
3. Expert MLP — local on each rank
Each rank receives a bucketed batch [T_local, H] where T_local is the sum of all incoming tokens for the E/N experts it owns. Runs the standard MoE GEMM (often a fused group-gemm kernel).
4. All2All Combine
Reverse dispatch: send each token's result back to its origin rank. Multiply by the router's gating weight and sum the top-k contributions (now collocated again).
5. Residual + LayerNorm — local on each rank
The output is added to the residual stream just like a dense MLP.

The total comm is two All2All per MoE layer, with payload proportional to k × B, not E or any weight dimension. The expert matmul itself is purely local.

All2All backends

vLLM supports several All2All implementations, picked via --all2all-backend:

BackendHow it worksWhen to use
allgather_reducescatter (default) Emulate All2All with AllGather followed by ReduceScatter. Pure NCCL, no special build required. The safe everywhere-fallback. Higher latency at scale but works on any hardware NCCL works on.
deepep_high_throughput DeepEP NVLink-optimized kernels. Bulk-tokens path, peer-to-peer with IPC handles, no NCCL collective. Single-node NVLink, large batches (prefill-heavy workloads). Highest throughput.
deepep_low_latency DeepEP RDMA-optimized for small messages. Fewer staging buffers, lower per-token overhead. Decode-heavy (small batches), multi-node with InfiniBand/RoCE.
mori AMD's MoE All2All kernels. MI300-class hardware.
nixl_ep NIXL / RIXL-based send-recv with staging buffers. Disaggregated deployments using NIXL.
flashinfer_nvlink_{one,two}_sided FlashInfer kernels for MN-NVL fabric (multi-node NVLink). GB200/MN-NVL pods.

Source: vllm/config/parallel.py:174-183 lists the enum; vllm/distributed/device_communicators/all2all.py has the manager classes (AgRsAll2AllManager, DeepEPHTAll2AllManager, etc.).

The imbalance problem and EPLB

Real routers don't pick experts uniformly. Some experts get 3× more tokens than others; some are essentially dead. The slow rank sets the pace, so MoE throughput is limited by the busiest expert's rank.

The imbalance tax: with 256 experts and naive placement, the 99th-percentile load on the busiest rank can be 2–3× the average. All other ranks idle, waiting for the slow one to finish its expert MLP.

EPLB: Expert-parallel load balancer

Set enable_eplb=True (requires enable_expert_parallel=True):

  1. Maintain online statistics of per-expert token counts.
  2. Periodically re-compute a placement that flattens the histogram.
  3. Migrate expert weights point-to-point using a separate EPLB communication group (so it doesn't contend with the forward-pass All2All). Backend is selectable: torch_nccl, torch_gloo, nixl, pynccl.
  4. Update the routing table so future tokens hit the new owners.

See vllm/distributed/eplb/. The dedicated group _EPLB is initialised alongside _EP in parallel_state.py.

EP + DP coupling

EP scope = TP-group × DP-group, when both are enabled. Reason: the All2All operates over the full set of tokens in flight, and that set spans DP. The MoE forward pass is one of the few places where DP ranks must coordinate (see DP — MoE coupling).

The product determines the EP world size. For DP=16, TP=4 on DeepSeek-V3: 

EP_world_size = DP × TP = 16 × 4 = 64
experts_per_rank = E / EP_world_size = 256 / 64 = 4

Each of the 64 ranks owns 4 of the 256 experts. The All2All operates over all 64 ranks; the AllReduce-style sync used by TP within a DP replica is not active inside the MoE layer.

Elastic EP

enable_elastic_ep=True uses stateless NCCL groups for DP/EP. The motivation: in a long-running serving deployment, you may need to add or remove DP replicas at runtime (scale up under load, scale down at night) without restarting the existing replicas.

  • Group membership is published via a coordinator (stateless_coordinator.py), backed by a TCPStore.
  • Workers re-form NCCL communicators on every membership change.
  • Requires enable_eplb=True because experts have to be migrated to/from the joining/leaving rank.

Config and code pointers

# Basic EP for a MoE model
vllm serve deepseek-ai/DeepSeek-V3 \
  --tensor-parallel-size 4 \
  --data-parallel-size 16 \
  --enable-expert-parallel

# With EPLB
vllm serve ... \
  --enable-expert-parallel \
  --enable-eplb \
  --all2all-backend deepep_high_throughput

# Save disk I/O on load by only reading local expert weights
vllm serve ... --enable-ep-weight-filter
