Context Parallelism — DCP and PCP

Shard the sequence dimension instead of the head dimension. vLLM has two flavours, and the names are about which dimension they shard, not which phase they run in: DCP shards the context (the cached KV of a sequence) and PCP shards the query tokens of a prefill. DCP actually runs in both decode and (chunked-)prefill — it just never parallelises the new query tokens. PCP is the follow-up that does exactly that, to cut TTFT on very long prompts. Both recombine partial attention with the same LSE (log-sum-exp) algebra; they differ in their collectives.

One-line model

TP fails when you can't divide the heads any further (e.g. only 8 KV heads, already at TP=8) or when the KV cache for a long sequence won't fit per-rank. CP solves both: shard tokens, recombine partial attention outputs using LSE algebra, paying one extra collective per attention layer.

The unlock for FlashInfer: CP can't merge partial attentions without each shard's LSE — the online-softmax correction needs the per-shard softmax denominators. FlashInfer only recently started returning LSE from its prefill/decode wrappers, which is why BatchDCPPrefillWrapper (vllm/v1/attention/backends/flashinfer.py:214) now exists and FlashInfer can be a DCP backend. Without LSE there is no merge_attn_states, so there was no way to stitch the rank-local outputs back into a correct global attention.

Why CP exists: the GQA wall and long context

TP shards along heads. That works until one of two things runs out:

The GQA wall

Modern models use Grouped-Query Attention (GQA) where num_kv_heads is small (e.g. Llama-3 70B has 8 KV heads, Mistral 8x22B has 8, DeepSeek-V3 has 1 via MLA). Once you reach TP = num_kv_heads, you can't shard the KV cache any further — the next rank would have to replicate KV, defeating the memory split.

DCP says: keep TP at the head limit, then shard tokens within each KV head. So a TP=16, DCP=2 setup gives effectively 32-way compute split while still respecting 8 KV heads.

The long-context wall

For a 1M-token prefill, even one TP rank's KV cache is enormous (KV is S × H_kv per layer). Storing the full sequence per rank during prefill is impossible. PCP splits the tokens themselves across ranks — each rank handles a contiguous chunk of the sequence.

PCP also enables ring-attention-style algorithms where K and V slabs cycle around the ring while Q stays put.

Larger interleaves are friendlier to KV-block alignment — you can store an entire KV block (typically 16 or 64 tokens) on one rank instead of striping tokens within a block. At interleave = block_size the layout is block-level: contiguous blocks rotate through ranks.

The LSE merge: how partial attentions combine

Attention output is a softmax-weighted sum of value vectors. For one query against keys/values indexed by k, with scores s_k = q·k_k / √d:

O = Σk   exp(sk)
Σj exp(sj)   · vk

What O and L actually are

This is the crux. When a shard i computes attention over only its slice of keys K_i, it returns a pair (O_i, L_i):

  • O_i is a fully-normalized attention output — the softmax taken over shard i's keys only, dotted with shard i's values. It is not a raw, un-normalized numerator; shard i already divided by its own local denominator. So O_i = Σk∈K_i exp(s_k) v_k / Σk∈K_i exp(s_k). It's a complete, valid attention output — just over a subset of the keys. Shape [tokens, heads, head_dim].
  • L_i is the log of that shard's softmax denominator: L_i = log Σk∈K_i exp(s_k) — the "log-sum-exp." This is the one scalar per (token, head) that O_i threw away when it normalized, and it's exactly what's needed to undo and re-do the normalization at merge time. Shape [heads, tokens].

Without L_i you could never recombine, because you'd have no idea how much softmax mass each shard's O_i represents. A shard that saw the single highest-scoring key and a shard that saw a thousand near-zero keys both return a unit-normalized O_i; only L_i tells you the first one should dominate.

The final softmax is reconstructed, not recomputed

There is no second pass over QKT. The merge never revisits scores — it only reweights the pre-normalized O_i's by each shard's share of the total softmax mass:

Given shards i, j with (O_i, L_i), (O_j, L_j):

L = log(exp(L_i) + exp(L_j))   // merged LSE
w_i = exp(L_i − L),  w_j = exp(L_j − L)   // note w_i + w_j = 1
O = w_i · O_i + w_j · O_j

The punchline: those weights w_i = exp(L_i − L) are themselves a softmax over the per-shard LSEs. So the global softmax factorizes cleanly into two nested softmaxes:

final O = Σi softmax(L)i · O_i
        ↑ softmax over shards       ↑ softmax within shard

Generalises to N shards as a tree-reduction (each step merges two partials, the merged (O, L) feeds the next step). This is the same identity FlashAttention uses to tile a long sequence in SRAM; DCP/PCP just lift it to the network.

Worked example. One query, four keys, scores s = [2, 1, 3, 0]. Split into shard A = keys{0,1} and shard B = keys{2,3}.
  • Shard A: denom = e²+e¹ = 10.11, so L_A = 2.31 and O_A = 0.731 v₀ + 0.269 v₁.
  • Shard B: denom = e³+e⁰ = 21.09, so L_B = 3.05 and O_B = 0.953 v₂ + 0.047 v₃.
  • Merge: L = log(10.11 + 21.09) = 3.44, w_A = 10.11/31.20 = 0.324, w_B = 0.676.
  • O = 0.324·O_A + 0.676·O_B. Expanding the v₀ coefficient: 0.324×0.731 = 0.237 — which is exactly e²/31.20, the global softmax weight on key 0. Every coefficient matches the one-shot softmax.
The two normalized partials, reweighted by their LSE share, reproduce the full-sequence softmax bit-for-bit.
In the kernel. vllm/v1/attention/ops/triton_merge_attn_states.py:118-161 does exactly this, with a max-subtraction for stability: max_lse = max(p_lse, s_lse); p_se = exp(p_lse − max_lse), likewise s_se; p_scale = p_se/(p_se+s_se) (= w_i); out = p_out·p_scale + s_out·s_scale. The DCP prefill path feeds it prefix = context region and suffix = new-tokens region. Tokens with no prefix context skip the merge entirely — their O is copied straight from the suffix (kernel lines 89-102).

Why this dictates the comm pattern (ag_rs vs a2a)

The collective has to do two things at once, and that's what shapes the two backend choices. After the per-rank attention, recall that Q was AllGathered across the DCP group, so every rank holds a partial output for all heads — but each partial only covers that rank's KV shard. The collective must:

  1. LSE-merge across ranks — combine the N partials per head into the true output (the math above).
  2. Scatter heads back — leave each rank holding only its own H/N heads, because the downstream o_proj is row-parallel TP and each rank owns just its head slice.

"Merge-then-scatter" is the shape of a reduce-scatter, not an all-reduce — nobody needs the whole answer, just their head slice. The two backends realize it differently:

ag_rs (default) — two collectives

common.py:212 cp_lse_ag_out_rs:

  1. AllGather the LSE only ([B,H], tiny) so every rank knows all shards' L_i per head.
  2. Local Triton kernel rescales this rank's output by its weight w_i = exp(L_i − L) — producing its weighted contribution.
  3. ReduceScatter the weighted outputs (dim=1): the sum-across-ranks is the LSE merge, and the scatter hands each rank its H/N heads.

Only the small LSE is broadcast; the big O moves once, via the reduce-scatter.

a2a — one collective

dcp_alltoall.py:393 dcp_a2a_lse_reduce:

  1. Pack each rank's (O_i, L_i) by destination — rank r's slice goes to whichever rank owns those heads.
  2. One All2All: now each rank holds, for its H/N heads, the partials from all N ranks.
  3. Do the full LSE combine locally — no further comm.

One NCCL call instead of two, and O + LSE travel together. Lower latency; favoured for MLA, where the head/comm layout suits it.

Same answer either way — a2a just folds the gather, merge, and scatter into a single exchange. This is the dcp_comm_backend knob (parallel.py:333), and it's also why a2a is rejected at DCP=1: with one rank there are no partials to exchange.

Worked 4-rank example

Take DCP=4 and (for simplicity) 1 head owned per rank, so 4 heads total. KV is sharded across the 4 ranks. After the Q-AllGather + per-rank attention, every rank has computed a partial (O, L) for all 4 heads, but each partial covers only its own KV shard. Call that partial p[r][h] = "head h attended over KV shard r". The state is a 4×4 matrix — each rank holds one full row:

head 0head 1head 2head 3
rank 0 (KV shard 0)p[0][0] ★p[0][1]p[0][2]p[0][3]
rank 1 (KV shard 1)p[1][0]p[1][1] ★p[1][2]p[1][3]
rank 2 (KV shard 2)p[2][0]p[2][1]p[2][2] ★p[2][3]
rank 3 (KV shard 3)p[3][0]p[3][1]p[3][2]p[3][3] ★

The goal: rank h must end up owning the merged column h — the LSE-combine of p[0][h], p[1][h], p[2][h], p[3][h] — because that's the head it carries into o_proj. (★ marks the diagonal partial each rank already has for its own head.)

ag_rs: reduce the columns in place

  1. AllGather the LSE. Each rank sends its row's 4 LSE scalars; afterwards every rank holds the full 4×4 LSE table (the outputs O stay put — only the tiny L is gathered).
  2. Local rescale. Rank r now knows, for each head h, the global L_h = logΣ_s exp(L[s][h]), so it scales its own row entry by w[r][h] = exp(L[r][h] − L_h). Rank 0's row becomes w[0][0]·O[0][0], w[0][1]·O[0][1], …
  3. ReduceScatter over the head dim. This sums each column across the 4 ranks (the sum is the merge, since the weights were already applied) and hands column h only to rank h:
    rank h receives   Σr w[r][h]·O[r][h]  =  merged head h

2 collectives. Big O moves once (the RS); the AG carries only LSE.

a2a: transpose, then merge locally

  1. Pack by destination. Rank r splits its row into 4 pieces and labels piece h for rank h — carrying O and L together. (The diagonal piece p[r][r] is destined for itself.)
  2. One All2All. This transposes the matrix: rank h collects the whole of column h — p[0][h], p[1][h], p[2][h], p[3][h] — from the four ranks. "Every rank has all heads / own shard" becomes "every rank has own head / all shards."
  3. Local LSE merge. Rank h now has all 4 partials for its head and combines them with no further comm → merged head h.

1 collective. Each rank ships 3 of its 4 pieces out (keeps the diagonal), O+L together.

Both land rank h with the identical merged column h. With k heads per rank instead of 1, every p[r][h] above is a k-head block and the picture is unchanged. The matrix view also makes the names literal: ag_rs = AllGather (LSE) + ReduceScatter (outputs); a2a = all-to-all (the transpose).

DCP: Decode Context Parallel

DCP stripes the cached KV of a sequence across the DCP ranks. The name says "decode" because long-context decode is the workload that motivated it, but the same machinery runs during (chunked) prefill too — what DCP never shards is the query.

DCP in decode

During decode each step produces 1 new token per request:

  • Q is the single new token's projection.
  • K, V are the entire past sequence (cached) plus this new token.

With DCP=4, the KV cache for a long context is striped across 4 ranks. Each rank computes attention against its slice of K/V, returning (O_i, L_i). The merge step turns into one collective:

DCP decode forward, one attention layer
  1. QKV projection (Q replicated since this is the new token; KV partial because each rank only updated its own slice last step).
  2. Per-rank flash-attention against the rank's K/V shard, returning LSE so we get both O_i and L_i.
  3. DCP collective:
    • "ag_rs": AllGather the (tiny) LSE, rescale locally, then ReduceScatter the weighted outputs to split heads for the next layer (vllm/v1/attention/ops/common.py:212, cp_lse_ag_out_rs). See comm pattern below.
    • "a2a": one All2All ships each partial to its head-owner + local merge (vllm/v1/attention/ops/dcp_alltoall.py:393, dcp_a2a_lse_reduce). Lower latency, MLA-friendly.
  4. o_proj (RowParallel) + AllReduce as in normal TP.

DCP in prefill: the two-region split

GQA (unlike MLA) doesn't structurally separate prefill and decode in the forward pass, so the DCP prefill path splits each sequence into two regions and computes attention separately for each:

|---------- context_len ----------|--- query_len ---|
|------ KV sharded across DCP -----|-- new tokens ---|

These two regions need different handling, which is exactly why FlashInfer's BatchDCPPrefillWrapper holds two inner wrappers (flashinfer.py:224-227):

  • Context region — a paged-KV wrapper (_context, causal=False). The cached KV is distributed across DCP ranks, so this follows the MLA-DCP recipe: AllGather Q across the DCP group so every rank holds all query heads, run attention against the rank-local KV shard, then combine the partial outputs/LSE with AllGather-out + ReduceScatter (or A2A).
  • New-tokens region — a ragged-KV wrapper (_new_tokens, causal=True). The new query tokens attend to themselves; this KV is purely local, so no DCP collective is required here at all.

The two partial results are then stitched with LSE (merge_attn_states):

BatchDCPPrefillWrapper.run() — flashinfer.py:281
  1. prefill_query_across_dcp = get_dcp_group().all_gather(prefill_query, dim=1) — gather Q over the head dim so each rank has the full query.
  2. output_context_tmp, lse_context_tmp = _context.run(..., return_lse=True) against the rank's paged KV shard.
  3. output_context, lse_context = _dcp_combine(...) — the AG-out+RS (or A2A) collective; this is the only cross-rank step.
  4. output_query, lse_query = _new_tokens.run(prefill_query, key, value, return_lse=True) — local causal attention, no comm.
  5. merge_attn_states(out, output_context, lse_context, output_query, lse_query) — LSE-merge the two regions.
Why AllGather Q at all? Inside a DCP group, Q and KV are sharded along different axes. The QKV projection is column-parallel over the whole TP group (DCP ranks included), so each rank natively produces Q for only its heads (num_qo_heads local). KV, by contrast, can't be split by head — that's the GQA wall DCP exists to get past — so it's sharded by context token: each rank holds the same KV heads but only ~1/N of the positions. The LSE merge needs every head to attend the whole context, i.e. every rank must compute a partial for all heads against its local KV shard (a full row of the matrix) — which requires every head's Q to be present everywhere. So we AllGather Q (cheap: 1 token in decode, query_len in prefill) rather than shuffle the KV cache (huge and pinned per-rank). Move Q to the data, not the data to Q. In the code, the local num_qo_heads becomes num_qo_heads × dcp_world_size for the context wrapper after the gather (flashinfer.py:254, 290).
The caveat that catches everyone: DCP only shards the context. Notice step 1 AllGathers Q — every rank ends up computing the new-token attention for the whole query, so that compute is replicated, not parallelised, across the DCP group. So running DCP through a prefill batch buys you (a) less KV duplication and (b) sharded context attention — genuinely useful for chunked-prefill of a request that already has a long cached context. It does not reduce the compute or TTFT of prefilling a long new sequence. If cutting TTFT on a long new prompt is the goal, that's PCP, below.
The constraint: tensor_parallel_size % decode_context_parallel_size == 0 (vllm/config/parallel.py:485). DCP reuses TP ranks; it doesn't add new ones — it is a subdivision within the TP domain. With TP=8 and DCP=4, each TP group of 8 ranks is split into two DCP groups of 4, where the DCP dimension shards context tokens and the remaining TP dimension shards heads. The world size stays 8. DCP size must lie in [1, tp_size / num_kv_heads].

PCP: Prefill Context Parallel

PCP is the follow-up to DCP that closes the gap above: it also shards the query tokens of a prefill, so the new-token compute is split across ranks and TTFT on a long new prompt drops. Sharding the query is harder than sharding the context because:

  • Q is a long sequence (not 1 token), partitioned across the PCP ranks.
  • K and V also span the whole prompt.
  • The attention is causal — rank r's tokens attend to all earlier tokens (which live on lower-rank shards), so somebody has to hold complete sequence information at merge time.

PCP doesn't replace DCP — it wraps it. Inside each PCP group it first runs the ordinary DCP computation, then, before the final LSE update, performs an AllGather within the PCP group to recover the complete-sequence information the causal merge needs. Because one side of the prefill merge must hold complete info, there are three implementation strategies:

  • AllGatherQ — gather the query shards (the GQA-backend approach).
  • AllGatherKV — gather the K/V shards (the MLA-backend approach).
  • Ring-Attention — cycle K/V around the ring instead of gathering all at once.

The ring-attention variant looks like:

PCP=4 ring attention (prefill, 4 sequence shards)
  1. Each rank starts with its own Q, K, V shard. Compute local attention Q_local · K_local → (O, L).
  2. Rotate K, V to the next rank in the ring. Receive K, V from the previous rank.
  3. Compute Q_local · K_received, merge into (O, L) via LSE update.
  4. Repeat N−1 times until every rank has seen every other rank's K/V.
  5. Apply causal masking based on global token positions: rank r only attends to tokens with positions ≤ its own.
PCP adds GPUs; DCP doesn't. This is the structural difference. DCP is a subdivision within the TP domain (TP % DCP == 0, world size unchanged). PCP's communication domain stands alongside DP/PP/TP — it is a factor of the world size: world_size = PP × TP × PCP (vllm/config/parallel.py:774-778). PCP=2 on a TP=8 job means 16 GPUs, not 8.

PCP can be combined with DCP — they are orthogonal axes (PCP outer, DCP inner). The total context-parallel rank is pcp_rank * dcp_world_size + dcp_rank over a total CP world of pcp_world_size * dcp_world_size (vllm/config/parallel.py:343-344), and the KV interleaving (below) is defined over that combined total-CP rank.

Status in this tree: the PCP config knob and process groups exist (parallel.py:116, parallel_state.py:1297/1628), but no attention backend calls get_pcp_group() yet — the query-sharding attention path (AllGatherQ / AllGatherKV / Ring) is still landing. DCP, including the FlashInfer prefill wrapper above, is fully wired.

KV cache interleaving

The KV cache layout has to agree with the CP layout. The relevant knob is cp_kv_cache_interleave_size:

interleave_sizeLayoutWhen to use
1 Token-level: token i on rank i % CP Most even load. Default for short blocks.
= block_size Block-level: whole KV blocks rotate through ranks Friendly to kernel alignment; one block per rank avoids strided gather.
between Stripe of interleave_size tokens per rank, repeating Tuning knob; rarely needed.

Constraint: cp_kv_cache_interleave_size ≤ block_size and block_size % cp_kv_cache_interleave_size == 0, enforced in vllm/config/vllm.py:2087-2091. The interleave is defined over the combined total-CP rank, so it governs both DCP and PCP striping at once.

Limits and gotchas

DCP reuses TP ranks

TP % DCP == 0 is hard-enforced. Increasing DCP doesn't add GPUs; it changes how the existing TP group's GPUs slice work. Memory for KV is what's saved, not memory for weights.

Mamba / linear-attention has no CP

Sequence sharding only works because softmax-attention has the LSE merge identity. Mamba's recurrence has no equivalent — the SSM state at position i depends causally on every earlier position, with no log-sum-exp decomposition.

a2a backend requires DCP > 1

With DCP=1 there's no merge to do. The validator catches misconfigurations: parallel.py:491.

Tail imbalance at small sequences

For decode at a small batch where the prefix is shorter than CP, some ranks have nothing to attend to. The collective still has to run; you pay latency for no work. CP shines when the typical context is ≥ 16k tokens.
