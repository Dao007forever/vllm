Data Parallelism (DP)

Each rank holds a full copy of the model and serves an independent batch. Communication is minimal — until you mix it with MoE/EP, at which point DP and EP become inseparable.

One-line model

DP replicates the model. In a dense-only deployment, DP ranks barely talk to each other. In a wide-MoE deployment, DP groups define the batch that an All2All has to dispatch and combine over — so DP and EP end up coupled through the MoE layers.

The shape of DP in vLLM

data_parallel_size = N means the engine launches N identical workers, each running the same model on the same weights. Where they differ:

  • Each DP rank gets its own requests. There is no shared batch.
  • Each DP rank has its own KV cache and its own scheduler.
  • DP ranks coordinate only at well-defined points (start of step, MoE dispatch, end of step).

The full grid is ExternalDP × DP × PP × PCP × TP (see parallel_state.py:1566). Each DP rank is itself a TP/PP slab. If you set DP=2, TP=4 on 8 GPUs you get two independent 4-GPU TP groups, each running the model.

Dense models — the easy case

For a non-MoE model, DP ranks are functionally independent during the forward pass. The only inter-DP communication is:

  1. A tiny AllReduce at the start of each step deciding whether to execute (so no rank deadlocks waiting for a peer that has no work). See vllm/v1/worker/dp_utils.py.
  2. Background gossip on the coordinator (request count, KV blocks free) used by the front-end's load balancer to pick which rank gets the next request.
The bandwidth advantage: dense DP does no per-layer communication. If your model fits on one GPU and you have many GPUs, DP gives you near-linear throughput scaling. This is why "TP=1, DP=N" is the right setting for small models on big boxes — latency stays single-GPU low, throughput grows with N.

MoE models — where DP gets coupled

A Mixture-of-Experts layer has many experts (say 256 for DeepSeek-V3). With expert parallelism, each expert lives on exactly one rank. A token sent to expert e must travel to whichever rank holds expert e.

The natural batch for that All2All is "all tokens currently in flight across all DP ranks." So at every MoE layer:

1. Local forward
Run attention + router locally on each DP rank.
2. All2All dispatch
Send each token to the rank holding its expert.
(Bandwidth: ~B×H per rank.)
3. Expert MLP
Each rank runs only the experts it owns, on the tokens it received.
4. All2All combine
Send results back to the rank that originated the token.

This couples the DP ranks: they must enter the MoE All2All together. So DP scheduling ends up coordinated even though the rest of the forward pass is independent.

DP synchronisation: padding and barriers

The All2All needs every DP rank to participate, even if a rank has zero real tokens at this step. The trick:

  • Each DP rank publishes its current token count to peers via a small AllReduce of an integer.
  • The max becomes the All2All bucket size; ranks pad with dummy tokens up to that size, then mask them out on the receiving side.

The slowest (largest) rank sets the pace. Long-tail latency comes from imbalance, not from comm cost.

The DP load balancer (next section) tries to keep batches even so that padding stays small.

Internal, external, hybrid LB

vLLM exposes three DP load-balancer modes:

Internal LB (default)

One API server fronts all DP ranks. It tracks queue depths and KV-block availability per rank, and routes incoming requests to the least-loaded.

Good for single-node deployments. Simple to operate.

External LB

Each DP rank gets its own API server. An external load balancer (the one in your Kubernetes ingress) decides which rank receives each request.

Pattern: one-pod-per-rank in K8s, useful for wide-EP deployments where DP and EP both span many nodes. Set data_parallel_external_lb=True.

Hybrid LB

Each node runs its own API server that balances among the local DP ranks on that node, but an external LB balances among the nodes.

Sweet spot for multi-node MoE: locality-aware on the node, infrastructure-aware across nodes.

Dual-batch overlap (DBO)

Bandwidth idle time during MoE All2All can be hidden by running two micro-batches concurrently — while batch A is doing its All2All, batch B runs its attention/MLP locally, and vice versa. 

// Without DBO
[attn][MLP][AllToAll][expert][AllToAll][attn][MLP][AllToAll]...
                ^ stall ^                 ^ stall ^

// With DBO (two ubatches A and B, interleaved)
A:  [attn][MLP][a2a]   [exp]   [a2a]   [attn]...
B:           [attn][MLP][a2a]   [exp]   [a2a]...
  • enable_dbo: bool = False — opt-in flag.
  • ubatch_size: int = 0 — microbatch size; auto if 0.
  • Thresholds at dbo_prefill_token_threshold = 512 and dbo_decode_token_threshold = 32 guard against splitting tiny batches (where the overlap doesn't pay back the overhead).

DP vs TP vs PP — the comm-vs-memory triangle

Memory per rankPer-step commLatencyBest when
DP full model (worst) minimal (or 2×A2A/MoE layer) single-rank fast model fits; want throughput
TP 1/N (best for big models) 2 AllReduce / layer lowest if NVLink available single-node; model doesn't fit one GPU
PP 1/N by layer 1 SendRecv / pipeline stage boundary bubble in pipeline costs steps multi-node; slow interconnect

They compose. A common DeepSeek-V3 deployment is DP=16, TP=4, EP=64 — meaning 64 GPUs total, each TP=4 box is one DP replica, and the EP dimension is "TP × DP" (since the routers span all 64 ranks for expert lookup).
