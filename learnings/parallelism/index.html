vLLM Parallelism — Learning Notes

Five flavours of parallelism plus the collective communication primitives they're built from. Read these to develop a mental model for how vLLM splits a single forward pass across many GPUs — and why the right choice depends on whether you're memory-bound, bandwidth-bound, or latency-bound.

What's here

Six docs: one foundation (collective communication primitives) and five parallelism strategies (TP, DP, EP, CP, PP). Each strategy doc has worked examples, the matching collective ops, and code pointers into vllm/distributed/ and vllm/model_executor/.

New here? Start with COMMUNICATION.html. Every parallel layer in vLLM is built out of seven NCCL/Gloo primitives; once those click, the rest of the docs read mechanically.

All notes

Foundation

  • COMMUNICATION.html Broadcast, AllReduce, AllGather, ReduceScatter, All2All, Send/Recv: the seven NCCL/Gloo primitives every parallelism strategy is built from. Cheat-sheet, worked examples, useful algebraic identities, and where vLLM exposes them.

Strategies

  • TP.html — Tensor Parallelism Column/Row parallel linear layers, the Q·K·V to o_proj & gate_up to down pattern, sequence-parallel optimisation, and why you can't shard softmax-attention by hidden dim.
  • DP.html — Data Parallelism Replicate the model. Trivial for dense, coupled for MoE (where DP defines the All2All batch). Internal/external/hybrid load balancers, padding, dual-batch overlap.
  • EP.html — Expert Parallelism Each rank owns whole experts. All2All dispatch/combine. The seven All2All backends. EPLB load balancing. Elastic EP for scale-up/down.
  • CP.html — Context Parallelism (DCP + PCP) Shard the sequence dimension. LSE merge algebra. DCP for decode reuses TP ranks; PCP for prefill uses ring attention. KV cache interleaving.
  • PP.html — Pipeline Parallelism & Chunked Pipeline Parallel Layer-block stages connected by Send/Recv. The bubble problem and microbatching. CPP narrow (rbln #5: async Send removes Bubble 0) vs broad (vLLM RFC #28912: four-bubble taxonomy, FLOPs-balanced chunking, async scheduler + async isend/irecv).

Each strategy is a different choice of what to shard and which collective handles the join. The communication doc enumerates the joins; the strategy docs use them.

Mental model

  • COMMUNICATION — the alphabet. Every parallel layer in vLLM is a sentence in this alphabet.
  • TP — shard within a layer; pay AllReduce per Column→Row pair.
  • DP — shard requests; pay almost nothing (until MoE forces a per-step All2All).
  • EP — shard experts; pay All2All dispatch and combine per MoE layer.
  • CP — shard tokens; pay one LSE merge per attention layer (DCP reuses TP ranks, PCP uses ring SendRecv).
  • PP — shard by layer-block; pay one Send/Recv per stage boundary, mind the bubble.

Where to find the code

  • Group construction: vllm/distributed/parallel_state.py — initialize_model_parallel() builds all five groups (TP/DP/PP/PCP/DCP, plus EP/EPLB for MoE) from one ExternalDP × DP × PP × PCP × TP rank grid.
  • Collectives: vllm/distributed/communication_op.py — wrapper functions like tensor_model_parallel_all_reduce.
  • Device-specific kernels: vllm/distributed/device_communicators/ — All2All managers (DeepEP, Mori, NIXL, FlashInfer, etc.) and the custom-AR CUDA kernel.
  • Parallel layers: vllm/model_executor/layers/linear.py — ColumnParallelLinear, RowParallelLinear, QKVParallelLinear, etc.
  • MoE: vllm/model_executor/layers/fused_moe/ — FusedMoE and its dispatcher.
  • Config: vllm/config/parallel.py — every knob mentioned in this folder, with its default and validation.
