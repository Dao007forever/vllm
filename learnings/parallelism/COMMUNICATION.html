Collective Communication Primitives

The seven NCCL/Gloo collectives that every parallelism strategy in vLLM is built out of. Read this first if the terms AllReduce, AllGather, ReduceScatter, and All2All are not yet second nature — the rest of the parallelism docs assume they are.

The mental model in one line

Every collective is a function of two tensor shapes (in → out) and a reduction (sum, none, concat). Knowing the shape transform lets you read any parallel layer mechanically: "this layer's input is sharded by hidden-dim, output is sharded by hidden-dim, so we need AllReduce."

Notation used throughout these docs
  • B — batch size (number of sequences in the step)
  • S — sequence length (tokens per sequence)
  • H — hidden dimension (model's residual-stream width). For Llama-3 70B, H = 8192.
  • I — MLP intermediate dimension (usually 3–4× H). For Llama-3 70B, I = 28672.
  • D — a generic tensor size in bytes, used in the cheat-sheet below.
  • N — number of ranks in the collective's group (e.g. TP size).
  • num_heads, head_dim — attention heads. Note H = num_heads × head_dim; for Llama-3 70B num_heads = 64, head_dim = 128. "Sharding by head" means slicing H into groups of head_dim.
  • num_kv_heads — KV heads under GQA (often much smaller than num_heads; e.g. 8 for Llama-3 70B). Hard ceiling on TP for the KV cache.
  • E — expert count for MoE (e.g. 256 for DeepSeek-V3); top-k = experts activated per token.

Cheat sheet

OpPer-rank input → per-rank outputReductionVolume / rank (ring)
Broadcast [D] on root → [D] on all copy D send (root only)
AllReduce [D] on all → [D] identical on all sum (or max/min/avg) ~2D (ring: RS + AG)
AllGather [D/N] per rank → [D] identical on all concat D − D/N
ReduceScatter [D] per rank → [D/N] shard per rank sum D − D/N
All2All [N, D/N] per rank → [N, D/N] per rank (transposed routing) no reduction; reshuffle by index D − D/N
Send/Recv [D] on src → [D] on dst copy D

"Ring" is the algorithm NCCL uses on most homogeneous topologies. N is the number of ranks in the group. The volume column is the per-rank bytes on the wire, which is what determines latency on a fixed-bandwidth link.

Broadcast

Definition

One rank (the root) sends its tensor to every other rank in the group. Used to seed identical weights, token IDs, or seeds at startup.

Before

rank 0 (root)[a, b, c, d]
rank 1—
rank 2—
rank 3—
↓ broadcast(src=0) ↓

After

rank 0[a, b, c, d]
rank 1[a, b, c, d]
rank 2[a, b, c, d]
rank 3[a, b, c, d]
vLLM uses: input-token broadcast from TP rank 0 to the rest of a TP group, so every rank sees the same prompt before the forward pass. See parallel_state.py: use_message_queue_broadcaster=True on the TP group.

AllReduce

Definition

Every rank starts with a tensor of the same shape. Every rank ends with the element-wise sum (or max/min/avg) of all input tensors. Conceptually AllReduce = ReduceScatter + AllGather, which is exactly how the ring algorithm implements it.

Before

rank 0[1, 2, 3, 4]
rank 1[1, 1, 1, 1]
rank 2[0, 1, 0, 1]
rank 3[1, 0, 1, 0]
↓ all_reduce(op=SUM) ↓

After (every rank holds the sum)

rank 0[3, 4, 5, 6]
rank 1[3, 4, 5, 6]
rank 2[3, 4, 5, 6]
rank 3[3, 4, 5, 6]

When it shows up in vLLM

  • End of every TP layer pair: after RowParallelLinear (sharded on input dimension), each rank holds a partial sum — AllReduce sums them to the full output.
    vllm/model_executor/layers/linear.py:1559 calls tensor_model_parallel_all_reduce(output_parallel).
  • DP synchronization barrier: a tiny AllReduce of a counter decides whether all DP ranks are ready for the next step. See vllm/v1/worker/dp_utils.py.
  • Custom all-reduce: for small tensors over NVLink, vLLM ships a hand-written kernel that beats NCCL by avoiding the ring. Toggle with --disable-custom-all-reduce.
The deceptive cost. AllReduce moves ~2× the message size on the wire vs AllGather or ReduceScatter alone. When the layer's output is already going to be sharded again immediately (e.g. fused TP + sequence parallelism), the right move is to skip the full AllReduce and use ReduceScatter, saving half the bandwidth.

AllGather

Definition

Each rank owns a shard of size D/N. After AllGather, every rank holds the full concatenated tensor of size D. The reduction is concatenation, not summation.

Before

rank 0[a0]
rank 1[a1]
rank 2[a2]
rank 3[a3]
↓ all_gather ↓

After

rank 0[a0, a1, a2, a3]
rank 1[a0, a1, a2, a3]
rank 2[a0, a1, a2, a3]
rank 3[a0, a1, a2, a3]

Where vLLM uses it

  • Vocab-parallel logits: each rank computes a slice of the vocabulary's logits, AllGather concatenates so that argmax/softmax sees the whole vocab.
  • MoE allgather-reducescatter backend: tokens are gathered to every rank before the experts dispatch them (used when DeepEP is not available). See all2all_backend = "allgather_reducescatter".
  • Sequence-parallel TP variant: after a ReduceScatter has sharded activations by sequence dimension, AllGather is needed before the next attention layer that consumes the full sequence.

ReduceScatter

Definition

Every rank starts with the same shape D. After ReduceScatter, every rank holds the element-wise sum — but only of its D/N slice. The natural inverse of AllGather.

Before

rank 0[1, 2, 3, 4]
rank 1[1, 1, 1, 1]
rank 2[0, 1, 0, 1]
rank 3[1, 0, 1, 0]
↓ reduce_scatter(op=SUM) ↓

After (each rank has the sum of its column)

rank 0[3]
rank 1[4]
rank 2[5]
rank 3[6]
Why it pairs with AllGather. In the ring algorithm, AllReduce is a ReduceScatter followed by an AllGather. When the next op only needs a shard (e.g. another ColumnParallel layer that splits along the same axis), the AllGather half is redundant — you can leave the activations sharded across the layer boundary, saving bandwidth. This is the heart of "sequence parallelism" optimisations.

All2All

Definition

The most-misunderstood primitive. Every rank sends a different payload to every other rank. Conceptually it transposes a 2-D tensor of payloads across the rank dimension.

If you imagine a matrix where row i = "what rank i wants to send" and column j = "what rank j wants to receive", All2All is the rank-by-rank transpose.

Before (each rank has a row, indexed by destination)

rank 0 sends[a→0, a→1, a→2, a→3]
rank 1 sends[b→0, b→1, b→2, b→3]
rank 2 sends[c→0, c→1, c→2, c→3]
rank 3 sends[d→0, d→1, d→2, d→3]
↓ all_to_all ↓

After (each rank has a column, indexed by source)

rank 0 has[a→0, b→0, c→0, d→0]
rank 1 has[a→1, b→1, c→1, d→1]
rank 2 has[a→2, b→2, c→2, d→2]
rank 3 has[a→3, b→3, c→3, d→3]

Worked example — MoE token dispatch

Each rank holds 4 tokens. The router decides each token's expert. With EP=4 (expert parallel size 4), expert e lives on rank e. To send each token to its expert's rank we need an All2All.

MoE dispatch: routing 16 tokens to 4 expert-ranks
RankLocal tokens (id, chosen-expert)After All2All this rank holds tokens for
0(t0, e1), (t1, e3), (t2, e0), (t3, e2)expert 0: t2, t5, t6, t9, t12
1(t4, e2), (t5, e0), (t6, e0), (t7, e3)expert 1: t0, t8, t11, t13, t15
2(t8, e1), (t9, e0), (t10, e3), (t11, e1)expert 2: t3, t4, t14
3(t12, e0), (t13, e1), (t14, e2), (t15, e1)expert 3: t1, t7, t10
After dispatch, each rank holds only tokens whose expert lives here. The reverse op (combine) is a second All2All in the opposite direction.

Variants in vLLM (all2all_backend)

  • allgather_reducescatter — the safe default: replace All2All with AG + RS. Higher latency, no special kernel needed.
  • deepep_high_throughput / deepep_low_latency — DeepEP kernels (NVLink + RDMA optimised).
  • mori, nixl_ep, flashinfer_nvlink_{one,two}_sided — specialised kernels per hardware target.
Token counts are uneven. All2All in MoE is actually a ragged All2All — each rank-pair sends a different number of tokens. NCCL has a vectorised all_to_all_single for this; DeepEP goes further and dispatches tokens with per-expert buckets.

Send / Recv

Definition

Point-to-point primitive: one rank sends a tensor to one peer. Unlike the collectives above, this is not symmetric — the sender and receiver run different operations.

Where vLLM uses it

  • Pipeline parallel hidden-state hand-off: PP stage k's last layer sends activations to stage k+1's first layer.
  • KV transfer connectors: prefill workers send KV-cache blocks to decode workers in disaggregated deployments (NIXL, Mooncake). See vllm/distributed/kv_transfer/.
  • EPLB weight migration: when experts are rebalanced, affected weights are point-to-point shipped to the new rank.

Useful identities

Knowing these lets you compose collectives into custom layers and predict where bandwidth will go: 

AllReduce         ≡  ReduceScatter + AllGather
AllGather + RS    ≡  AllReduce (round-trip cancels out)
AllGather         ≡  N concurrent Broadcasts, each rank acting as root for its shard
ReduceScatter     ≡  N concurrent Reduces, rank i as the root for slice i
All2All           ≡  N concurrent SendRecvs, scheduled to avoid contention
Broadcast         ≡  Trivial AllGather where only root has the payload

The first identity is why sequence parallelism works. AllReduce leaves a full [B, S, H] tensor on every rank — but the next LayerNorm + residual after an attention block doesn't need the full tensor; those ops are elementwise along S. So we can stop the collective halfway: do only the ReduceScatter, run LayerNorm/residual on the rank-local sequence shard, and AllGather just before the next TP region needs the full sequence again.

Concrete example: one Llama-style transformer block, TP=4

StepWithout SPWith SP
Block input X [B, S, H] replicated on all 4 ranks X [B, S/4, H] — each rank holds 1/4 of the sequence
Pre-attn LayerNorm Each rank redundantly normalises the full [B, S, H] Each rank normalises its [B, S/4, H] shard. 1/4 the compute, 1/4 the activation memory.
Enter TP region (attention) Input already full AllGather along S → [B, S, H]
Attention compute Sharded by head; output is partial sum [B, S, H] Same
End of TP region AllReduce → full [B, S, H] replicated ReduceScatter along S → [B, S/4, H] shard of the sum
Residual + LayerNorm (pre-MLP) Each rank redundantly runs it on full [B, S, H] Each rank runs it on its [B, S/4, H] shard.
MLP: same as attention AllReduce at end AllGather at start, ReduceScatter at end

Compare the comm cost. The identity says AllReduce ≡ ReduceScatter + AllGather, so per block: 

Without SP:  2 × AllReduce              = 2 × (RS + AG)  bytes on the wire
With SP:     2 × (RS at end + AG at start) = 2 × (RS + AG)  bytes on the wire

Identical bandwidth. What SP actually buys you is:

  • Activation memory drops by 1/N between TP regions, because between the RS and the next AG, each rank holds only its sequence shard. For a 70B model at B·S = 8192, that's ~6 GB of activation memory saved per layer at TP=4.
  • Elementwise compute (LayerNorm, residual, dropout) drops by 1/N, since each rank only processes its shard instead of redundantly computing the full tensor.

Why was a bandwidth claim plausible? Because if you don't split RS and AG across the layer boundary — if you do the full AllReduce, then later explicitly shard the activations — you've paid for the AG and thrown it away. SP just doesn't pay for the throw-away half: it leaves activations sharded right up until the next TP layer genuinely needs them.

The third identity (AllGather as N concurrent Broadcasts) explains the per-rank bandwidth D − D/N: each rank receives (N−1) · D/N bytes (everyone else's shard) and sends (N−1) · D/N bytes (its own shard to the rest). On a fully-connected topology those N broadcasts run in parallel in a single hop; on a ring NCCL pipelines them across N−1 steps.

How vLLM exposes them

Every parallel layer in vLLM uses the helper wrappers in vllm/distributed/communication_op.py rather than calling torch.distributed directly. The wrappers:

  • Pick the right group (TP, DP, PP, DCP, PCP, EP, EPLB).
  • Dispatch to the right backend: NCCL, custom-all-reduce, Gloo, PyNccl, or a device-specific manager (CUDA / XPU / CPU).
  • Handle CUDA-graph capture safely (some backends require special staging buffers).

Common helpers

  • tensor_model_parallel_all_reduce(x) — TP group AllReduce
  • tensor_model_parallel_all_gather(x, dim) — TP group AllGather
  • get_dp_group().all_reduce(x)
  • get_ep_group().all_to_all(send_buf, recv_buf)
  • get_pp_group().send_tensor_dict(...) / recv_tensor_dict(...)

Group coordinators

  • TP: tensor-parallel rank inside a layer.
  • PP: pipeline stage, used for hand-off only.
  • DP: data-parallel replicas, sync at step boundaries.
  • EP: expert-parallel ranks for MoE All2All.
  • DCP / PCP: decode / prefill context-parallel groups.
  • EPLB: separate group for expert-weight migration.
Source: vllm/distributed/parallel_state.py builds every group at startup, then exposes get_*_group() accessors. The pinout is: ExternalDP × DP × PP × PCP × TP — see the comment around parallel_state.py:1566.
