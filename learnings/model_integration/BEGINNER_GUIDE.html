A Beginner's Guide to vLLM Model Integration

The deepseek2_kv_cache_wiring note is a chronological trace — it follows time, not structure. This guide is the missing top-down view of the class hierarchy: which class owns what, how they call each other, and exactly where the KV cache plugs in. Read this first; the trace then reads as a confirmation.

TL;DR — five classes, one spine

Every text model in vLLM is the same five-level nesting: *ForCausalLM (the thing the engine holds) → *Model (the decoder stack) → *DecoderLayer (one transformer block) → *Attention (Q/K/V projections + RoPE) → a leaf Attention / MLAAttention module (the only class that owns KV cache). Forward calls flow down this chain; hidden states flow back up. The leaf never receives a cache tensor as an argument — it registers itself at construction and the engine binds a tensor onto it later. Models differ only at the leaves: Gemma-3 swaps in sliding-window vs full layers (→ many KV groups, column-packed memory); DeepSeek-V3.2 swaps in latent MLA + a sparse indexer (→ one group, per-layer tensors). The spine above is identical.

1. The spine: five classes, top to bottom

Open any model file in vllm/model_executor/models/ — llama.py, gemma3.py, deepseek_v2.py — and you will find the same skeleton, just with different names. Using Llama as the reference (it is the simplest complete example):

The nesting (each line constructs the next)  vllm/model_executor/models/llama.py
1 LlamaForCausalLM  # the engine holds THIS object
    owns .model + .lm_head + .logits_processor
2   → LlamaModel  # the decoder stack
    owns .embed_tokens + .layers[] + .norm
3     → LlamaDecoderLayer  # one transformer block, built N times
    owns .self_attn + .mlp + 2 layernorms
4       → LlamaAttention  # projections + RoPE
    owns .qkv_proj + .o_proj + .rotary_emb + .attn
5         → Attention  layers/attention/attention.py  # ★ THE KV CACHE OWNER
The naming convention is load-bearing. The HF architecture string (e.g. "LlamaForCausalLM" in config.json) is looked up in the ModelRegistry to find the top class. From there the *Model / *DecoderLayer / *Attention names are just convention — but every model follows it, which is why once you know one model you can read any of them.

2. Who is responsible for what

ClassOwnsResponsible forDoes NOT do

the "runner-facing" object		 model, lm_head (ParallelLMHead), logits_processor The outer contract: forward() → hidden states, compute_logits() → vocab logits, load_weights() → maps checkpoint tensors onto submodules. Declares which capabilities it supports (SupportsLoRA, SupportsPP, …). No attention math, no KV cache. It's a thin shell that delegates everything to .model.

Built num_hidden_layers times by make_layers, each with a unique prefix. No projections itself — it owns the sub-modules that do them.
LlamaAttention
the attention wrapper		 qkv_proj (QKVParallelLinear), o_proj (RowParallelLinear), rotary_emb, attn Project hidden → Q,K,V; apply RoPE; call the leaf; project the result back. Computes head counts under tensor parallelism (num_heads // tp_size). Does not store K/V or run the attention kernel — it hands Q,K,V to self.attn.
Attention ★
the leaf — KV cache owner		 kv_cache tensor, attn_backend, layer_name, kv_cache_dtype Everything KV-cache. Registers itself in static_forward_context; declares its get_kv_cache_spec(); on forward, writes new K/V into its slots and runs the backend kernel to read the cached range. No projections (those are its parent's job). It receives Q,K,V already computed.
The one sentence that unlocks the hierarchy. Levels 1–4 are plumbing — they move tensors around and own weights. Level 5 (Attention) is the only place state persists across decode steps, and it is the only class the KV-cache machinery knows or cares about. Everything in the wiring trace happens at level 5.

3. How they call each other (the forward chain)

A single decode step calls model.forward() once. That call threads down the spine carrying (input_ids, positions), and hidden states thread back up. Here is the whole chain with the actual method bodies (condensed from llama.py):

# LEVEL 1 — LlamaForCausalLM.forward  (the engine calls this)
hidden = self.model(input_ids, positions, intermediate_tensors, inputs_embeds)
return hidden                       # logits computed separately, see §7

# LEVEL 2 — LlamaModel.forward
hidden = self.embed_tokens(input_ids)    # ids → vectors (first PP rank only)
residual = None
for layer in self.layers:           # the decoder stack
    hidden, residual = layer(positions, hidden, residual)
hidden, _ = self.norm(hidden, residual)  # final RMSNorm (last PP rank only)
return hidden

# LEVEL 3 — LlamaDecoderLayer.forward  (residual-stream choreography)
hidden, residual = self.input_layernorm(hidden, residual)
hidden = self.self_attn(positions, hidden)             # ↓ into level 4
hidden, residual = self.post_attention_layernorm(hidden, residual)
hidden = self.mlp(hidden)
return hidden, residual

# LEVEL 4 — LlamaAttention.forward  (projections + RoPE)
qkv, _ = self.qkv_proj(hidden)
q, k, v = qkv.split([self.q_size, self.kv_size, self.kv_size], dim=-1)
q, k = self.rotary_emb(positions, q, k)
attn_out = self.attn(q, k, v)                          # ↓ into level 5 — the cache
out, _ = self.o_proj(attn_out)
return out

# LEVEL 5 — Attention.forward  (the KV cache owner)
# writes k,v into self.kv_cache at this step's slots, then
# the backend kernel reads the whole cached range and attends.

The block loop (level 3) runs N times per forward. Inside each block the leaf is the only step that touches persistent memory — every other box is pure projection / normalization on the current step's tokens.

4. The supporting cast (you'll see these everywhere)

The five spine classes are built out of a small, reused toolbox. You don't need to memorize their internals, but recognizing them removes 80% of the noise in a model file:

Linear layers

QKVParallelLinear — fused Q/K/V projection, sharded by head across TP ranks.
MergedColumnParallelLinear — fused gate+up for the MLP.
RowParallelLinear — the output projections (o_proj, down_proj); does the all-reduce.
These three are why load_weights has a "stacked_params_mapping" — it splits checkpoint q_proj/k_proj/v_proj into one fused qkv_proj.

Embedding & head

VocabParallelEmbedding — token-id → vector, vocab sharded across TP.
ParallelLMHead — hidden → vocab logits; optionally tied to the embedding weights.
LogitsProcessor — applies the head and any logit scaling.

Norm, RoPE, MLP

RMSNorm / GemmaRMSNorm — the layernorms (fused add-residual variant).
get_rope(...) — builds the rotary embedding.
LlamaMLP — gate_up_proj → SiluAndMul → down_proj. (MoE models swap this for a FusedMoE block — the attention path is unchanged.)

How the N blocks get built: make_layers 
# vllm/model_executor/models/utils.py — called by every *Model.__init__
self.start_layer, self.end_layer, self.layers = make_layers(
    config.num_hidden_layers,
    lambda prefix: LlamaDecoderLayer(vllm_config, prefix=prefix),
    prefix=f"{prefix}.layers",        # "model.layers.0", "model.layers.1", ...
)
The growing prefix is what gives every leaf Attention its globally-unique layer_name — and that name is the key it registers under in §5. make_layers also handles pipeline parallelism: ranks that don't own a layer get a cheap PPMissingLayer placeholder.

5. Where the KV cache plugs in

This is the crux, and it's the part the class hierarchy hides until you look at level 5. The model code never allocates KV memory and never passes a cache tensor as an argument. Instead the leaf Attention module does three things across its lifetime, and the engine meets it halfway.

A. AT CONSTRUCTION (inside Attention.__init__)  layers/attention/attention.py
  self.layer_name = prefix  # its unique id from make_layers
  compilation_config.static_forward_context[prefix] = self  # ← REGISTERS itself
  self.kv_cache = torch.tensor([])  # placeholder; real tensor bound later

B. AT STARTUP (engine asks each registered layer)  v1/engine/core.py
  spec = attn.get_kv_cache_spec(vllm_config)  # "how big is one token of my cache?"
  # engine profiles memory → computes num_blocks → allocates one tensor per layer
  forward_context[layer_name].kv_cache = real_tensor  # bind_kv_cache — the wiring moment

C. AT RUN TIME (inside Attention.forward)
  write this step's K/V into self.kv_cache at the scheduler-assigned slots, then
  the backend kernel reads the cached range and returns the attention output.
Register, don't allocate. Because the leaf registers itself by name, the engine can discover every cache owner in the model generically — it never needs the model to hand its caches over. This is exactly why adding a new model rarely means touching the KV-cache code: as long as your attention is built on the Attention (or MLAAttention) base, registration and binding are inherited for free.

The single decision the leaf makes that ripples through the entire cache system is its spec. Attention.get_kv_cache_spec returns a different spec class depending on how the layer was configured:

# Attention.get_kv_cache_spec — attention.py:566 (condensed)
if self.sliding_window is not None:
    return SlidingWindowSpec(block_size, num_kv_heads, head_size, ...)   # windowed lifecycle
else:
    return FullAttentionSpec(block_size, num_kv_heads, head_size, ...)    # lives until request ends

The spec is what the cache machinery groups, sizes, and allocates by. That is the bridge from "model integration" into the KV-cache world — and it's exactly where Gemma-3 and DeepSeek-V3.2 diverge.

→ full byte-level walk in deepseek2_kv_cache_wiring.html; the cache-side design in kv_cache/BEGINNER_GUIDE.html.

6. At run time: how the kernels read & write the cache

§5 ended with "the leaf writes its K/V and the backend kernel reads the cached range" — true for a vanilla model, but the shape of that read/write is exactly what an exotic attention type changes. Here are the three families you'll meet, each a different answer to two questions: what does self.kv_cache physically hold, and how does the kernel address it. All three run inside the same Attention.forward slot of the spine (§3) — only the leaf class differs.

The forward splits into two ops. Look at Attention.forward (attention.py:437): it first calls a KV-cache update op (the WRITE), then an attention op (the READ). The backend implements both — for FlashAttention they are do_kv_cache_update → reshape_and_cache_flash (write) and flash_attn_varlen_func (read). Both are handed self.kv_cache and the per-step addressing metadata (slot mapping for the write, block table for the read).

6.1 First: where block_table and slot_mapping come from

The two ops above are handed two pieces of addressing metadata — the WRITE takes a slot_mapping (one physical slot per token of this step) and the READ takes a block_table (the per-request list of block ids). Neither is built by the model. Both are constructed by the GPUModelRunner in _prepare_inputs, before the forward runs, out of the block ids the scheduler already allocated (wiring note §6). It happens in two phases.

PHASE A — build the block_table (CPU, persists & grows across steps)  v1/worker/gpu_model_runner.py · block_table.py
1 scheduler's allocate_slots popped block ids → carried on SchedulerOutput.new_block_ids
2 runner _update_states writes one row per request into the CPU block-table buffer:
    • new request → block_table.add_row(block_ids, row)  # the whole row
    • running request → block_table.append_row(new_block_ids, row)  # ONLY the new ids (the shortfall)
3 commit_block_table(num_reqs) → copy_to_gpu()  # the kernel reads it on GPU

PHASE B — compute the slot_mapping (GPU, rebuilt fresh every step)
4 compute_slot_mapping(num_reqs, query_start_loc, positions) launches a Triton kernel that, per scheduled token:
    block_index = pos // block_size
    block_id    = block_table[req, block_index]  # the row from Phase A
    offset     = pos %  block_size
    slot = block_id · block_size + offset
5 tail of the buffer (CUDA-graph padding tokens) is filled with PAD_SLOT_ID = -1  # backends skip those writes
Worked example — request r0, block_size = 16, persistent block-table row [40, 41, 42]

One persistent row of block ids (Phase A). The slot_mapping below is recomputed each step from this row + only the positions scheduled that step. Slot for any position = block_id·16 + (pos%16).

Here is what "covers only the scheduled tokens" means concretely. The length of slot_mapping equals the number of tokens being written this step — not the sequence length. Watch it change between prefill and decode for the same request:

STEP 1 — PREFILL r0's 33-token prompt → all 33 positions (0–32) are scheduled
  positions = [0, 1, …, 32]  (33 tokens processed this step)
  pos 0  → 40·16 + 0  = slot 640
  pos 16 → 41·16 + 0  = slot 656
  pos 32 → 42·16 + 0  = slot 672
  slot_mapping = [640, 641, …, 671, 672]  # length 33 — one per prompt token

STEP 2 — DECODE r0 generates position 33 → only 1 token is scheduled
  positions = [33]  (1 token processed this step)
  block_index = 33 // 16 = 2 → block_id = block_table[r0, 2] = 42
  offset = 33 % 16 = 1 → slot = 42·16 + 1 = 673
  slot_mapping = [673]  # length 1 — even though the sequence is now 34 tokens long
That's the whole point. At decode step 2 the sequence has 34 tokens, but slot_mapping has one entry, because positions 0–32 were already written to slots 640–672 back in step 1. Those cached tokens are never re-written — they are only read, and the read finds them through the block_table, not the slot_mapping. So:
  • slot_mapping = the WRITE targets = this step's new tokens. Grows to the prompt length at prefill, shrinks to 1 (or a few, with spec decode) at each decode step.
  • block_table = the READ map = every token the request has ever cached. Only grows, by one block id every block_size tokens.
Chunked prefill: yes, only the chunk. "Scheduled tokens" is literally num_scheduled_tokens[req] for that step. A decode request schedules 1; an unchunked prefill schedules the whole prompt; a chunked prefill schedules only its chunk (e.g. a 4000-token prompt at a 2048 budget → step A schedules positions 0–2047, step B schedules 2048–3999). So in step A slot_mapping has 2048 entries, in step B it has 1952 — each covering only that chunk. The positions are absolute (pos = num_computed_tokens + i, gpu_model_runner.py:1390), so block_index = pos // block_size still lands each chunk's tokens in the correct already-allocated blocks. The block_table row is unchanged by chunking — it's the same growing list of blocks regardless of how the prefill is sliced.

A real step batches many requests, and slot_mapping is a single flat 1-D tensor over the whole batch's scheduled tokens, partitioned per request by query_start_loc (the Triton kernel launches one program per request and fills that request's slice — block_table.py:354). E.g. r0 decoding (1 token) batched with r1 prefilling a 4-token prompt into row [50, …]:

The batch dimension is "scheduled tokens," not "requests" — a decode request contributes 1 entry, a prefilling request contributes its whole prompt. query_start_loc is the only thing that says where one request's tokens end and the next begins. The block_table meanwhile has one row per request (r0's [40,41,42], r1's [50,…]) — different shape, different lifetime.
One translation table, three readers. The block_table is dereferenced by the write (to build slot_mapping), by the read (to gather cached K/V), and by any sparse-index conversion (DeepSeek's indexer, §6.4) — all in units of block_size. That shared unit is why the cache block size must equal the kernel's block granularity (wiring note §9).
Two different lifetimes, and one per group. The block_table grows — each step append_rows only the newly allocated ids onto a persistent row. The slot_mapping is rebuilt from scratch every step and covers only the tokens scheduled this step. And there is one block_table per KV-cache group (a MultiGroupBlockTable) but one slot_mapping per layer in the forward context — so Gemma-3's full and sliding groups each get their own block table, while DeepSeek's 122 same-group layers all share one (§8). With those two tensors in hand, the per-family ops below are just "scatter to slot_mapping" and "gather via block_table."

6.2 Gemma-3 paged K/V — scatter write, block-table gather read

The standard Attention leaf. self.kv_cache is a tensor shaped (2, num_blocks, block_size, num_kv_heads, head_size) — dim 0 is K vs V (the kernel does key_cache, value_cache = kv_cache.unbind(0)). Each token-slot stores a real K and V vector. (In a vanilla model that tensor is one private allocation per layer; in hybrid Gemma-3 it's a per-column allocation shared across layers — see §6.2.1 below, which is exactly your question.)

# WRITE — do_kv_cache_update → reshape_and_cache_flash  (flash_attn.py:874)
for t in new_tokens:                        # this step's tokens only

    key_cache[slot]   = key[t]              # scatter: one token → one slot
    value_cache[slot] = value[t]

# READ — flash_attn_varlen_func  (flash_attn.py:796)

6.2.1 Column-packing: self.kv_cache is bound per layer, but allocated per column

Your question, precisely. For a vanilla single-group model, each layer gets its own private buffer (shared_by=[layer_name]). Gemma-3 is different: its full and sliding layers form multiple equal-sized groups, and the allocator takes the general column-packed branch. The physical allocation count is per column, not per layer — but each layer's self.kv_cache is still set individually (it's a view that aliases its column's buffer).

# get_kv_cache_config_from_groups — the general (column-packed) branch  (kv_cache_utils.py:1289)
group_size = max(len(g.layer_names) for g in kv_cache_groups)   # = layers per group
for i in range(group_size):                 # ONE tensor per column i, NOT per layer
    shared_by = [groups[j].layer_names[i] for j in range(len(groups))]  # one layer from EACH group
    kv_cache_tensors.append(KVCacheTensor(size=page_size * num_blocks, shared_by=shared_by))

# _allocate_kv_cache_tensors — one raw buffer per KVCacheTensor  (gpu_model_runner.py:6833)
for kv_cache_tensor in config.kv_cache_tensors:
    tensor = torch.zeros(kv_cache_tensor.size, dtype=torch.int8)   # the single physical buffer
    for layer_name in kv_cache_tensor.shared_by:
        kv_cache_raw_tensors[layer_name] = tensor                  # SAME object → every layer in the column

Two physical buffers for five layers — count = group_size (layers-per-group), not layer count. Each layer's self.kv_cache is its own view object, but full.0 / sw.0 / sw.1 alias the same storage T0.
Why aliasing is safe — the slots never collide. full.0, sw.0, sw.1 share T0's bytes, but they're in different groups with different block tables, and every block id comes from one global pool (§6.1). So a given request gets distinct block ids per group → distinct slot ranges → the three layers write to disjoint regions of T0. The shared buffer is just a memory pool; the global block-id space is what keeps the columns from stepping on each other. This is the cross-group physical pooling win: when SWA layers free blocks early (windowing), those slots return to the shared pool and a full layer can reuse them. → kv_cache/BEGINNER_GUIDE.html §7 has the byte-level walk-through.
So, to answer directly: the binding (layer.kv_cache = …) is per layer — every layer object has its own self.kv_cache view, and the forward pass looks itself up by name as always. The allocation is per column — there are group_size physical buffers, each aliased by one layer from each group. Contrast DeepSeek-V3.2 (§8): it's one group, so no aliasing — 122 genuinely separate per-layer tensors, differing in byte width.

6.3 Kimi Linear (KDA) a fixed-size recurrent state — read-modify-write, no paging

Kimi Linear interleaves KimiMLAAttention (full, §6.4) with KimiDeltaAttention — a linear-attention leaf that inherits MambaBase, not the paged Attention. Its self.kv_cache is not a paged token cache; it's a tuple of fixed-size state tensors (conv_state_q, conv_state_k, conv_state_v, recurrent_state), with one slot per request (indexed by state_indices, not by token position). There is no block table and no per-token slot mapping — the whole history is compressed into that one state.

# READ-MODIFY-WRITE — KimiDeltaAttention.forward  (kda.py:309-452)
idx = state_indices[req]                          # one slot for the WHOLE sequence
s   = recurrent_state[idx]                         # READ: fixed-size state, size independent of context length
out, s_new = chunk_kda(q, k, v, initial_state=s)   # prefill kernel  (or fused_recurrent_kda for decode)
recurrent_state[idx] = s_new                       # WRITE BACK in place — overwrites, never grows
# conv_state_{q,k,v} updated the same way via causal_conv1d_update

No block_table, no slot_mapping, no growth. A 100-token and a 100k-token request use the same state bytes. The flip side: you can't ask "what was K at position 47?" — it's been irreversibly mixed into S, so partial prefix sharing works only at coarse checkpoints. This is why linear-attention layers get a MambaSpec, not a FullAttentionSpec. → kv_cache/LINEAR_ATTENTION.html, MAMBA.html.

6.4 DeepSeek-V3.2 (MLA) one latent vector/token — write all, read a sparse subset

The MLAAttention leaf. self.kv_cache is paged like Gemma-3, but each token-slot holds a single latent vector (num_kv_heads = 1, 656 B in the fp8_ds_mla layout), not separate per-head K and V. Because index_topk is set, a second cache — the indexer's k_cache (132 B/token) — rides alongside in the same block ids.

# WRITE — do_kv_cache_update (MLA variant)  (mla_attention.py:565)
slot = slot_mapping[t]
kv_cache[slot] = pack(kv_c_normed[t], k_pe[t])     # latent KV + RoPE, one slot per token
indexer.k_cache[slot] = quantized_index_k[t]       # the 132 B scorer key

# READ — two stages (sparse)
# 1) indexer scores EVERY cached token, emits ≤ index_topk (2048) selected logical ids
sel = indexer.topk(query[t], indexer.k_cache, block_table[req])
# 2) sparse MLA kernel converts sel → global slots via block_table, gathers ONLY those
out[t] = sparse_attend(query[t], kv_cache, selected_slots=to_global(sel, block_table))

7. The contract: what the engine requires of a model

"Integrating a model" really means: implement a handful of methods on the top class so the GPUModelRunner can drive it. The runner only ever calls these:

Capabilities are opt-in mixins. The base classes SupportsPP, SupportsLoRA, SupportsEagle, … in models/interfaces.py are Protocols. Adding pipeline-parallel support, for example, is "inherit SupportsPP and return IntermediateTensors at the PP boundary" — both visible in LlamaModel.forward above.

8. Contrast: Gemma-3 vs DeepSeek-V3.2

Both models use the identical spine from §1 — *ForCausalLM → *Model → *DecoderLayer → *Attention. If you diff their files, the ForCausalLM/Model levels are nearly copy-paste. All the interesting divergence is at the leaf and in the spec it returns — and the two models diverge in opposite directions, which makes them the perfect pair to study.

Gemma-3 heterogeneous lifecycles

Gemma-3 alternates attention types: config.layer_types[i] is either "sliding_attention" or "full_attention" (typically 5 sliding : 1 full). Gemma3Attention.__init__ reads that and sets per_layer_sliding_window on its leaf accordingly. It also adds q_norm/k_norm and a per-layer RoPE base — but those are local details.

KV-cache consequence: sliding layers return SlidingWindowSpec, full layers return FullAttentionSpec. Different spec types ⇒ multiple KV-cache groups ⇒ the column-packed tensor layout, NULL blocks for expired window positions, and per-group block tables. This is the canonical hybrid case.

DeepSeek-V3.2 heterogeneous byte widths

DeepSeek keeps one attention type everywhere, but it's an exotic one: the leaf is MLAAttention (latent KV, num_kv_heads=1, head_size=576), reached through a DeepseekV2MLAAttention → MultiHeadLatentAttentionWrapper wrapper. Because index_topk is set, each layer also builds a sparse Indexer whose DeepseekV32IndexerCache registers a second cache owner.

KV-cache consequence: all caches are MLAAttentionSpec (a FullAttentionSpec subclass) at block_size 64. Same type ⇒ one uniform-type group ⇒ one block table, but per-layer tensors (main rows 656 B, indexer rows 132 B). NOT column-packed.

The mental model. Two independent axes can make a model "non-vanilla":
  • Lifecycle (when is a position's K/V still needed?) — if layers disagree, you get multiple groups and column packing. Gemma-3
  • Byte width (how many bytes per token-slot?) — if layers agree on lifecycle but differ in width, you stay in one group with per-layer tensors. DeepSeek-V3.2
A plain Llama varies on neither axis: one type, one width ⇒ one group, one uniform tensor per layer. Read the two contrast cases and the vanilla case and you've seen every layout vLLM has.

9. The big takeaways

  1. One spine, five levels. *ForCausalLM → *Model → *DecoderLayer → *Attention → leaf Attention. Learn it once, read any model.
  2. Levels 1–4 are plumbing; level 5 is the only stateful one. The leaf Attention is the sole owner of KV cache.
  3. Forward threads down, hidden states bubble up. Cache tensors are never arguments — the leaf finds its own via self.kv_cache, bound by name.
  4. Register, don't allocate. The leaf registers itself in static_forward_context at construction; the engine discovers it, asks for its get_kv_cache_spec(), and binds a tensor.
  5. The spec is the bridge. Which spec class a leaf returns decides grouping and memory layout — it's where "model integration" hands off to "KV cache".
  6. Models differ only at the leaves. Gemma-3 varies the lifecycle (sliding vs full → many groups, column-packed); DeepSeek-V3.2 varies the byte width (MLA + indexer → one group, per-layer tensors). The spine above is identical.
