DeepSeek-V3.2 KV Cache — Engine Execution Trace

A chronological walk from EngineCore.__init__ to the fourth decode step, using the real deepseek-ai/DeepSeek-V3.2 config: read the config, build the 61 layers, derive KV cache specs, allocate and bind the cache, then watch the scheduler hand out blocks and step() drive the model. DeepSeek-V3.2 runs through the shared V2-family path (DeepseekV2* classes) plus a sparse indexer.

Config: huggingface.co/deepseek-ai/DeepSeek-V3.2 · config.json · Engine: vllm/v1/engine/core.py · Model: vllm/model_executor/models/deepseek_v2.py · MLA layer: vllm/model_executor/layers/attention/mla_attention.py · Runner: vllm/v1/worker/gpu_model_runner.py

Short version. Startup is one method — EngineCore._initialize_kv_caches — running in four phases: collect specs → profile free memory → compute num_blocks → allocate + bind. The model never allocates KV memory itself: each of V3.2's 61 MLAAttention layers (and, because index_topk is set, 61 indexer caches too) registers itself in static_forward_context under a unique name and exposes a get_kv_cache_spec() returning an MLAAttentionSpec. For V3.2 the main MLA spec is num_kv_heads=1, head_size = 512+64 = 576, stored in the fp8_ds_mla layout at 656 bytes/token; the indexer spec is 132 bytes/token. The runner collects all 122 specs, the planner turns free GPU bytes into a block count, the worker allocates one tensor per layer, and bind_kv_cache writes each tensor back onto the layer's .kv_cache. At run time the scheduler allocates block ids per request and step() maps positions to physical slots with slot = block_id·64 + offset.

1. Startup — EngineCore builds the executor and loads the model

The V1 engine core constructs an executor, which spins up a worker, which loads the model onto the device. This is all synchronous and happens before a single request is accepted.

    → UniProcExecutor._init_executor → WorkerWrapperBase → init_device() → load_model()

        → GPUModelRunner.load_model → builds DeepseekV2ForCausalLM

Two distinct objects carry the cache-shaping inputs the whole way through:

vllm_config.model_config

Wraps the HF config. Supplies num_hidden_layers, the MLA dims (kv_lora_rank, qk_rope_head_dim, …), max_model_len, and use_mla.

vllm_config.cache_config

Runtime cache knobs: block_size, cache_dtype, gpu_memory_utilization, prefix-caching flags. num_gpu_blocks is filled in later by startup phase 3.

compilation_config
.static_forward_context

The registry every attention layer writes itself into, keyed by layer name. This is how the runner later discovers all cache owners without the model handing them over explicitly.

The spec is not built at construction time — it is a method the runner calls later, so it always reads the current cache_config (after any block-size defaulting). For MLA it returns an MLAAttentionSpec:

# MLAAttention.get_kv_cache_spec
def get_kv_cache_spec(self, vllm_config) -> KVCacheSpec:
    kv_cache_dtype = kv_cache_dtype_str_to_dtype(

    )
    return MLAAttentionSpec(
        block_size=vllm_config.cache_config.block_size,
        num_kv_heads=1,
        head_size=self.head_size,                       # kv_lora_rank + qk_rope_head_dim
        dtype=kv_cache_dtype,
        cache_dtype_str=vllm_config.cache_config.cache_dtype,
    )

V3.2 adds a second cache per layer. Because the config has index_topk, DeepseekV2MLAAttention also builds an Indexer whose DeepseekV32IndexerCache registers a second name — model.layers.{i}.self_attn.indexer.k_cache — into static_forward_context and returns its own MLAAttentionSpec:

    head_dim=self.head_dim + self.head_dim // self.quant_block_size * 4,  # 128 + 1·4 = 132
    dtype=torch.uint8,                                                  # 132 bytes/token
    prefix=f"{prefix}.indexer.k_cache",
    cache_config=cache_config,
)
So the 61-layer V3.2 model registers 122 cache owners: 61 main MLA caches (656 B/token) and 61 indexer caches (132 B/token). Both are MLAAttentionSpec (a subclass of FullAttentionSpec) with block_size = 64, so is_uniform_type folds all 122 into one KV-cache group → one block table, one block pool, one shared block-id space (main and indexer get the same ids for a request — they are not separate pools). Storage is still per-layer (§5.1): 122 distinct tensors, main rows 656 B, indexer rows 132 B. The indexer scores all cached tokens and emits up to index_topk = 2048 logical token ids; the main MLA read uses only those.

4. Startup — the four-phase KV cache initialization

With weights resident, EngineCore._initialize_kv_caches turns specs into allocated memory. It is one method with four clear phases.

# vllm/v1/engine/core.py — EngineCore._initialize_kv_caches (condensed)
# PHASE 1: collect every layer's spec from the runner
kv_cache_specs = self.model_executor.get_kv_cache_specs()

# PHASE 2: profile peak activation memory → free bytes available for KV
available_gpu_memory = self.model_executor.determine_available_memory()

# PHASE 3: turn free bytes + specs into a concrete plan (groups, num_blocks, tensors)
kv_cache_configs = get_kv_cache_configs(
    vllm_config, kv_cache_specs, available_gpu_memory
)
scheduler_kv_cache_config = generate_scheduler_kv_cache_config(kv_cache_configs)
vllm_config.cache_config.num_gpu_blocks = scheduler_kv_cache_config.num_blocks

# PHASE 4: allocate the tensors and bind them onto the layers
self.model_executor.initialize_from_config(kv_cache_configs)

Phase 1 — collect specs

The runner walks every module registered as an AttentionLayerBase in static_forward_context (exactly the layers from §3) and asks each for its spec.

# vllm/v1/worker/gpu_model_runner.py — GPUModelRunner.get_kv_cache_spec
attn_layers = get_layers_from_vllm_config(self.vllm_config, AttentionLayerBase)
kv_cache_spec = {}
for layer_name, attn_module in attn_layers.items():
    if spec := attn_module.get_kv_cache_spec(self.vllm_config):
        kv_cache_spec[layer_name] = spec
return kv_cache_spec   # {"model.layers.0.self_attn.attn": MLAAttentionSpec(...), ...}

Phase 2 — profile available memory

Worker.determine_available_memory runs a dummy forward under a memory-profiling context to measure peak activation usage, then reports how many bytes remain for the cache given gpu_memory_utilization.

Phase 3 — compute groups and num_blocks

get_kv_cache_configs groups layers by cache type, optionally auto-fits max_model_len to the memory, and divides free bytes by the per-block page size to get num_blocks. V3.2's 122 layers do not share an identical spec (656 vs 132 B), but they are the same type (all MLAAttentionSpec, block_size 64), so is_uniform_type folds them into one group with per-layer page sizes — see §5.1.

# vllm/v1/core/kv_cache_utils.py (shape of the calculation)
num_blocks = available_memory // per_block_bytes_across_all_layers
kv_cache_tensors = [
    KVCacheTensor(size=per_layer_specs[name].page_size_bytes * num_blocks,
                  shared_by=[name])
    for name in group.layer_names
]

The per-block byte cost is the sum across every layer tensor a single block id reserves. For V3.2 with block_size = 64:

main MLA (fp8_ds_mla): 64 tokens × 656 B/token = 41,984 B per block per layer
indexer K (uint8): 64 tokens × 132 B/token =  8,448 B per block per layer
one block id, all 61 layers: 61 × (41,984 + 8,448) = 61 × 50,432 ≈ 2.93 MiB
# worked example: 40 GiB of KV budget
num_blocks = 42,949,672,960 B // 3,076,352 B ≈ 13,961 blocks → ~893,500 cached tokens

The page size for the main MLA cache is not the naive block_size · 1 · 576 · sizeof(dtype): under fp8_ds_mla, MLAAttentionSpec.real_page_size_bytes hard-codes the custom 656-byte per-token layout (512 latent + 64 RoPE + scales). The resulting num_blocks is written to cache_config.num_gpu_blocks — the single number the scheduler's block pool hands out from.

5. Startup — allocate the tensors and bind them to the layers

Phase 4 reaches the worker, which delegates to the runner. The runner sets up attention backends, allocates raw byte tensors, views them at the right dtype/shape, then binds them back onto each layer object.

for layer_name, kv_cache in kv_caches.items():
    forward_context[layer_name].kv_cache = kv_cache   # the §3 placeholder is replaced
This is the wiring. The same layer object the model built in §2–3, registered in static_forward_context in §3, declared its spec in Phase 1, now has its real tensor written onto self.kv_cache. At forward time the layer reads/writes this tensor by looking itself up in the forward context — no tensor is ever passed down through the module call graph.

5.1 Per-layer vs column-packed: which path V3.2 takes

vLLM has three allocation layouts. Which one runs is decided by get_kv_cache_config_from_groups from the number and types of groups:

LayoutTriggerTensors
uniform specall layers identical spec (most dense models)one per layer (shared_by=[name]).
single-group uniform-typeall layers same type, different page sizesone per layer, each sized by its own page size. V3.2 path
column-packedmultiple groups (requires a SlidingWindowMLASpec)one tensor shared by one layer per group.

V3.2 lands in the middle row, so each layer has its own physical tensor — 122 of them (main rows 656 B, indexer rows 132 B). It is not column-packed. Two facts force this:

  • group_and_unify_kv_cache_specs (the column-packed/DeepSeek-V4 entry) returns None unless some layer is a SlidingWindowMLASpec. V3.2 has none — its indexer is a plain MLAAttentionSpec — so the multi-group path is never taken.
  • MLAAttentionSpec subclasses FullAttentionSpec, and all 122 layers use block_size 64, so UniformTypeKVCacheSpecs.is_uniform_type is True → all 122 collapse into one group → the single-group branch allocates one tensor per layer.

5.2 "Uniform type" ≠ "uniform page size"

The obvious objection: if main is 656 B/block and indexer is 132 B/block, how can they live in one UniformTypeKVCacheSpecs? Because is_uniform_type never checks page size. It requires only two things (kv_cache_interface.py:688):

  • same block_size (all 64), and
  • same spec class (all MLAAttentionSpec ⊂ FullAttentionSpec).

The docstring states the intent — "same type means always need the same number of token slots." That is the real invariant: every layer addresses the same number of token positions with the same block size, so one block table and one block-id namespace serve all of them. Byte width per slot is free to differ.

The allocator never packs the differing bytes together. It gives each layer its own tensor sized by its own page size; the group-level page_size_bytes is just the sum, used only to compute num_blocks:

# UniformTypeKVCacheSpecs.page_size_bytes — kv_cache_interface.py:678
return sum(spec.page_size_bytes for spec in self.kv_cache_specs.values())

# get_kv_cache_config_from_groups — single-group branch, kv_cache_utils.py:1262
num_blocks = available_memory // group.kv_cache_spec.page_size_bytes      # ← the SUM
per_layer_specs = group.kv_cache_spec.kv_cache_specs
kv_cache_tensors = [
    KVCacheTensor(
        size=per_layer_specs[name].page_size_bytes * num_blocks,         # ← each layer's OWN size
        shared_by=[name],                                                # ← one layer per tensor
    )
    for name in group.layer_names
]
one block id costs: 61·41,984 (main) + 61·8,448 (indexer) = 3,076,352 B ≈ 2.93 MiB
num_blocks = available_memory // 3,076,352

per-layer tensors — all num_blocks rows, different row widths:
  main_Li.kv_cache : (num_blocks, 64, 656) → row = 41,984 B
  indexer_Li.kv_cache : (num_blocks, 64, 132) → row =  8,448 B

So block id 42 is row 42 in every tensor — but that row is 41,984 B in a main tensor and 8,448 B in an indexer tensor. Same row index, different byte width. The differing page sizes are fine because nothing is shared: each KVCacheTensor is its own allocation, so its width is independent. This is exactly why column packing needs uniform page size (multiple layers reinterpret the same raw bytes, so the byte count must match — get_uniform_page_size asserts it at kv_cache_utils.py:960) while the single-group per-layer path does not.

One group, but still per-layer storage. Because there is only one group, there is one block table and one block-id space shared by all 122 layers — a request's main and indexer caches use the same block ids. The bytes are still separate per layer. So block id 42 names row 42 in every one of the 122 tensors at once: main_L0.kv_cache[42] (656 B), indexer_L0.kv_cache[42] (132 B), … one slot index, 122 distinct physical rows. Column packing is the opposite axis — it shares one tensor across groups (one layer from each), and only ever appears in multi-group hybrids (Gemma-3 SWA+full, DeepSeek-V4). It never shares storage between two layers of the same group.

6. A request arrives — the scheduler allocates blocks

Startup is over. A request enters the waiting queue. On the next step(), Scheduler.schedule() decides how many tokens of this request to run and asks the KV cache manager for blocks to hold their KV.

How this connects to §5. bind_kv_cache and allocate_slots never touch the same objects — they sit on opposite sides of a boundary and meet only through a shared block-id space [0, num_blocks). Think physical RAM vs. page allocator:
bind_kv_cache (§5)allocate_slots (§6)
sideworker / GPUscheduler / CPU
ownsthe bytes: tensors (num_blocks, 64, 656) per layerthe ids: KVCacheBlock(0…num_blocks−1), no tensors
whenonce, at startupevery step, per request
rolefixed arena of num_blocks physical pagesmalloc returning page indices, not pointers
The single thing that couples them is num_blocks: cache_config.num_gpu_blocks = scheduler_kv_cache_config.num_blocks (set in _initialize_kv_caches) sizes both the worker's tensors and the scheduler's BlockPool(num_gpu_blocks). A block id 42 from allocate_slots is meaningful only because §5 bound a tensor whose row 42 exists. The id rides SchedulerOutput to the worker, where compute_slot_mapping turns (block_id, offset) into the slot that indexes layer.kv_cache — the block table is the page table between the two halves.

The actual "how many blocks" arithmetic lives in the per-type manager. It is a ceiling division minus what the request already holds:

# vllm/v1/core/single_type_kv_cache_manager.py — allocate_new_blocks
num_required_blocks = cdiv(num_tokens, self.block_size)        # ceil(total / block_size)
num_new_blocks      = num_required_blocks - len(req_blocks)    # only the shortfall
if num_new_blocks > 0:
    new_blocks = self.block_pool.get_new_blocks(num_new_blocks) # pops from the free list
    req_blocks.extend(new_blocks)
What a block id means. A block id is an index into the cache tensor's first dimension, not a token position. For a tensor shaped (num_blocks, block_size, …), block id 12 names row kv_cache[12], which holds block_size consecutive tokens.

7. step() — schedule, execute, update

The engine loop is three calls. Everything in §6 happens inside the first; the model runs in the second; sampled tokens are folded back into request state in the third.

# vllm/v1/engine/core.py — EngineCore.step()
if not self.scheduler.has_requests():
    return {}, False
scheduler_output = self.scheduler.schedule()                       # §6: pick reqs, allocate blocks
future = self.model_executor.execute_model(scheduler_output, non_block=True)
grammar_output = self.scheduler.get_grammar_bitmask(scheduler_output)
model_output = future.result()
if model_output is None:
    model_output = self.model_executor.sample_tokens(grammar_output)
engine_core_outputs = self.scheduler.update_from_output(           # advance num_computed_tokens,
    scheduler_output, model_output                                 #   append sampled tokens
)

Inside execute_model: positions → slots

GPUModelRunner.execute_model first reconciles request state (_update_states: new requests call InputBatch.add_request and write a full block-table row; running requests append only the newly allocated ids). Then _prepare_inputs moves the block table to GPU and computes the slot mapping.

# vllm/v1/worker/gpu_model_runner.py — _prepare_inputs (key calls)
self.input_batch.block_table.commit_block_table(num_reqs)          # CPU block ids → GPU
...
self.input_batch.block_table.compute_slot_mapping(                 # one flat slot per scheduled token
    num_reqs,
    self.query_start_loc.gpu[: num_reqs + 1],
    self.positions[:total_num_scheduled_tokens],
)
block_index = position // block_size
block_id = block_table[request_row, block_index]
offset_in_block = position % block_size
slot = block_id * block_size + offset_in_block

The slot mapping (one int per token, keyed by layer) goes into the forward context; the block table (keyed by KV cache group) goes into the attention metadata. Then the model runs.

# the forward call
with set_forward_context(attn_metadata, self.vllm_config,
                         slot_mapping=slot_mappings_by_layer):
    model_output = self._model_forward(input_ids=input_ids, positions=positions, ...)

Inside each MLA layer, the write happens before the read: the layer looks itself up in the forward context, gets its slot mapping, and writes the current tokens' latent KV into self.kv_cache (the tensor bound in §5), then attention reads back over the cached range using the block table.

# vllm/model_executor/layers/attention/mla_attention.py (conceptual write)
slot = slot_mapping[t]
if slot >= 0:
    block_id, off = slot // block_size, slot % block_size
    kv_cache[block_id, off] = pack(kv_c_normed[t], k_pe[t])   # latent KV + RoPE

8. Four consecutive steps, with real V3.2 numbers

Real constants: block_size = 64, one request r0 with a 190-token prompt (chosen to sit two slots short of a block boundary), no prefix-cache hit, index_topk = 2048. All 122 layers share one block table (§5.1), so the main and indexer caches use the same block ids and the same slot values — the only difference is which per-layer tensor each slot indexes.

After the waiting-queue allocation (§6), the single pool gives r0 three blocks (ceil(190/64) = 3):

block table[r0] = [40, 41, 42] # shared by all 122 layers (main MLA + indexer)
slot = block_id·64 + offset # one value per token; indexes row in EACH layer's own tensor

9. Why the cache block size must equal the kernel's K-block granularity

FlashMLA sparse advertises exactly one supported block size — get_supported_kernel_block_sizes() → [64] — and the cache is laid out with block_size = 64 to match. This is not arbitrary; it is forced by how sparse attention dereferences the cache.

The chain that turns a selected token into a byte address has three parties, and all three multiply by the same B = block_size:

1 write (compute_slot_mapping): slot = block_table[pos//B]·B + pos%B
2 logical→physical (triton_convert_req_index_to_global_index): for each top-k token, global = block_table[tok//B]·B + tok%B
3 read (the FlashMLA kernel): gathers K from the paged cache, indexing the block table in units of B

The block table is the only map from logical position to physical block, and its unit is one block = B tokens. If any party used a different B, its arithmetic would land on the wrong row of the cache tensor. Three concrete failure modes if they disagreed:

ReasonWhat breaks if cache B ≠ kernel B
Shared address mathThe conversion in step 2 computes tok//B and tok%B with the cache's B. The kernel re-derives a block index from the same global value using its B. Different B → the kernel indexes a different block-table entry → reads another request's tokens (or out of bounds).
Paged non-contiguityTokens are physically contiguous within a 64-token block but blocks are scattered across the pool (e.g. ids 42 then 43 may be far apart). A kernel tile that pulled K in a granularity larger than 64 would straddle two blocks that are not adjacent in memory → it reads garbage across the gap.
Gather unit = page unitSparse MLA still fetches every selected token through the block table. One block-table lookup yields one block's worth of contiguous K — exactly B tokens. The kernel's inner loop must consume K in that same B-sized stride, or successive lookups desynchronize from the data they describe.
In one sentence. The block table is a shared translation table whose unit is block_size; the write path, the top-k logical→physical conversion, and the kernel's K gather all dereference it, so they must agree on that unit. The kernel's K-block granularity is its hard-coded block_size — FlashMLA sparse compiles tiles assuming 64-token pages — so the cache must be allocated with the identical 64, which is why EngineCore._initialize_kv_caches sets cache_config.block_size from min(group block sizes) and vllm_config.validate_block_size() rejects a mismatch.

(The C128A "compressed" decode path scales this by a fixed compress_ratio — kernel_B = block_size // compress_ratio — which is the same coupling expressed as an exact integer ratio rather than equality. Equality is the compress_ratio = 1 case.)
