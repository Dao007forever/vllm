Buddy Allocator & Decoupled Hybrid Pages

How VLLM_USE_BUDDY_BLOCK_POOL=1 changes KV-cache allocation for hybrid models (full attention + mamba/linear), and why it helps prefix caching.

One-sentence summary. Baseline forces every KV-cache group to share one block size and one padded page, so the small-page group (full attention) gets inflated up to the big-page group (mamba state). The buddy mode decouples them: each group keeps its native page, and a buddy allocator manages a single base-page address space so the groups can have different block sizes and block counts.

1. The two groups, and why their pages differ

A hybrid model like Kimi-Linear has two kinds of KV-cache layers:

Full attention (MLA)

Stores a per-token KV slab. Page size grows linearly with block size: page = block_size × bytes_per_token. A 64-token block is a small page; caching works at block granularity, so a small block means fine-grained prefix reuse.

Mamba / linear (KDA)

Stores a fixed-size recurrent state (~530 KB/layer for Kimi-Linear), independent of block size. Here block_size is the snapshot interval: one full state is written every block_size tokens. Bigger block = fewer snapshots = less memory.

The tension: the mamba native page (fixed, large) is much bigger than a small full-attention page. The allocator has to reconcile two groups whose natural page sizes differ by orders of magnitude.

2. Baseline: couple everything into one uniform pool

Baseline can only build a uniform block pool — every group shares one block size and one page size. To make the small full-attention page line up with the large mamba page, it inflates the attention block until its page is ≥ the mamba page (vllm/platforms/interface.py · _align_hybrid_block_size), then pads the smaller side up to a common page.

full-attn page mamba state padding (wasted)

Kimi-Linear, user --block-size 64 → attn inflated to 512 tokens so the attn page matches the mamba native page. Both groups now use block_size 512.

1 attention block = 512 tokens (one coarse cache unit)
1 mamba block = 512-token snapshot interval (page already this big)
→ ~2,685 large blocks total. A prompt shorter than 512 tokens never fills a block, so it is never cached.
The cost. The block size is welded to the mamba page. Tiny prompts fall below the 512-token caching granularity and get a 0% hit. You cannot make attention finer without also making every mamba snapshot finer (and vice-versa) — there is one knob for both.

3. Buddy: decouple the pages, share one address space

Buddy mode lets each group keep its native page. Instead of one uniform page, the allocator defines a base page and measures every group's page as a number of base pages — its chunk_order (2**chunk_order base pages per block). A buddy allocator then splits/merges power-of-two runs of base pages out of one shared address space.

Why attn must be chunk_order 0. Attention layers read through a kernel-block-expansion strided view that assumes a uniform per-block stride. If an attention group landed at chunk_order > 0, the kernel pages inside one logical block would no longer sit at a uniform stride and the view couldn't represent them. Anchoring base_page to the smallest attention page guarantees every attention group collapses to chunk_order 0. Mamba handles its own stride in a separate reshape branch and tolerates base_page > its native page.

4. How allocation works (the code path)

vllm/v1/core/kv_cache_utils.py — when VLLM_USE_BUDDY_BLOCK_POOL=1 and groups have more than one distinct page size:

# 1. Anchor the unit to the smallest ATTENTION page (not min of all groups).
attn_page_sizes = [g.page_size_bytes for g in groups if not isinstance(g, MambaSpec)]
base_page = min(attn_page_sizes)

# 2. Each group's chunk_order = ceil(log2(page / base_page)), so 2**order >= ratio.
#    Groups with page <= base_page collapse to 0 (they pad up by the gap).
for g in groups:
    ratio = g.page_size_bytes / base_page
    g.chunk_order = max(0, ceil(log2(ratio))) if ratio > 1 else 0
    overhang = max(overhang, g.page_size_bytes - base_page)

# 3. num_blocks counts BASE units. One block_id per base page.
num_blocks = get_num_blocks(available_memory, base_page)

# 4. Each layer-tuple slab = num_blocks * base_page, plus a tail overhang so the
#    strided read of a big (mamba) page at the last block_id stays in bounds.
tensor_size = base_page * num_blocks + max_overhang_bytes

The address space is one block_id per base page. The buddy allocator hands out runs of 2**chunk_order contiguous base pages and merges them back on free (VLLM_BUDDY_MAX_ORDER caps the largest run). Attention blocks are single base pages; a mamba block is a power-of-two run that covers its state plus the padding gap.

The page-size inversion fix (Option B)

Bug: at large user batch sizes the MLA page can grow past the mamba native page (e.g. bs 512: MLA 589,824 B > mamba 542,720 B). An earlier version picked base_page = min(all pages), which then chose the mamba page and forced MLA to chunk_order 1 — breaking the uniform-stride requirement and crashing with an out-of-bounds setStorage.

Fix: anchor base_page to the smallest attention page and let mamba pad up. Attention always lands at chunk_order 0. This mirrors what baseline already does post-inflation (larger natural page becomes the anchor, smaller side pads up) — buddy just applies it per-group instead of globally.

5. Interaction with mamba_cache_mode

The mode controls how much of the mamba state is snapshotted for prefix reuse (MambaSpec.max_memory_usage_bytes):

ModeMamba block sizeMamba memoryWhat is prefix-cached
none= max_model_len (1 block)1 page / seqnothing — state recomputed
alignforced = attn block~2 pages / seq (bounded)attention KV only; mamba recomputed
allcoarse, chunk-alignedcdiv(max_len, mamba_bs) × pageattention KV and mamba snapshots
Why decoupling makes all usable. In baseline, all still inflates the attn block to ~512 (coupled) and grows the mamba snapshot pool — you pay more memory and still can't cache tiny prompts. In buddy mode, attention stays at 64 (fine caching) while mamba snapshots at the coarse lcm(chunk_size, kernel_align) boundary, and --mamba-block-size directly controls the snapshot count cdiv(max_len, mamba_bs) — so you dial mamba memory down to whatever headroom the attention pool can spare. Baseline has no such lever; both block sizes are welded together.

6. Net effect (measured)

+36 ppprefix-cache hit, tiny-prompt workload (attn now caches sub-512 prompts)
≈49×more blocks at bs=64 (148,763 vs 3,003) for slightly less memory
≈ baselinethroughput at bs=512 (within run-to-run noise)

The win is concentrated where baseline's coarse block size hurts — small prompts and high prefix reuse. Where the working set already fits in baseline's coarse blocks, the two converge: extra blocks are unused headroom, not extra hits.
